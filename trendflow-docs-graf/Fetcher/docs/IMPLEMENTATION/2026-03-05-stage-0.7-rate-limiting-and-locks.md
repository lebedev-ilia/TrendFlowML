## 2. Технические детали

### 2.1. Rate limiting

- Ключи:
  - `rate:youtube:metadata:{ip_or_proxy_id}`
  - `rate:youtube:download:{ip_or_proxy_id}`
- Упрощённый fixed‑window подход:
  - `redis.incr(key)` + `expire(key, window_sec)` при первом вызове;
  - проверка `count <= limit` для разрешения запроса.
- Конфигурация:
  - лимиты и окна задаются через конфиг Fetcher (отдельный документ по конфигурации будет добавлен позже);
  - рекомендуемые значения взяты из практических оценок YouTube rate limits (см. `plan.md`).
- Поведение при превышении лимита:
  - адаптеры трактуют это как “rate limited” с нормализованным `error_code` (например, `YOUTUBE_RATE_LIMITED`);
  - шаг либо откладывается (retry/backoff), либо завершает run/задачу с контролируемой ошибкой (см. будущее требование).

### 2.2. Distributed locks

- Lock для `download_video`:
  - ключ: `lock:video:{platform}:{platform_video_id}`;
  - используется `SET key value NX EX ttl` для атомарного получения lock’а;
  - предотвращает параллельное скачивание одного и того же видео разными воркерами;
  - при уже установленном lock’е шаг:
    - либо ждёт появления артефакта (`artifacts`);
    - либо завершает работу, переиспользуя уже готовый артефакт.
- Lock для upload артефактов:
  - ключ: `lock:artifact:{video_id}:{artifact_type}`;
  - защищает от двойного upload’а одного и того же артефакта при retry/resume;
  - перед upload’ом проверяется, что в `artifacts` нет завершённой записи.

### 2.3. Интеграция

- Orchestrator:
  - может использует rate‑лимитер для принятия решений о постановке задач (особенно тяжёлых download’ов);
  - интеграция с backpressure и circuit breaker будет добавлена позже.
- Platform adapters:
  - обязаны вызывать rate‑лимитер перед любым внешним запросом (YouTube API, `yt-dlp`);
  - обязаны использовать video‑locks перед скачиванием видео.

## 4. Известные ограничения и TODO

- **Нет реализации rate‑лимитера и lock’ов в коде**:
  - в этом этапе зафиксирован только контракт и алгоритмы;
  - требуется реализовать обёртки над Redis и интегрировать их с Orchestrator’ом и адаптерами.
- **Упрощённая модель rate limiting**:
  - используется fixed‑window подход; для production может понадобиться более гибкая схема (token bucket/leaky bucket).
- **Нет конфигурационного документа Fetcher для лимитов**:
  - значения лимитов/окон пока не вынесены в отдельный ENV/CONFIG‑документ.
