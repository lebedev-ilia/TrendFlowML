## Stage 1.5 — Реализация YouTube comments worker (MVP)

**Дата**: 2026-03-05  
**Статус**: ✅ реализовано (ограниченный MVP)  
**Фаза**: Phase 1 — Core Ingestion Logic → Comments ingestion

---

## 1. Выполненные задачи

- Реализован базовый вариант `YouTubeAdapter.fetch_comments()`:
  - использует `yt-dlp` для получения `info_dict` и читает поле `comments` (если доступно);
  - создаёт записи в таблице `comments` для top‑N комментариев (ограничение `limit`);
  - сохраняет агрегированные поля (author, text, like_count, reply_count, published_at) и `raw_json`.
- Добавлен `run_comments_worker(run_id: str, limit: int = 100)`:
  - выбирает source по `video_sources` (аналогично metadata/video workers);
  - вызывает `YouTubeAdapter.fetch_comments(...)`.

---

## 2. Технические детали реализации

### 2.1. YouTubeAdapter.fetch_comments

Файл: `fetcher/platforms/youtube/adapter.py`

- Шаги:
  1. Через `yt_dlp.YoutubeDL({"skip_download": True, "quiet": True})` вызывается `extract_info(source, download=False)` и извлекается `info_dict`.
  2. Из `info_dict` берётся:
     - `platform_video_id = info["id"]`;
     - `comments = info.get("comments") or []` (если это поле присутствует).
  3. Список комментариев обрезается до `limit`.
  4. Через `session_scope()`:
     - ищется/создаётся `Video(platform="youtube", platform_video_id=platform_video_id)`;
     - для каждого комментария `c` создаётся запись `Comment`:
       - `author`, `text`, `like_count`, `reply_count`;
       - `published_at` восстанавливается из `timestamp` (если это int, трактуется как UNIX‑время UTC);
       - `raw_json=c`.

- Ограничения:
  - yt-dlp не всегда возвращает поле `comments` без дополнительной конфигурации; на этом этапе мы используем простейший вариант и считаем, что конфигурация yt-dlp будет адаптирована позже;
  - comments.json в storage пока не создаётся (вся нагрузка на таблицу `comments`).

### 2.2. Comments worker

Файл: `fetcher/workers/comments.py`

- `_get_source_for_run(db, run_id)`:
  - такая же логика, как в metadata/video workers;
  - выбирает `VideoSource` по `run_id` и возвращает `normalized_video_id` или `url`.
- `run_comments_worker(run_id: str, limit: int = 100)`:
  - валидирует, что `Run` существует;
  - получает `source` через `_get_source_for_run`;
  - создаёт `YouTubeAdapter` и вызывает `fetch_comments(source, run_id=run_id, limit=limit)`.

Worker экспортирован в `fetcher/workers/__init__.py`.

## 3. Тестирование

На этом этапе:

- автоматических тестов нет (требуется окружение с Postgres и реальным yt-dlp);
- выполнена логическая проверка:
  - согласованности с `DATABASE.md` (таблица `comments`);
  - использования тех же паттернов, что и metadata/video workers (поиск `Run`, `VideoSource`).

Для последующих стадий потребуются:

- unit‑тесты с моками `yt_dlp` и `session_scope()`;
- интеграционные тесты для проверки загрузки реальных/тестовых комментариев.

---

## 4. Известные ограничения и TODO

- **Зависимость от поля `comments` в info_dict**:
  - текущий код полагается на то, что yt-dlp вернёт `info["comments"]`;
  - для production потребуется:
    - явная настройка yt-dlp для извлечения комментариев;
    - или переход на YouTube Data API.
- **Нет сохранения comments.json в storage и артефакта `comments_file`**:
  - сейчас комментарии хранятся только в таблице `comments`;
  - в будущем следует:
    - писать компактный `comments.json` в `video-analytics-raw`;
    - регистрировать артефакт `comments_file` в `artifacts`.
- **Нет PII‑фильтрации**:
  - цели и требования к PII‑фильтрации описаны в чеклисте (Phase 1/Phase 3), но в коде пока не реализованы;
  - фильтрация будет добавлена как дополнительный слой поверх сохранения комментариев.
