# Централизованное логирование для Fetcher

**Дата**: 2026-03-05  
**Стадия**: Phase 6 — DevOps & Infrastructure  
**Статус**: ✅ Реализовано

## Обзор

Реализована интеграция с централизованными системами логирования: Grafana Loki, Elasticsearch и AWS CloudWatch. Логи отправляются в централизованное хранилище для индексации, поиска и анализа.

## Требования

Из чеклиста Phase 6:
- Интеграция с ELK Stack (Elasticsearch, Logstash, Kibana)
- Или интеграция с Loki (Grafana Loki)
- Или интеграция с Cloud logging (AWS CloudWatch, GCP Cloud Logging, Azure Monitor)
- Настройка отправки structured logs в JSON формате
- Индексация и поиск по логам

## Реализация

### 1. Модуль logging_handlers

**Файл**: `fetcher/logging_handlers.py`

**Реализованные handlers**:

1. **LokiHandler**:
   - Отправка логов в Grafana Loki через HTTP API
   - Использует Loki Push API: `POST /loki/api/v1/push`
   - Формат: Prometheus labels + log lines
   - Поддержка динамических labels (run_id, stage, platform)

2. **ElasticsearchHandler**:
   - Отправка логов в Elasticsearch через HTTP API
   - Использует Elasticsearch Index API: `POST /{index}/_doc`
   - Структурированные документы с полями (@timestamp, level, message, run_id, stage, etc.)

3. **CloudWatchHandler**:
   - Отправка логов в AWS CloudWatch Logs
   - Использует boto3 для отправки
   - Автоматическое создание log group и stream при необходимости

**Функция setup_centralized_logging**:
- Универсальная функция для настройки любого backend'а
- Обработка ошибок (не ломает основной поток)
- Опциональная зависимость (не требует установки всех библиотек)

### 2. Конфигурация

**Файл**: `fetcher/config.py`

**Добавленные настройки**:

- `logging_backend`: Backend для логирования ("loki", "elasticsearch", "cloudwatch")
- `logging_loki_url`: URL Grafana Loki API
- `logging_elasticsearch_url`: URL Elasticsearch API
- `logging_elasticsearch_index`: Имя индекса Elasticsearch
- `logging_cloudwatch_log_group`: Имя log group в CloudWatch
- `logging_cloudwatch_region`: AWS регион для CloudWatch

**Пример конфигурации**:
```python
# Loki
logging_backend = "loki"
logging_loki_url = "http://loki:3100"

# Elasticsearch
logging_backend = "elasticsearch"
logging_elasticsearch_url = "http://elasticsearch:9200"
logging_elasticsearch_index = "fetcher-logs"

# CloudWatch
logging_backend = "cloudwatch"
logging_cloudwatch_log_group = "/aws/fetcher"
logging_cloudwatch_region = "us-east-1"
```

### 3. Интеграция в setup_logging

**Файл**: `fetcher/logging.py`

**Изменения**:
- `setup_logging()` теперь проверяет `settings.logging_backend`
- Если backend настроен, создаётся соответствующий handler
- Handler добавляется к root logger вместе со стандартным StreamHandler
- Используется тот же formatter (JSON или text)

**Логика работы**:
1. Всегда создаётся StreamHandler для stdout
2. Если настроен централизованный backend, создаётся дополнительный handler
3. Оба handler'а используют один и тот же formatter
4. Ошибки при отправке в централизованное хранилище не ломают основной поток

### 4. Формат логов

**JSON формат** (StructuredFormatter):
```json
{
  "timestamp": "2026-03-05T12:00:00Z",
  "level": "INFO",
  "message": "Starting metadata worker for run_id=550e8400-...",
  "logger": "fetcher.workers.metadata",
  "run_id": "550e8400-...",
  "stage": "fetch_metadata",
  "platform": "youtube"
}
```

**Loki формат**:
- Labels: `{job="fetcher", level="info", logger="fetcher.workers.metadata", run_id="...", stage="fetch_metadata", platform="youtube"}`
- Log line: JSON строка

**Elasticsearch формат**:
- Документ с полями: `@timestamp`, `level`, `message`, `logger`, `run_id`, `stage`, `platform`, `platform_video_id`

**CloudWatch формат**:
- Log event с timestamp и message (JSON строка)

## Использование

### Настройка через переменные окружения

```bash
# Loki
export LOGGING_BACKEND=loki
export LOGGING_LOKI_URL=http://loki:3100

# Elasticsearch
export LOGGING_BACKEND=elasticsearch
export LOGGING_ELASTICSEARCH_URL=http://elasticsearch:9200
export LOGGING_ELASTICSEARCH_INDEX=fetcher-logs

# CloudWatch
export LOGGING_BACKEND=cloudwatch
export LOGGING_CLOUDWATCH_LOG_GROUP=/aws/fetcher
export LOGGING_CLOUDWATCH_REGION=us-east-1
```

### Настройка в Kubernetes ConfigMap

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: fetcher-config
data:
  logging-backend: "loki"
  logging-loki-url: "http://loki:3100"
```

### Зависимости

**Для Loki**:
```bash
pip install httpx
```

**Для Elasticsearch**:
```bash
pip install httpx
```

**Для CloudWatch**:
```bash
pip install boto3
```

## Известные ограничения

- **GCP Cloud Logging**: Не реализован (можно добавить через `google-cloud-logging`)
- **Azure Monitor**: Не реализован (можно добавить через `azure-monitor-opentelemetry`)
- **Batch отправка**: Логи отправляются по одному (можно оптимизировать через batch отправку)
- **Retry логика**: Нет автоматических retry при ошибках (можно добавить)
- **Buffering**: Нет буферизации логов (можно добавить для снижения нагрузки)
