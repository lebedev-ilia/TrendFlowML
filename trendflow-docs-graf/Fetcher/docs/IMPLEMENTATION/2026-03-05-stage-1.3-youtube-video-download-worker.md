## Stage 1.3 — Реализация YouTube video download worker

**Дата**: 2026-03-05  
**Статус**: ✅ реализовано (MVP)  
**Фаза**: Phase 1 — Core Ingestion Logic → Video download pipeline

---

## 1. Выполненные задачи

- Реализован базовый `YouTubeAdapter.download_video()`:
  - использует `yt-dlp` для скачивания лучшего видео ≤720p;
  - загружает результат в `video-analytics-raw` по layout из `STORAGE_LAYOUT.md`;
  - создаёт/обновляет артефакт `video_file` в таблице `artifacts`.
- Добавлен `run_video_download_worker(run_id: str)`:
  - выбирает source по `video_sources` (URL или normalized_video_id);
  - вызывает `YouTubeAdapter.download_video(...)`.
- Обновлена структура кода в `CODE_STRUCTURE.md` для отражения новой логики.

## 2. Технические детали реализации

### 2.1. YouTubeAdapter.download_video

Файл: `fetcher/platforms/youtube/adapter.py`

Шаги:

1. **Получение info_dict**
   - аналогично metadata‑методу:
     - `yt_dlp.YoutubeDL({"skip_download": True, "quiet": True}).extract_info(source, download=False)`;
   - извлекается `platform_video_id = info["id"]`.

2. **Проверка кеша артефакта**
   - через `session_scope()`:
     - ищется `Video(platform="youtube", platform_video_id=platform_video_id)` или создаётся новая запись;
     - проверяется наличие `Artifact(video_id, artifact_type="video_file", status="COMPLETED")`;
     - если такой артефакт уже есть — метод завершается (reuse артефакта).

3. **Скачивание через yt-dlp**
   - временный каталог: `/tmp/fetcher_video` (можно переопределить `FETCHER_TMP_DIR`);
   - шаблон имени: `{tmp_dir}/{video_id}.%(ext)s`;
   - опции:
     - `format="bestvideo[height<=720]+bestaudio/best"`;
     - `outtmpl=tmp_template`;
     - `quiet=True`;
     - `merge_output_format="mp4"`.
   - вызывается `extract_info(source, download=True)`, после чего итоговый путь берётся через `ydl.prepare_filename(info)` с заменой расширения на `.mp4`.

4. **Upload в Object Storage**
   - строится ключ:
     - `raw/youtube/YYYY/MM/DD/VIDEO_ID/video.mp4` (UTC‑дата);
   - используется `storage_client.upload_file(downloaded_path, bucket=settings.bucket_raw, key=storage_key)`;
   - после успешного upload’а временный файл удаляется.

5. **Регистрация/обновление артефакта**
   - второй `session_scope()`:
     - находится `Video` по `(platform, platform_video_id)`;
     - создаётся или обновляется `Artifact(video_file)`:
       - `storage_path = storage_key`;
       - `status = "COMPLETED"`.

### 2.2. Video download worker

Файл: `fetcher/workers/video.py`

- `_get_source_for_run(db, run_id)`:
  - ищет `VideoSource` для `run_id` (первый по `created_at`);
  - возвращает `normalized_video_id` при наличии, иначе `url`.
- `run_video_download_worker(run_id: str)`:
  - приводит строковый `run_id` к `UUID`;
  - убеждается, что `Run` существует;
  - извлекает source через `_get_source_for_run`;
  - вызывает `YouTubeAdapter.download_video(source, run_id=run_id)`.

## 4. Тестирование

На данном этапе:

- автоматические тесты не добавлялись (понадобится окружение с Postgres/S3/yt-dlp);
- проведена проверка:
  - согласованности путей с layout’ом (`STORAGE_LAYOUT.md`);
  - соответствия используемых таблиц схеме (`DATABASE.md`);

Для последующих этапов требуется:

## 5. Известные ограничения и TODO

- **Нет distributed lock для download**:
  - пока не используется ключ `lock:video:{platform}:{platform_video_id}` из `RATE_LIMITING_AND_LOCKS.md`;
  - требуется добавить lock‑обёртку вокруг скачивания, чтобы исключить параллельные скачивания.
- **Нет rate limiting и proxy‑пула**:
  - вызовы yt-dlp идут напрямую без учёта внешнего лимита запросов;
  - интеграция с rate‑лимитером и прокси будет выполнена в следующих стадиях.
- **Нет метрик и логирования**:
  - не пишутся `FetchLog` и метрики по длительности/успешности скачиваний;
  - будет добавлено в Phase 2 (Observability).
