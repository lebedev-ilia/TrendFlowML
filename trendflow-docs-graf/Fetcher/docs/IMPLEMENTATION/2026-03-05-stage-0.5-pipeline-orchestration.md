## Stage 0.5 — Дизайн pipeline orchestration и state machine (Phase 0 — Foundation)

**Дата**: 2026-03-05  
**Статус**: ✅ завершено (на уровне архитектурного дизайна)  
**Чеклист**: Fetcher `Phase 0 — Foundation → Pipeline orchestration`

---

## 1. Выполненные задачи

- Сформирован и задокументирован целевой дизайн orchestration‑уровня Fetcher:
  - state machine для ingestion‑pipeline’а (набор состояний и таблица переходов);
  - event‑driven постановка задач (логические Celery/Redis/Kafka‑tasks для каждого шага);
  - подход к resume после сбоев на основе идемпотентности и данных в БД.
- Создан документ `Fetcher/docs/PIPELINE_ORCHESTRATION.md`, описывающий:
  - общий pipeline (`NORMALIZE_SOURCE` → `CHECK_CACHE` → `FETCH_*` → `FINALIZE`);
  - список состояний (`PENDING` … `COMPLETED`/`FAILED`);
  - требования к задачам orchestration’а и правила переходов.

## 2. Технические детали

### 2.1. State machine

- Список состояний run’а:
  - `PENDING`, `NORMALIZING_SOURCE`, `CHECKING_CACHE`, `FETCHING_METADATA`, `FETCHING_CHANNEL`,
    `FETCHING_COMMENTS`, `DOWNLOADING_VIDEO`, `UPLOADING_ARTIFACTS`, `FINALIZING`, `COMPLETED`, `FAILED`.
- Базовая таблица разрешённых переходов:
  - линейная цепочка от `PENDING` до `FINALIZING` с возможностью прыгать в `FINALIZING` при cache hit;
  - любой промежуточный статус может перейти в `FAILED` при фатальной ошибке.
- Хранение состояния:
  - основное поле `status` в таблице `runs` (см. `DATABASE.md`);
  - дополнительные поля (`error`, `started_at`, `finished_at`) фиксируют таймстемпы и причины сбоев.

### 2.2. Orchestrator и задачи

- Orchestrator реализуется как набор задач очереди:
  - `normalize_source(run_id)`
  - `check_cache(run_id)`
  - `fetch_metadata(run_id)`
  - `fetch_channel(run_id)`
  - `fetch_comments(run_id)`
  - `download_video(run_id)`
  - `upload_artifacts(run_id)` (опционально, может быть частью download)
  - `finalize(run_id)`
- Общий жизненный цикл:
  - Backend вызывает “входную” операцию `fetch_video(run_id)` → ставится `normalize_source`;
  - каждая задача:
    - читает текущий статус из БД;
    - валидирует переход (соответствует ожидаемому состоянию);
    - выполняет свою работу (с опорой на PlatformAdapter и storage‑клиент);
    - обновляет статус и, при необходимости, ставит следующую задачу.

### 2.3. Event‑driven triggering

- Дизайн предполагает **event‑driven подход**:
  - завершение шага порождает обновление статуса в БД и публикацию события (см. `BACKEND_CONTRACTS.md` → схема событий);
  - эти события могут использоваться:
    - самим Fetcher (для orchestration и мониторинга);
    - Backend (WebSocket‑события для UI).
- Разделение ответственности:
  - Fetcher отвечает за постановку и выполнение ingestion‑задач;
  - Backend использует события Fetcher для агрегации статуса run’а на стороне пользователя.

### 2.4. Resume support

- Основной принцип: **идемпотентные шаги + источник истины в БД**.
- Перед выполнением любого шага:
  - задача проверяет по БД и storage, выполнен ли шаг уже (наличие артефактов, записей `fetch_jobs`, `artifacts`);
  - если данные уже есть и консистентны:
    - шаг считается выполненным;
    - статус обновляется без повторной тяжёлой работы (например, без повторного скачивания видео).
- Это позволяет:
  - безопасно ретраить задачи очереди;
  - переживать рестарты воркеров;
  - не бояться дубликатов задач (at‑least‑once delivery).

## 3. Тестирование

На этом этапе **не добавлялся рабочий код orchestration’а**, поэтому тестирование ограничено:

- логическим ревью:
  - проверка, что все состояния run’а, перечисленные в чеклисте Phase 0, отражены в state machine;
  - сверка с `plan.md` и ранее описанным pipeline.
- проверкой согласованности с другими контрактами:
  - согласование статусов/шагов с `BACKEND_CONTRACTS.md` (run_id lifecycle, события);
  - соответствие использованных сущностей таблицам из `DATABASE.md`.

Unit‑тесты для конкретной реализации state machine/очереди будут добавлены на стадиях Phase 1+.

---

## 4. Известные ограничения и TODO

- **Не выбран и не реализован конкретный queue‑backend**:
  - Celery/Redis/Kafka упоминаются как варианты, но конкретная реализация orchestration’а будет зафиксирована в следующих этапах;
  - не описаны детали retry‑политики/visibility timeout’ов и т.п.
- **Нет формальной enum‑типизации статусов/шагов**:
  - текущий дизайн опирается на строки; в будущем рекомендуется ввести enum‑классы (по аналогии с `RunStatus` в DataProcessor API).
- **Нет реального кода Orchestrator’а**:
  - не реализованы сервисы уровня `FetchOrchestrator`/`TaskManager` для Fetcher;
  - это будет отдельной задачей Phase 1 (Core Ingestion Logic).

  - описать и реализовать backpressure, distributed locks и rate‑limiter (см. соответствующие разделы чеклиста).
