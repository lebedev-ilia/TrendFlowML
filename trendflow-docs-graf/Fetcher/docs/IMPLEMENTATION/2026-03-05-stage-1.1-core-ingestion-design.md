## Stage 1.1 — Дизайн Core Ingestion Logic (Metadata/Video/Comments/Artifacts)

**Дата**: 2026-03-05  
**Статус**: ✅ завершено (дизайн и документация)  
**Фаза**: Phase 1 — Core Ingestion Logic

---

## 1. Выполненные задачи

- Сконсолидирован и задокументирован дизайн основных ingestion‑воркеров Fetcher:
  - Metadata worker;
  - Video download worker;
  - Comments worker;
  - Artifact Builder и lifecycle артефактов.
- Создан документ `Fetcher/docs/CORE_INGESTION.md`, описывающий:
  - цели каждого worker’а;
  - входы/выходы;
  - ключевые инварианты (идемпотентность, кеш‑reuse, управление артефактами).

---

## 2. Технические детали

### 2.1. Metadata worker

- Обязан:
  - извлекать метаданные и данные по каналу с платформы (YouTube в первой версии);
  - заполнять:
    - `video_metadata` (title, description, language, duration_seconds, published_at, raw_json);
    - `channel_metadata` (channel_id, channel_title, subscriber_count, video_count, view_count_channel, raw_json);
  - опционально записывать `meta.json` в object storage и регистрировать его в `artifacts`.
- Инварианты:
  - идемпотентность: повторный вызов не должен дублировать записи;
  - уважение rate‑лимитов и обработка ошибок (на основе `RATE_LIMITING_AND_LOCKS.md` и `PLATFORM_ADAPTERS.md`).

### 2.2. Video download worker

- Реализует:
  - проверку кеша в `artifacts` (`video_file` / `COMPLETED`);
  - получение distributed lock `lock:video:{platform}:{platform_video_id}` перед скачиванием;
  - скачивание и нормализацию видео (yt-dlp + ffmpeg);
  - upload в `video-analytics-raw` и регистрацию в `artifacts` (storage_path, size_bytes, checksum, status).
- Важные инварианты:
  - отсутствие “висящих” tmp‑файлов в `/tmp`;
  - отсутствие дублирующих скачиваний для одного видео;
  - корректное восстановление после частично завершённых upload’ов (status PENDING/FAILED, файл в storage).

### 2.3. Comments worker

- Обязанности:
  - загрузка top‑N (≤100 по умолчанию) релевантных комментариев с платформы;
  - реализация pagination streaming и stopping conditions (лимит по времени, страницам, количеству);
  - сохранение комментариев в:
    - таблицу `comments` с агрегированными полями и `raw_json`,
    - либо `comments.json` + запись в `artifacts`.
- Закладывается возможность:
  - добавить PII‑фильтрацию (email/phone/url) на следующих фазах без изменения интерфейса worker’а.

### 2.4. Artifact Builder и lifecycle

- Artifact Builder:
  - ожидает завершения обязательных этапов (metadata, video, comments);
  - строит `manifest.json` (`FetcherManifest`), используя layout из `STORAGE_LAYOUT.md`;
  - записывает `manifest.json` в storage и регистрирует его как артефакт;
  - инициирует запуск DataProcessor (через backend / DP API).
- Lifecycle артефактов:
  - статусы: `PENDING` → `UPLOADING` → `COMPLETED` / `FAILED`;
  - корректная обработка частично скачанных/загруженных артефактов при resume;
  - логирование инцидентов в `fetch_logs`.

## 3. Тестирование

На этом этапе реализован только дизайн (без кода воркеров).

Проведена логическая валидация:

- согласование с:
  - `plan.md` (описание ролей workers и artifact builder);
  - `DATABASE.md` (используемые таблицы и связи);
  - `STORAGE_LAYOUT.md` и `BACKEND_CONTRACTS.md` (требования к артефактам и manifest.json).
- проверены инварианты:
  - idempotency для каждого шага;
  - cache‑reuse по `(platform, platform_video_id)` и `artifacts`.

Фактические unit/integration‑тесты появятся при реализации Python‑кода workers.

---

## 4. Известные ограничения и TODO

- **Нет реализаций воркеров в коде**:
  - metadata/video/comments workers и Artifact Builder пока существуют только как дизайн;
  - потребуется отдельный этап для реализации на Celery/Redis или другой очереди.
- **Нет формального описания retry‑политики и backoff’а для каждого worker’а**:
  - в этом этапе зафиксированы только высокоуровневые требования;
  - детали retry/backoff будут связаны с rate limiting и circuit breaker’ом.
