# Интеграция Kafka событий в существующий код

**Дата**: 2026-03-05  
**Стадия**: Phase 4 — Scalability Engineering  
**Статус**: ✅ Реализовано

## Обзор

Интегрирована публикация Kafka событий в существующий код Fetcher. События автоматически публикуются при изменении статусов run'ов, изменении stages и начале/завершении/ошибках job'ов.

## Требования

Из чеклиста Phase 4:
- Интеграция Kafka событий в существующий код (tasks.py, orchestrator.py)
- Автоматическая публикация событий при изменении статусов
- Автоматическая публикация событий при начале/завершении job'ов

## Реализация

### 1. Инициализация Kafka Producer

**Файл**: `fetcher/api.py`

**Изменения**:
- Добавлена инициализация Kafka producer при старте приложения
- Проверка `kafka_enabled` и `kafka_bootstrap_servers` в настройках
- Обработка ошибок при инициализации (не ломает старт приложения)

```python
# Инициализация Kafka producer (если включено)
if settings.kafka_enabled and settings.kafka_bootstrap_servers:
    try:
        bootstrap_servers = settings.kafka_bootstrap_servers
        if isinstance(bootstrap_servers, str):
            bootstrap_servers = [s.strip() for s in bootstrap_servers.split(",")]
        init_producer(
            bootstrap_servers=bootstrap_servers,
            topic_prefix=settings.kafka_topic_prefix,
        )
        logger.info(f"Kafka producer initialized: bootstrap_servers={bootstrap_servers}")
    except Exception as e:
        logger.warning(f"Failed to initialize Kafka producer: {e}")
```

### 2. Интеграция в Orchestrator

**Файл**: `fetcher/orchestrator.py`

**Добавленные события**:

1. **При изменении статуса на NORMALIZING_SOURCE**:
   - `publish_status_changed()` - событие изменения статуса

2. **При изменении статуса на CHECKING_CACHE**:
   - `publish_status_changed()` - событие изменения статуса
   - `publish_stage_changed()` - событие изменения stage (normalize_source → check_cache)

3. **При cache hit (переход на FINALIZING)**:
   - `publish_status_changed()` - событие изменения статуса
   - `publish_stage_changed()` - событие изменения stage (check_cache → finalize)

4. **При cache miss (переход на FETCHING_METADATA)**:
   - `publish_status_changed()` - событие изменения статуса
   - `publish_stage_changed()` - событие изменения stage (check_cache → fetch_metadata)

**Особенности**:
- Получение `platform` и `platform_video_id` из `video_source` для публикации в событиях
- Публикация событий после успешного обновления статуса в БД

### 3. Интеграция в Tasks

**Файл**: `fetcher/tasks.py`

**Добавленные события для всех задач**:

1. **fetch_metadata_task**:
   - `publish_job_started()` - при начале задачи
   - `publish_job_finished()` - при успешном завершении (с duration_ms)
   - `publish_job_failed()` - при ошибке (с error_code и error_message)

2. **download_video_task**:
   - `publish_job_started()` - при начале задачи
   - `publish_job_finished()` - при успешном завершении (с duration_ms)
   - `publish_job_failed()` - при ошибке (с error_code и error_message)

3. **fetch_comments_task**:
   - `publish_job_started()` - при начале задачи
   - `publish_job_finished()` - при успешном завершении (с duration_ms)
   - `publish_job_failed()` - при ошибке (с error_code и error_message)

4. **finalize_task**:
   - `publish_status_changed()` - при изменении статуса на COMPLETED
   - Получение `platform` и `platform_video_id` из `video_source` для публикации

**Особенности**:
- Получение `platform` и `platform_video_id` из БД для каждой задачи
- Измерение `duration_ms` для job'ов (время выполнения)
- Извлечение `error_code` из исключений для публикации в событиях ошибок

## Использование

### Настройка

```bash
# Включить Kafka
export FETCHER_KAFKA_ENABLED=true
export FETCHER_KAFKA_BOOTSTRAP_SERVERS="kafka1:9092,kafka2:9092"
export FETCHER_KAFKA_TOPIC_PREFIX="fetcher"
```

### Автоматическая публикация

События публикуются автоматически при:
- Изменении статусов run'ов (в orchestrator.py)
- Изменении stages (в orchestrator.py)
- Начале/завершении/ошибках job'ов (в tasks.py)

### Просмотр событий

События публикуются в Kafka topic `{prefix}.events` (по умолчанию `fetcher.events`).

Пример события:
```json
{
  "event_version": "1.0",
  "source": "fetcher",
  "run_id": "550e8400-...",
  "type": "run.status_changed",
  "ts": "2026-03-05T12:00:00.000Z",
  "platform": "youtube",
  "platform_video_id": "dQw4w9WgXcQ",
  "stage": "fetch_metadata",
  "status": "FETCHING_METADATA",
  "payload": {
    "old_status": "CHECKING_CACHE",
    "new_status": "FETCHING_METADATA"
  }
}
```

## Известные ограничения

- **Только основные события**: Публикуются основные события (status_changed, stage_changed, job.*), но не все возможные события (например, artifact.uploaded)
- **Ошибки при публикации**: Ошибки при публикации событий логируются, но не прерывают основной поток
- **Опциональность**: Kafka опционален, если не включен, события не публикуются (без ошибок)
