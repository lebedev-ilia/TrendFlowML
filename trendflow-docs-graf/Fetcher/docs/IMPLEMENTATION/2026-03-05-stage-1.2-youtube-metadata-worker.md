## 2. Технические детали реализации

### 2.1. YouTubeAdapter.fetch_metadata

Файл: `fetcher/platforms/youtube/adapter.py`

Основные шаги:

1. **Вызов yt-dlp**
   - Используется Python‑API `yt_dlp.YoutubeDL` с опциями:
     - `skip_download=True`;
     - `quiet=True`.
   - Выполняется `extract_info(source, download=False)` для получения `info_dict`.
   - Извлекается `platform_video_id = info["id"]`.

2. **Обновление БД**
   - Через `session_scope()`:
     - ищется или создаётся `Video(platform="youtube", platform_video_id=platform_video_id)`;
     - обновляются поля:
       - `Video.channel_id`, `Video.duration_seconds`;
       - `VideoMetadata` (title, description, language, duration_seconds, published_at, raw_json);
       - `ChannelMetadata` (channel_id, channel_title, subscriber_count, video_count, view_count, raw_json).
   - `published_at` восстанавливается из `upload_date` (`YYYYMMDD`) при наличии.

3. **Запись meta.json в storage**
   - Формируется ключ:
     - `raw/youtube/YYYY/MM/DD/VIDEO_ID/meta.json`, где `YYYY/MM/DD` — текущая UTC‑дата;
   - временный файл пишется в `/tmp/fetcher_meta/{video_id}_meta.json`;
   - используется `storage_client.upload_file(tmp_path, bucket=settings.bucket_raw, key=storage_key)`;
   - временный файл удаляется после upload’а.

4. **Регистрация артефакта**
   - Через `session_scope()`:
     - находится `Video` по `(platform, platform_video_id)`;
     - создаётся или обновляется `Artifact(video_id, artifact_type="metadata_file", storage_path=storage_key, status="COMPLETED")`.

### 2.2. Metadata worker

Файл: `fetcher/workers/metadata.py`

- Вспомогательная функция `_get_source_for_run(db, run_id)`:
  - выбирает первый `VideoSource` для `run_id`;
  - если `normalized_video_id` не пуст — использует его;
  - иначе использует исходный `url`.
- Основная функция `run_metadata_worker(run_id: str)`:
  - конвертирует `run_id` в `UUID`;
  - через `session_scope()`:
    - проверяет наличие `Run` в таблице `runs`;
    - определяет `source` через `_get_source_for_run`;
  - создаёт `YouTubeAdapter` и вызывает `fetch_metadata(source, run_id=run_id)`.

Это синхронный worker‑скелет, который позже можно обернуть в Celery‑задачу.

## 4. Тестирование

На этом этапе:

- интеграционные тесты не добавлялись (требуют настроенного Postgres/S3/yt-dlp);
- проведена базовая ручная валидация логики (по коду):
  - согласованность полей с `DATABASE.md` и `BACKEND_CONTRACTS.md`;
  - корректность путей storage (совместимость со `STORAGE_LAYOUT.md`);
  - отсутствие сильного связывания с queue‑backend (Celery/Redis можно добавить позже).

Для production‑использования потребуется:

- unit‑тесты для:
  - маппинга `info_dict` → `Video`/`VideoMetadata`/`ChannelMetadata`;
  - генерации `storage_key` и работы с `Artifact`;
  - `run_metadata_worker` (обработка отсутствующих `Run`/`VideoSource`).
- интеграционные тесты в docker‑окружении с реальной БД и MinIO.

## 5. Известные ограничения и TODO

- **Нет rate limiting и proxy‑pool в fetch_metadata**:
  - текущая реализация не использует слой из `RATE_LIMITING_AND_LOCKS.md`;
  - необходимо будет обернуть вызов yt-dlp в rate‑лимитер и прокси‑роутинг.
- **Нет логирования и событий**:
  - не пишутся `FetchLog` и события `FetcherEvent` при успешном/неуспешном выполнении;
  - это будет добавлено на этапах Phase 2 (Observability) и Phase 5 (Failure handling).
- **Ошибки yt-dlp не классифицированы**:
  - любые исключения yt-dlp сейчас пробрасываются как есть;
  - требуется маппинг на `error_code` и retry‑политику.
