## 2. Технические детали

### 2.1. Роль БД Fetcher

- БД Fetcher **отдельна** от БД backend и DataProcessor:
  - концентрируется на ingestion‑потоках, кешах и артефактах;
  - не хранит тяжёлые двоичные данные (видео, большие JSON), только метаданные и ссылки.
- Основные задачи схемы:
  - отслеживать жизненный цикл ingestion run’ов (`runs`, `fetch_jobs`, `fetch_logs`);
  - обеспечивать глобальный кеш по видео (`videos`, `artifacts`, `video_metadata`, `comments`, ...);
  - хранить временные снэпшоты для temporal‑фичей (`video_snapshots`).

### 2.2. Ключевые таблицы и связи

- `runs`:
  - хранит `run_id` (UUID из backend), статус ingestion, таймстемпы и текст ошибки;
  - является точкой входа для всех связанных данных Fetcher.
- `video_sources`:
  - связывает run c исходным URL и нормализованным `platform_video_id`;
  - позволяет поддерживать повторный ingestion для одного `run_id` с уже известным видео.
- `videos`:
  - глобальный кеш по `(platform, platform_video_id)` с уникальным ограничением;
  - позволяет reuse артефактов и метаданных между разными run’ами.
- `video_metadata`, `channel_metadata`:
  - хранят как нормализованные поля (`title`, `duration_seconds`, ...), так и `raw_json` для будущего анализа.
- `video_snapshots`:
  - предназначена для временных метрик (views/likes/comments/subs), используемых в ML‑фичах.
- `comments`:
  - отдельная строка на комментарий, с `raw_json` и агрегированными полями;
  - потенциальный кандидат на partitioning в будущем.
- `artifacts`:
  - регистрирует все артефакты, находящиеся в S3/MinIO;
  - хранит `artifact_type`, `storage_path`, `status`, `checksum`, `size_bytes`.
- `fetch_jobs` и `fetch_logs`:
  - обеспечивают observability и отладку ingestion‑pipeline’а;
  - позволяют восстанавливать ход выполнения по `run_id`.

### 2.3. Инварианты и связи с контрактами

- Глобальный кеш:
  - `UNIQUE(platform, platform_video_id)` в `videos` гарантирует отсутствие дубликатов;
  - таблица `artifacts` ссылается на `video_id`, что упрощает reuse.
- Идемпотентность:
  - любые idempotency‑проверки в коде Fetcher будут опираться на наличие записей в `videos`/`artifacts`/`video_metadata`/`comments`;
  - схема спроектирована так, чтобы повторный запуск ingestion не создавал конфликтов по ключам.
- Совместимость с `manifest.json`:
  - поля `video_id` и артефакты в `artifacts` логически соответствуют полям `video_id` и `artifacts.*.path` в `FetcherManifest`;
  - `storage_path` в `artifacts` должен быть совместим с layout’ом, описанным в `plan.md` и используемым в manifest.

## 3. Тестирование

На этом этапе **не создавались реальные миграции/таблицы**, поэтому тестирование ограничено:

- логической проверкой схем:
  - на отсутствие циклических зависимостей;
  - на корректность ссылок (`REFERENCES videos(id)`, `REFERENCES runs(id)` и т.п.);
  - на совместимость с ранее описанными таблицами в `plan.md`.
- ревью инвариантов:
  - проверка, что все инварианты из `plan.md` (глобальный кеш, separation of concerns, отсутствие тяжёлых данных в БД) отражены в схеме.

Unit‑тесты для ORM‑моделей и Alembic‑миграций будут добавлены, когда будет реализована фактическая интеграция с Postgres.

## 4. Известные ограничения и TODO

- **Нет Alembic‑миграций**:
  - SQL‑контракт зафиксирован только в документации, без миграционных файлов;
  - структура `backend/alembic` может быть использована как референс при добавлении миграций для Fetcher.
- **Нет индексов и partitioning**:
  - в MVP‑схеме не описаны вторичные индексы (кроме `UNIQUE(platform, platform_video_id)`);
  - для production‑нагрузки потребуется:
    - индексы по `video_id`, `run_id`, `created_at` в крупных таблицах;
    - partitioning/шардинг для `comments` и `fetch_logs`.
- **Нет формальной типизации статусов**:
  - статусы и стадии (`status`, `job_type`, `artifact_type`, `stage`) представлены как `VARCHAR`;
  - в будущем можно ввести enum‑типы или константы на уровне ORM‑моделей.
