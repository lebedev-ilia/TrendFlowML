## 2. Технические детали

### 2.1. Backpressure Detection

**Файл**: `fetcher/backpressure.py`

**Функции**:
- `get_processor_queue_size(processor_api_url: Optional[str] = None) -> int`:
  - Получает размер очереди DataProcessor через HTTP API
  - Возвращает 0 при ошибке (fail-open подход)
  - Пока заглушка (требует endpoint в DataProcessor API)
- `check_backpressure(threshold: Optional[int] = None, processor_api_url: Optional[str] = None) -> bool`:
  - Проверяет, превышен ли порог размера очереди
  - Логирует warning при обнаружении backpressure
  - Возвращает True если очередь переполнена

**Исключение**:
- `BackpressureError`: выбрасывается при обнаружении backpressure, содержит `retry_after` (по умолчанию 300 секунд)

### 2.2. Интеграция в finalize_task

**Изменённый файл**: `fetcher/tasks.py`

**Изменения**:
- В `finalize_task`: перед обработкой артефактов вызывается `check_backpressure()`
- При обнаружении backpressure:
  - Логируется warning
  - Выбрасывается `BackpressureError` с `retry_after=300`
  - Celery retry выполняется через 5 минут

### 2.3. Конфигурация

## 3. Технические детали

### 3.1. Fail-Open Approach

- При ошибке получения размера очереди (timeout, connection error) система считает, что backpressure нет
- Это предотвращает блокировку pipeline при недоступности DataProcessor API
- Логируются warning'и для мониторинга

### 3.2. Retry Strategy

- При backpressure задача retry через 5 минут (300 секунд)
- Это даёт DataProcessor время обработать очередь
- После нескольких retry задача может быть помечена как failed (если backpressure долго не снимается)

## 5. Известные ограничения

- Нет автоматического снижения скорости ingestion при backpressure (только полная пауза)
