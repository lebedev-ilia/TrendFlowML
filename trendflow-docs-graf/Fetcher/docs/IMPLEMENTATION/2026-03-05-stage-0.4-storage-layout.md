## 2. Технические детали

### 2.1. Бакеты и их роли

- `video-analytics-raw`:
  - хранит raw‑видео (`video.mp4`) и сырые JSON (`meta.json`, `comments.json`);
  - является основным источником данных для формирования `manifest.json`.
- `video-analytics-processed`:
  - хранит артефакты DataProcessor (NPZ, derived features, результаты моделей);
  - Fetcher не пишет сюда напрямую, но layout должен быть совместим с ожиданиями DataProcessor/backend.
- `video-analytics-temp`:
  - предназначен для временных артефактов, если они нужны;
  - должен иметь агрессивные lifecycle‑политики очистки.

### 2.2. Layout raw‑данных

Layout внутри `video-analytics-raw`:

- `raw/youtube/YYYY/MM/DD/VIDEO_ID/video.mp4`
- `raw/youtube/YYYY/MM/DD/VIDEO_ID/meta.json`
- `raw/youtube/YYYY/MM/DD/VIDEO_ID/comments.json`

Свойства:

- осознанное использование даты (`YYYY/MM/DD`) для равномерного распределения и лёгкой очистки;
- `VIDEO_ID` совпадает с `video_id`/`platform_video_id` в БД Fetcher и manifest;
- `meta.json` и `comments.json` являются “сырыми” источниками для любых downstream‑процессов.

### 2.3. Связь с `FetcherManifest`

- Поля `artifacts.video_file.path`, `artifacts.meta_file.path`, `artifacts.comments_file.path` в `FetcherManifest` (`Fetcher/schemas/manifest.py`) являются:
  - **относительными ключами** внутри `video-analytics-raw`;
  - прямым контрактом между Fetcher и DataProcessor.
- Layout спроектирован так, чтобы DataProcessor не знал:
  - имен бакетов;
  - внутренней логики хранения tmp‑файлов;
  - он опирается только на manifest.

### 2.4. Storage‑клиент и конфигурация

- Определён минимальный интерфейс `StorageClient` (логический, без кода реализации в этом этапе):
  - `upload_file(local_path, bucket, key)` — загрузка файла;
  - `download_file(bucket, key, local_path)` — скачивание;
  - `object_exists(bucket, key)` — проверка наличия объекта.
- В документации приведён пример реализации на `boto3` для MinIO/S3 и связь с конфигурацией:
  - endpoint (`MINIO_ENDPOINT` / `S3_ENDPOINT`);
  - креденшелы (`AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`);
  - имена бакетов.

### 2.5. Lifecycle‑политики

- Для `video-analytics-raw`:
  - удалить или перевести в cold storage через N дней (например, 30).
- Для `video-analytics-processed`:
  - хранить долго, как основной источник фич и результатов.
- Для `video-analytics-temp`:
  - TTL порядка 7 дней или меньше.

Эти политики должны быть реализованы на уровне конфигурации S3/MinIO, а не логикой приложения Fetcher (Fetcher только уважает факт возможного удаления raw‑данных через время).

## 3. Тестирование

На этом этапе ограничились **дизайном и документацией**, без реализации кода:

- проверена консистентность layout’а с:
  - `Fetcher/docs/plan.md` (раздел 6);
  - контрактом `FetcherManifest` (`Fetcher/schemas/manifest.py`);
  - общим подходом к layout’у в backend (`backend/docs/STORAGE_LAYOUT.md`).
- верифицировано, что:
  - пути, указанные в примере manifest, соответствуют описанному layout’у;
  - DataProcessor может использовать только manifest‑ключи, не зная деталей бакетов.

## 4. Известные ограничения и TODO

- **Нет фактической реализации `StorageClient`**:
  - в этом этапе описан только интерфейс и пример использования `boto3`;
  - реализация клиента и интеграция с Fetcher workers будет отдельной задачей.
- **Не задокументированы переменные окружения Fetcher для storage**:
  - требуется отдельный документ по переменным окружения (аналог `ENVIRONMENT_VARIABLES.md` в DataProcessor API);
  - туда нужно вынести список необходимых ключей (endpoints, креденшелы, имена бакетов).
- **Lifecycle‑политики не описаны в виде конкретных JSON‑policy для S3/MinIO**:
  - в этом этапе зафиксированы только high‑level требования.
