## Stage 1.4 — Реализация Artifact Builder и сборки manifest.json

**Дата**: 2026-03-05  
**Статус**: ✅ реализовано (MVP)  
**Фаза**: Phase 1 — Core Ingestion Logic → Artifact builder

## 1. Выполненные задачи

- Реализован `run_artifact_builder(run_id: str)`:
  - собирает данные из БД (`runs`, `video_sources`, `videos`, `artifacts`, `comments`);
  - формирует `FetcherManifest` (из `Fetcher/schemas/manifest.py`);
  - записывает `manifest.json` в `video-analytics-raw` по layout’у;
  - создаёт/обновляет артефакт `manifest` в таблице `artifacts`.
- Экспортирован Artifact Builder через `fetcher/workers/__init__.py`.
- Обновлена документация по структуре кода (`CODE_STRUCTURE.md` будет при необходимости дополнен).

## 2. Технические детали реализации

Файл: `fetcher/workers/artifacts.py`

### 2.1. Определение видео по run_id

- Функция `_get_video_for_run(db, run_id)`:
  - находит `VideoSource` для заданного `run_id` (первый по `created_at`);
  - использует `platform` и `normalized_video_id` для поиска `Video` в таблице `videos`;
  - возвращает кортеж `(Video, platform)`, либо поднимает `ValueError`, если данные не найдены/неполны.

### 2.2. Сборка артефактов

- Внутри `run_artifact_builder`:
  - по `video.id` ищутся артефакты:
    - `video_file` (обязателен, статус `COMPLETED`);
    - `metadata_file` (обязателен, статус `COMPLETED`);
    - `comments_file` (опционален, статус `COMPLETED`).
  - при отсутствии `video_file` или `metadata_file` поднимается `ValueError` (builder предполагает, что pipeline ещё не закончен).
  - для `comments_file`, если он есть:
    - считается `comment_count` по таблице `comments` (`COUNT(*) WHERE video_id = ...`).

### 2.3. Формирование FetcherManifest

- Используется `FetcherManifest` из `Fetcher/schemas/manifest.py`:
  - `run_id` — UUID run’а;
  - `video_id` — `Video.platform_video_id`;
  - `platform` — из `VideoSource.platform`;
  - `duration_seconds` — из `Video.duration_seconds` (если нет, используется 0.0);
  - `artifacts` — объект `ManifestArtifacts`:
    - `video_file` → `ArtifactInfo` с `path`, `checksum`, `size_bytes`;
    - `meta_file` → `ArtifactInfo` с `path`, `checksum`, `size_bytes`;
    - `comments_file` (если есть) → `ArtifactInfo` с `path`, `checksum`, `size_bytes`, `comment_count`.
- `manifest_version` и `storage_layout_version` берутся по умолчанию (`"1.0"`).

### 2.4. Запись manifest.json в storage и регистрация артефакта

- Формируется ключ:
  - `raw/{platform}/YYYY/MM/DD/{video_id}/manifest.json` (UTC‑дата);
- временный файл:
  - записывается в `/tmp/fetcher_manifest/{video_id}_manifest.json` (или `FETCHER_TMP_DIR`);
  - содержит JSON `manifest.model_dump()` (Pydantic v2).
- `storage_client.upload_file(tmp_path, bucket="video-analytics-raw", key=storage_key)`;
- после upload’а временный файл удаляется.
- В БД:
  - ищется/создаётся `Artifact(video_id, artifact_type="manifest")`;
  - `storage_path` обновляется на `storage_key`, `status="COMPLETED"`.

## 4. Известные ограничения и TODO

- **Нет учёта retry/resume сценариев**:
  - builder предполагает, что вызов происходит после завершения ingestion‑шагов;
  - не реализовано повторное построение manifest после частичных неудач.
- **Нет интеграции с state machine и событиями**:
  - не обновляется статус run’а на `COMPLETED`/`FAILED`;
  - не публикуются события `run.status_changed`/`job.finished`.
- **Bucket для manifest зашит строкой**:
  - используется `"video-analytics-raw"` напрямую, а не `settings.bucket_raw`;
  - это будет вынесено в конфигурацию при последующей итерации.
