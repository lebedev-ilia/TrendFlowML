### ✅ 2. Rate limiting настроен

**Статус**: ✅ РЕАЛИЗОВАНО

**Реализация**:
- Настроен `slowapi` для rate limiting в `api/main.py`
- Rate limit: **100 запросов в час** на backend instance
- Применён к `POST /api/v1/process` endpoint
- Использует заголовок `X-Backend-ID` для идентификации backend instance
- Fallback на IP адрес клиента если заголовок отсутствует
- Exception handler для `RateLimitExceeded` возвращает 429 Too Many Requests
- Graceful degradation: если slowapi не установлен, rate limiting отключается с предупреждением

**Файлы**:
- `api/main.py` - инициализация Limiter и exception handler
- `api/endpoints/process.py` - применение `@rate_limit_decorator`
- `requirements-api.txt` - зависимость `slowapi>=0.1.9`

### ✅ 3. Graceful shutdown работает корректно

**Статус**: ✅ РЕАЛИЗОВАНО

**Реализация для API**:
- Signal handlers для SIGTERM и SIGINT в `api/main.py`
- Lifespan context manager управляет graceful shutdown
- Шаги graceful shutdown:
  1. Остановка приёма новых запросов (через `shutdown_event`)
  2. Ожидание завершения текущих запросов (5 секунд таймаут)
  3. Очистка ресурсов (закрытие Redis клиента)
- Логирование всех шагов shutdown

**Реализация для Worker**:
- Signal handlers для SIGTERM и SIGINT в `api/worker.py`
- Метод `Worker.stop()` реализует graceful shutdown:
  1. Остановка приёма новых задач (через `shutdown_event`)
  2. Завершение текущих задач с таймаутом 5 минут
  3. Обновление состояния активных run'ов на "recovering"
  4. Удаление heartbeat записей из Redis
  5. ACK pending сообщений в очереди для перераспределения другим worker'ам
- Отслеживание активных задач через `active_tasks` dict

**Файлы**:
- `api/main.py` - graceful shutdown для API
- `api/worker.py` - graceful shutdown для Worker
- `api/services/worker.py` - метод `stop()` с полной логикой shutdown

**Ссылка**: Этап 4.3 - Graceful Shutdown

   - Протестировать shutdown в production-like окружении

   - Обновить контактную информацию в OpenAPI
