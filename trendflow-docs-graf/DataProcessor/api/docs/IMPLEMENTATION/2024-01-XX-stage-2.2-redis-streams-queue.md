## Выполненные задачи

### ✅ Реализовать enqueue_run() с Redis Streams

Создан модуль `api/services/queue.py` с функцией `enqueue_run()`:

1. **Поддержка приоритетов (high, normal, low)**
   - Валидация приоритета
   - Выбор соответствующей очереди (`queue:high`, `queue:normal`, `queue:low`)

2. **Использование XADD вместо LPUSH**
   - Использование `redis.xadd()` для добавления в stream
   - Сохранение run_id, timestamp и приоритета в stream

3. **Ограничение размера stream (maxlen)**
   - Параметр `maxlen=10000` для предотвращения переполнения
   - Использование `approximate=True` для производительности

4. **Дополнительные функции**
   - `get_queue_length()` - получение длины очередей
   - `get_pending_count()` - получение количества pending сообщений
   - Сохранение метаданных в Redis (`run:meta:{run_id}`)
   - Сохранение приоритета в Redis (`run:priority:{run_id}`)

### ✅ Реализовать worker loop с consumer groups

Создан модуль `api/services/worker.py` с классом `Worker`:

1. **Создание consumer group**
   - Метод `_ensure_consumer_groups()` создает группы для всех очередей
   - Использование `xgroup_create()` с `mkstream=True`
   - Обработка ошибки BUSYGROUP (группа уже существует)

2. **Чтение из всех приоритетных очередей**
   - Чтение из `queue:high`, `queue:normal`, `queue:low`
   - Использование ">" для чтения только новых сообщений
   - Блокирующее чтение с таймаутом 5 секунд

3. **Использование XREADGROUP**
   - Использование `xreadgroup()` для чтения из consumer group
   - Поддержка нескольких consumer'ов для масштабирования
   - Уникальный consumer_name для каждого worker'а

4. **ACK механизм**
   - Метод `_ack_message()` для подтверждения обработки
   - ACK только после успешной обработки
   - Сообщения остаются в pending при ошибках для retry

5. **Retry логика**
   - Сообщения не ACK'аются при ошибках
   - Сообщения остаются в pending и могут быть обработаны повторно
   - В будущем можно добавить счетчик retry и DLQ

6. **Интеграция с ProcessorService и TaskManager**
   - Использование ProcessorService для запуска обработки
   - Обновление статуса через TaskManager
   - Создание ProcessRequest из метаданных

### ✅ Интеграция queue в endpoint process.py

Обновлен endpoint `POST /api/v1/process`:

1. **Использование Redis Streams если доступен**
   - Проверка доступности Redis
   - Вызов `enqueue_run()` для добавления в очередь
   - Сохранение метаданных в Redis

2. **Fallback на MVP режим**
   - Если Redis не доступен, используется старый способ
   - Запуск через `asyncio.create_task()` и `_run_processing_background()`
   - Обратная совместимость с MVP

## Технические детали

### Созданные файлы

1. **`api/services/queue.py`** (новый файл)
   - Функция `enqueue_run()` для добавления задач в очередь
   - Функции `get_queue_length()` и `get_pending_count()` для мониторинга
   - Константы для имен очередей и настроек

2. **`api/services/worker.py`** (новый файл)
   - Класс `Worker` для обработки задач из очереди
   - Методы `start()`, `stop()`, `_worker_loop()`
   - Методы `_ensure_consumer_groups()`, `_process_message()`, `_ack_message()`
   - Функция `start_worker()` для запуска worker'а

### Измененные файлы

1. **`api/endpoints/process.py`**
   - Добавлен импорт `enqueue_run` из `api.services.queue`
   - Добавлена функция `_enqueue_fallback()` для MVP режима
   - Обновлена логика endpoint для использования Redis Streams с fallback

### Архитектурные решения

1. **Гибридный режим**
   - Использование Redis Streams если доступен
   - Fallback на MVP режим (in-memory + asyncio.create_task) если Redis не доступен
   - Обеспечивает плавный переход от MVP к production

2. **Consumer Groups**
   - Один consumer group "workers" для всех очередей
   - Уникальный consumer_name для каждого worker'а
   - Поддержка масштабирования через несколько worker'ов

3. **Приоритеты**
   - Три уровня приоритета: high, normal, low
   - Чтение из всех очередей одновременно
   - Redis обрабатывает очереди в порядке приоритета

4. **ACK механизм**
   - ACK только после успешной обработки
   - Сообщения остаются в pending при ошибках
   - Автоматический retry через повторное чтение pending сообщений

5. **Метаданные**
   - Сохранение метаданных в Redis для быстрого доступа
   - TTL 7 дней для метаданных
   - Включение метаданных в stream для надежности


## Известные проблемы и ограничения

1. **Worker не запускается автоматически**
   - Worker нужно запускать отдельно
   - В будущем будет отдельный Docker контейнер для worker'ов

2. **Retry логика**
   - Нет ограничения на количество retry
   - Нет DLQ (Dead Letter Queue) для сообщений с множественными ошибками
   - TODO: Добавить счетчик retry и DLQ

3. **Приоритеты**
   - Приоритет не передается из запроса
   - Всегда используется "normal"
   - TODO: Добавить поддержку приоритета в ProcessRequest

4. **Pending сообщения**
   - Нет автоматической обработки pending сообщений
   - Нужно вручную обрабатывать или использовать XPENDING
   - TODO: Добавить обработку pending сообщений
