### ✅ Redis Streams queue работает с consumer groups

**Статус**: ✅ РЕАЛИЗОВАНО И ПРОВЕРЕНО

**Проверка**:
- ✅ Consumer groups создаются автоматически при старте worker'а

- ✅ Используется `XREADGROUP` для чтения из consumer groups
- ✅ ACK механизм реализован через `XACK`
- ✅ Поддержка нескольких worker'ов через consumer groups

**Код**:
- `api/services/worker.py`:
  - `CONSUMER_GROUP_NAME = "workers"` (строка 45)
  - `_ensure_consumer_groups()` создает группы для всех очередей (строки 109-134)
  - `xreadgroup()` используется для чтения (строки 154-155)
  - `XACK` используется для подтверждения обработки (строка 460)

**Детали**:
- Consumer group создается с начальным ID "0" (читать все сообщения)
- Обработка ошибок при создании группы (уже существует, другие ошибки)
- Поддержка приоритетных очередей: high, normal, low

**Архитектура**:
```
Backend → API → Redis Streams (consumer groups) → Worker(s)
```

### ✅ Subprocess isolation реализован

**Статус**: ✅ РЕАЛИЗОВАНО И ПРОВЕРЕНО

**Проверка**:
- ✅ Каждый run запускается в отдельном subprocess через `asyncio.create_subprocess_exec`
- ✅ Опциональный мониторинг памяти через `psutil`
- ✅ Автоматическое завершение subprocess при превышении лимита памяти
- ✅ Изоляция stdout/stderr для каждого subprocess
- ✅ Обработка ошибок и exit codes

**Код**:
- `api/services/processor.py`:
  - `_run_main_py_sync()` использует `asyncio.create_subprocess_exec` (строки 190-201)
  - `_monitor_subprocess_memory()` опционально мониторит память (строки 295-361)
  - Лимит памяти настраивается через `config.subprocess_memory_limit_mb`
  - Docker resource limits установлены в `docker-compose.yml`

**Детали**:
- Subprocess запускается с изоляцией stdout/stderr
- Опциональный мониторинг памяти каждые 5 секунд
- Автоматическое завершение при превышении лимита через `process.kill()`
- Классификация ошибок по exit codes (killed_by_memory_limit, killed_by_signal, и т.д.)

**Docker конфигурация**:
- Worker: 4G memory limit, 4 CPUs
- API: 2G memory limit, 2 CPUs
- Subprocess memory limit настраивается через `SUBPROCESS_MEMORY_LIMIT_MB`

### ✅ Heartbeat и recovery работают

**Статус**: ✅ РЕАЛИЗОВАНО И ПРОВЕРЕНО

**Проверка**:
- ✅ Heartbeat loop реализован в worker для каждого активного run'а
- ✅ Heartbeat отправляется каждые 30 секунд с TTL 60 секунд
- ✅ Проверка heartbeat в StateReader при чтении статуса
- ✅ Автоматическое обнаружение crashed run'ов (отсутствие heartbeat)
- ✅ Recovery механизм для восстановления crashed run'ов
- ✅ Re-enqueue crashed run'ов обратно в очередь

**Код**:
- `api/services/worker.py`:
  - `_heartbeat_loop()` отправляет heartbeat каждые 30 секунд (строки 412-445)
  - Heartbeat запускается параллельно с обработкой (строки 315-318)
  - Автоматическая остановка heartbeat после завершения (строки 323-328)

- `api/services/recovery.py`:
  - `check_and_recover_run()` проверяет heartbeat и восстанавливает (строки 31-75)
  - `recover_run()` восстанавливает crashed run (строки 76-137)
  - `recover_all_crashed_runs()` массовое восстановление (строки 136-177)

- `api/services/state_reader.py`:
  - Проверка heartbeat для running run'ов (строки 267-275)
  - Автоматический вызов recovery при отсутствии heartbeat

**Детали**:
- Heartbeat хранится в `run:heartbeat:{run_id}` с TTL 60 секунд
- Проверка heartbeat выполняется при чтении статуса через StateReader
- Recovery обновляет статус на RECOVERING и re-enqueue в очередь
- Добавление события "recovery_started" в event stream

**Сценарий**:
1. Worker запускает обработку → heartbeat loop запускается
2. Heartbeat обновляется каждые 30 секунд
3. Worker падает → heartbeat перестает обновляться
4. API читает статус → обнаруживает отсутствие heartbeat
5. Recovery автоматически запускается → run re-enqueue в очередь

### ✅ Backpressure защищает от перегрузки

**Статус**: ✅ РЕАЛИЗОВАНО И ПРОВЕРЕНО

**Проверка**:
- ✅ Проверка длины очереди перед принятием нового run'а
- ✅ Конфигурация `max_queue_length` (по умолчанию 100)
- ✅ Возврат 503 Service Unavailable при превышении лимита
- ✅ Retry-After header для указания времени повтора
- ✅ Проверка как в TaskManager, так и в Redis Streams

**Код**:
- `api/config.py`:
  - `max_queue_length: int = 100` (строка 34)

- `api/endpoints/process.py`:
  - Проверка `task_manager.can_accept_new_run()` (строка 170)
  - Проверка `get_total_queue_length()` для Redis (строка 217)
  - Выброс `BackpressureError` при превышении лимита (строки 172-175, 219-222)

- `api/services/queue.py`:
  - `get_total_queue_length()` суммирует длины всех очередей (строки 130-150)

- `api/utils/errors.py`:
  - `BackpressureError` с поддержкой `retry_after` (строки 69-78)

**Детали**:
- Проверка выполняется в двух местах:
  1. TaskManager (in-memory registry) - для MVP fallback
  2. Redis Streams (если доступен) - для production режима
- Retry-After header устанавливается в 300 секунд (5 минут)
- Логирование предупреждений при backpressure

**Сценарий**:
1. Очередь заполнена (100+ задач)
2. Новый запрос на обработку → проверка длины очереди
3. Превышение лимита → BackpressureError
4. API возвращает 503 Service Unavailable с Retry-After header
5. Клиент должен повторить запрос через указанное время

### ✅ Можно запустить несколько worker'ов

**Статус**: ✅ РЕАЛИЗОВАНО И ПРОВЕРЕНО

**Проверка**:
- ✅ Отдельный Dockerfile для worker (`docker/worker/Dockerfile`)
- ✅ Отдельный сервис `dataprocessor-worker` в docker-compose.yml
- ✅ Поддержка масштабирования через `docker-compose up --scale`
- ✅ Consumer groups обеспечивают распределение задач между worker'ами
- ✅ Уникальный worker_id для каждого worker'а

**Код**:
- `docker-compose.yml`:
  - Сервис `dataprocessor-worker` определен (строки 93-152)
  - Комментарий о масштабировании: `docker-compose up --scale dataprocessor-worker=3` (строка 152)
  - Resource limits установлены (4G memory, 4 CPUs)

- `docker/worker/Dockerfile`:
  - Отдельный Dockerfile для worker процесса
  - Все необходимые зависимости установлены

- `api/worker.py`:
  - Entry point для worker процесса
  - Генерация уникального worker_id

- `api/services/worker.py`:
  - `worker_id` используется для идентификации в consumer group
  - `consumer_name = f"worker-{worker_id}"` (строка 76)

**Детали**:
- Каждый worker имеет уникальный ID (HOSTNAME-PID-UUID)
- Consumer groups обеспечивают распределение задач между worker'ами
- Каждый worker читает из одной и той же consumer group
- ACK механизм предотвращает дублирование обработки

**Команды для масштабирования**:
```bash
# Запустить 3 worker'а
docker-compose up --scale dataprocessor-worker=3

# Запустить 5 worker'ов
docker-compose up --scale dataprocessor-worker=5
```

**Архитектура**:
```
Backend → API → Redis Streams (consumer group: "workers")
                              ↓
                    ┌─────────┴─────────┐
                    ↓         ↓         ↓
                Worker 1  Worker 2  Worker 3
```

## Рекомендации для production

1. **Мониторинг**:
   - Добавить метрики для длины очереди
   - Мониторинг heartbeat и recovery событий
   - Алерты при превышении backpressure лимита

2. **Масштабирование**:
   - Настроить auto-scaling для worker'ов на основе длины очереди
   - Настроить resource limits в зависимости от нагрузки

3. **Надежность**:
   - Настроить периодический recovery loop для crashed run'ов
   - Добавить ограничение на количество recovery попыток
   - Настроить alerting для crashed run'ов

4. **Производительность**:
   - Настроить TTL cache в зависимости от нагрузки
   - Оптимизировать чтение из Storage для cold path
   - Рассмотреть использование Redis Cluster для масштабирования
