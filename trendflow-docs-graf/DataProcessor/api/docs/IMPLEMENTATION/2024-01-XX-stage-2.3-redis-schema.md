## Выполненные задачи

### ✅ Реализовать структуры данных в Redis

Создан модуль `api/services/redis_schema.py` с функциями для работы со всеми структурами данных:

1. **`run:meta:{run_id}` (TTL 7 дней)**
   - Функции: `save_run_metadata()`, `get_run_metadata()`
   - Хранение метаданных run'а в JSON формате
   - Используется для быстрого доступа к метаданным без обращения к Storage

2. **`run:state:{run_id}` (TTL 1 день)**
   - Функции: `save_run_state()`, `get_run_state()`
   - Кэш состояния для hot path
   - Автоматическое добавление `updated_at` при сохранении
   - Используется для быстрого получения статуса без чтения из Storage

3. **`run:heartbeat:{run_id}` (TTL 60 сек)**
   - Функции: `update_run_heartbeat()`, `get_run_heartbeat()`, `is_run_alive()`
   - Хранение timestamp последнего heartbeat
   - Используется для обнаружения crashed run'ов

4. **`run:lock:{run_id}` (TTL 3600 сек)**
   - Функции: `acquire_run_lock()`, `release_run_lock()`, `is_run_locked()`
   - Idempotency lock для предотвращения дублирования обработки
   - Использует SET с NX для атомарной блокировки

5. **`run:priority:{run_id}`**
   - Функции: `save_run_priority()`, `get_run_priority()`
   - Хранение приоритета run'а
   - TTL 7 дней

6. **`queue:high`, `queue:normal`, `queue:low` (Streams)**
   - Уже реализованы в `api/services/queue.py`
   - Используются для приоритетных очередей

7. **`stream:events:{run_id}` (TTL 1 день)**
   - Функции: `add_run_event()`, `get_run_events()`
   - Redis Stream для хранения событий run'а
   - Ограничение размера stream (maxlen=1000)
   - Автоматический TTL через EXPIRE

### Архитектурные решения

1. **Hot Path / Cold Path**
   - Redis cache используется для быстрого доступа (hot path)
   - Storage используется как fallback (cold path)
   - Автоматическое объединение данных из обоих источников

2. **Idempotency Lock**
   - Атомарная блокировка через SET NX
   - Автоматическое освобождение через TTL
   - Защита от дублирования обработки

3. **Heartbeat**
   - Обновление heartbeat при обработке
   - Использование для обнаружения crashed run'ов
   - TTL 60 секунд для автоматической очистки

4. **Events Stream**
   - Использование Redis Streams для событий
   - Ограничение размера для предотвращения переполнения
   - TTL для автоматической очистки старых событий

5. **TTL Management**
   - Все структуры данных имеют TTL
   - Автоматическая очистка старых данных
   - Разные TTL для разных типов данных

## Известные проблемы и ограничения

1. **Hot Path / Cold Path объединение**
   - Текущая реализация только проверяет Redis cache
   - TODO: Реализовать объединение данных из Redis и Storage

2. **Events Stream**
   - Нет consumer groups для событий
   - TODO: Добавить поддержку consumer groups для событий

3. **Heartbeat Recovery**
   - Нет автоматического recovery для crashed run'ов
   - TODO: Реализовать recovery механизм на основе heartbeat

4. **Lock Timeout**
   - Блокировка освобождается только через TTL
   - TODO: Добавить возможность ручного освобождения блокировки
