## Выполненные задачи

### ✅ Реализовать heartbeat loop в worker

Добавлен heartbeat loop в `Worker` класс:

1. **Отправка heartbeat каждые 30 секунд**
   - Метод `_heartbeat_loop()` запускается параллельно с обработкой сообщения
   - Heartbeat отправляется каждые 30 секунд через `update_run_heartbeat()`
   - TTL heartbeat установлен в 60 секунд (определен в `redis_schema.py`)

2. **Управление жизненным циклом heartbeat**
   - Heartbeat loop запускается при начале обработки run'а
   - Останавливается автоматически после завершения обработки (успех или ошибка)
   - Проверяет статус run'а перед отправкой heartbeat
   - Останавливается если run больше не активен

3. **Интеграция с обработкой сообщений**
   - Heartbeat loop запускается в `_process_message()` перед запуском обработки
   - Останавливается после завершения обработки (успех или ошибка)
   - Хранится в словаре `active_heartbeats` для управления

### ✅ Реализовать проверку heartbeat в API

Добавлена проверка heartbeat в `StateReader`:

1. **Проверка при чтении статуса**
   - Проверка heartbeat выполняется в `get_run_status()` для run'ов со статусом "running"
   - Используется функция `check_and_recover_run()` из recovery service
   - Автоматически запускает recovery если heartbeat отсутствует

2. **Обнаружение crashed run'ов**
   - Проверка выполняется только для run'ов со статусом "running"
   - Если heartbeat отсутствует → run считается crashed
   - Автоматически запускается recovery процесс

### ✅ Реализовать recovery логику

Создан новый модуль `api/services/recovery.py`:

1. **Функция `check_and_recover_run()`**
   - Проверяет heartbeat run'а
   - Запускает recovery если heartbeat отсутствует
   - Возвращает True если run был восстановлен

2. **Функция `recover_run()`**
   - Обновляет статус на "recovering"
   - Получает приоритет run'а из Redis
   - Получает метаданные run'а из Redis
   - Возвращает run в queue с сохранением приоритета
   - Добавляет событие "recovery_started" в event stream
   - Логирует процесс восстановления

3. **Функция `recover_all_crashed_runs()`**
   - Находит все run'ы со статусом "running"
   - Проверяет heartbeat для каждого
   - Восстанавливает все crashed run'ы
   - Возвращает количество восстановленных run'ов
   - Может использоваться для периодической очистки

## Технические детали

### Измененные файлы

1. **`api/services/worker.py`**
   - Добавлено поле `active_heartbeats: Dict[str, asyncio.Task]` для управления heartbeat tasks
   - Добавлен метод `_heartbeat_loop()` для отправки heartbeat
   - Интеграция heartbeat loop в `_process_message()`
   - Остановка heartbeat loop после завершения обработки

2. **`api/services/state_reader.py`**
   - Добавлена проверка heartbeat в `get_run_status()`
   - Автоматический вызов recovery при обнаружении crashed run'а
   - Обновление статуса после recovery

3. **`api/services/recovery.py`** (новый файл)
   - `check_and_recover_run()` - проверка и восстановление одного run'а
   - `recover_run()` - восстановление crashed run'а
   - `recover_all_crashed_runs()` - массовое восстановление

### Архитектурные решения

1. **Heartbeat Loop**
   - Запускается как отдельная asyncio task
   - Работает параллельно с обработкой сообщения
   - Автоматически останавливается при завершении обработки
   - Проверяет статус run'а перед отправкой heartbeat

2. **Recovery Process**
   - Проверка heartbeat выполняется при чтении статуса
   - Recovery запускается автоматически при обнаружении crashed run'а
   - Сохраняется приоритет и метаданные при восстановлении
   - Добавляется событие в event stream для отслеживания

3. **Обработка ошибок**
   - Graceful handling если Redis недоступен
   - Логирование всех операций recovery
   - Обработка исключений при восстановлении

## Использование

### Автоматическое восстановление

Recovery запускается автоматически при чтении статуса run'а:

```python
# При вызове get_run_status() для running run'а без heartbeat
# автоматически запускается recovery
status = await state_reader.get_run_status(run_id)
```

### Массовое восстановление

Для восстановления всех crashed run'ов:

```python
from api.services.recovery import recover_all_crashed_runs

# Восстановить все crashed run'ы
recovered_count = await recover_all_crashed_runs()
print(f"Recovered {recovered_count} run(s)")
```

### Периодическая очистка

Можно добавить периодическую задачу для восстановления crashed run'ов:

```python
import asyncio

async def periodic_recovery():
    while True:
        await asyncio.sleep(300)  # Каждые 5 минут
        await recover_all_crashed_runs()

# Запустить в фоне
asyncio.create_task(periodic_recovery())
```

## Известные проблемы и ограничения

1. **Heartbeat Timing**
   - Heartbeat отправляется каждые 30 секунд
   - TTL heartbeat - 60 секунд
   - Run может быть помечен как crashed если worker упал между отправками heartbeat
   - TODO: Рассмотреть уменьшение интервала heartbeat для более быстрого обнаружения

2. **Recovery Race Conditions**
   - Если несколько запросов одновременно проверяют heartbeat, может быть несколько recovery попыток
   - TODO: Добавить блокировку для предотвращения дублирования recovery

3. **Storage vs Redis**
   - Recovery использует только Redis для определения статуса
   - Storage может содержать более актуальную информацию
   - TODO: Проверять Storage перед recovery для более точного определения статуса

4. **Recovery Retry Limit**
   - Нет ограничения на количество recovery попыток
   - Run может бесконечно восстанавливаться если постоянно падает
   - TODO: Добавить счетчик recovery попыток и ограничение
