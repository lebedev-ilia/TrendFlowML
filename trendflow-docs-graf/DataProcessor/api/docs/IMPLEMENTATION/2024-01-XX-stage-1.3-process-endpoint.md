### ✅ Реализация endpoint POST /api/v1/process

Реализован полный endpoint с следующей функциональностью:

1. **Проверка существования run_id (Idempotency)**
   - Проверка через `task_manager.is_run_active()`
   - Возврат 409 Conflict если run уже существует

2. **Проверка лимита активных run'ов (Backpressure)**
   - Проверка через `task_manager.can_accept_new_run()`
   - Возврат 503 Service Unavailable с Retry-After header если лимит превышен

3. **Регистрация run в TaskManager**
   - Сохранение метаданных run'а
   - Установка статуса "queued"

4. **Запуск обработки через ThreadPoolExecutor**
   - Использование `ProcessorService.run_processing()`
   - Запуск в фоне через `asyncio.create_task()`
   - Использование semaphore для ограничения параллелизма

5. **Фоновая задача обработки**
   - Получение слота через semaphore
   - Обновление статуса на "running" при старте
   - Обновление статуса на "success" или "error" при завершении

### ✅ Обработка ошибок

Добавлены обработчики для всех типов ошибок:

1. **RunNotFoundError** (404) - уже был в main.py
2. **InvalidPayloadError** (400) - уже был в main.py
3. **ProcessingError** (500) - уже был в main.py
4. **RunAlreadyExistsError** (409) - добавлен новый обработчик
5. **BackpressureError** (503) - добавлен новый обработчик с Retry-After header

### ✅ Улучшения сервисов

1. **TaskManager**
   - Исправлен метод `acquire_slot()` для правильной работы с async context manager
   - Semaphore возвращается напрямую как async context manager

2. **ProcessorService**
   - Улучшена конвертация ProcessRequest в CLI args
   - Добавлена поддержка всех опциональных параметров
   - Определение пути к main.py относительно структуры проекта

### Архитектурные решения

1. **Idempotency**
   - Проверка существования run_id перед регистрацией
   - Возврат 409 Conflict если run уже активен

2. **Backpressure**
   - Проверка лимита активных run'ов перед принятием нового
   - Возврат 503 с Retry-After header для клиента

3. **Асинхронная обработка**
   - Endpoint возвращает 202 Accepted сразу после постановки в очередь
   - Обработка выполняется в фоне через asyncio.create_task()
   - Semaphore ограничивает параллелизм на уровне выполнения

4. **Управление состоянием**
   - TaskManager отслеживает все активные run'ы
   - Статусы обновляются в фоновой задаче
   - In-memory registry для MVP (будет заменён на Redis в Этапе 2)

## Известные проблемы и ограничения

1. **Profile config**
   - Profile config передаётся в запросе, но не конвертируется в CLI args
   - TODO: Реализовать сохранение profile_config в временный файл и передачу через --profile-path

2. **Оценка длительности**
   - estimated_duration_seconds захардкожен на 300 секунд
   - TODO: Рассчитывать на основе истории обработки

3. **In-memory registry**
   - Активные run'ы хранятся в памяти
   - При рестарте сервера информация о активных run'ах теряется
   - Будет заменён на Redis в Этапе 2

4. **ThreadPoolExecutor**
   - Используется для MVP
   - В Этапе 2 будет заменён на Redis Queue + Worker процессы
