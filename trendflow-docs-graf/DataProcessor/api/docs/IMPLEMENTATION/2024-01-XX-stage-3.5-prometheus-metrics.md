## Выполненные задачи

### ✅ Реализовать `GET /api/v1/metrics`

Endpoint реализован в `api/endpoints/metrics.py`:

1. **Основной endpoint**
   - Путь: `GET /api/v1/metrics`
   - Возвращает метрики в формате Prometheus text format
   - Использует `prometheus_client.generate_latest()`
   - Content-Type: `text/plain; version=0.0.4; charset=utf-8`

2. **Обработка ошибок**
   - Возврат пустых метрик при ошибке генерации
   - Логирование ошибок без прерывания работы API

### ✅ Queue метрики

Реализованы метрики для очереди:

1. **`dataprocessor_queue_length` (Gauge)**
   - Метрика длины очереди по приоритетам
   - Labels: `priority` (high, normal, low)
   - Обновляется в `get_total_queue_length()` при чтении длин очередей
   - Обновляется в `enqueue_run()` при добавлении в очередь

2. **`dataprocessor_queue_wait_seconds` (Histogram)**
   - Метрика времени ожидания в очереди
   - Buckets: [10, 30, 60, 300, 600] секунд
   - Измеряется в `worker.py` при получении сообщения из очереди
   - Использует timestamp из сообщения для расчета времени ожидания

### ✅ Processing метрики

Реализованы метрики для обработки:

1. **`dataprocessor_processing_seconds` (Histogram)**
   - Метрика времени обработки run'а
   - Labels: `processor`, `component`
   - Buckets: [60, 300, 600, 1800, 3600] секунд
   - Измеряется в `worker.py` при обработке run'а
   - Использует время начала и окончания обработки

2. **`dataprocessor_failures_total` (Counter)**
   - Метрика общего количества ошибок
   - Labels: `processor`, `component`, `error_type`
   - Увеличивается при ошибках обработки в `worker.py` и `process.py`
   - Поддерживает различные типы ошибок (exception, killed_by_memory_limit, и т.д.)

### ✅ Resource метрики

Реализованы метрики для ресурсов:

1. **`dataprocessor_memory_bytes` (Gauge)**
   - Метрика использования памяти на run
   - Labels: `run_id`
   - Обновляется в `processor.py` при мониторинге памяти subprocess
   - Обновляется при превышении лимита памяти перед завершением процесса

2. **`dataprocessor_active_runs` (Gauge)**
   - Метрика текущего количества активных run'ов
   - Обновляется в `task_manager.py` при регистрации и обновлении статуса run'ов
   - Обновляется в `health.py` при health check
   - Считает run'ы со статусами: queued, running, recovering

3. **`dataprocessor_crashed_runs_total` (Counter)**
   - Метрика общего количества crashed run'ов
   - Увеличивается в `recovery.py` при обнаружении crashed run'а (отсутствие heartbeat)

### ✅ Интеграция метрик в код

Метрики интегрированы в следующие модули:

1. **`api/services/queue.py`**
   - Обновление `queue_length` при изменении длины очереди
   - Обновление в `enqueue_run()` и `get_total_queue_length()`

2. **`api/services/worker.py`**
   - Измерение `queue_wait_time` при получении сообщения
   - Измерение `processing_time` при обработке run'а
   - Увеличение `failure_rate` при ошибках обработки

3. **`api/services/task_manager.py`**
   - Обновление `active_runs` при регистрации и обновлении статуса run'ов

4. **`api/services/recovery.py`**
   - Увеличение `crashed_runs` при обнаружении crashed run'а

5. **`api/services/processor.py`**
   - Обновление `memory_bytes` при мониторинге памяти subprocess

6. **`api/endpoints/process.py`**
   - Увеличение `failure_rate` при ошибках в fallback режиме

7. **`api/endpoints/health.py`**
   - Обновление `active_runs` при health check

### Архитектурные решения

1. **Использование prometheus_client**
   - Использование стандартной библиотеки `prometheus_client`
   - Поддержка всех типов метрик: Counter, Histogram, Gauge
   - Автоматическая генерация формата Prometheus

2. **Обновление метрик**
   - Метрики обновляются в местах, где происходят соответствующие события
   - Обработка ошибок при обновлении метрик (не блокирует основную логику)
   - Логирование ошибок на уровне debug

3. **Labels для метрик**
   - Использование labels для детализации метрик
   - Поддержка фильтрации и агрегации в Prometheus
   - Оптимизация количества метрик через labels

4. **Buckets для Histogram**
   - Настроены buckets для типичных значений времени обработки
   - Позволяют анализировать распределение времени обработки

### Настройка Prometheus

```yaml
# prometheus.yml
scrape_configs:
  - job_name: 'dataprocessor-api'
    scrape_interval: 15s
    static_configs:
      - targets: ['dataprocessor-api:8001']
    metrics_path: '/api/v1/metrics'
```

### Пример запросов PromQL

```promql
# Среднее время обработки за последний час
rate(dataprocessor_processing_seconds_sum[1h]) / rate(dataprocessor_processing_seconds_count[1h])

# Количество ошибок за последний час
rate(dataprocessor_failures_total[1h])

# Текущая длина очереди
sum(dataprocessor_queue_length)

# Количество активных run'ов
dataprocessor_active_runs

# Процент успешных run'ов
(1 - (rate(dataprocessor_failures_total[1h]) / rate(dataprocessor_processing_seconds_count[1h]))) * 100
```

## Известные проблемы и ограничения

1. **Определение processor и component**
   - В некоторых местах используется "unknown" для processor и component
   - Метрики могут быть неточными без правильных labels
   - TODO: Извлекать processor и component из результата обработки или метаданных

2. **Обновление метрик**
   - Метрики обновляются асинхронно, могут быть задержки
   - Некоторые метрики обновляются только при определенных событиях
   - TODO: Добавить периодическое обновление метрик для актуальности

3. **Memory метрика**
   - Обновляется только при мониторинге памяти subprocess
   - Не обновляется для всех run'ов
   - TODO: Добавить периодическое обновление memory метрики для всех активных run'ов

4. **Queue wait time**
   - Измеряется только если есть timestamp в сообщении
   - Может быть неточным для старых сообщений
   - TODO: Улучшить измерение времени ожидания в очереди

5. **Error types**
   - Типы ошибок могут быть неполными
   - Некоторые ошибки классифицируются как "unknown"
   - TODO: Улучшить классификацию типов ошибок
