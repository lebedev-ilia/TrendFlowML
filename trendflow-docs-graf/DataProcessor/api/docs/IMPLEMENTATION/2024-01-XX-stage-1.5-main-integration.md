## Выполненные задачи

### ✅ Создать services/processor.py

Файл уже был создан в этапе 1.1, но был значительно улучшен:
- Добавлена полная реализация всех методов
- Интеграция с main.py через subprocess

### ✅ Реализовать запуск через thread pool

1. **ThreadPoolExecutor с ограничением**
   - Используется `ThreadPoolExecutor(max_workers=config.max_concurrent_runs)`
   - Ограничение параллелизма на уровне executor

2. **Конвертация payload в CLI args**
   - Реализован метод `_convert_to_cli_args()`
   - Поддержка всех параметров из ProcessRequest
   - Добавлена поддержка `--rs-base`, `--run-audio`
   - Автоматическое добавление `--profile-path` после сохранения профиля

3. **Запуск subprocess**
   - Определение пути к main.py относительно структуры проекта
   - Использование переменной окружения PYTHON для выбора интерпретатора
   - Установка рабочей директории (cwd) в корень DataProcessor

### ✅ Сохранение profile_config в временный YAML файл

Реализован метод `_save_profile_config()`:
- Создание директории `_profiles_cache/{run_id}/` в корне проекта
- Сохранение profile_config в YAML формат через `yaml.safe_dump()`
- Путь к файлу передаётся в main.py через `--profile-path`
- Файл сохраняется для возможной отладки (cleanup закомментирован)

### ✅ Обработка stdout/stderr из subprocess

Улучшена обработка вывода subprocess:
- Логирование stdout на уровне DEBUG
- Логирование stderr на уровне WARNING
- Сохранение stdout/stderr в результате для анализа ошибок
- Извлечение основной ошибки из последней строки stderr

### ✅ Обработка exit code

Реализована детальная обработка exit code:
- Определение типа ошибки по exit code:
  - `0`: успех
  - `1`: общая ошибка
  - `2`: неправильное использование команд
  - `>128`: завершение по сигналу
- Извлечение сообщения об ошибке из stderr
- Возврат структурированного результата с типом ошибки

4. **Логирование**
   - Команда логируется на уровне DEBUG
   - stdout логируется на уровне DEBUG
   - stderr логируется на уровне WARNING

## Известные проблемы и ограничения

1. **Cleanup профилей**
   - Временные файлы профилей не удаляются автоматически
   - Могут накапливаться со временем
   - TODO: Реализовать cleanup через некоторое время или при завершении обработки

2. **Зависимость от yaml**
   - Требуется установка PyYAML
   - Не добавлен в requirements-api.txt (может быть уже в основном проекте)

3. **Путь к main.py**
   - Определяется относительно структуры проекта
   - Может не работать если структура изменена
   - В production лучше использовать абсолютный путь или переменную окружения

4. **Обработка stdout/stderr**
   - Весь вывод сохраняется в памяти
   - Для длительных процессов может быть проблемой
   - В будущем можно использовать streaming или ограничение размера
