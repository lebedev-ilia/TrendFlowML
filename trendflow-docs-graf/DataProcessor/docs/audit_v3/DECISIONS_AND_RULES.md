# Audit v3 — Decisions & Working Rules (source-of-truth)

Этот документ фиксирует **решения (FINAL)** и **рабочие правила** для Audit v3.
Все изменения компонентов/контрактов в рамках аудита должны соответствовать этим правилам.

## Контекст

- **Цель**: довести каждый компонент DataProcessor до **MVP+** качества по **логике/фичам/контрактам** так, чтобы после аудита **не менять алгоритмы**, а только включать/отключать фичи/компоненты по надобности (baseline + transformer + production).
- **Важно**: сейчас оптимизации и время выполнения не приоритет (ресурсы считаем почти неограниченными). Приоритет — качество, польза, корректность контрактов и воспроизводимость.

## Канонические инварианты проекта (напоминание)

См. `DataProcessor/docs/contracts/*`. Ключевое:

- **NPZ — source-of-truth**, JSON/HTML — presentation/debug.
- **Segmenter — единственный источник sampling** (`frame_indices` per-component, `union_timestamps_sec` как time-axis).
- **No-fallback policy** (для hard deps): отсутствующие зависимости/`frame_indices` → fail-fast.
- **Empty outputs валидны** (через маски + NaN + `empty_reason`), но должны быть строго определены.
- **Storage per-run** + единый `manifest.json`.

## РЕШЕНИЯ (FINAL) — Round 0

### 1) Downstream: baseline + transformer одновременно

**FINAL**: делаем сразу оба потребителя.

- Храним и/или предоставляем:
  - **sequence / time-series outputs** (для Encoder/v1 Transformer),
  - **tabular summaries** (для baseline/quality gate),
  - при необходимости оба слоя в одном NPZ (или отдельные NPZ) — но контракт должен быть явным.

### 2) Политика удаления/шумных фич

**FINAL**:

- Если считаем, что фича/алгоритм **мешает моделям больше чем помогает** → **удаляем смело** (с bump версии схемы/producer).
- Если фича **только для аналитики/дебага** и шумная/эвристическая → **можно не удалять**, но:
  - явно документируем как **heuristic/noisy/debug-only**,
  - маркируем как **не model_facing**.

### 3) Граница model_facing vs analytics vs debug

**FINAL**: **явная маркировка** (вариант “B” из вопросов).

- В документации компонента фиксируем:
  - **model_facing** (идет в baseline/v1),
  - **analytics** (для интерпретации/аналитики),
  - **debug-only** (локальный контроль качества, не часть прод контракта).

### 3.1) Schema system (machine-readable) — обязательное требование аудита

**FINAL (Audit v3)**: каждый audited компонент обязан иметь **четкую схему** и быть включён в **общую систему схем**.

- **NPZ остаётся source-of-truth**, но для каждого `schema_version` должен существовать:
  - **human schema**: `SCHEMA.md` рядом с компонентом (как читать ключи, dtype/shape, tiers),
  - **machine schema**: реестр схем (для VisualProcessor: `VisualProcessor/schemas/<schema_version>.json`).
- Машиночитаемая схема обязана фиксировать:
  - ключи NPZ,
  - `dtype`/`shape`,
  - `tier`: `model_facing | analytics | debug`,
  - required vs optional ключи (и policy для отсутствующих).
- **Валидация схемы в runtime**:
  - если `schema_version` известен системе схем → валидатор должен fail-fast при расхождении,
  - неизвестные схемы допускаются только как временный rollout (warning), но audited компоненты должны быть “известны”.
- **Правило обратной совместимости**:
  - любое существенное изменение ключей/dtype/shape → bump `schema_version`,
  - в audited состоянии рекомендуется `allow_extra_keys=false`, чтобы изменения контракта не проходили “тихо”.

Дополнение (глобально для Audit v3):
- **Schema bump по умолчанию**: если правка затрагивает контракт артефакта, мы **делаем bump `schema_version` сразу**, не пытаясь “втиснуть” изменение в старую версию.

### 4) Политика render (HTML/JSON) и “что отдаём на backend”

**FINAL**:

- В **production** рендеры (HTML) не используются.
- В **dev** рендеры сохраняем для вашей проверки качества.
- Уже сейчас в документации компонента фиксируем:
  - **что именно и в каком виде будет отдавать backend** (структура/поля/агрегаты),
  - рендеры — **временный debug**, который позже будет выключен/удалён при переносе на сайт.

#### 4.1) Render как критерий аудита (персонализация + human-first)

**FINAL (Audit v3)**: рендер должен быть **персонализирован под компонент** и **понятен человеку**, который:
- не знаком с проектом,
- может не знать Python,
- не разбирается в ML,
но запустил прогон и получил `render.html`.

Минимальное требование к **каждому** audited компоненту (общие правила “mini-dashboard”):

- **README обязан содержать большой раздел “Render (dev-only)”**, где описано:
  - какие файлы рендера создаются (`_render/render_context.json`, `_render/render.html`);
  - **как читать выход**: что означают ключевые графики/таблицы/поля;
  - какие **типовые распределения** ожидаем и какие **аномалии** искать;
  - как рендер связан с NPZ (source-of-truth);
  - где смотреть **время выполнения** (`meta.stage_timings_ms`);
  - какие **параметры конфига** сильнее всего меняют результат и стоимость.
- **render.html не должен быть “одним универсальным шаблоном”**.
  Он должен содержать компонент-специфические визуализации, иначе человек не сможет сделать QA качества.

Дополнительный **UX-стандарт** (то, что мы считаем “рендером уровня аудита”):

- **Страница = мини-дашборд**:
  - сверху: краткое “что это за компонент” + что именно он пишет в NPZ (1–3 предложения);
  - блок **Key facts**: `schema_version`, `producer_version`, `models_used`, `N frames/segments`, время выполнения (из `meta.stage_timings_ms`);
  - блок **Config highlights**: 5–10 наиболее важных параметров (и как они влияют на стоимость/качество);
  - блок **How to QA**: что считать нормой/аномалией (по данным компонента).
- **Навигация и секции**:
  - оглавление/якоря по секциям (Overview / QA / Tables / Examples / Meta);
  - кликабельные ссылки “jump-to” внутри страницы (например к конкретному кадру/событию/строке в таблице).
- **Интерактивность (без внешнего интернета)**:
  - таблицы с **поиском** и **сортировкой** (vanilla JS OK; без CDN-зависимостей);
  - фильтры по ключевым порогам (confidence/score/area/duration/… — по смыслу компонента);
  - “copy-friendly” значения (например JSON snippets / sha256 prefix / ids).
- **Top / Anti-top (обязательно)**:
  - минимум 1–2 метрики, по которым показываем **топ-пики** (лучшее/самое сильное) и **анти-топ** (хужее/подозрительное);
  - для каждого пика должна быть “точка входа” в **пример** (frame/event card, ссылка на картинку, строка в таблице).
- **Примеры / визуальные превью**:
  - если компонент привязан к кадрам/сегментам/событиям → рендер обязан показывать **несколько конкретных примеров** (K штук, равномерно/по пикам),
    чтобы человек мог оценить качество “глазами”, а не только по числам.
- **Assets policy (обязательно, если есть превью)**:
  - все генерируемые картинки/миниатюры сохраняем в `_render/assets/` и ссылаемся на них из HTML;
  - имена файлов должны быть детерминированы (чтобы ререндер не раздувал папку мусором);
  - рендер должен работать offline (открытие `render.html` из файловой системы).
- **Privacy (если есть персональные данные/текст/лица)**:
  - явный баннер/пояснение, что именно сохранено (raw vs hashed/len),
  - настройка в конфиге, которая это контролирует, должна быть указана в “Config highlights”.

Чеклист аудита (для ревью рендера):

- открываем `render.html` без запуска сервера → всё читаемо, секции/навигация работают;
- есть top/anti-top + быстрый переход к примерам;
- таблицы ищутся/сортируются, есть фильтры по порогам;
- есть “Key facts” и “Config highlights”, видно время выполнения;
- при наличии превью есть `_render/assets/` и они подхватываются.

Компонент-специфический минимум (пример обязательных targets для 5 core visual; расширяем аналогично для других компонентов по смыслу данных):

- `core_object_detections`: несколько кадров (K≥12) с **bbox overlay** + подписи классов/score; отдельный блок `text_region`.
- `core_depth_midas`: несколько кадров (K≥10) с **depth-map overlay** (нормализованная карта + легенда).
- `core_optical_flow`: несколько пар кадров (K≥10) с **motion heatmap** (`preview_flow_mag_map_norm`) + пики на timeline.
- `ocr_extractor`: несколько кадров с bbox + **распознанным текстом**, распределения confidence, явное предупреждение про privacy.
- `core_clip`: thumbnails sampled кадров + пики/провалы cosine similarity, интерпретация top сцен/промптов (без raw embeddings).

**Важно**: NPZ остаётся source-of-truth, а рендер — это “человеческая интерпретация” NPZ, не отдельный контракт данных.

### 5) Soft deps vs error vs valid empty — решаем per-component

**FINAL**:

- Для каждого компонента отдельно фиксируем:
  - что является **hard dependency** (нет → error),
  - что является **soft dependency** (нет → не error, но возможен `empty`/warning/quality drop),
  - какие причины считаются **valid empty** vs **error**.
- Примерная логика:
  - отсутствие `ocr_extractor` для `franchise_recognition` может быть **валидным empty/quality drop**,
  - но “компонент вообще не выдал распределение вероятностей” — скорее **error** (если по смыслу обязан всегда выдавать).

### 6) Sampling requirements: обязательные требования компонента

**FINAL**:

- Компоненты имеют приоритет над Segmenter: каждый компонент должен **требовать** нужную выборку для качества.
- Требования фиксируются как **обязательные** (fail-fast, если Segmenter не обеспечил).
- Segmenter будет аудирован **в конце**, когда соберём требования всех компонентов и синтезируем global sampling policy.

### 7) Empty_reason: строгий словарь + расширения

**FINAL**: строгий словарь + расширения через явное документирование (вариант “B”).

- Базовый словарь: см. `docs/contracts/ARTIFACTS_AND_SCHEMAS.md`.
- Если нужен новый `empty_reason`, добавляем его:
  - в документацию компонента,
  - (и при необходимости) в общий словарь контрактов.

### 8) Time-axis/alignment

**FINAL**: сохраняем привязку к time-axis где применимо (вариант “A”).

- Visual: `union_timestamps_sec` / индексы в union domain.
- Audio: секунда/сэмплы по `audio/segments.json`.
- Text: если есть alignment — фиксируем; если нет — явно определяем empty/error семантику.

### 9) ModelManager / no-network enforcement

**FINAL**: жёстко — все модели/веса через `dp_models` (ModelManager), без сетевых загрузок.

- В аудите допускается, что некоторые модели пока не заведены в ModelManager → тогда **дописываем ModelManager** вместе с фиксом компонента.
- Зафиксировано, что потенциально будет 1–2 “сложных” компонента — решим на месте.

### 10) Definition of Done компонента

**FINAL**: оба требования одновременно (вариант “C”):

- формализм/контракты/README/схемы/empty semantics/manifest/meta,
- плюс реальное качество/польза фич (подтверждение рендерами/статистикой).

### 11) Набор видео для валидации

**FINAL**: минимум **10–20 видео** разных типов (сначала можно smoke на 3–5).

### 12) Порядок старта

**FINAL**:

- Segmenter — в конце.
- Начинаем с `core_clip`.

## Sampling requirements — зафиксированные решения по компонентам (draft)

Segmenter будет аудирован **в конце**, но требования компонентов фиксируем по ходу аудита.

### Важно для sampling policy (train-time reality)

Для обучения downstream моделей реальный разброс по числу кадров в исходном видео (до sampling) примерно:

- **min ~120 кадров** (≈ 4 секунды при 30 fps)
- **max ~72000 кадров** (≈ 20 минут при 60 fps)

Это сильно влияет на shape/coverage и на то, как мы синтезируем global sampling curve (Segmenter в конце аудита).

### `core_clip` (primary visual sampling group)

**FINAL (Audit v3)**: для `core_clip` фиксируем требование к выборке, зависящее от длительности:

- **Если `duration_sec <= 120` (≤ 2 минуты)**: stride по кадрам = **7** (берём каждый 7-й кадр на analysis timeline)
- **Если `duration_sec > 120`**: непрерывная кривая budgets до 20 минут, и на **20 минут cap = 3000 кадров**

Пример непрерывной формы (для реализации в Segmenter; параметры можно уточнять при синтезе global policy):

- $N_{target}(T)=round(3000 * (min(T,1200)/1200)^{\\alpha})$, для $T>120$, где $\\alpha\\approx0.78$
