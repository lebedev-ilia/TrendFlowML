## 5. Сверка с кодом

- **Алгоритм:** `classic` backend → `librosa.pyin` + `librosa.yin` → очистка NaN → выбор метода по взвешенному score → по `enable_basic_stats` скаляры и `_calc_additional_metrics` (smoothness, jumps, octave histogram, skew/kurtosis).
- **`run_segments`:** на каждый сегмент `_extract_pitch_features`; `has_pitch` если `f0_mean` truthy и > 0; в агрегат идёт только **одно число `f0_mean` с сегмента** (не полный pyin-ряд). Поэтому метрики контура/jumps/skew на выходе относятся к **ряду из ~12 точек**, а не к кадрам STFT.
- **Валидация `run()`** требует `f0_mean` в `[fmin, fmax]`; для сегментного режима часть проверок мягче на уровне сегмента (пустые сегменты дают `segment_mask=false` или полный empty).

## 6. Вердикт

**Сильные стороны**

- Стабильный контракт v2: meta,tabular + каноническая ось сегментов + optional octave object.
- Скаляры f0 и производные от агрегата годятся для **tabular / baseline** и отчётов «средний тон / разброс / скачки».
- На этом run нет NaN в числовых рядах сегментов; маска полная — данные читаемые.

**Риски / ограничения**

- Без `--pitch-enable-time-series` в NPZ **нет плотного f0 по времени** (только npy debug при включении) — для **AudioEncoder** как dense sequence нужен либо другой артефакт, либо явное включение time-series и политика хранения.
- Смысл «контур/скачки» на **12 точках** слабее, чем на полном pyin; корреляции с другими модулями нужно проверять на diversity set.
- До фикса: мусорный NaN в таблице от строкового `backend`.
