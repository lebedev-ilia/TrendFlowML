## YOLO fine-tune plan v1 (41 класс, кадры из 100k видео, GPU: Tesla T4 16GB)

**⚠️ ВАЖНО**: Финальный набор классов (v1.0) утверждён и задокументирован в `yolo_fine_tune/YOLO_CLASSES_V1_FINAL.md`.

Цель: дообучить детектор для `core_object_detections` так, чтобы он был **сильным proposal layer** под popularity‑модель.
То есть детектор должен стабильно находить “регионы интереса” (bbox) и базовые объекты, а **бренды/конкретные люди/конкретные модели**
распознаются отдельными semantic heads (`core_brand_semantics`, future `core_face_identity`, etc.).

### Ключевая идея архитектуры (чтобы не перепутать роли)

- `core_object_detections` (YOLO): “где что находится” + треки + *грубая* семантика уровня объекта/региона.
- `core_brand_semantics` (CLIP‑matching): “какой бренд/логотип/эмблема”.
- `core_face_landmarks` (+ future identity): “кто именно этот человек”.
- `scene_classification` / future landmarks: “какое место/контекст”.

Поэтому в YOLO v1 мы **не** стараемся учить 500 брендов и 1000 знаменитостей — это отдельные ветки.

## 1) BBox vs segmentation: что выбираем в v1

### Рекомендация для v1: **bbox‑детектор**
Причины:
- разметка bbox дешевле в разы → быстрее выйти на рабочее качество
- проще QA и быстрее итерации (active learning)
- downstream head’ам (CLIP/OCR) чаще всего достаточно bbox‑кропа

### Отдельно про `text_region`: нужен ли он в YOLO, если есть “компонент для текста”
В текущем репо `VisualProcessor/modules/text_scoring` **не выполняет OCR**, а является consumer’ом **внешнего OCR NPZ** (`ocr_npz`).
Если OCR артефакт отсутствует — модуль возвращает валидный empty результат и не падает.

Поэтому для v1 рекомендация такая:
- **Да, держим `text_region` в YOLO** как proposal‑класс (это дешёво и даёт ранний сигнал “в кадре есть крупный текст”).
- В будущем делаем отдельный core/head OCR (det+rec), но **гейтим его через `text_region`**:
  - OCR работает по bbox‑кропам `text_region`, а не по всему кадру → сильно дешевле и стабильнее по latency.

Когда можно НЕ держать `text_region` в YOLO:
- если у тебя уже есть обязательный OCR/text-detection компонент, который всё равно запускается всегда, и ты готов держать его cost отдельно.
Но даже в этом случае `text_region` в YOLO часто полезен как “fast gating” и для shared sampling group.

### Когда в v2 имеет смысл добавить segmentation (точечно)
Если увидим, что bbox слишком “грязные” и ломают downstream:
- `logo_region` на сложном фоне
- `text_region` с большим количеством постороннего

Тогда сегментацию добавляем **не для всех классов**, а только для 1–3 “больных” регион‑классов.

## 2) Таксономия v1 для YOLO (40 классов): принципы

### Принцип 1: классы должны быть “объектного уровня”, а не брендов
Например:
- `phone` / `laptop` / `tablet` — да
- `iphone` / `macbook` — лучше делать в semantic head (бренд/модель) поверх `phone/laptop` proposals

Если “iPhone” как сущность критична, делаем компромисс:
- YOLO: `phone` + `apple_logo_region` (как регион‑класс) → дальше `core_brand_semantics` уточняет.

### Принцип 2: обязательно иметь **region‑proposal классы**
Они дают максимальную ценность:
- `logo_region`
- `text_region`
- `product_closeup`
- `package`

### Принцип 3: люди/машины/одежда — как “контекст/наличие”
Для popularity модели часто важны presence/coverage/частота:
- `person`
- `car`, `bus`, `truck`, `motorcycle`, `bicycle`
- `clothing_*` (см. ниже)

## 4) Данные: как из 100k видео сделать датасет под разметку (кадры)

### 4.1. Sampling strategy (v1)
Не размечаем “все кадры”, делаем репрезентативный сэмплинг:

- **Uniform baseline**: 1 кадр каждые \(X\) секунд (например 2–5 сек) для части видео.
- **Diversity sampling**: кластеризация по `core_clip` embeddings и выбор центроидов (меньше дубликатов).
- **Hard-negative sampling**: кадры, где базовый детектор даёт FP на logos/text/screens.

Практично для старта:
- взять **~10k видео** (из 100k) с максимальным разнообразием
- извлечь **2–4 кадра/видео** → **20–40k кадров** (это уже серьёзный датасет)

### 4.2. Active learning loop (обязательно, чтобы не утонуть в разметке)
Итерации:
1) Train0: взять небольшой seed (например 2–5k кадров), размеченных bbox.
2) Run0: прогнать Train0 по большой пачке неразмеченных видео → собрать:
   - FP кандидаты (`logo_region/text_region/screen_*`) с высокой уверенностью
   - FN кандидаты (там где люди руками говорят “логотип был, но не найден”)
3) Label1: размечаем только “информативные” кадры (ошибки/новизна).
4) Train1, Train2 … пока метрики и визуальный QA не стабилизируются.

Это типично экономит **2–5×** разметки.

## 5) Разметка: bbox pipeline, QA и стоимость

### 5.1. Что размечаем bbox
- все 40 классов в идеале, но на старте можно сделать “ядро”:
  - `logo_region`, `text_region`, `screen_*`, `phone`, `laptop`, `person`, `car`, `clothing_*`
- остальное добавлять после первой итерации, когда пайплайн работает

### 5.2. Инструменты разметки
Подойдут CVAT / Label Studio / Roboflow / any bbox tool.
Важно:
- экспорт в YOLO format (txt per image: class x_center y_center w h normalized)
- строгий class_id mapping (stable ids)

### 5.3. QA разметки (иначе модель не взлетит)
- двойная проверка на “тонких” классах:
  - `logo_region`, `text_region`, `screen_phone`
- правила:
  - bbox должен плотно обнимать объект/регион
  - для logo/text: **лучше чуть tight**, чем захватить фон

### 5.4. Оценка времени разметки (порядок величины)
Сильно зависит от плотности объектов, но ориентир:
- 20–40k кадров bbox → обычно **1–4 недели** работы небольшой команды разметки + QA
- с active learning можно начать с **2–5k** (1–3 дня) и быстро получить Train0.

## 6) Тренировка на Tesla T4 16GB: реалистичный план

### 6.1. Модель/размер
На T4:
- комфортнее fine-tune **small/medium** модели на 640 (batch умеренный)
- очень большие (x) будут медленнее и/или потребуют маленький batch

Рекомендация:
- начать с того, что у нас уже есть в проде/репо (`yolo11x_640` как baseline), но для T4 может быть тяжело.
- для итераций Train0/Train1 использовать **smaller** (s/m) для скорости, затем финально перетренить “большую” если надо.

### 6.2. Время тренировки (очень грубо)
На 640, 10k картинок:
- small/medium: **несколько часов – ~1 день** на 100–200 эпох (зависит от batch/augs)
На 30–40k:
- **~1–3 дня** на тех же настройках

Главный тайм‑синк — разметка, не тренировка.

### 6.3. Экспорт/деплой
- export → ONNX (фиксируем opset, dynamic axes как договоримся)
- собрать Triton model repo branch (`triton/models/<name>`)
- оформить dp_models spec:
  - `weights_digest`, `runtime_params`, `model_name/version`

## 7) Метрики и критерии “готово”

### 7.1. Offline метрики
- mAP@0.5 и mAP@0.5:0.95 на вал‑сете
- per-class recall для ключевых:
  - `logo_region`, `text_region`, `screen_phone`, `person`, `car`, `phone`, `laptop`

### 7.2. Product QA (важнее mAP для popularity)
На наборе видео (50–200):
- логотипы/текст/экраны находятся устойчиво в динамике
- FP на “похожих паттернах” (узоры, вывески, фон) приемлем
- downstream head’ы (CLIP/OCR) реально улучшаются от этих proposals

## 8) Что делаем прямо сейчас (конкретный v1 roadmap по датасету)

1) **Зафиксировать финальные 40 классов** и ids в `TAXONOMY_V1.yaml`.
2) Собрать seed‑датасет:
   - выбрать 2–5k кадров из разнообразных 1–2k видео
   - разметить bbox по “ядру” классов (logo/text/screen/person/phone/laptop/car/clothing)
3) Train0 на T4, быстро проверить на видео QA.
4) Active learning: собрать 5–10k “ошибочных/новых” кадров, разметить, Train1.
5) Когда proposals стабильны — расширить разметку до полного списка 40 классов и сделать финальный TrainFinal.
