## Фаза 1 — Полный анализ фичей из прогона (300 видео)

Цель владельца дословно: "берёт результаты этого прогона и анализирует их, строит распределения,
прогоняет через модели, находит корреляции — вообщем полный анализ фичей".

- [ ] **Распределения**: по каждой фиче в 8 NPZ-файлах на видео (per-component) — гистограмма/базовая
      статистика (mean/std/min/max/% NaN/% constant). Отдельно пометь: сколько фичей **константны по
      всем 300 видео** (кандидаты на удаление — как уже делалось для 6-видео выборки в
      `feature_audit_cleaning`, exp_0002, но теперь на несравнимо большей и разнообразной выборке —
      результат может отличаться, не копируй старый список вслепую, пересчитай).
- [ ] **NaN-аудит**: сверь с уже известными dead-фичами из
      `DataProcessor/docs/COMPONENT_DEEP_DIVE_PORTFOLIO_ASSESSMENT.md` §3 (напр. `loudness_lufs` 100%
      NaN, `video_pacing` config-drift NaN и т.д. — но эти компоненты частично пофикшены с тех пор,
      проверь текущее состояние на реальных 300 видео, не по старому отчёту).
- [ ] **Корреляции**: feature-feature (найти дублирующиеся/избыточные семейства — портфолио §3.8 уже
      называет кандидатов: audio "rhythm trio" tempo/onset/rhythmic, "tonality" chroma/hpss/key/
      spectral_entropy/spectral_flatness, text 3 независимых similarity computations — но это пока
      теоретические подозрения автора портфолио, не измеренные; измерь фактическую корреляцию, когда
      дойдёшь до Audio/Text данных). Feature-target корреляции — только если реальные таргеты
      присоединены (см. Фазу 0/п.1 ниже).
- [ ] **Прогон через модели**: используй существующий пайплайн `DataProcessor/DatasetBuilder/` (`
      build_training_table.py`, `add_targets.py`, `build_full_dataset.py`) + `Models/baseline/Training/
      train_baseline.py` — не изобретай новый тренировочный код с нуля, расширяй существующий по тому
      же паттерну, что exp_0001-exp_0004.
- [ ] **Стратифицированный анализ по видео-признакам**: используй `strata`/`cell` из `corpus300.json`
      (формат `duration_bucket/view_bucket/language`) — сравни распределение/качество фичей и (когда
      появятся) permutation importance по стратам. Владелец прямо просил: "на каких видео качество фичей
      лучше, а где хуже (например на коротких видео, или без людей)". Для "без людей" — придётся
      косвенно определять признак присутствия людей (пока `core_face_landmarks` не прогнан — можешь
      попросить Agent A прогнать его на подвыборке в приоритете, если для тебя это критично, через
      его доску/VK; либо оценить по имеющимся полям, если где-то в scene_classification/emotion есть
      косвенный сигнал).
- [ ] **Анализ работы Segmenter** (явно запрошено владельцем): сколько кадров реально сэмплируется на
      видео разной длительности, соответствует ли это заявленным бюджетам по компонентам
      (`cut_detection: 400-1500`, `core_clip: 200-800`, `core_depth_midas: 120-400`, `shot_quality:
      200-1000` — см. `DataProcessor/docs/contracts/SEGMENTER_CONTRACT.md`), есть ли видео, где
      сэмплирование выглядит аномально (слишком мало/много кадров относительно длительности), как
      влияет `drift_tol_sec` (недавно поднятый с 1с до 10с — сколько видео реально имеют drift 1-10с,
      это нормальный диапазон или маскирует более серьёзную проблему синхронизации у части видео?).

**Deliverable**: `Models/state/analysis/FEATURE_ANALYSIS_300CORPUS.md` (+ вспомогательные CSV/PNG в той

      применялся для DataProcessor GPU-пода в только что завершённой работе.

## Общие правила работы (коммуникация, git, бюджет)

- **VK**: коротко, отчёты по завершении Фаз, не по каждому шагу. Вопросы владельцу — редко и только
  реально важные (например: "нашёл, что реальные таргеты у корпуса Agent A ЕСТЬ/НЕТ, дальше делаю X").
- **Git**: работай в `Models/**` и `DataProcessor/DatasetBuilder/**` (твоя зона по уже сложившемуся
  разделению в `training_tasks.json`: "TrendFlow Bot (Agent A) works the DataProcessor side"). НЕ трогай
  код компонентов внутри `DataProcessor/VisualProcessor/`, `AudioProcessor/`, `TextProcessor/` — это
  зона Agent A, даже если ты там нашёл проблему — задокументируй находку и передай ему, не правь сам.
  Коммить часто, маленькими шагами, `git pull --rebase` перед push (второй агент коммитит параллельно).
- **Синхронизация с Agent A**: обновляй `training_tasks.json` своими статусами как обычно; для
  сообщений "для Agent A" — либо VK, либо явная пометка в своих файлах анализа (не редактируй его
  `AGENT_A_WORKPLAN_2026-07-21.md`/его отчёты).
- **Качество превыше скорости**: если данных недостаточно для честного вывода — так и напиши, не
  дописывай правдоподобную догадку как факт (та же дисциплина, что уже видна в твоих exp_0001-exp_0004
  записях — продолжай в том же духе, это хорошая практика).
