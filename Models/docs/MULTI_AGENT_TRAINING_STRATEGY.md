# Стратегия: несколько агентов + первый baseline на реальных данных

**Дата:** 2026-07-18. Отвечает на прямой запрос владельца — как организовать работу нескольких
автономных агентов над `Models/` (сбор фичей → обработка → обучение baseline) параллельно с уже
идущей работой над `DataProcessor/` (TrendFlow Bot / `deepdive_agent.py`).

## Что уже есть в `Models/` (проверено по коду, не по докстрингам)

- **Контракты полностью описаны**: таргет `y = log(1 + Δ)` для views/likes на 7/14/21д (`snapshot_0`
  как точка отсчёта), гибридный сплит время+канал (защита от утечки по каналу), метрики — Spearman
  как north-star + MAE + разбивка по 8 возрастным бакетам, golden-сеты (holdout=2000, regression_mini=200).
- **Код обучения baseline УЖЕ НАПИСАН и ни разу не запускался**: `Models/baseline/Training/
  train_baseline.py` (CatBoost/LightGBM/HistGradientBoosting, 6 моделей views/likes×7/14/21д, сплит,
  метрики, манифест) + `evaluate_baseline.py`, `generate_golden_sets.py`, `predict_baseline.py`.
  Директорий `artifacts/`/`golden_sets/` не существует — ни разу не прогонялось.
- **Реальный блокер — не компоненты, а сборщик датасета.** Контракты описывают `DataProcessor/
  DatasetBuilder/` (`feature_spec.yaml`, `build_training_table.py`, `add_targets.py`,
  `build_full_dataset.py`) как единую точку сборки NPZ→плоская таблица+таргеты — **этого кода нет
  вообще**, только описание в доках. Это единственное, что реально должно быть написано с нуля.
- **Замороженный список фичей v1.0 (`BASELINE_MODEL.md`) содержит компоненты, которых сейчас нет**:
  `core_brand_semantics`, `core_car_semantics`, `core_place_semantics`, `core_face_identity` (все —
  stub-базы, только что решили не трогать) и полагается на `core_object_detections` (сейчас на
  COCO-весах, идёт переразметка). **Если обучать строго по замороженному списку — обучение будет на
  шуме/константах по 4-5 из ~19 групп фичей.** Это первое, что нужно решить (см. §4/§7).
- Таргеты реальны — снапшоты просмотров/лайков лежат в HF-датасете владельца (`Ilialebedev/*`),
  Fetcher уже это собирает.

## 1. Как правильно описать задачи агентам

Задача агента = его системный промпт + ссылки на канонические документы (протоколы, контракты,
чеклисты), а НЕ пересказ этих документов своими словами внутри промпта — иначе при любом изменении
протокола придётся править код промпта отдельно, и они разъедутся (ровно то, что уже видели в отчётах
как «фикс есть, но не там, где ожидали»). Рабочий шаблон описания задачи (уже применён к
`deepdive_agent.py`, использовать тот же):

1. **Роль одним предложением** — не «помогай с моделями», а «ты собираешь плоскую таблицу фичей и
   таргетов из NPZ-артефактов согласно `DatasetBuilder`».
2. **Источник истины** — конкретный путь к документу-протоколу, который агент читает целиком в
   системный промпт (как `COMPONENT_DEEP_DIVE_PROTOCOL.md` сейчас). Если протокола ещё нет —
   первая задача агента ИМЕННО написать его (не код), и только потом писать код по нему.
3. **Границы (что НЕ трогать)** — явный список: не менять `core_object_detections`, не тратить время
   на brand/car/place/franchise, не резать другие ветки пайплайна.
4. **Definition of done** — не «сделай хорошо», а проверяемый факт: файл создан, метрика посчитана,
   строка в чеклисте/логе появилась. Мы уже используем этот принцип для автопродолжения очереди
   компонентов (сверка чеклиста до/после) — тот же приём работает и для обучения: «готово» = появился
   `metrics.json` с непустыми числами, не текстовое заявление агента.
5. **Когда звать владельца** — явный, короткий список триггеров (данные, деньги/лицензии, юр. вопросы,
   развилка с высокой ценой ошибки), и явный запрет спрашивать про рутину. Это уже сформулировано в
   промпте `deepdive_agent.py` — использовать тот же список, не изобретать заново.

## 2. Как выстроить взаимодействие между агентами

У Claude-агентов нет прямого канала друг с другом — только файлы в общем репозитории и (если нужно)
отдельные VK-боты для человека. Не пытаться делать «агент А пишет агенту Б» — вместо этого:

- **Общий файл-очередь заданий** (тот же паттерн, что `state/queue.json`/`claim.json` у
  `agent_runner.py` и чеклист у `deepdive_agent.py`) — один агент кладёт туда задачи, другой их
  забирает, помечает статус. Никакого прямого сообщения между агентами не нужно.
  Ты уже держишь это в голове с чеклистом компонентов; ровно тот же принцип — новый файл
  `Models/state/training_tasks.json` (или аналог) со статусами `⬜/🔄/✅` по каждому шагу пайплайна
  (`dataset_builder`, `feature_audit`, `baseline_v0_train`, `baseline_v0_eval`, …).
- **Общий лог переписки** (как `agent1_chat.log`/`fetcher_chat.log`) — каждый агент дописывает свои
  действия туда; следующий агент читает хвост лога перед стартом, чтобы понимать контекст без диалога.
- **Git — единственная реальная точка синхронизации кода.** Если два агента правят код одновременно —
  риск конфликтов. Простое правило: agents НЕ работают параллельно над одним и тем же файлом/этапом
  пайплайна; порядок этапов (§4) и так последователен, значит естественным образом только один агент
  одновременно «держит перо» на конкретном шаге.
- **VK-боты — для человека, не для агентов.** Если хочешь видеть прогресс каждого агента отдельно —
  да, отдельный VK-бот на агента (по образцу Fetcher-watchdog, третий бот). Но это канал «агент →
  ты», не «агент → агент».

## 3. Какие результаты должны получиться (учитывая неготовые компоненты)

Реалистичная цель первого прохода — **не** «финальная точная модель», а **работающий сквозной
конвейер + первое честное число, от которого можно отталкиваться**:

- Готовый `DatasetBuilder` (код, не только доки) — воспроизводимо строит таблицу фичей+таргетов из
  того, что реально есть в `result_store` сейчас.
- Первый обученный baseline (**v0-схема фичей**, см. §7) — набор Spearman/MAE по 6 таргетам на
  реальном (пусть небольшом) холдауте, плюс feature importance — как диагностика, не как продукт.
- Явный список «что не вошло и почему» (semantic-identity семья, `core_object_detections`-семантика
  текста/логотипов) — чтобы у оценки метрики был честный контекст, а не завышенное или заниженное
  ожидание.
- План **v1**: как только семантические базы/детектор дозреют — просто пере-собрать `DatasetBuilder`
  с расширенным `feature_spec.yaml` и переобучить, сравнить с v0 — это уже воспроизводимая процедура,
  не разовая работа.

Не ждать «идеальных» метрик с первого прогона — цель первого прогона: пайплайн работает, числа не
NaN/не подозрительно идеальны (см. утечку в §7), дальше есть с чем сравнивать последующие итерации.

## 4. Порядок действий (пересмотрен относительно предложенного)

Ты предложил: сначала доработать компоненты → тестирование → обучение → масштаб. По факту (после
аудита 75 компонентов) большинство компонентов АРХИТЕКТУРНО готовы — не хватает именно сборщика
датасета и реального прогона батча, а не доработки алгоритмов. Поэтому предлагаю не ждать 100%
готовности всех компонентов, а вести две ветки частично параллельно:

1. **`DatasetBuilder` — писать уже сейчас**, не дожидаясь fix'ов компонентов. Он работает с ЛЮБЫМ
   реальным NPZ на диске — качество входа улучшится позже само, когда компоненты дозреют, пере-прогон
   `DatasetBuilder` — не отдельная разработка, а повторный вызов той же функции.
2. Параллельно — TrendFlow Bot продолжает то, что уже поручено: пересчёт батчей после committed-фиксов
   (портфолио-оценка §7.1), подбор реального тест-корпуса (`AUTOMATED_TEST_CORPUS_PROTOCOL.md`).
3. Как только есть таблица фичей+таргеты хотя бы на подобранном тест-корпусе (150-250 видео) —
   **первый прогон baseline** на v0-схеме (без stub-компонентов). Это самая ранняя точка, где можно
   получить реальную обратную связь: работает ли вообще пайплайн, есть ли утечка, разумны ли метрики.
4. Только после этого — расширение: больше видео (ближе к 100k корпусу), больше фичей (по мере
   готовности семантических баз/детектора), сравнение v0 vs v1.
5. Более тяжёлые модели (Encoder/transformer, v1 из `V1_TRANSFORMER_MODEL.md`) — после того, как
   baseline даст осмысленный ориентир по метрикам; не раньше.

Логика: не растягивать «сначала всё почини» на месяцы, когда первый честный прогон можно получить
за один цикл работы одного агента на существующих данных.

## 5. Сколько агентов и с какими моделями

Предложенное разделение (аналитик / кодер по указаниям / тестировщик) вводит координационные издержки
без реальной выгоды параллелизма — сборка датасета → обучение → оценка это **последовательный**
конвейер, не три независимых потока работы. Разбивать последовательный процесс на 3 агента означает,
что два из трёх всегда простаивают, ожидая передачи эстафеты, а передача через файлы/лог всё равно
нужна на каждом шаге.

**Предлагаю 2 агента, не 3:**

- **Агент A — продолжение текущего TrendFlow Bot** (`deepdive_agent.py`), занимается DataProcessor-
  стороной: пересчёт батчей, подбор корпуса, дозревание компонентов по мере появления данных.
  Модель — Opus (уже так настроено), это работа с суждениями/архитектурными решениями.
- **Агент B — новый «Models Bot»**, отдельный процесс (можно на том же VK_TOKEN, что и A, но НЕ
  одновременно — либо третий бот со своим VK_TOKEN, если нужно видеть оба параллельно). Ведёт
  `DatasetBuilder` → baseline train → eval как один непрерывный конвейер, а не отдельные роли —
  именно потому что шаги строго последовательны, играть в «кодер ждёт тестировщика» тут не нужно.
  Модель — тоже Opus для первого прохода (проектирование схемы фичей, решения про утечку/сплиты —
  это суждения, не рутина); можно понизить до Sonnet на повторных прогонах, когда дизайн устоялся.

**Третий агент — не сейчас, но по конкретному триггеру**: если/когда метрики Агента B станут
основанием для реальных продуктовых решений (не просто «работает ли пайплайн», а «выбираем ли эту
версию модели в прод») — тогда есть смысл в независимой проверке (аналог `/verify` у
`agent_runner.py` — тот же паттерн независимой валидации, что уже есть для компонентов). До этого
момента одного Агента B, который сам считает golden/regression_mini сеты и явно рапортует числа,
достаточно — не плодить процессы ради формы.

## 6. Стоит ли дорабатывать baseline-код (`train_baseline.py` и соседи)

**Нет, не сейчас.** Код написан осмысленно (CatBoost/LightGBM/HistGB, метрики по контракту, манифест
запуска) и ни разу не проверялся на реальных данных — любые доработки вслепую рискуют чинить то, что
не было проблемой, и пропустить то, что реально сломается только при первом реальном запуске
(например: как `select_numeric_features` поведёт себя на реальной v0-схеме, не потащит ли она
случайно ID-колонки или что-то с утечкой). Правило: **сначала прогнать как есть на первой реальной
таблице, фиксировать РЕАЛЬНЫЕ ошибки/трения, только тогда чинить** — не оптимизировать заранее.

## 7. Как обрабатывать большой объём фичей — методология

- **Отбор фичей на входе — используй уже готовый аудит.** У нас уже есть 75 FINAL_REPORT.md, где
  прямо названы мёртвые/константные/дублирующие фичи (портфолио-оценка §3.8 и раздел по каждому
  компоненту). Не нужно заново считать корреляции, чтобы понять «rhythm-трио», «tonality-семья»,
  `mfcc`≈производная `mel` и т.д. — просто исключи их из `feature_spec.yaml` v0 сразу, это уже
  сделанная работа.
- **Нормализация — низкий приоритет для первой итерации.** Градиентный бустинг (CatBoost/LightGBM/
  HistGB) не чувствителен к масштабу признаков (сплиты по порогу, не по расстоянию) — не трать на это
  время сейчас. Понадобится, только если позже добавишь линейную/нейросетевую модель поверх тех же
  фичей.
- **NaN — не заполнять бездумно.** У бустингов (особенно LightGBM/CatBoost) есть нативная обработка
  пропусков с обучаемым направлением сплита — она часто лучше, чем impute нулём/средним, потому что
  сам факт «пропущено» может быть частью сигнала (например «лиц не обнаружено» коррелирует с
  ретеншеном иначе, чем «лицо есть, но эмоция X»). Заполнять руками только то, что по природе НЕ
  missing, а «отсутствие явления» (например `count=0`, а не `count=NaN`).
- **Корреляция/избыточность — досчитать после первого прогона, не до.** Убрать вручную только уже
  ИЗВЕСТНЫЕ дубли (аудит выше); остальное — посмотреть на feature importance первого прогона, потом
  сокращать по факту, а не по догадке.
- **Таргет** — уже log1p(Δ) по контракту, оставить как есть, это разумный выбор для тяжёлых хвостов
  распределения просмотров.
- **Сплит** — гибридный время+канал уже специфицирован, использовать как есть; для подбора
  гиперпараметров внутри тренировочной части — `GroupKFold` по `channel_id`, не обычный k-fold
  (иначе повторится та же утечка, от которой защищает основной сплит).
- **Метрики** — Spearman как north-star уже верный выбор для предсказания просмотров (устойчив к
  выбросам и монотонным нелинейностям лучше, чем MAE/RMSE) — оставить как специфицировано.
- **Модели** — начать с CatBoost или LightGBM (нативные категориальные + NaN), дефолтные/слегка
  подобранные гиперпараметры на первом прогоне — цель первого прогона понять, есть ли сигнал вообще,
  не выжать максимум. Тюнинг (Optuna/RandomizedSearch) и ансамблирование (усреднение сидов,
  блендинг CatBoost+LightGBM) — только после того, как одиночная модель показала осмысленный сигнал;
  ансамблировать шум бессмысленно.
- **Обязательный шаг перед тем, как доверять числам — аудит утечки.** Явно проверить, что ни одна
  фича не построена из данных, недоступных на момент `snapshot_0` (ровно как уже поймали в отчётах
  `comments_embedder`/`comments_aggregator` — публикация → комментарии постфактум). Подозрительно
  идеальная метрика на первом прогоне — почти всегда сигнал утечки, а не успеха.
- **SHAP/feature importance с первого прогона** — не только для интерпретируемости аналитику, но и
  как встроенная проверка на здравый смысл (если топ-фича по важности выглядит как утечка — это и есть
  способ её поймать).

## 8. Что ещё предложить

- **Сухой прогон на маленьком корпусе прежде масштаба.** Прогнать весь конвейер (DatasetBuilder →
  train → eval) на тех же 150-250 видео из `AUTOMATED_TEST_CORPUS_PROTOCOL.md`, прежде чем пытаться
  на 100k — дешёвая проверка интеграции перед дорогой попыткой на всём корпусе.
- **Версионирование схемы фичей уже заложено в контрактах (`feature_schema_version` v0/v1)** —
  начать явно с `v0-real` (без stub-компонентов) вместо того, чтобы держать «замороженный v1.0» из
  доков как единственную цель — переименовать/пересмотреть его как только identity/detector реально
  дозреют, а не тянуть несовместимый с реальностью список сейчас.
- **Простой лог экспериментов** (даже CSV/markdown: id, версия схемы, модель, метрики, дата) —
  при нескольких агентах и долгой работе легко потерять историю, кто что уже пробовал; не нужен
  тяжёлый MLflow, достаточно дисциплины одного файла.
- **Переиспользовать уже готовую инфраструктуру процессов** — systemd + git credential store +
  VK-бот, всё это уже настроено для Fetcher/deepdive; Models Bot ставится по тому же шаблону, не
  изобретать заново.
