# ADR 0001: Celery, Redis (брокер + pub/sub) и manifest как source of truth

- **Статус:** принято  
- **Дата:** 2026-04-02  
- **Связанные документы:** [RUNS_AND_WORKERS.md](../RUNS_AND_WORKERS.md), [EVENTS_AND_LOGGING.md](../EVENTS_AND_LOGGING.md), [STORAGE_LAYOUT.md](../STORAGE_LAYOUT.md), [OVERVIEW.md](../OVERVIEW.md), `app/worker.py`, `app/services/events.py`, `app/tasks/`

---

## Контекст

Обработка видео через DataProcessor — **длительная** и **ресурсоёмкая**: HTTP-запрос не должен удерживать соединение до завершения пайплайна. Нужно:

1. Отложить работу в фоне и масштабировать исполнителей независимо от процесса API.
2. Доставлять **прогресс и статусы** клиенту (в т.ч. WebSocket по ingestion run) без опроса БД на каждый шаг.
3. Однозначно определить **итог работы**: какие компоненты посчитаны, где лежат NPZ/HTML, как связать это с предсказаниями в продукте.

PostgreSQL в схеме `core.*` нужен как **индекс** (пользователи, workspace, статусы job, ссылки), но не как единственное хранилище тяжёлых результатов.

---

## Решение

### 1. Celery для фоновых задач

- **Брокер и backend результатов:** один и тот же **Redis** (`TF_BACKEND_REDIS_URL`), см. `app/worker.Celery`.
- **Задачи:** пакет `app/tasks/` — в первую очередь `process_analysis_job`, `process_ingestion_run`, периодическая `sync_ingestion_run_status` (расписание beat в `worker.py`).
- **API** ставит задачу и возвращает быстрый ответ (например, 202/201); дальнейшая работа выполняется в **worker-процессах**.

### 2. Redis pub/sub для событий run (отдельно от очереди Celery)

- **Канал:** `run:{run_id}` — см. `app/services/events.py` (`run_channel`, `publish_run_event`, `subscribe_run_events`).
- Worker и смежный код публикуют JSON-события прогресса; **WebSocket** `/api/runs/{run_id}/events` подписывается на тот же канал и транслирует клиенту (после проверки JWT и владельца run).
- Таким образом, **два использования Redis:** транспорт задач Celery и **шина событий** для real-time UI; протокол pub/sub простой и достаточный для текущего масштаба.

### 3. Файловое хранилище и `manifest.json` как source of truth

- **Канонический результат** пайплайна: каталог **result store** (структура platform_id / video_id / run_id), файл **`manifest.json`** и артефакты (NPZ, отчёты и т.д.) — см. [STORAGE_LAYOUT.md](../STORAGE_LAYOUT.md), контракт в DataProcessor.
- После успешного прохода task **читает manifest**, синхронизирует сводку в `AnalysisJob` / **Prediction** и регистрирует артефакты — логика в `app/tasks/manifest.py` и связанных вызовах из `app/tasks/analysis.py`.
- **БД** отражает состояние для запросов API и UI; при расхождении приоритет у **файлов и manifest** как у полного описания прогона.

## Последствия

**Плюсы**

- Предсказуемое разделение: API быстрый, тяжёлая работа в worker.
- Горизонтальное масштабирование workers; Redis остаётся общей точкой.
- UI получает события без жёсткой привязки к реализации DataProcessor.
- Воспроизводимость и отладка: manifest и артефакты на диске.

**Минусы / риски**

- Нужны **как минимум два процесса** в проде: API + worker; для расписания sync ingestion — ещё **beat** (в docker-compose по умолчанию не включён).
- Потеря или рассинхрон Redis влияет и на очередь, и на pub/sub — операционно Redis становится **критичной** зависимостью (уже отражено в `/health/ready`).
- Deprecation в клиенте Redis (`publish_run_event` / subscribe): планово заменить на `aclose()` и актуальный API клиента (см. предупреждения в тестах).

**Альтернативы (отклонены для текущей версии)**

- Синхронный HTTP «дожаться всего» в одном запросе — неприемлемо по времени и таймаутам.
- Только БД для прогресса без pub/sub — лишняя нагрузка и сложнее real-time.
- Другой брокер (RabbitMQ, SQS) — избыточно для текущего этапа при уже принятом Redis для Celery.

## Итог

Архитектура **Celery + Redis + manifest/файлы как SoT** осознанно выбрана для разделения concerns, масштабирования воркеров и честного хранения результатов анализа. Детали сценариев и статусов — в [RUNS_AND_WORKERS.md](../RUNS_AND_WORKERS.md) и [EVENTS_AND_LOGGING.md](../EVENTS_AND_LOGGING.md).
