# E2E Worklog 2026-03-13

This note captures the concrete fixes that were applied after the full `Backend -> Fetcher -> DataProcessor` chain was already passing, plus the runtime checks that confirmed the remaining observability and usability issues were resolved.

## Scope

Goal of this pass:

- make `DataProcessor /status` reflect real processor state and progress during and after a run;
- reduce backend sync noise from stale `Fetcher` ingestion runs;
- make `backend/scripts/e2e_run_to_complete.py` print useful progress immediately in non-interactive execution;
- document the new storage/runtime expectations for local E2E.

## Changes Applied

### 1. `DataProcessor` status/progress aggregation

Files:

- `DataProcessor/api/services/state_reader.py`
- `DataProcessor/api/tests/unit/test_state_reader_cache.py`

What changed:

- `StateReader` now derives status/progress from aggregated `run_state.json` even when Redis hot-path is incomplete.
- The reader now uses `profile_config.processors` from Redis metadata to determine the effective processor set.
- Disabled processors (`audio`, `text`, `visual` in the lightweight E2E profile) are now exposed as `skipped` instead of stale `waiting`.
- Overall progress is now calculated only from enabled processors, so a `segmenter-only` E2E run reaches `overall=1.0` on success instead of being diluted by disabled processors.
- `started_at` / `finished_at` are backfilled from processor snapshots when top-level run fields are missing.
- After cold-path aggregation, the result is written back into Redis to refresh the hot cache.

Why:

- Before this change, the API could return `status=success` with `overall=0.0`, `stage=null`, and no useful component detail because it was effectively reading a thin Redis state instead of the persisted run state.

### 2. E2E storage root alignment

File:

- `backend/scripts/e2e_env.sh`

What changed:

- Local E2E now sets `STORAGE_ROOT` to the repository `storage/` directory instead of `/tmp/dataprocessor-storage`.
- `ALLOWED_VIDEO_PATHS` and `VIDEO_URL_CACHE_DIR` are aligned with that same repo-local storage root.

Why:

- `DataProcessor/main.py` already writes `state/` and `result_store/` under repo storage for ingestion runs.
- `DataProcessor API` was previously reading its state from `/tmp/dataprocessor-storage`, so `/status` could not see the persisted run files that actually existed.

Operational note:

- Required directories for local E2E are now:

```bash
mkdir -p ./storage/videos ./storage/uploads ./storage/videos/_url_cache
```

- `backend/scripts/start_e2e_stack.sh` already creates `STORAGE_ROOT/videos` and `STORAGE_ROOT/uploads`; when restarting pieces manually, create them yourself first.

### 3. Backend sync noise reduction

Files:

- `backend/app/config.py`
- `backend/app/tasks.py`

What changed:

- Added `TF_BACKEND_INGESTION_SYNC_LOOKBACK_HOURS` (default `6`).
- `sync_ingestion_run_status` now polls only recent pending/running runs inside that lookback window.
- The polling query now prioritizes newest runs first with `updated_at.desc()`.

Why:

- Older stale runs were being polled ahead of fresh ones, generating avoidable noise and occasionally contributing to bursty `Fetcher` traffic.
- After the change, recurring sync cycles during validation completed in milliseconds with no `429` noise.

### 4. `e2e_run_to_complete.py` stabilization

File:

- `backend/scripts/e2e_run_to_complete.py`

What changed:

- All output now uses flushed printing, so progress is visible immediately in non-interactive/background execution.
- Added optional DataProcessor progress polling via `--dataprocessor-url` (defaults from `TF_BACKEND_DATAPROCESSOR_API_URL`).
- In full E2E mode, the script can now print both Fetcher progress and DataProcessor status/stage/overall progress on the same line.

Why:

- Previously the script could appear silent for too long in automation even though the run had already started.
- Backend-only output also hid the true DataProcessor stage; now the script can surface `segmenter` and `overall=0.50` directly from `DataProcessor /status`.

#### D. SSE and Redis idle-noise follow-up

Files:

- `DataProcessor/api/endpoints/runs.py`
- `DataProcessor/api/services/redis_client.py`
- `DataProcessor/api/services/redis_schema.py`

What changed:

- `DataProcessor /events` now uses `StreamingResponse` for already-formatted SSE lines instead of wrapping them in `EventSourceResponse`.
- Redis client socket timeout was raised above the Streams `BLOCK` timeout so normal idle waiting does not surface as `TimeoutError`.
- Duplicate writes of the same final status in Redis hot-path are now treated as benign refreshes instead of invalid transitions.

## Current Recommended Local E2E Procedure

```bash
./backend/scripts/setup_e2e_infra.sh
mkdir -p ./storage/videos ./storage/uploads ./storage/videos/_url_cache
./backend/scripts/start_e2e_stack.sh
```

Then run:

```bash
cd backend
source scripts/e2e_env.sh
.venv/bin/python -u scripts/e2e_run_to_complete.py \
  --source-url "https://www.youtube.com/watch?v=DYor3e2effY" \
  --with-dataprocessor \
  --fetcher-url http://localhost:8000
```

For the broader but still practical DataProcessor-only smoke profile:

```bash
python DataProcessor/main.py \
  --video-path ./storage/videos/_url_cache/<run>.mp4 \
  --output ./storage/frames_dir \
  --rs-base ./storage/result_store \
  --global-config DataProcessor/configs/e2e_audio_smoke.yaml \
  --no-run-visual
```

For manual teardown:

```bash
./backend/scripts/stop_e2e_stack.sh
```
