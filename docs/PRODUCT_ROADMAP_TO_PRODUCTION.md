# Дорожная карта до продакшена (TrendFlowML)

Документ фиксирует **согласованный глобальный план эволюции системы** от текущего состояния (Audit v3 Audio, E2E в разработке) до масштабной обработки и продакшена. Порядок фаз можно смещать, но зависимости между ними стоит учитывать.

---

## Фаза 0 — Текущая база (коротко)

- Audit v3 и smoke/full-тесты **AudioProcessor** (21 экстрактор).
- Подготовка HF cache для emotion_diarization (`prepare_hf_cache.sh`), документация тестирования.
- E2E **Backend → Fetcher → DataProcessor (audio path)** — доводится до стабильного прогона (см. `backend/docs/E2E_RUNBOOK.md`), в т.ч. env Fetcher / YouTube Data API vs mock.

**Критерий выхода:** повторяемый E2E с AudioProcessor + валидация артефактов по контракту.

---

## Фаза 1 — TextProcessor: Audit v3 + тесты

**Цель:** привести TextProcessor к тому же уровню дисциплины, что и Audio: схемы NPZ/JSON, валидаторы, smoke/full по профилям, privacy-first контракты (где применимо).

**Рекомендации:**

- Явно зафиксировать **контракт входа** из ASR/token-only пути vs сырого текста (уже задекларировано в Audit v3 для Backend/DataProcessor).
- Параллельно готовить **минимальный E2E-режим** с текстом: либо автоген из ASR артефакта, либо фикстура `VideoDocument`.
- Документировать отличия от Audio: нет Segmenter-окон в том же виде, но есть зависимость от upstream артефактов.
- **Изолированный dev-смок по 22 TextProcessor-экстракторам** на JSON-сценариях Audit v3 зафиксирован в репозитории: скрипт `DataProcessor/TextProcessor/scripts/smoke_each_extractor_audit_v3.py`, описание и флаги — `example/text_audit_v3_smoke/scenarios/README.md`; запись прогона — `DataProcessor/docs/audit_v3/RUN_LOG.md`.

**Критерий выхода:** smoke/full по текстовому пайплайну + раздел в общем Testing/E2E чеклисте.

---

## Фаза 2 — Сквозной E2E и «полный выход» всех процессоров

**Цель:** после стабилизации Audio E2E и Audit Text — прогон **полной цепочки** (Visual + Audio + Text по выбранному профилю) и осознанный разбор **всех артефактов** (manifest, NPZ, HTML/render, meta).

**Рекомендации:**

- Не смешивать в одном прогоне «всё включено» без профиля: завести **эталонный audit-профиль** «full_multimodal» с явным списком процессоров и extractors.
- Фиксировать **матрицу**: какой процессор что пишет в storage и что потребляет downstream (таблица зависимостей).
- Добавить **автоматическую сверку**: наличие ожидаемых путей в manifest, версии schema_version, отсутствие required keys.

**Критерий выхода:** один документированный сценарий «full run» + скрипт или чеклист валидации полного выхода.

---

## Фаза 3 — Segmenter + материализация фич в БД / табличном слое

**Цель:** унифицировать сэмплинг (Segmenter как single owner) и **сохранение агрегированных/плоских фич** для аналитики и обучения — «в базу или в таблицу (pandas/parquet)».

**Рекомендации:**

- **Не тащить сырые NPZ в SQL построчно** как единственный источник: нормальная схема — Npz/Parquet в object storage + **индекс в БД** (run_id, component, schema_version, uri, hash, размеры).
- Если нужен pandas — опереться на **Parquet/DuckDB** как промежуточный слой для 100k+ видео; pandas — для ноутбуков и небольших срезов.
- Любые изменения Segmenter оформлять как **версия sampling_policy** (уже в контракте), чтобы старые ряды не смешивались с новыми.

**Критерий выхода:** спецификация таблицы/хранилища фич + миграция или ETL из существующего result_store.

---

## Фаза 4 — Подгонка алгоритмов и лёгких моделей под фичи

**Цель:** на разнообразном наборе видео прогнать фичи, обучить **лёгкие** модели (baseline), смотреть **вклад признаков**, убирать шум, корректировать экстракторы/флаги.

**Рекомендации:**

- Ввести **единый feature store / датасет-версию** (`dataset_v1`, seed, список video_id), иначе сравнение «до/после» бессмысленно.
- Отделить **продуктовые метрики** (качество предсказания) от **инженерных** (стабильность фич, доля NaN, время обработки).
- Любое удаление/изменение фичи — через **deprecation в schema** и версию контракта, а не «тихий» сдвиг.

**Критерий выхода:** отчёт по feature importance + список изменений в профилях/extractors с обоснованием.

---

## Фаза 5 — Инфраструктура под масштаб (~100k видео)

**Цель:** очереди, воркеры, идемпотентность, наблюдаемость, стоимость, отказоустойчивость.

**Рекомендации (обязательно учесть заранее):**

- **DataProcessor API vs worker:** общее состояние задач в Redis/БД, а не только in-memory TaskManager.
- **Горизонтальное масштабирование** воркеров с разделением по типу задач (GPU/CPU).
- **Квоты и приоритеты**, лимиты длительности видео, политика retry и dead-letter.
- **Стоимость:** батчирование (DynamicBatch), отдельные пулы под тяжёлые экстракторы.
- **Бэкапы storage**, политика TTL для временных файлов и кэша.

**Критерий выхода:** нагрузочный план + минимальный production runbook (уже частично в `backend/docs/OPERATIONS.md` и аналогах — расширить под 100k).

---

## Зависимости между фазами (схема)

```
Фаза 0 (E2E Audio стабилен)
    → Фаза 1 (Text Audit v3)
    → Фаза 2 (Full E2E + полный выход)
    → Фаза 3 (Segmenter + материализация фич)
    → Фаза 4 (ML/feature tuning на стабильном датасете)
    → Фаза 5 (Масштаб и прод-инфра)
```

Фазы 3 и 4 **частично можно вести параллельно** (например, прототип parquet-слоя до финальной заделки Segmenter), но **версионирование sampling** должно быть согласовано до масштабного обучения.

## Наблюдаемость: удобный мониторинг (сквозной минимум)

Цель — чтобы **одним взглядом** было видно: жив ли пайплайн, где узкое место, и не деградирует ли качество артефактов/моделей. Не обязательно поднимать всё в Фазе 0; базу закладывают рано, **разворачивают по мере роста нагрузки**.

### Рекомендуемый каркас

| Слой | Назначение | Практика |
|------|------------|----------|
| **Логи** | Разбор инцидентов, трассировка одного видео/run | Структурированный JSON; обязательные поля: `service`, `level`, `correlation_id` (например `run_id` / `celery_task_id`), `video_id` где есть; единый формат для Backend, Fetcher, воркеров, DataProcessor. |
| **Метрики** | Дашборды, алерты, тренды | Prometheus-совместимый scrape (или OTLP → коллектор → тот же бэкенд). RED для HTTP: rate, errors, duration (p50/p95). Для Celery: длина очереди, время ожидания в очереди, время выполнения задачи по типу. |
| **Трейсы** | E2E «где застряло» (опционально до Фазы 2+) | OpenTelemetry: spans от API до воркера и ключевых стадий DataProcessor; экспорт в Jaeger/Tempo/Grafana Tempo. |
| **Дашборды** | Операционный и «качество контракта» | Grafana (или аналог): отдельные панели по фазам — см. таблицу ниже. Имеет смысл **два уровня**: *Ops* (латентность, ошибки, очереди) и *Data quality* (полнота manifest, доля валидных NPZ). |
| **Алерты** | Не ловить проблемы глазами в логах | Пороги на: рост 5xx, p95 латентности API, глубина очереди Celery, отсутствие heartbeat воркеров, диск/квота storage, ошибки Fetcher (auth/quota). Для Фазы 4 — отдельно порог на деградацию валидационной метрики (если автопрогон). |

### Что не смешивать

- **Продуктовые метрики ML** (AUC, MAE, калибровка) — в experiment tracking или в отдельном дашборде «ML», не в одних панелях с CPU без контекста.
- **Служебные метрики инфраструктуры** (PostgreSQL connections, Redis memory) — включать с Фазы 5 или при первых признаках узких мест в staging.

## Метрики по фазам (что мерить и зачем)

Ниже — **минимальный набор**, который можно расширять. Имена метрик лучше зафиксировать в коде/конвенции, чтобы дашборды не «плыли».

| Фаза | Фокус | Метрики и сигналы (инженерные) | Метрики и сигналы (качество данных / продукта) |
|------|--------|--------------------------------|--------------------------------------------------|
| **0** | Стабильный E2E и Audio | Длительность E2E по стадиям; `http_requests_total` / errors по Backend; задержки и ошибки Celery по типу задачи; ошибки Fetcher (в т.ч. «API key», timeout). | Прохождение контрактной валидации артефактов audio; счётчик «runs с полным набором ожидаемых путей в manifest». |
| **1** | Text Audit v3 | То же для текстового пути; время выполнения text-extractors; доля retry. | Доля run с валидным текстовым NPZ/JSON; расхождение schema_version с эталоном; опционально длина/пустые тексты (sanity). |
| **2** | Полный мультимодальный выход | Суммарное wall-time по процессору/экстрактору; очередь задач при full-профиле; ошибки OOM/GPU (если есть). | **Manifest completeness**: доля обязательных ключей; автоматический «score» полноты; число предупреждений валидаторов на run. |
| **3** | Segmenter + материализация | Latency записи в индекс/Parquet; размер батча; ошибки ETL; задержка «артефакт есть → строка в feature index». | Доля успешных materialization; контроль `sampling_policy` / `schema_version` в записанных рядах; дубликаты/пропуски `video_id,run_id`. |
| **4** | Модели и фичи | Время обучения/инференса на батче; использование GPU в экспериментах. | Train/val/test метрики модели; **доля NaN/constant features**; распределения ключевых фич (drift vs baseline dataset); feature importance / SHAP на срезе (офлайн). |
| **5** | ~100k и прод | Celery queue depth, age of oldest task; throughput видео/час; error rate по стадиям; utilization CPU/GPU; диск и Object storage; latency БД (p95). | SLO: «% run завершён успешно за N часов»; SLA по восстановлению; cost proxy (GPU-часы, объём egress); инциденты с привязкой к `run_id`. |

### Удобство для команды

- **Один «сводный» дашборд** на продукт (зелёный/жёлтый/красный по последнему nightly E2E + по очереди).
- **Deep-link**: из алерта или метрики — шаблон запроса в логах по `run_id` / `video_id`.
- **Runbooks**: краткие шаги под каждый тип алерта (ссылка на `backend/docs/OPERATIONS.md` и E2E runbook).

## Базы данных и хранилища по фазам

Идея — **не умножать БД без нужды**, но заранее разделить роли: *операционное состояние*, *метаданные артефактов*, *аналитика/ML слой*.

### Роли и что куда

| Роль | Типичное решение | Содержимое |
|------|------------------|------------|
| **Операционная БД** | PostgreSQL | Пользователи, проекты, runs, статусы задач, ссылки на файлы, миграции Alembic (или аналог). |
| **Кэш / брокер** | Redis | Celery broker/result backend, кратковременные блокировки, rate-limit (если используется). |
| **Файлы и NPZ** | Локальный диск / S3-совместимое хранилище | Крупные блобы; в Postgres — только URI, размер, `content_hash`, `schema_version`. |
| **Индекс фич / артефактов** (с Фазы 3) | Те же Postgres (таблицы индекса) или отдельная схема | Плоские ключи для джойна с `video_id`, `run_id`, версия сэмплинга; не хранить внутри таблицы многомегабайтные массивы без необходимости. |
| **Аналитика и тренировочные срезы** | Parquet + DuckDB / облачный warehouse (по мере роста) | Тяжёлые сканирования, ноутбуки, выгрузки для обучения; версия датасета как артефакт (manifest + хеш). |
| **Experiment tracking** (опционально, Фаза 4+) | MLflow, Weights & Biases или тонкие таблицы `experiment_run` в Postgres | Метрики, git commit, profile hash, список фич; не дублировать сырые веса модели в БД без политики хранения. |

### Окружения

| Окружение | Минимум | Заметки |
|-----------|---------|---------|
| **Dev** | Postgres + Redis + папка или MinIO для файлов | Мониторинг можно урезать до логов и одного скрипта «health». |
| **Staging** | Как прод, но меньше ресурсов | Включить метрики + один дашборд; прогонять E2E nightly; приблизить сетевую политику к прод (timeouts, TLS). |
| **Prod (Фаза 5)** | Managed Postgres (бэкапы, при необходимости PITR), Redis HA, object storage с lifecycle/TTL | Read-replica или выделенный аналитический путь для тяжёлых отчётов, чтобы не забивать OLTP. |

### Миграции и версии

- Любое изменение индекса фич или контракта — **миграция + совместимость** (двойная запись или версионирование таблиц `features_v2`).
- В документации run/runbook указать: **какой профиль и schema_version** считаются эталоном для nightly и для прод.

## Риски и типичные ошибки

| Риск | Митигация |
|------|-----------|
| «Все фичи в одну SQL-таблицу» | Широкие таблицы + смена схемы; лучше вложенные/JSONB или колонки только для индекса + файлы в storage. |
| E2E без фиксированного профиля | Неповторяемые результаты; зафиксировать эталонный профиль и хеш конфига. |
| Обучение до стабилизации E2E | Утечки и несопоставимые строки; сначала контракт и воспроизводимый pipeline. |
| 100k без observability | Невидимые зависания и перерасход GPU; метрики и алерты — часть Фазы 5, но проектировать с Фазы 2. |
