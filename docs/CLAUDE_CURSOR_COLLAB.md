# Модель взаимодействия Claude ⇄ Cursor (валидация компонентов)

Claude не может прогонять стек (нет GPU/моделей/видео в песочнице). Cursor гоняет.
Делим роли и обмениваемся через файлы в репозитории.

## Роли

**Claude (аналитик/архитектор):**
- статический разбор логики компонента (код, контракты, зависимости, model-fit);
- пишет **run-spec** (что прогнать, сколько видео, длины, профиль, что собрать);
- анализирует артефакты после прогона → пишет **отчёт** (числа+слова+рекомендации);
- проектирует правки алгоритма/Segmenter/контрактов; ведёт чеклист и версии.

**Cursor (исполнитель):**
- выполняет run-spec на стеке (реальные видео, GPU, модели);
- складывает артефакты в оговорённые пути + короткий **run-report** (статусы, пути, аномалии);
- вносит правки кода по согласованному diff (или Claude пишет патч, Cursor прогоняет);
- гоняет `e2e_validate_output_quality.py`, `feature_quality_audit.py`, `batch_runs_feature_report.py`.

## Цикл (на компонент)

```
Claude: static-review + RUN_SPEC.md  ──▶  Cursor: прогон  ──▶  артефакты + RUN_RESULT.md
       ▲                                                                    │
       └──── отчёт + рекомендации + правки  ◀── Claude анализ артефактов ◀──┘
   (повтор: доработка компонента/Segmenter → новый прогон, пока владелец не ок)
   → штамповка: версия + единый doc-шаблон + чеклист.
```

## Где что лежит (per component)

```
DataProcessor/docs/component_reports/<component>/
  RUN_SPEC.md         # Claude → Cursor: задание на прогон (версионируется)
  RUN_RESULT.md       # Cursor → Claude: что получилось, пути к артефактам, аномалии
  artifacts/          # Cursor кладёт: *_batch.csv, *_health.md, §0.2-json, логи, ключевые NPZ-срезы
  REPORT_<date>.md    # Claude: анализ + вердикт + рекомендации (шаблон в PROTOCOL §3)
```

## Формат RUN_SPEC (что Claude даёт Cursor)

1. **Компонент** + версия кода/схемы.
2. **Профиль**: какие компоненты включить (целевой + все зависимости по цепочке).
3. **Видео-матрица**: список/датасет, длины, кол-во, требования к контенту
   (напр. для action_recognition — ролики с людьми/движением).
4. **Команды**: точные вызовы (прогон + сбор метрик).
5. **Что собрать**: NPZ (или срезы), health-CSV, §0.2-json, тайминги, логи, VRAM/CPU.
6. **На что смотреть**: конкретные поля/аномалии (пусто/NaN/деградация по длине).
7. **Формат ответа**: заполнить `RUN_RESULT.md` по шаблону + пути к `artifacts/`.

## Формат RUN_RESULT (что Cursor возвращает Claude)

- Статус каждого видео (ok/empty/fail + empty_reason), `duration_sec` фактический.
- Пути к артефактам в `artifacts/`.
- Тайминги стадий + VRAM/CPU пики (для ветки оптимизаций).
- Замеченные аномалии/ошибки/патчи, которые пришлось применить (как в LOGIC_ERRORS).
- Версии: `dataprocessor_version`, `sampling_policy_version`, `model_signature`.

## Правило по цепочкам зависимостей и контрактам

Компоненты запускаются **в порядке зависимостей** (один читает выход другого:
Segmenter → core_* → modules/heads). Поэтому:
- в RUN_SPEC всегда перечисляем **всю цепочку** зависимостей целевого компонента;
- по мере валидации фиксируем **контракт между компонентами** (вход/выход + валидация)
  — см. `DataProcessor/docs/COMPONENT_CONTRACTS.md` (реестр межкомпонентных контрактов).
