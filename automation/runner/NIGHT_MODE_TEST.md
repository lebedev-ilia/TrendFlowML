# 🌙 Ночной режим — Тестовый сценарий

## Предусловия
- ✅ Chrome запущен с `--remote-debugging-port=9222`
- ✅ Вы залогинены в claude.ai
- ✅ Очередь компонентов не пуста (или есть подготовленные компоненты)
- ✅ Демон лимитов работает: `python limits_daemon.py`

---

## Сценарий проверки

### Шаг 1: Запустить ночной режим (5 авто-сессий)
```
/sas 5
```
Ожидаемо:
- 🤖 Раннер запущен
- ▶️ Старт: component_1
- Агент начинает работу над компонентом

### Шаг 2: Дождитесь 90% лимита (или сымитируйте)
Если лимит реально не достигает 90%:
```
/settings limit_pct_stop 50    # временно снизить порог для теста
```

Ожидаемо при достижении порога:
- ⚠️ Лимит Claude 51% ≥ 50% — СВОРАЧИВАЙСЯ...
- Агент **завершает** текущий шаг, фиксирует состояние
- 🛑 компонент готово (≈$...)
- Поды гасятся

### Шаг 3: Проверить, что агент НЕ ждёт команды
**Важно:** должно быть **МОЛЧАНИЕ** (~10-20 сек), затем:
- 🌙 Лимит 91%≥50%, жду сброса (5ч: Reset in ...). Пауза 10 мин.

Это означает:
- ✅ Агент **не ждёт** /start-session
- ✅ Агент **сам ждёт** сброса в автоматическом режиме
- ✅ **НЕ требует** вашего подтверждения

### Шаг 4: Имитация сброса лимита

Способ 1: Ждать реального сброса (5 часов, обычно быстрее)

Способ 2: Сымитировать ручное обновление лимитов
```
/scanlimits      # пересканировать claude.ai вручную
```
После обновления, если лимит упал ниже 50%:
- 🌙 Лимит сброшен, продолжаю (осталось авто-сессий: 4).
- ▶️ Старт: component_2

Это означает:
- ✅ Работа **продолжилась сама**
- ✅ **Без вашего вмешательства**
- ✅ Компонент выбран из очереди автоматически

### Шаг 5: Повторить 4 раза (5 авто-сессий)
После 5-й сессии:
- ✅ Очередь пуста или лимиты выбиты
- Агент остановится и ждёт нового задания

### Шаг 6: Отменить ночной режим (если нужно)
```
/sas-stop
```

## Критические проверки (контролировать ЛЬ вы видите)

| Что происходит | Должно быть так | Если не так | Статус |
|---|---|---|---|
| Лимит достигает 90% | Сессия завершается, под гасится | Сессия продолжается или под не гасится | ❌ |
| Агент начинает ждать | **Молчание** ~10 сек, затем "Лимит ... жду сброса" | Агент пишет "Пришли /start-session" | ❌ |
| Лимит сбрасывается | "Лимит сброшен, продолжаю" + новый компонент | Ничего или ошибка | ❌ |
| Работа продолжилась | Агент автоматически берёт следующий компонент | Требует вашей команды | ❌ |
| Повтор N раз | N компонентов обработано в цикле | Остановился раньше | ❌ |

## Восстановление подов (bonus test)

Если во время ночного режима 2 пода вышли из строя (EXITED):

### Автоматическое восстановление
Каждые ~6 минут watchdog проверяет и вызывает:
```python
podmanager.self_recovery_check()
```

Ожидаемо:
- 🔄 Self-recovery: deleted pod_1, created new pod pod_2
- Агент продолжает работу с новым подом

### Ручное восстановление (если нужно)
```bash
python podmanager.py list                    # посмотреть все поды
python podmanager.py terminate <pod_id>      # удалить неработающий
python podmanager.py self-recovery           # автосброс
```

## Отладка при проблемах

### "Агент зависает на сбросе лимита"
1. Проверь логи: `tail -f automation/runner/state/agent_runner.log`
2. Проверь limits.json: `cat state/limits.json`
3. Проверь, что Chrome с CDP работает: `python claude_limits_scraper.py --cdp 9222`

### "Лимит не обновляется"
1. Проверь демон: `python limits_daemon.py`
2. Проверь Chrome на порту 9222
3. Попробуй `/scanlimits` вручную

### "Поды не создаются при recovery"
1. Проверь RunPod API ключ в .env
2. Проверь наличие свободных GPU
3. Смотри: `python podmanager.py list`
