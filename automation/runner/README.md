# TrendFlow Runner — автономные агенты валидации компонентов

Первый агент (Claude, `agent_runner.py`) крутится в цикле: берёт следующий незаштампованный
компонент, доводит его до прод-готовности на GPU-поде и штампует, СРАЗУ переходя к следующему —
непрерывно через ВСЕ компоненты очереди, без пауз между ними. Вопросы — сначала в интернете, потом
Второму агенту (быстрый авто-ответчик `supervisor.py`); Второй агент (`assistant.py`) параллельно
контролирует Первого и общается с владельцем в VK. Поды — только через `pod_control`/`manager`,
автономно (без подтверждения в VK на каждое действие). НЕТ дневного $-бюджета — единственные лимиты:
% Claude 5ч/неделя (95%/97%), баланс RunPod (предупреждение) и ценовой потолок пода ($0.30/$0.60ч).
См. `AGENT_CONTEXT.md` раздел 0 и 5.3 для деталей.

## 0. Предпосылки
- `claude -p "say ok"` уже отвечает `ok` (Claude Code авторизован твоим Max). ✅ у тебя есть.
- Python 3.10+.

## 1. Установка
```bash
cd ~/Рабочий\ стол/TrendFlowML/automation/runner
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env      # затем впиши VK_TOKEN и RUNPOD_API_KEY
```

## 2. Проверка каналов (без модели и без пода)
```bash
source .venv/bin/activate
python messenger.py "проверка канала"   # тебе придёт сообщение в VK
python runpod_api.py                      # покажет статус подов
```

## 3. Пробный прогон логики (VK-вопросы есть, под не трогаем)
```bash
python agent_runner.py --once --dry-pod
```
Агент возьмёт один компонент, но реального управления подом не будет (`pod_control` в dry-режиме
просто сообщает). Проверь, что доходят сообщения и (при необходимости) вопросы в VK.

## 4. Боевой одиночный прогон
```bash
python agent_runner.py --once
```
Агент через подтверждение в VK поднимет под, отработает компонент, погасит под, пришлёт резюме.

## 5. Непрерывный режим (автозапуск)
```bash
sudo cp runner.service /etc/systemd/system/trendflow-runner.service
sudo systemctl daemon-reload
sudo systemctl enable --now trendflow-runner
journalctl -u trendflow-runner -f     # лог живьём
```
Остановить: `sudo systemctl stop trendflow-runner`.
Пауза без остановки сервиса: `touch state/PAUSED` (агент встанет после текущего компонента).
Снять паузу: `rm state/PAUSED` и перезапусти цикл.

## Порядок работы (протокол сессии)
Полностью описан в `AGENT_CONTEXT.md` (раздел 1 «Системный промпт: рабочий агент», раздел 3 «Онбординг»,
агент читает их первым делом каждой сессии). Кратко, на каждый компонент:
1. агент читает онбординг → 2. файл прогресса → 3. код/доки/взаимодействия компонента →
4. присылает тебе в VK **брифинг** (как понял задачу, прогресс, план, советы, вопросы) и ЖДЁТ ответа
(GPU на этом шаге не поднимается) → 5. по твоему ответу работает, поды создаёт/мигрирует/удаляет с
подтверждением → 6. вопросы по ходу → 7. пишет **ИТОГ** с предлагаемым вердиктом и ЖДЁТ →
8. по твоему ОК штампует и гасит под → 9. следующий компонент — только после твоего `/start-session`.

Так как шаги 1–3 не требуют GPU, агент может утром всё прочитать, прислать брифинг и ждать тебя весь
день без затрат; вечером ты отвечаешь — и начинается работа с подом.

## Причины остановки пода
1. Сторож простоя: ~1 час без активности на поде (в 20 мин — предупреждение).
2. Ты явно просишь остановить (`/start-session Останови поды` или в любом ответе агенту).
3. Нет твоего ответа на вопрос дольше 30 мин при поднятых подах — система гасит их до ответа, затем
   агент поднимает заново и продолжает.

## Жизненный цикл сессии
Работа идёт **непрерывно через ВСЕ компоненты очереди** — по завершении компонента раннер сам помечает
его закрытым (`state/done.json`, очередь двигается, не дожидаясь твоего штампа) и берёт следующий. Нет
дневного $-бюджета, который бы это останавливал. Стопы — только: рабочий лимит Claude 95% (сворачивание,
жёсткий стоп на 97%), запрос Второго агента (увидел проблему — просит аккуратный стоп с сохранением
контекста), или твоя явная команда. На стопе: агент фиксирует состояние, гасит под, пишет
`state/last_session.md`, сессия встаёт до `/start-session` или `/ssc` (или до авто-рестарта от Второго
агента / после сброса лимита).

## Управление из VK (команды)
- `/logs`, `/stop-logs` — дублировать логи модели в чат вкл/выкл.
- `/sss` (=`/stop-session`) — стоп + сохранение + стоп пода.
- `/sps` — стоп без сохранения и **без** стопа пода.
- `/start-session [текст]` — старт (пусто = следующий компонент из очереди).
- `/ssc [текст]` — старт с чтением контекста прошлой сессии (`last_session.md`).
- `/verify <компонент>` — независимая верификация.
- `/model [opus|sonnet|haiku|fable|...]` — сменить модель (со следующей сессии).
- `/settings [ключ значение]` — показать/изменить настройки: `limit_pct_stop`, `limit_pct_hard`,
  `max_pod_hourly_usd[_short]`, `runpod_balance_warn_usd`, `pod_hourly_usd`, `model`.
- `/status`, `/help`.

Второй агент (`assistant.py`, отдельный VK-бот VK_TOKEN2) добавляет `/monitor [on|off|<минуты>]` —
управление периодической проверкой Первого агента.

## Живой индикатор в консоли
Каждая строка модели помечается тегом `[Opus 4.8][~3.2k][345s]` = модель · приблизит. токены сессии ·
секунды с начала сессии. Токены точны в конце сессии (из отчёта SDK), в процессе — оценка.

## Сторож простоя пода
Если под RUNNING, но активности нет: >20 мин — предупреждение в VK; >60 мин — автоматический стоп пода.

## Прогресс компонента (возобновление)
На каждый компонент ведётся файл прогресса `state/progress/<component>.md`. Агент читает его в начале
сессии и продолжает с места остановки (не начинает заново). После `/stop-session` раннер НЕ берёт
следующий компонент сам — ждёт `/start-session`; пустой `/start-session` продолжает текущий компонент
по файлу прогресса, `/start-session <текст>` — запускает ручное задание.

## Создание/миграция пода с выбором GPU
`pod_control migrate|create` (или авто-вызов при недоступности GPU): агент присылает в VK список
доступных GPU с ценами ($/ч, через RunPod GraphQL), ты отвечаешь номером — создаётся новый под с нашим
Network Volume (данные сохраняются). Прямые изменения подов через curl заблокированы.

## Как это работает
- `agent_runner.py` — цикл; на каждый компонент запускает `ClaudeSDKClient` с системным промптом,
  собранным из `AGENT_CONTEXT.md` (секции `common`+`worker`, или `common`+`verify` для `/verify`),
  инструментами и хуком.
- `component_queue.py` — очередь: приоритет + компоненты со статусом ✅ в `COMPONENT_VALIDATION_CHECKLIST.md`
  считаются готовыми и пропускаются. Текущий компонент — в `state/claim.json`.
- `tools.py` — `ask_human` (вопрос → Второй агент отвечает синхронно) и `pod_control`/`manager`
  (status/start/stop_all/create/migrate, автономно, ценовой потолок $0.30 или $0.60 для коротких
  прогонов). `hooks.py` блокирует прямые `curl .../pods/<id>/start|stop` мимо `pod_control` и логирует
  вердикты по опасным командам в `state/hook_decisions.log`.
- `budget.py` — ТОЛЬКО учёт (не лимит): `state/spend_log.csv`, `pod_ledger.csv`, `sessions.csv` + баланс
  RunPod (`runpod_api.account_balance()`).
- `messenger.py` — VK send + group long poll (проверено на твоём сообществе, group_id 240071492); каждое
  исходящее/входящее сообщение Первого агента также пишется в `state/agent1_chat.log` для Второго агента.

## Учёт денег (модель + GPU) — БЕЗ дневного лимита
Файлы в `state/`: `spend_log.csv` (модель по компонентам), `pod_ledger.csv` (аренда подов: gpu, цена/час,
время, $), `sessions.csv` (все сессии: время, компонент, модель$, gpu$, токены) — это УЧЁТ для
информации, не лимит. Реальные ограничители: (1) ценовой потолок пода — `max_pod_hourly_usd` ($0.30) /
`max_pod_hourly_usd_short` ($0.60 для коротких прогонов); (2) фактический баланс RunPod — сторож
предупреждает в VK при `< runpod_balance_warn_usd` ($1 по умолчанию), НЕ останавливает работу. Инструмент
`budget_status` показывает траты за сегодня, ставку пода и на сколько часов хватит баланса RunPod.

## Верификация (независимая проверка)
`/verify <component>` запускает отдельную сессию со своим ТЗ (`AGENT_CONTEXT.md`, разделы 2 и 4):
она перепроверяет выход объективными инструментами (валидаторы,
`feature_quality_audit.py`, `golden_batch_compare.py`), сверяет с универсальными хард-гейтами и
`component_reports/<component>/CRITERIA.md`, и пишет `VERIFICATION_*.md` с сырыми числами. Штампует владелец.

## Критерии приёмки (двухуровневые)
Универсальные хард-гейты (валидатор rc=0, ось времени, health, expected-empty, golden, разные длины) +
per-component `CRITERIA.md`, которые агент предлагает в брифинге и владелец одобряет (числа ИЛИ явные
исключения вроде «NaN by design»). Так пороги честные и согласованные, а не выдуманы заранее.

## Защита от опасных команд
`hooks.py` требует вердикта Второго агента (быстрый авто-ответчик `supervisor.py`) на потенциально
разрушительные/дорогие команды (`rm -rf`, `snapshot_download` без allow_patterns, `git push`, работа с
дисками и т.п.) — решение логируется в `state/hook_decisions.log`. Мутации подов через curl запрещены
полностью — только `pod_control`/`manager`.

## Память между сессиями
`AGENT_CONTEXT.md` (раздел 7 «Решения и уроки») — агент читает его на старте и дополняет в конце (грабли,
решения, соглашения), чтобы не переучиваться и не повторять ошибки.

## Деньги / лимиты (важно) — НЕТ дневного $-бюджета
- Headless-вызовы Max идут из отдельного кредит-пула Max-подписки, сверх — по API-тарифам; единственный
  лимит на это — % 5ч/недели (`limit_pct_stop=95`, `limit_pct_hard=97`).
- Единственный реальный денежный ограничитель — БАЛАНС RunPod: сторож предупреждает в VK, если
  `< runpod_balance_warn_usd` ($1 по умолчанию), но НЕ останавливает работу — пополни баланс вручную.
- Ценовой потолок пода: `max_pod_hourly_usd` ($0.30 обычно) / `max_pod_hourly_usd_short` ($0.60 для
  явно коротких/быстрых прогонов).
- Под гасится после каждого компонента, при паузе и при аварии. Если авто-стоп не сработал — раннер
  пишет тебе в VK «проверь runpod.io вручную».
