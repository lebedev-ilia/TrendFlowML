## 2. Технические детали

### 2.1. Run_id lifecycle

- Backend остаётся **source of truth** для `run_id` (UUID) и бизнес‑сущности run’а.
- Fetcher заводит собственную запись `runs` с внешним ключом на backend `run_id` и ведёт детальный ingestion state machine:
  - статусы: `PENDING`, `NORMALIZING_SOURCE`, `CHECKING_CACHE`, `FETCHING_METADATA`, `FETCHING_CHANNEL`, `FETCHING_COMMENTS`, `DOWNLOADING_VIDEO`, `UPLOADING_ARTIFACTS`, `FINALIZING`, `COMPLETED`, `FAILED`;
  - каждый статус идемпотентен, повторный запуск шага не ломает артефакты (используется кеш по `(platform, platform_video_id)` и таблице `artifacts`).
- Для Backend определён минимальный маппинг статусов Fetcher → агрегированный `ingestion_status` (`running` / `success` / `error`) без необходимости знать внутренние детали стейтов.

### 2.2. Pipeline events schema

- Определён envelope события с жёстко зафиксированными полями:
  - `event_version`, `source`, `run_id`, `type`, `ts` (обязательные);
  - `platform`, `platform_video_id`, `stage`, `status`, `payload` (рекомендуемые).
- Согласованы базовые типы событий:
  - `run.status_changed`, `run.stage_changed`, `job.started`, `job.finished`, `job.failed`, `log.line`.
- Схема событий спроектирована так, чтобы быть:
  - совместимой по смыслам с текущей системой событий Backend (`backend/docs/EVENTS_AND_LOGGING.md`);
  - расширяемой (возможность добавлять новые `type` и поля в `payload` без ломающих изменений).

### 2.3. Manifest.json contract

- Зафиксирован минимальный обязательный набор полей манифеста:
  - `manifest_version`, `run_id`, `video_id`, `platform`, `duration_seconds`, `artifacts.*.path`.
- Добавлены рекомендованные поля для production‑грейд сценариев:
  - `storage_layout_version`, `checksum`, `size_bytes`, `comment_count`.
- Описаны инварианты:
  - manifest создаётся и становится видимым только после успешного завершения всех обязательных ingestion‑шагов;
  - manifest атомарен: либо отсутствует, либо ссылается на корректные и проверенные артефакты;
  - checksum и size синхронизированы с фактическими данными в S3/MinIO.

### 2.4. PlatformAdapter interface

- Уточнён базовый интерфейс:
  - `fetch_metadata(source, *, run_id)`, `download_video(source, *, run_id)`, `fetch_comments(source, *, run_id, limit=100)`.
- Зафиксированы общие инварианты:
  - идемпотентность операций по `(platform, platform_video_id)`;
  - явное разделение retryable и non‑retryable ошибок с нормализованными `error_code`;
  - обязательное уважение rate‑лимитов и proxy‑pool (будет реализовано в Phase 0/1).
- Определён контракт включения/выключения платформ через конфигурацию (`ENABLED_PLATFORMS`, feature‑флаги на платформы).

## 4. Известные ограничения и TODO

- **Нет формальных схем**:
  - JSON Schema / pydantic‑модели для manifest и событий пока не созданы — это отдельная подзадача Phase 0.
- **Нет реализации транспорта событий**:
  - схема событий описана логически; конкретные каналы (Redis Streams / pubsub, формат хранения) будут определены при реализации orchestration и observability.
- **Нет связки с backend БД**:
  - договорённости по полям `ingestion_status` и таблицам backend описаны на уровне контракта, но не отражены в коде Backend.
