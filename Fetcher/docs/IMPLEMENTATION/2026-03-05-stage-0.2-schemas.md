## 2. Технические детали

### 2.1. Схема manifest.json (`FetcherManifest`)

Файл: `Fetcher/schemas/manifest.py`

- `ArtifactInfo`:
  - `path: str` — относительный путь внутри bucket’а (`video-analytics-raw`), как в `plan.md`.
  - `checksum: Optional[str]` — строка вида `sha256:...`, опциональна для MVP.
  - `size_bytes: Optional[int]` — размер файла в байтах.
  - `comment_count: Optional[int]` — количество комментариев (используется для `comments_file`).
- `ManifestArtifacts`:
  - `video_file: ArtifactInfo`
  - `meta_file: ArtifactInfo`
  - `comments_file: Optional[ArtifactInfo]` — допускает отсутствие комментариев (флаги хранения).
- `FetcherManifest`:
  - `manifest_version: str = "1.0"` — версия контракта (мажорная часть ломает совместимость).
  - `run_id: UUID` — идентификатор run’а (из Backend).
  - `video_id: str` — нормализованный идентификатор видео (совместим с `platform_video_id`).
  - `platform: str` — `"youtube"` и т.п.
  - `duration_seconds: float` — длительность видео.
  - `storage_layout_version: str = "1.0"` — версия layout’а бакетов.
  - `artifacts: ManifestArtifacts` — группа артефактов, необходимых DataProcessor.

Эта модель напрямую реализует раздел “3. Контракт manifest.json и версионирование” из `BACKEND_CONTRACTS.md` и может использоваться:

- для валидации манифеста перед записью в storage;
- для генерации JSON Schema через `.model_json_schema()` в следующих этапах;
- как тип ответа в будущих API Fetcher.

### 2.2. Схемы событий Fetcher (`FetcherEvent*`)

Файл: `Fetcher/schemas/events.py`

- Тип `FetcherEventType`:
  - `Literal["run.status_changed", "run.stage_changed", "job.started", "job.finished", "job.failed", "log.line"]`.
- `FetcherEventBase`:
  - Обязательные поля:
    - `event_version: str = "1.0"`;
    - `source: Literal["fetcher"]` (по умолчанию `"fetcher"`);
    - `run_id: UUID`;
    - `type: FetcherEventType`;
    - `ts: datetime` (UTC).
  - Рекомендуемые поля:
    - `platform: Optional[str]`;
    - `platform_video_id: Optional[str]`;
    - `stage: Optional[str]`;
    - `status: Optional[str]`.
- Payload‑модели:
  - `FetcherRunStatusChangedPayload` — `old_status`, `new_status`, `reason`, `error_code`.
  - `FetcherRunStageChangedPayload` — `old_stage`, `new_stage`.
  - `FetcherJobStartedPayload` — `job_type`, `job_id`.
  - `FetcherJobFinishedPayload` — `job_type`, `job_id`, `duration_ms`.
  - `FetcherJobFailedPayload` — `job_type`, `job_id`, `error_code`, `error_message`.
  - `FetcherLogLinePayload` — `level` (`debug|info|warning|error`), `message`.
- Специализированные события:
  - `FetcherRunStatusChangedEvent(FetcherEventBase)` с `type="run.status_changed"` и `payload: FetcherRunStatusChangedPayload`.
  - Аналогично для `FetcherRunStageChangedEvent`, `FetcherJobStartedEvent`, `FetcherJobFinishedEvent`, `FetcherJobFailedEvent`, `FetcherLogLineEvent`.
- Union‑тип:
  - `FetcherEvent = Union[...все специализированные события...]` — удобно для типизации потребителей (Backend, логи, тесты).

Эти модели соответствуют разделу “2. Схема событий pipeline” в `BACKEND_CONTRACTS.md` и позволяют:

- типизировать и валидировать события на стороне Fetcher;
- при необходимости генерировать JSON Schema для интеграции с Backend;
- использовать единый union‑тип `FetcherEvent` для сериализации/десериализации в тестах и сервисах.

## 3. Тестирование

На этом этапе не добавлялся исполняемый код пайплайна, поэтому тестирование ограничено:

- валидацией базовых кейсов через pydantic (ручная проверка в REPL/скриптах):
  - создание `FetcherManifest` с минимально обязательными полями;
  - сериализация/десериализация `FetcherRunStatusChangedEvent` и `FetcherLogLineEvent`;
  - проверка работы ограничений (`ge=0` для `size_bytes`, `duration_ms`, `comment_count`).
- проверкой согласованности полей с описанием в `BACKEND_CONTRACTS.md` и `plan.md`.

Формальные unit‑тесты для схем будут добавлены позже, вместе с кодом, который их использует (API/worker/producer событий).

## 4. Известные ограничения и TODO

- **JSON Schema пока не зафиксированы в отдельных файлах**:
  - генерация `.json`‑схем из pydantic (через `.model_json_schema()` и последующее сохранение в репозиторий) будет отдельной задачей;
  - не заведены machine‑readable схемы для событий/manifest в `docs/` или `reference/`.
- **Нет enum‑ов для статусов/стейтов**:
  - статусы и стадии пока представлены строками (`str`), без строгих enum‑классов;
  - это упростит первые итерации, но в следующих стадиях рекомендуется вынести значения в `Enum` (аналог `RunStatus` в DataProcessor).
- **Нет интеграции с Backend/DP кодом**:
  - Backend и DataProcessor пока не используют эти схемы напрямую; интеграция будет выполняться позже (через отдельные задачи/PR).
