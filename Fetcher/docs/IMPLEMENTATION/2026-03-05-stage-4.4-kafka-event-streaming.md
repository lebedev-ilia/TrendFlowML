# Kafka Event Streaming для Fetcher

**Дата**: 2026-03-05  
**Стадия**: Phase 4 — Scalability Engineering  
**Статус**: ✅ Реализовано

## Обзор

Реализована интеграция с Apache Kafka для event streaming в production. Поддерживается публикация событий pipeline в Kafka topics и обработка задач через Kafka consumers (опционально, Celery + Redis остаётся для MVP).

## Требования

Из чеклиста Phase 4:
- Producer для отправки событий в Kafka
- Consumer для обработки задач из Kafka
- Конфигурация Kafka (brokers, topics, partitions)
- Поддержка обеих систем (Celery для MVP, Kafka для production)
- Миграция с Celery на Kafka (опционально)

## Реализация

### 1. Kafka Producer

**Файл**: `fetcher/kafka_producer.py`

**Класс `FetcherKafkaProducer`**:
- Отправка событий pipeline в Kafka topics
- Отправка задач для обработки consumer'ами
- Поддержка различных topic'ов для разных типов событий/задач
- Обработка ошибок (не ломает основной поток)

**Методы**:
- `publish_event()`: Публикация события Fetcher в topic `{prefix}.events`
- `publish_task()`: Публикация задачи в topic `{prefix}.tasks.{task_type}`

**Глобальный producer**:
- `init_producer()`: Инициализация глобального producer
- `get_producer()`: Получение глобального producer
- `publish_event()`: Удобная функция для публикации событий

### 2. Kafka Consumer

**Файл**: `fetcher/kafka_consumer.py`

**Класс `FetcherKafkaConsumer`**:
- Обработка задач из Kafka topics
- Поддержка consumer groups для балансировки нагрузки
- Автоматический commit offset'ов

**Функции**:
- `create_consumer_for_task_type()`: Создание consumer для конкретного типа задачи
- `consume_tasks()`: Обработка задач с помощью handler'а

### 3. Модуль событий

**Файл**: `fetcher/events.py`

**Функции для публикации событий**:
- `publish_status_changed()`: Событие изменения статуса run'а
- `publish_stage_changed()`: Событие изменения stage
- `publish_job_started()`: Событие начала job'а
- `publish_job_finished()`: Событие завершения job'а
- `publish_job_failed()`: Событие ошибки job'а

**Интеграция**:
- Автоматическая публикация в Kafka, если `kafka_enabled=True`
- Fallback на логирование, если Kafka не включен

### 4. Конфигурация

**Файл**: `fetcher/config.py`

**Добавленные настройки**:
- `kafka_enabled`: Включить Kafka event streaming (по умолчанию False)
- `kafka_bootstrap_servers`: Kafka brokers (например, "localhost:9092" или ["kafka1:9092", "kafka2:9092"])
- `kafka_topic_prefix`: Префикс для topics (по умолчанию "fetcher")

**Пример конфигурации**:
```python
kafka_enabled = True
kafka_bootstrap_servers = ["kafka1:9092", "kafka2:9092"]
kafka_topic_prefix = "fetcher"
```

### 5. Topics

**События**:
- `{prefix}.events` - Все события pipeline (run.status_changed, run.stage_changed, job.*)

**Задачи**:
- `{prefix}.tasks.fetch_metadata` - Задачи для metadata worker
- `{prefix}.tasks.download_video` - Задачи для video download worker
- `{prefix}.tasks.fetch_comments` - Задачи для comments worker
- `{prefix}.tasks.finalize` - Задачи для finalize worker

## Использование

### Настройка через переменные окружения

```bash
export FETCHER_KAFKA_ENABLED=true
export FETCHER_KAFKA_BOOTSTRAP_SERVERS="kafka1:9092,kafka2:9092"
export FETCHER_KAFKA_TOPIC_PREFIX="fetcher"
```

### Инициализация Producer

```python
from fetcher.kafka_producer import init_producer

# Инициализация при старте приложения
producer = init_producer(
    bootstrap_servers=["kafka1:9092", "kafka2:9092"],
    topic_prefix="fetcher",
)
```

### Публикация событий

```python
from fetcher.events import publish_status_changed

# Публикация события изменения статуса
publish_status_changed(
    run_id="550e8400-...",
    old_status="PENDING",
    new_status="FETCHING_METADATA",
    platform="youtube",
    platform_video_id="dQw4w9WgXcQ",
)
```

### Создание Consumer

```python
from fetcher.kafka_consumer import create_consumer_for_task_type

# Создание consumer для metadata tasks
consumer = create_consumer_for_task_type(
    task_type="fetch_metadata",
    bootstrap_servers=["kafka1:9092", "kafka2:9092"],
    topic_prefix="fetcher",
)

# Обработка задач
def handle_task(task_message):
    run_id = task_message["run_id"]
    # Обработка задачи
    ...

consumer.consume_tasks(handle_task)
```

## Интеграция с существующим кодом

### Публикация событий при изменении статусов

События публикуются автоматически через функции из `fetcher/events.py`. Для интеграции нужно вызывать эти функции в местах, где происходят изменения статусов:

```python
from fetcher.events import publish_status_changed

# При изменении статуса run'а
old_status = run.status
run.status = new_status
publish_status_changed(
    run_id=run.id,
    old_status=old_status,
    new_status=new_status,
    platform=platform,
    platform_video_id=platform_video_id,
)
```

### Использование Kafka для задач (опционально)

Для использования Kafka вместо Celery для обработки задач:

1. Настроить `kafka_enabled=True`
2. Создать consumers для каждого типа задачи
3. Публиковать задачи через `producer.publish_task()` вместо Celery tasks

## Известные ограничения

- **Только события**: Публикация событий реализована, но интеграция в существующий код (tasks.py, orchestrator.py) требует дополнительной работы
- **Consumer для задач**: Consumer создан, но не интегрирован в workers (можно использовать вместо Celery)
- **Миграция**: Миграция с Celery на Kafka требует дополнительной настройки и тестирования
- **Error handling**: Ошибки при публикации событий логируются, но не прерывают основной поток
