# Реализация интеграции с DataProcessor API для Backpressure

**Дата**: 2026-03-05  
**Стадия**: Phase 4 — Scalability Engineering  
**Статус**: ✅ Реализовано

## Обзор

Реализована полная интеграция с DataProcessor API для проверки размера очереди и контроля backpressure. Используются два endpoint'а DataProcessor: `/api/v1/health` (предпочтительно) и `/api/v1/metrics` (fallback).

## Требования

Из чеклиста Phase 4:
- Реализовать реальный запрос к DataProcessor API для проверки размера очереди
- Добавить endpoint в DataProcessor API или использовать существующие метрики Prometheus
- Обработать случаи недоступности DataProcessor API (fallback на разрешение)

## Реализация

### 1. Улучшение `get_processor_queue_size()`

**Файл**: `fetcher/backpressure.py`

**Изменения**:

- **Вариант 1: Health endpoint** (предпочтительно):
  - Запрос к `/api/v1/health`
  - Извлечение `metrics.queue_length` из JSON ответа
  - Простой и быстрый способ

- **Вариант 2: Metrics endpoint** (fallback):
  - Запрос к `/api/v1/metrics` (Prometheus формат)
  - Парсинг метрики `dataprocessor_queue_length` по приоритетам
  - Суммирование значений для всех приоритетов

- **Обработка ошибок**:
  - `httpx.TimeoutException` → возврат 0 (разрешение)
  - `httpx.RequestError` → возврат 0 (разрешение)
  - Ошибки парсинга → возврат 0 (разрешение)
  - Логирование всех ошибок

**Пример использования**:
```python
queue_size = get_processor_queue_size("http://dataprocessor-api:8000")
# queue_size = 150 (например)
```

### 2. Метрики для мониторинга

**Файл**: `fetcher/metrics.py`

**Добавленные метрики**:

- `fetcher_backpressure_detected_total` (Counter):
  - Общее количество обнаружений backpressure
  - Инкрементируется при `queue_size > threshold`

- `fetcher_backpressure_check_errors_total` (Counter):
  - Общее количество ошибок при проверке размера очереди
  - Labels: `error_type` (timeout, request_error, parse_error, etc.)

- `fetcher_processor_queue_size` (Gauge):
  - Текущий размер очереди DataProcessor
  - Обновляется при каждой проверке

**Интеграция**:
- Метрики обновляются в `get_processor_queue_size()` и `check_backpressure()`
- Позволяют мониторить состояние backpressure в Grafana

### 3. Логика работы

**Последовательность проверки**:

1. Проверка наличия `dataprocessor_api_url` в настройках
2. Попытка получить размер очереди через `/api/v1/health`
3. Если health endpoint недоступен → fallback на `/api/v1/metrics`
4. Если оба endpoint'а недоступны → возврат 0 (разрешение)
5. Обновление метрик
6. Сравнение с threshold

**Пример логирования**:
```
DEBUG: DataProcessor queue size from health endpoint: 150
WARNING: Backpressure detected: processor queue size=150, threshold=100
```

## Конфигурация

**Настройки** (`fetcher/config.py`):

- `dataprocessor_api_url`: URL DataProcessor API (например, `http://dataprocessor-api:8000`)
- `backpressure_threshold`: Порог размера очереди (по умолчанию 1000)

**Пример**:
```python
dataprocessor_api_url = "http://dataprocessor-api:8000"
backpressure_threshold = 1000
```

## Интеграция с DataProcessor

**DataProcessor endpoints**:

1. **`GET /api/v1/health`**:
   ```json
   {
     "status": "healthy",
     "metrics": {
       "queue_length": 150,
       "active_runs": 2
     }
   }
   ```

2. **`GET /api/v1/metrics`** (Prometheus формат):
   ```
   dataprocessor_queue_length{priority="high"} 50.0
   dataprocessor_queue_length{priority="normal"} 100.0
   ```

## Обработка ошибок

**Стратегия fallback**:

- При недоступности DataProcessor API → разрешение (возврат 0)
- Логирование всех ошибок для мониторинга
- Метрики для отслеживания проблем с API

**Типы ошибок**:
- `timeout` - таймаут запроса
- `request_error` - ошибка HTTP запроса
- `parse_error` - ошибка парсинга ответа
- `metrics_endpoint_error` - ошибка metrics endpoint
- `unexpected_error` - неожиданная ошибка

## Тестирование

**Рекомендуемые тесты**:

1. Успешное получение размера очереди через health endpoint
2. Fallback на metrics endpoint при недоступности health
3. Обработка таймаутов
4. Обработка ошибок HTTP
5. Парсинг Prometheus метрик
6. Обновление метрик

## Известные ограничения

- **Таймаут**: Фиксированный таймаут 5 секунд (можно сделать конфигурируемым)
- **Retry**: Нет автоматических retry при ошибках (можно добавить)
- **Кэширование**: Нет кэширования размера очереди (можно добавить для снижения нагрузки)
