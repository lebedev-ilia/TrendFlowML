# Kubernetes Deployment манифесты для Fetcher

**Дата**: 2026-03-05  
**Стадия**: Phase 6 — DevOps & Infrastructure  
**Статус**: ✅ Реализовано

## Обзор

Созданы полные Kubernetes манифесты для развёртывания всех компонентов Fetcher в production, включая отдельные deployment'ы для каждого типа worker'а, HPA для авто-масштабирования, ConfigMap и Secrets.

## Требования

Из чеклиста Phase 6:
- Отдельные deployment'ы для каждого типа worker'а
- Правильные resource limits согласно чеклисту
- Service для orchestrator API
- ConfigMap и Secrets
- HPA для авто-масштабирования

## Реализация

### 1. Компоненты

**Созданные манифесты**:

1. **Fetcher Orchestrator** (API сервис):
   - `orchestrator-deployment.yaml` - Deployment с resource limits (CPU: 0.2-1, RAM: 256MB-512MB)
   - `orchestrator-service.yaml` - Service для доступа к API
   - Health check probes (liveness и readiness)
   - 2 replicas для HA

2. **Metadata Worker**:
   - `metadata-worker-deployment.yaml` - Deployment (CPU: 0.5-1, RAM: 512MB-1GB)
   - `metadata-worker-hpa.yaml` - HPA (1-20 replicas, CPU 70%, Memory 80%)
   - Queue: `fetch.metadata`

3. **Download Worker**:
   - `download-worker-deployment.yaml` - Deployment (CPU: 2-4, RAM: 2GB-4GB)
   - `download-worker-hpa.yaml` - HPA (1-10 replicas, CPU 70%, Memory 80%)
   - Queue: `fetch.video`

4. **Comments Worker**:
   - `comments-worker-deployment.yaml` - Deployment (CPU: 1-2, RAM: 1GB-2GB)
   - `comments-worker-hpa.yaml` - HPA (1-15 replicas, CPU 70%, Memory 80%)
   - Queue: `fetch.comments`

5. **Finalize Worker**:
   - `finalize-worker-deployment.yaml` - Deployment (CPU: 0.5-1, RAM: 512MB-1GB)
   - Queue: `fetch.finalize`
   - 1 replica (фиксированное)

6. **Celery Beat**:
   - `beat-deployment.yaml` - Deployment (CPU: 0.1-0.5, RAM: 128MB-256MB)
   - 1 replica (singleton)

### 2. Конфигурация

**ConfigMap** (`configmap.yaml`):
- S3/MinIO конфигурация
- DataProcessor API URL
- Logging уровень
- Platform конфигурация
- Rate limiting параметры
- Backpressure threshold
- Lifecycle policies
- Feature flags

**Secrets** (`secrets-example.yaml`):
- PostgreSQL DSN
- Redis URL
- S3/MinIO credentials
- Proxy credentials (опционально)
- Примеры для External Secrets Operator

### 3. Авто-масштабирование

**HPA конфигурация**:
- Масштабирование по CPU (70% utilization)
- Масштабирование по памяти (80% utilization)
- Scale up: сразу, до 100% или +2 pods
- Scale down: через 5 минут, максимум 50% за раз

**Диапазоны replicas**:
- Metadata Worker: 1-20
- Download Worker: 1-10
- Comments Worker: 1-15

### 4. Health Checks

**Liveness Probe**:
- Orchestrator: HTTP GET `/health`
- Workers: Celery ping command
- Beat: Process check

**Readiness Probe**:
- Orchestrator: HTTP GET `/health`
- Workers: Celery ping command

### 5. Resource Limits

**Согласно чеклисту**:

| Component | CPU Request | CPU Limit | RAM Request | RAM Limit |
|-----------|-------------|-----------|-------------|-----------|
| Orchestrator | 0.2 | 1 | 256MB | 512MB |
| Metadata Worker | 0.5 | 1 | 512MB | 1GB |
| Download Worker | 2 | 4 | 2GB | 4GB |
| Comments Worker | 1 | 2 | 1GB | 2GB |
| Finalize Worker | 0.5 | 1 | 512MB | 1GB |
| Beat | 0.1 | 0.5 | 128MB | 256MB |

## Развёртывание

### Быстрый старт

```bash
# 1. Создать Secrets
kubectl create secret generic fetcher-secrets \
  --from-literal=postgres-dsn="..." \
  --from-literal=redis-url="..." \
  --from-literal=s3-access-key="..." \
  --from-literal=s3-secret-key="..."

# 2. Создать ConfigMap
kubectl apply -f k8s/fetcher/configmap.yaml

# 3. Развернуть компоненты
kubectl apply -f k8s/fetcher/orchestrator-deployment.yaml
kubectl apply -f k8s/fetcher/orchestrator-service.yaml
kubectl apply -f k8s/fetcher/metadata-worker-deployment.yaml
kubectl apply -f k8s/fetcher/download-worker-deployment.yaml
kubectl apply -f k8s/fetcher/comments-worker-deployment.yaml
kubectl apply -f k8s/fetcher/finalize-worker-deployment.yaml
kubectl apply -f k8s/fetcher/beat-deployment.yaml

# 4. Развернуть HPA
kubectl apply -f k8s/fetcher/metadata-worker-hpa.yaml
kubectl apply -f k8s/fetcher/download-worker-hpa.yaml
kubectl apply -f k8s/fetcher/comments-worker-hpa.yaml
```

## Известные ограничения

- **Metrics-based HPA**: HPA использует только CPU и память. Для более точного масштабирования можно добавить custom metrics (размер очереди Celery).
- **Pod Disruption Budget**: Не создан (рекомендуется добавить для production).
- **Network Policies**: Не созданы (рекомендуется добавить для production).
- **Resource Quotas**: Не созданы (рекомендуется добавить для production).

## Следующие шаги
