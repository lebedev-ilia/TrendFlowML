## TikTok

| Подход                    | Авторизация и лимиты            | Поля и возможности                              | Сложность внедрения                 | Надёжность   | Риск блокировки |
|---------------------------|---------------------------------|-----------------------------------------------|-------------------------------------|--------------|-----------------|
| **Официальный API (Display API)** | OAuth2 (Client ID/Secret, Access Token); лимиты: 600 запросов/мин. | Извлечение видео по `user_id` или `item_id`. Возвращает JSON с `id`, `create_time`, `video_description`, `duration`, `like_count`, `comment_count`, `share_count`, `view_count` и пр.. | Средняя. Необходима регистрация приложения, получение токена, настройка OAuth. | Высокая (официальная схема). | Низкий (в рамках лимитов, токен). |
| **Неофициальный Python-сборщик (TikTokApi)** | Не требуется официальная авторизация; эмулирует поведение браузера (Playwright, msToken). | Позволяет получать тренды, данные пользователя, видео, комменты. Возвращает объекты с полями (аналогично API). | Высокая. Требует обхода блокировок (ротация токенов, эмуляция браузера). | Средняя. Устойчивость зависит от обновлений сайта. | Средний–высокий (бот-защита, можно получить 403). |
| **Веб-скрейпинг**        | Нет (но часто требуется авторизация). | Парсинг HTML/JSON выдачи сайта или API-запросов сайта. Поля – то же, что в API, но доступны через динамический JS. | Очень высокая. У TikTok динамическая подгрузка, сложные подписи запросов. | Низкая. Требуется поддерживать сложный скрипт, подвержен изменениям. | Очень высокий (CAPTCHA, блокировка по IP или UA). |
| **Реверс-инжиниринг мобильных запросов** | Анализ протоколов мобильных приложений. | Можно получить данные видео, комментариев, лайков и пр., но требуется вычислять параметры (`X-Tt-Params`, подпись). | Очень высокая. Нужно декомпилировать приложения, повторять алгоритмы подписи. | Нестабильная (легко сломается при обновлении). | Очень высокий (нарушает TOS, может привести к бану приложения/IP). |

- **Официальный Display API**: Платформа TikTok предоставляет *Display API* (часть TikTok for Developers) для чтения профиля и видео пользователей. С помощью эндпоинтов `/v2/user/info/` и `/v2/video/list/` или `/v2/video/query/` можно получать метаданные видео (ID, время публикации, описание, длительность, статистику: просмотры, лайки, комментарии). Для работы требуется пройти OAuth-авторизацию, получить *Client Access Token*, **токен пользователя**; лимит – 600 запросов в минуту на каждый эндпоинт. Пример получения списка видео пользователя (Python):  

```python
import requests

ACCESS_TOKEN = "ВАШ_ТОКЕН"
open_id = "USER_OPEN_ID"  # из TikTok OAuth
url = "https://open.tiktokapis.com/v2/video/list/"
headers = {"Access-Token": ACCESS_TOKEN}
params = {"open_id": open_id, "count": 5}
resp = requests.get(url, headers=headers, params=params)
data = resp.json()
for video in data.get("data", []):
    print(video["id"], video.get("video_description"), video.get("view_count"))
```
*Источник:* Документация TikTok API. Поля и их типы описаны в спецификации `video object` (см. `id`, `create_time`, `video_description`, `duration`, `like_count`, `comment_count`, `view_count` и др.).

- **Неофициальные библиотеки (SDK)**: Самой популярной является [TikTokApi](https://github.com/davidteather/TikTok-Api) (Python). Она запускает браузер через Playwright, эмулирует запросы к *web* API TikTok, используя ваше `msToken` (кука). Пример:  

```python
from TikTokApi import TikTokApi
import os, asyncio

async def example():
    ms_token = os.getenv("MS_TOKEN")  # cookie из браузера
    async with TikTokApi() as api:
        await api.create_sessions(ms_tokens=[ms_token], num_sessions=1, sleep_after=3)
        async for video in api.trending.videos(count=5):
            print(video.id, video.as_dict["stats"]["playCount"], video.as_dict["desc"])

asyncio.run(example())
```
Тут метод `api.trending.videos()` возвращает объекты видео с атрибутами (ID, статистика, описание). Библиотека снимает часть ограничений, но при этом следует обойтись средствами антибота (использовать прокси, менять `User-Agent`, `msToken` и т.п.).  

- **Веб-скрейпинг**: Можно попробовать парсить данные с публичных страниц TikTok. Например, выполнить HTTP-запрос к `https://www.tiktok.com/@username` (с нужными заголовками и куками) и извлечь JSON из встроенных скриптов. Однако TikTok сильно защищён: страница загружается через React, запросы к его внутреннему API подпадают под проверку подписи (`X-Tt-Params`, `X-Bogus` и т.д.). Часто требуется использование *headless*-браузеров (Selenium, Playwright) и обход капчей. Надёжность метода низкая: частые блокировки и изменения сайта.  

- **Реверс-инжиниринг**: Можно анализировать мобильное приложение TikTok (Android/iOS) и перехватывать его HTTPS-трафик. Известно, что API-запросы (на `api2.musical.ly` или новые TikTok-инстансы) требуют вычисления сложных параметров (подписи, шифрования параметров через hmac и т.д.). Это чрезвычайно сложно реализовать и постоянно поддерживать. Такой подход имеет высокий риск: TikTok может блокировать IP, аккаунты и активно обновляет алгоритмы подписи.  

**Оценка и рекомендации:** Официальный API предпочтителен с точки зрения стабильности и легальности. Для небольших задач можно воспользоваться `TikTokApi` (есть примеры и обсуждения на GitHub, но нужно быть готовым к поломкам). Скрытые методы через скрейпинг или реверс – ненадёжны, сложны и потенциально запрещены TOS. Храните ключи доступа (`Client Secret`, токены) в защищённом виде (например, в переменных окружения). Для обхода лимитов API используйте кэширование, паузы и batch-запросы; при неофициальных методах – ротацию сессий и прокси.

## Instagram

| Подход                    | Авторизация и лимиты         | Поля и возможности                                        | Сложность внедрения         | Надёжность    | Риск блокировки |
|---------------------------|-----------------------------|----------------------------------------------------------|-----------------------------|---------------|-----------------|
| **Официальный Graph API (Instagram Graph)** | OAuth (страничные токены). Лимит ~200 запросов/час/аккаунт. | Эндпоинты `/IG_ID/media` и `/media_ID`. Можно получить `id`, `caption` (описание), `media_type`, `media_url` (фото/видео), `thumbnail_url`, `timestamp`, `like_count`, `comments_count` и т.д.. | **Сторонние SDK и библиотеки** | Не требуется офиц. токены, но может потребоваться логин. | `Instaloader`, `instagramy`, `InstagramAPI` и др. Работают парсингом веб-данных или мобильных API. Дают `caption`, `likes`, `comments`, `video_url`, `thumbnail_url`, `timestamp` и пр. | Средняя. Библиотеки упрощают (есть примеры в интернете), но со временем могут сломаться. | Средняя. Подвержены капчам и изменению HTML/API. | Средний (Instagram блокирует). |
| **Веб-скрейпинг**         | -                           | Парсинг `window._sharedData` или других встроенных JSON. Можно получить `username`, `bio`, `followers_count`, список постов с `caption`, `like_count`, `comments_count`, `media_url`. | Средняя–высокая. Учитывать защиту (User-Agent, задержки, капчи). | Низкая. API меняется, страницы динамические. | Высокий (IP/аккаунт-блокировки, капчи). |
| **Реверс-инжиниринг**     | Анализ API мобильного приложения (неофициальный). | Приватный API Instagram (как в мобильном приложении) позволяет получать подробные данные (включая приватные аккаунты) через эндпоинты `/api/v1/*`. Требует параметров аутентификации и подписи запроса. | Очень высокая. Нужно изучать мобильный трафик, обновления. | Нестабильная. Быстро ломается при обновлениях. | Очень высокий (нарушает правила, риск правовых последствий). |

- **Официальный Instagram Graph API**: Доступен только для **Business/Creator аккаунтов**, требует привязки к Facebook App. После получения *long-lived* токена (до 60 дней) можно делать GET-запросы к эндпоинту: 

  ```
  GET /{ig-user-id}/media?fields=id,caption,media_type,media_url,thumbnail_url,timestamp,like_count,comments_count
  ```
  
  Пример Python-запроса:  

  ```python
  import requests

  token = "ВАШ_LONG_LIVED_TOKEN"
  user_id = "IG_USER_ID"
  fields = "id,caption,media_type,media_url,thumbnail_url,timestamp,like_count,comments_count"
  url = f"https://graph.facebook.com/v17.0/{user_id}/media"
  resp = requests.get(url, params={"fields": fields, "access_token": token})
  data = resp.json()
  for media in data.get("data", []):
      print(media["id"], media["caption"], media.get("media_url"), media.get("like_count"))
  ```
  *Источник:* официальная документация Instagram Graph API (перечень полей: `media_type`, `media_url`, `thumbnail_url`, `caption`, `like_count`, `comments_count`). Лимит: около **200 запросов в час на аккаунт**. Рекомендуется указывать только нужные поля (`fields`) для экономии лимитов, кэшировать неизменяемые данные (описание, URL) и использовать пагинацию. Для операций в реальном времени можно применять Webhook от Facebook.

- **Instaloader и другие библиотеки**: [Instaloader](https://instaloader.github.io/) — популярный инструмент на Python. Он умеет скачивать посты (картинки и видео) вместе с метаданными: `caption`, `likes`, `comments`, `video_url`, `thumbnail_url`, `timestamp`. Пример использования Instaloader:  

  ```python
  import instaloader

  L = instaloader.Instaloader()
  profile = instaloader.Profile.from_username(L.context, "instagram")
  for post in profile.get_posts():
      print(post.date, post.caption, post.likes, post.comments)
      if post.is_video:
          print("Video URL:", post.video_url)
  ```
  
  *Источник:* документация Instaloader. Instaloader автоматически обходит многие защитные меры, но при частых запросах может потребовать логин или столкнуться с капчей.

- **Веб-скрейпинг**: Прямой запрос к сайту Instagram (`requests.get("https://www.instagram.com/имя_профиля/")`) можно проанализировать через `BeautifulSoup` или прямой парсинг JSON, встроенного в тег `<script>`. Например, есть скрипт с `type="application/ld+json"` или `window._sharedData`, из которых вытаскивается JSON с информацией о профиле и постах. Фрагмент кода (упрощённо):  

  ```python
  import requests, json
  from bs4 import BeautifulSoup

  headers = {"User-Agent": "Mozilla/5.0", "Referer": "https://www.instagram.com/"}
  url = f"https://www.instagram.com/ladygaga/"
  res = requests.get(url, headers=headers)
  soup = BeautifulSoup(res.text, "html.parser")
  script = soup.find("script", text=lambda t: t and "window._sharedData" in t)
  data = json.loads(script.text.split(" = ",1)[1].rstrip(";"))
  profile = data["entry_data"]["ProfilePage"][0]["graphql"]["user"]
  print(profile["full_name"], profile["edge_followed_by"]["count"], profile["edge_owner_to_timeline_media"]["count"])
  ```
  
  По итогам можно получить `full_name`, `biography`, количество подписчиков, а также список постов с описаниями и статистикой. *Важно:* Instagram активно противодействует такому сканированию (CAPTCHA, блокировки), кроме того, подобный парсинг **нарушает Условия использования** (Terms of Service) Instagram. Библиотека `requests` должна использовать адекватные заголовки и прокси при необходимости.

- **Реверс-инжиниринг**: Подразумевает использование приватного Instagram API (например, как это делает мобильное приложение через `api/v1/`). Этот метод открывает доступ даже к закрытым аккаунтам (при наличии учётных данных), но крайне сложен: требуется реверсить приложения, генерировать подписи запросов, удерживать сессию. Такой подход технически реализуем (существуют Python-репозитории, повторяющие приватные API), но он **нелегален и нестабилен**. Рекомендуется избегать его: помимо риска блокировки, это нарушает правила Instagram.  

**Оценка и рекомендации:** Если нужна поддержка и стабильность, лучше использовать официальный Graph API (требует бизнес-аккаунт и токен). Для массового сбора данных (при отсутствии бизнес-аккаунта) популярны Instaloader и аналогичные библиотеки, но следует внимательно читать предупреждения (никакой коммерческой выгоды, обрабатывать ошибки 429, хранить куки). Все ключи/токены храните безопасно (в защищённых файлах или переменных окружения, как советует Meta). Чтобы не превысить лимиты, выбирайте только нужные поля, применяйте кеширование ответов (Instagram рекомендует кешировать статичные поля до суток) и используйте экспоненциальную задержку при ошибках 429.

## RuTube

| Подход                    | Авторизация и лимиты     | Поля и возможности                     | Сложность        | Надёжность | Риск блокировки |
|---------------------------|--------------------------|----------------------------------------|------------------|------------|-----------------|
| **Официальный API**       | Неофициально. Нету открытой публичной документации. Возможно, требуется договор с RuTube. | Не документировано. Предположительно: `id`, `title`, `description`, `duration`, `views`, `likes`, `comments`, `author`, `upload_date`, ссылки на файлы (HLS). Без официальных данных сложно. | Н/д (скрытый). | Н/д | Н/д |
| **youtube-dl / yt-dlp**   | Не требует (работает с публичными URL). | Извлекает почти все метаданные видео: `title`, `uploader`, `upload_date`, `duration`, `view_count`, `like_count`, `comment_count`, `formats` (URL видео) и пр. Работает через встроенные парсеры. | Низкая–средняя. Сама библиотека абстрагирует детали. | Высокая. Поддерживается сообществом (в т.ч. `yt-dlp`). | Низкий (легален, эмитирует обычный доступ). |
| **Веб-скрейпинг**        | —                        | Прямой HTTP GET страницы `rutube.ru/video/<id>`. Можно парсить HTML (заголовок, описание, счетчики) и/или искать JSON-данные в `<script>`. | Средняя. Статический HTML (частично AJAX). Могут понадобиться прокси. | Средняя. Структура сайта мало меняется, но капчи/блокировки возможны. | Средний (аналог YouTube, возможно защита). |
| **Реверс-инжиниринг**    | Анализ Android/iOS-приложения RuTube. | После аутентификации (JWT-токен) приложение общается с API. Можно получить список видео, метаданные, комментарии. Требуется анализ запросов и ключей шифрования. | Очень высокая (нет документации, зашита запросов). | Низкая (лёгко сломать, при обновлении приложения менять всё). | Очень высокий (нарушает TOS, развёрнутые системы защиты). |

- **Официального публичного API нет**: RuTube не предоставляет открытой документации по API для чтения данных видео. Существует лишь API для загрузки видео (как видно из неофициальных источников), но для получения видео по ID публичных методов нет. Обычно рекомендуют обращаться к разделу Embed или через сотрудничество с RuTube.  

- **youtube-dl / yt-dlp**: Самый быстрый путь – использовать [youtube-dl](https://ytdl-org.github.io/youtube-dl/) или форк [yt-dlp](https://github.com/yt-dlp/yt-dlp). Эти инструменты поддерживают RuTube (см. список поддерживаемых сайтов). Они умеют скачивать видео и возвращать метаданные. Пример использования `yt-dlp` в Python:  

  ```python
  from yt_dlp import YoutubeDL

  url = "https://rutube.ru/video/1234567890abcdef/"  # ссылка на видео
  ydl_opts = {"skip_download": True, "quiet": True}
  with YoutubeDL(ydl_opts) as ydl:
      info = ydl.extract_info(url, download=False)
  print(info["title"], info.get("view_count"), info.get("upload_date"))
  if "formats" in info:
      print("Видео URL:", info["formats"][0]["url"])
  ```
  
  `yt_dlp` вернёт словарь с ключами: `title`, `uploader`, `upload_date`, `duration`, `view_count`, `like_count`, `comment_count`, `formats` (URL видео разных качеств) и др. Поскольку библиотека регулярно обновляется сообществом, этот метод наиболее надёжен и прост.

- **Веб-скрейпинг**: Можно выполнить HTTP-запрос к странице видео (например, через `requests` с подходящими заголовками) и спарсить нужные элементы. Например, в HTML RuTube-заголовок может быть в теге `<h1 class="video-player-title">`, описание в `<div>`, количество просмотров и лайков – в счётчиках (например, классы `.view-count`). Также в странице часто embed’ится JSON с более полной информацией. Пример простого скрейпинга (псевдокод):  

  ```python
  res = requests.get("https://rutube.ru/video/ABCDEFG/", headers={"User-Agent": "Mozilla/5.0"})
  soup = BeautifulSoup(res.text, "html.parser")
  title = soup.select_one(".video-player-title").text
  stats = soup.select_one(".video-player-statistic")  # class фигурный для блоков
  views = stats.find_all("span")[0].text
  likes = stats.find_all("span")[2].text
  desc = soup.select_one(".video-player-desc").text
  print(title, views, likes, desc)
  ```
  
  Такой подход требует поддержки: структура HTML может меняться. Кроме того, может быть защита (ограничение числа запросов, проверка заголовков).

- **Реверс-инжиниринг**: Теоретически можно декомпилировать мобильное приложение RuTube и повторять его API-запросы к серверу (обычно видеохосты используют JWT или прочие токены). Однако без официальной поддержки это очень затратно. Предположительно, приложение передаёт авторизационные заголовки (например, `Authorization: Bearer <token>`), а ответы приходят в JSON. Понимание формата запроса и шифрования (если есть) крайне сложно. Данный метод ненадёжен и по сути запрещён.

**Оценка и рекомендации:** Для RuTube нет простого официального API, поэтому практично использовать готовые средства вроде `yt-dlp` или писать скрипт на `requests+BeautifulSoup`. `yt-dlp` обеспечивает максимально полный парсинг метаданных и автоматически обходит ряд защит. При скрейпинге стоит соблюдать умеренные интервалы между запросами. Все ключи (если есть авторизация) – хранить безопасно. Ограничения RuTube по частоте запросов недокументированы, но при превышении возможен ответ 429 или блокировка. Примеры использования `yt-dlp` и скрейпинга упоминались в сообществах (см. документацию youtube-dl).

- **Официальный Twitch API (Helix)**: В Twitch есть полноценный [REST API Helix](https://dev.twitch.tv/docs/api/) (заменяет устаревший Kraken). Нужны `Client-ID` и `Bearer` токен (получаемые через OAuth или client credentials). Пример запроса видео:  

  ```
  GET https://api.twitch.tv/helix/videos?id=123456789
  ```
  заголовки: `Client-ID: ...`, `Authorization: Bearer ...`.  

  Ответ (`data` - массив видеозаписей) включает поля: `id`, `user_id`, `user_name`, `title`, `description`, `created_at`, `published_at`, `url`, `thumbnail_url`, `view_count`, `language`, `type`, `duration` и др. Например:  
  ```json
  {
    "data": [
      {
        "id": "987654321",
        "user_name": "CoolStreamer",
        "title": "Stream Highlights",
        "description": "Лучшие моменты стрима",
        "created_at": "2022-07-08T16:58:46Z",
        "url": "https://www.twitch.tv/videos/987654321",
        "view_count": 395246,
        "duration": "6h26m14s",
        ...
      }
    ]
  }
  ```  
  *Источник:* документация Twitch. К API также относятся `/helix/streams` (получить текущие стримы), `/helix/clips` (клипы), `/helix/users` и др.  

  Лимиты Twitch: примерно **800 запросов в минуту на приложение** (client credentials) и на пользователя. Ответы содержат заголовки `Ratelimit-Remaining` и `Ratelimit-Reset`. При превышении – HTTP 429. Рекомендуется мониторить эти заголовки и делать экспоненциальную задержку при достижении лимита.

- **Python-библиотеки**: Например, [twitchAPI](https://pytwitchapi.dev/) (поддерживает Helix и Chat). Упрощает авторизацию и запросы. Пример (асинхронный):  

  ```python
  from twitchAPI.twitch import Twitch
  import asyncio

  async def get_videos_example():
      twitch = await Twitch('APP_ID', 'APP_SECRET')  # создаёт и app-токен
      videos = await twitch.get_videos(user_ids=['123456'])  # список видео
      for vid in videos:
          print(vid.id, vid.title, vid.view_count)
  
  asyncio.run(get_videos_example())
  ```
  
  Этот код подключается к Twitch, а затем `get_videos` возвращает список объектов `Video`, содержащих нужные поля (см. пример документации).  

- **Веб-скрейпинг**: Twitch не имеет простого API без авторизации, но его страница клиента может быть скопирована. Например, можно загрузить HTML страницы канала или видео и попытаться вытащить метаданные (заголовок стрима, количество зрителей в `<meta>` и JSON-скриптах, список фрагментов HLS и т.д.). Однако сайт динамический и требует запуска JS, поэтому чаще используют headless-браузер (Puppeteer, Selenium). Этот метод ненадёжен: Twitch часто меняет верстку, а антибот (Cloudflare) может затруднить парсинг. 

- **Реверс-инжиниринг**: Для получения метаданных реверсинг не требуется (API открыт). Ранее существовал Twitch API v5, но сейчас его заменили Helix и привязали к OAuth.  

**Оценка и рекомендации:** Helix API – оптимальное решение для Twitch. Он предоставляет все требуемые данные (см. поля) и достаточно гибок (фильтрация по типу, языку, периоду, сортировка). Используйте официальную библиотеку, храните `Client Secret` и токены безопасно (в конфиге или env-файле). Следите за `rate limit`: Twitch даёт ~800 очков в минуту, пункты 151-159 документа упоминают заголовки лимита. При работе с API применяйте кеширование и batch-запросы (Helix поддерживает передачу нескольких ID). Риск блокировки невелик при соблюдении лимитов и корректном использовании токенов.
