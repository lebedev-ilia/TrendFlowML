## Выполненные задачи

### ✅ Реализовать idempotency lock

Idempotency lock уже был реализован в этапе 2.3 (Redis Schema), но был проверен и улучшен:

1. **Функция `acquire_run_lock()`** в `api/services/redis_schema.py`
   - Использует `SET run:lock:{run_id} NX EX 3600` для атомарной блокировки
   - Возвращает `True` если блокировка получена, `False` если уже заблокировано
   - TTL блокировки: 3600 секунд (1 час)

2. **Функция `release_run_lock()`** в `api/services/redis_schema.py`
   - Освобождает блокировку через `DELETE run:lock:{run_id}`
   - Вызывается после завершения обработки (успешной или с ошибкой)

3. **Проверка перед enqueue**
   - Lock проверяется в `POST /api/v1/process` перед регистрацией в TaskManager
   - Lock проверяется перед enqueue в Redis Streams
   - Порядок операций:
     1. Проверка TaskManager (is_run_active)
     2. Проверка Redis lock (acquire_run_lock)
     3. Проверка backpressure (can_accept_new_run)
     4. Регистрация в TaskManager
     5. Enqueue в Redis Streams

4. **Обработка конфликта (409)**
   - Используется `RunAlreadyExistsError` для обработки конфликта
   - Exception handler в `api/main.py` возвращает 409 Conflict
   - Exception handler в `api/endpoints/process.py` также возвращает 409 Conflict

### ✅ Интегрировать в `POST /api/v1/process`

Интеграция выполнена в `api/endpoints/process.py`:

1. **Проверка lock перед регистрацией**
   ```python
   # Попробовать получить блокировку в Redis (idempotency lock)
   lock_acquired = await acquire_run_lock(request.run_id)
   if not lock_acquired:
       raise RunAlreadyExistsError(
           f"Run with run_id={request.run_id} is already being processed",
           run_id=request.run_id
       )
   ```

2. **Освобождение lock после обработки**
   - В worker (`api/services/worker.py`): lock освобождается после успешной обработки и при ошибке
   - В fallback режиме (`_run_processing_background`): lock освобождается после завершения обработки

3. **Обработка ошибок**
   - Если произошла ошибка до enqueue, lock освобождается в exception handler
   - Если произошла ошибка во время обработки, lock освобождается в worker

   - Добавлено освобождение lock в exception handler при неожиданной ошибке
   - Улучшена обработка ошибок с освобождением lock

2. **`api/services/worker.py`** (уже было реализовано ранее)
   - Lock освобождается после успешной обработки (строка 366)
   - Lock освобождается при ошибке (строка 402)

3. **`api/services/redis_schema.py`** (уже было реализовано ранее)
   - `acquire_run_lock()` - получение блокировки
   - `release_run_lock()` - освобождение блокировки
   - `is_run_locked()` - проверка блокировки

### Архитектурные решения

1. **Атомарная блокировка**
   - Используется `SET NX` для атомарной установки блокировки
   - TTL 3600 секунд предотвращает зависание блокировок
   - Блокировка автоматически истекает через 1 час

2. **Порядок проверок**
   - Сначала проверяется TaskManager (быстрая проверка)
   - Затем проверяется Redis lock (надежная проверка)
   - Затем проверяется backpressure
   - Только после всех проверок выполняется регистрация и enqueue

3. **Освобождение lock**
   - Lock освобождается после успешной обработки
   - Lock освобождается при ошибке обработки
   - Lock освобождается в fallback режиме после завершения
   - Lock освобождается при неожиданной ошибке до enqueue

4. **Обработка ошибок**
   - `RunAlreadyExistsError` используется для конфликтов
   - Exception handler возвращает 409 Conflict
   - Lock не освобождается при 409 (lock не был установлен)

## Использование

### Автоматическое использование

Idempotency lock работает автоматически:

```python
# При запросе POST /api/v1/process:
# 1. Проверяется lock через acquire_run_lock()
# 2. Если lock не получен - возвращается 409 Conflict
# 3. Если lock получен - выполняется enqueue
# 4. Lock освобождается после завершения обработки
```

### Ручное использование

Для ручной работы с lock:

```python
from api.services.redis_schema import acquire_run_lock, release_run_lock

# Получить блокировку
lock_acquired = await acquire_run_lock(run_id)
if not lock_acquired:
    # Run уже обрабатывается
    raise RunAlreadyExistsError(f"Run {run_id} is already being processed", run_id=run_id)

# Выполнить обработку
try:
    # ... обработка ...
    pass
finally:
    # Освободить блокировку
    await release_run_lock(run_id)
```

## Известные проблемы и ограничения

1. **TTL блокировки**
   - TTL 3600 секунд может быть недостаточно для длительных обработок
   - Если обработка длится дольше часа, lock истечет
   - TODO: Обновлять TTL lock во время обработки (heartbeat)

2. **Fallback режим**
   - В fallback режиме lock освобождается только после завершения обработки
   - Если обработка зависнет, lock не будет освобожден до истечения TTL
   - TODO: Добавить мониторинг зависших обработок

3. **Race condition**
   - Теоретически возможна race condition между проверкой TaskManager и Redis lock
   - На практике это маловероятно из-за атомарности Redis операций
   - TODO: Рассмотреть использование транзакций Redis
