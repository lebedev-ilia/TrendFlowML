## Выполненные задачи

### ✅ Реализовать subprocess isolation

Улучшен `ProcessorService` для лучшей изоляции subprocess:

1. **Запуск subprocess для каждого run**
   - Использование `asyncio.create_subprocess_exec()` вместо `subprocess.run()`
   - Лучший контроль над процессом
   - Асинхронное чтение stdout/stderr

2. **Мониторинг процесса**
   - Асинхронное чтение stdout/stderr через `process.communicate()`
   - Правильная обработка завершения процесса
   - Логирование вывода процесса

3. **Обработка exit code**
   - Детальная классификация ошибок по exit code
   - Определение типа ошибки (general_error, misuse_of_shell_commands, signal_terminated, killed_by_memory_limit)
   - Извлечение основной ошибки из stderr

4. **Улучшения**
   - Использование `sys.executable` для определения Python интерпретатора
   - Правильная установка рабочей директории
   - Копирование переменных окружения для изоляции

### ✅ Добавить memory limits в Docker

Добавлены resource limits в docker-compose.yml:

1. **Memory limit для API контейнера**
   - Лимит: 2GB
   - Резервирование: 512MB

2. **CPU limits**
   - Лимит: 2 CPUs
   - Резервирование: 0.5 CPU

3. **Resource reservations**
   - Гарантированные ресурсы для стабильной работы
   - Предотвращение перегрузки контейнера

### ✅ Реализовать memory monitoring

Добавлен опциональный мониторинг памяти subprocess:

1. **Использование psutil**
   - Добавлена зависимость `psutil>=5.9.0` в requirements-api.txt
   - Опциональный импорт (graceful degradation если не установлен)

2. **Мониторинг памяти subprocess**
   - Функция `_monitor_subprocess_memory()` для мониторинга
   - Проверка каждые 10 секунд
   - Логирование использования памяти

3. **Kill при превышении лимита**
   - Конфигурируемый лимит через `SUBPROCESS_MEMORY_LIMIT_MB`
   - Автоматическое убийство процесса при превышении
   - Определение типа ошибки как "killed_by_memory_limit"

4. **Интеграция**
   - Мониторинг запускается как отдельная задача
   - Автоматическая отмена при завершении процесса
   - Обработка ошибок мониторинга

## Известные проблемы и ограничения

1. **Memory Monitoring**
   - Требует установки psutil
   - Мониторинг работает только если psutil доступен
   - TODO: Добавить fallback на другие методы мониторинга

2. **Subprocess Isolation**
   - Изоляция на уровне процесса, но не на уровне контейнера
   - Для полной изоляции нужны отдельные worker контейнеры
   - TODO: Реализовать отдельные worker контейнеры в Этапе 2.6

3. **Resource Limits**
   - Docker limits применяются ко всему контейнеру
   - Subprocess limits применяются к отдельному процессу
   - Может быть конфликт между двумя уровнями

4. **Memory Limit Enforcement**
   - psutil мониторинг не гарантирует точное соблюдение лимита
   - Процесс может превысить лимит между проверками
   - TODO: Использовать cgroups для более точного контроля
