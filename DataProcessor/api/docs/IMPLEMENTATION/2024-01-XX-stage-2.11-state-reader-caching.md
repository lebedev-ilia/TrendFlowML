## Выполненные задачи

### ✅ Добавить Redis клиент в `StateReader`

1. **Параметр `redis_client` в конструкторе**
   - Добавлен опциональный параметр `redis_client: Optional[Redis]`
   - Автоматическое получение через `get_redis_client()` если не указан
   - Сохранение в `self.redis_client` для использования в методах

2. **Поле `cache_ttl`**
   - Добавлено поле `self.cache_ttl = 300` (5 минут)
   - Используется для установки TTL при обновлении cache

### ✅ Реализовать hot path (Redis cache)

1. **Проверка Redis cache в `get_run_status()`**
   - Проверка cache в начале метода через `get_run_state_from_redis()`
   - Если cache доступен и содержит достаточно данных, возвращается упрощенный статус
   - Проверка heartbeat для running run'ов с автоматическим recovery

2. **Оптимизация для простых запросов**
   - Если `include_components=False` и `include_events=False`, возвращается упрощенный статус из cache
   - Это уменьшает нагрузку на Storage для частых запросов статуса

### ✅ Реализовать cold path (Storage)

1. **Чтение из Storage**
   - Если cache не доступен или не содержит нужных данных, читается из Storage
   - Загрузка `run_state.json` и состояний процессоров
   - Агрегация прогресса и определение текущей стадии

2. **Поиск метаданных**
   - Сначала проверяется Redis metadata через `get_run_metadata_from_redis()`
   - Если не найдено, используется `_find_run_metadata()` для поиска в Storage

### ✅ Реализовать обновление cache

1. **Функция `_update_cache_from_storage()`**
   - Обновляет Redis cache данными из Storage
   - Сериализует сложные структуры (processors, progress) в JSON
   - Использует `save_run_state_to_redis()` для сохранения

2. **Автоматическое обновление**
   - Cache обновляется автоматически после чтения из Storage
   - Это обеспечивает актуальность cache для последующих запросов

### ✅ Настроить TTL для cache

1. **TTL через `cache_ttl`**
   - Установлен в 300 секунд (5 минут)
   - Применяется через `save_run_state_to_redis()` (использует TTL_STATE из redis_schema)

2. **TTL для событий**
   - События в Redis Streams имеют TTL 1 день (TTL_EVENTS)
   - Устанавливается автоматически при добавлении события

### ✅ Обновить `get_events()` для использования Redis Streams

1. **Реализация `get_events()`**
   - Проверка Redis Streams через `get_run_events_from_redis()` для hot path
   - Fallback на чтение из Storage (state_events.jsonl) для cold path
   - Фильтрация событий по времени (since parameter)
   - Ограничение количества событий (limit parameter)

2. **Обработка ошибок**
   - Graceful degradation: если Redis не доступен, читается из Storage
   - Логирование предупреждений при ошибках чтения из Redis

### Архитектурные решения

1. **Hot Path (Redis Cache)**
   - Проверка cache в начале каждого запроса
   - Возврат упрощенного статуса из cache для простых запросов
   - Проверка heartbeat для running run'ов

2. **Cold Path (Storage)**
   - Чтение из Storage если cache не доступен
   - Полная загрузка данных из Storage для детальных запросов
   - Автоматическое обновление cache после чтения

3. **Обновление Cache**
   - Cache обновляется автоматически после чтения из Storage

4. **События**
   - Hot path: чтение из Redis Streams
   - Cold path: чтение из Storage (state_events.jsonl)
   - Фильтрация и ограничение количества событий

## Использование

### Автоматическое использование

Кэширование работает автоматически:

```python
# StateReader автоматически использует Redis cache
state_reader = StateReader(storage, key_layout, task_manager, redis_client)

# При запросе статуса:
# 1. Проверяется Redis cache (hot path)
# 2. Если cache не доступен, читается из Storage (cold path)
# 3. Cache автоматически обновляется после чтения из Storage

status = await state_reader.get_run_status(run_id)
events = await state_reader.get_events(run_id, since="2024-01-01T00:00:00Z", limit=50)
```

### Ручное использование

Для ручной работы с cache:

```python
from api.services.redis_schema import get_run_state, save_run_state

# Получить из cache
cached_state = await get_run_state(run_id)

# Обновить cache
await save_run_state(run_id, {
    "status": "running",
    "progress": json.dumps({"overall": 0.5}),
    "updated_at": datetime.now().isoformat()
})
```

## Известные проблемы и ограничения

1. **Сериализация данных**
   - Сложные структуры (processors, progress) сериализуются в JSON
   - Это может привести к потере типов при десериализации
   - TODO: Использовать более структурированный формат для cache

2. **Обновление cache**
   - Cache обновляется только при чтении из Storage
   - Если данные изменяются в Storage без чтения через StateReader, cache может устареть
   - TODO: Добавить механизм инвалидации cache при изменении данных

3. **TTL cache**
   - TTL установлен статически в 300 секунд
   - Для завершенных run'ов можно использовать больший TTL
   - TODO: Динамический TTL в зависимости от статуса run'а

4. **События из Storage**
   - События читаются из state_events.jsonl построчно
   - Для больших файлов это может быть медленно
   - TODO: Добавить индексацию событий по времени
