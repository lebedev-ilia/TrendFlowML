# Этап 6.3: Hybrid подход (Webhook + Polling Fallback)

**Дата**: 2024-01-XX  
**Этап**: 6.3 из [API_DEVELOPMENT_CHECKLIST.md](../../../docs/API_DEVELOPMENT_CHECKLIST.md)  
**Статус**: ✅ Завершено

## Обзор

Реализован production-grade hybrid подход для интеграции backend с DataProcessor API:
- **Webhook endpoint** для получения real-time уведомлений
- **SSE listener** для streaming обновлений
- **Polling fallback** если webhook/SSE не работают

## Выполненные задачи

### ✅ Реализовать webhook endpoint в backend

Создан endpoint `POST /api/webhooks/dataprocessor` в `backend/app/routers/webhooks.py`:

**Функциональность**:
- Валидация webhook signature через заголовок `X-Webhook-Signature`
- Обновление статуса AnalysisJob в БД
- Отправка WebSocket событий через Redis pubsub
- Обработка всех типов финальных статусов (success, error, empty, skipped, cancelled)

**Валидация подписи**:
- Для MVP используется простая проверка через API Key
- В production можно использовать HMAC-SHA256
- Graceful degradation: разрешает webhook без подписи в development mode

**Обновление БД**:
- Маппинг статусов DataProcessor → AnalysisJob:
  - `success` → `completed`
  - `error` → `failed`
  - `cancelled` → `cancelled`
  - `empty`, `skipped` → `completed`
- Обновление `completed_at` для финальных статусов
- Сохранение `error_message` при ошибках

### ✅ Реализовать SSE listener

Создана функция `stream_run_events_sse()` в `backend/app/services/dataprocessor.py`:

**Функциональность**:
- Подключение к SSE endpoint DataProcessor API (`/api/v1/runs/{run_id}/events`)
- Парсинг SSE формата (event:, data:)
- Обработка финальных событий (complete, error)
- Timeout защита
- Обработка ошибок с fallback на polling

**SSE формат**:
```
event: progress
data: {"run_id": "...", "progress": 0.5, ...}

event: complete
data: {"run_id": "...", "status": "success", ...}
```

### ✅ Реализовать polling fallback

Создана функция `wait_for_run_completion_hybrid()` в `backend/app/services/dataprocessor.py`:

**Стратегия**:
1. **Попытка 1**: SSE listener для real-time обновлений
   - Если SSE работает → получаем события в реальном времени
   - Если SSE завершается с финальным событием → возвращаем статус
   - Если SSE завершается без финального события → fallback на polling

2. **Fallback**: Polling (если SSE не работает)
   - Использует существующую функцию `poll_run_status()`
   - Периодические запросы к `/api/v1/runs/{run_id}/status`
   - Обработка всех типов ошибок

**Преимущества**:
- Real-time обновления через SSE (если доступно)
- Надежность через polling fallback
- Graceful degradation при ошибках

### ✅ Интегрировать hybrid подход в `process_analysis_job`

Обновлена Celery задача в `backend/app/tasks.py`:

**Было** (polling):
```python
final_status = asyncio.run(
    poll_run_status(
        run_id=payload.run_id,
        timeout_seconds=settings.dataprocessor_timeout_seconds,
        poll_interval=settings.dataprocessor_poll_interval
    )
)
```

**Стало** (hybrid):
```python
final_status = asyncio.run(
    wait_for_run_completion_hybrid(
        run_id=payload.run_id,
        webhook_timeout=30,  # Ждать webhook 30 секунд
        timeout_seconds=settings.dataprocessor_timeout_seconds,
        poll_interval=settings.dataprocessor_poll_interval
    )
)
```

## Технические детали

### Webhook signature

Для MVP используется простая проверка:
```python
expected_signature = settings.dataprocessor_api_key
return hmac.compare_digest(signature, expected_signature)
```

В production рекомендуется использовать HMAC-SHA256:
```python
expected_signature = hmac.new(
    settings.dataprocessor_api_key.encode(),
    payload_body,
    hashlib.sha256
).hexdigest()
```

### SSE парсинг

SSE формат:
```
event: <event_type>
data: <json_data>

```

Парсинг:
```python
if line.startswith("event: "):
    event_type = line[7:].strip()
if line.startswith("data: "):
    event_data = json.loads(line[6:])
```

### Hybrid стратегия

1. **SSE listener** (приоритет):
   - Real-time обновления
   - Меньше нагрузка на API
   - Лучший UX

2. **Polling fallback**:
   - Надежность
   - Работает даже если SSE недоступен
   - Простота реализации

## Тестирование

### Ручное тестирование

1. **Запустить DataProcessor API**:
   ```bash
   cd DataProcessor
   docker-compose up dataprocessor-api dataprocessor-worker
   ```

2. **Запустить backend**:
   ```bash
   cd backend
   export TF_BACKEND_DATAPROCESSOR_API_URL=http://localhost:8001
   export TF_BACKEND_DATAPROCESSOR_API_KEY=your-api-key
   uvicorn app.main:app --reload
   ```

3. **Создать задачу обработки** через API backend

4. **Проверить webhook**:
   ```bash
   curl -X POST http://localhost:8000/api/webhooks/dataprocessor \
     -H "Content-Type: application/json" \
     -H "X-Webhook-Signature: your-api-key" \
     -d '{
       "run_id": "...",
       "status": "success",
       "timestamp": "2024-01-01T12:00:00Z"
     }'
   ```

5. **Проверить SSE**:
   - Открыть SSE stream в браузере или через curl
   - Убедиться что события приходят в реальном времени

## Известные проблемы

1. **Webhook signature**: Для MVP используется простая проверка
   - Решение: В production использовать HMAC-SHA256

2. **SSE timeout**: SSE stream может зависнуть при длительных обработках
   - Решение: Использовать keepalive сообщения и timeout защиту

3. **Webhook URL**: DataProcessor API должен знать URL webhook'а
   - Решение: Передавать `webhook_url` в ProcessRequest (TODO)
