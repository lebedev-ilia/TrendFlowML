## Выполненные задачи

### ✅ Создать `api/worker.py`

Создан модуль `api/worker.py` как точка входа для отдельного worker процесса:

1. **Функция `main()`**
   - Настраивает обработчики сигналов (SIGTERM, SIGINT)
   - Запускает worker loop через `asyncio.run()`
   - Обрабатывает KeyboardInterrupt и другие исключения

2. **Функция `run_worker()`**
   - Инициализирует Redis клиент
   - Получает Storage и KeyLayout для checkpoint support
   - Создает Worker экземпляр
   - Запускает worker loop (блокирующий вызов)

3. **Функция `get_worker_id()`**
   - Генерирует уникальный ID worker'а
   - Использует переменную окружения `WORKER_ID` или hostname + PID

4. **Обработка сигналов**
   - `signal_handler()` для graceful shutdown
   - Обрабатывает SIGTERM и SIGINT
   - Корректно останавливает worker и закрывает Redis клиент

### ✅ Реализовать worker loop

Worker loop уже был реализован в `api/services/worker.py`:

1. **Чтение из Redis Streams**
   - Использует consumer groups для масштабирования
   - Читает из очередей high, normal, low
   - Использует `XREADGROUP` для получения сообщений

2. **Запуск subprocess для каждого run**
   - Использует `ProcessorService.run_processing()`
   - Subprocess isolation через `asyncio.create_subprocess_exec`
   - Опциональный мониторинг памяти через psutil

3. **Heartbeat loop**
   - Отправляет heartbeat каждые 30 секунд
   - TTL heartbeat: 60 секунд
   - Автоматическая остановка после завершения обработки

4. **Обработка ошибок**
   - Обработка исключений в `_process_message()`
   - Обновление статуса на error при ошибке
   - Освобождение lock даже при ошибке
   - Сообщения остаются в pending для retry

### ✅ Создать Dockerfile для worker

Создан `docker/worker/Dockerfile`:

1. **Базовый образ**
   - Использует `python:3.11-slim`
   - Те же системные зависимости что и API (ffmpeg, libsm6, libxext6, libxrender-dev, libgomp1)

2. **Зависимости**
   - Использует `requirements-api.txt` (те же зависимости что и API)
   - Устанавливает все необходимые Python пакеты

3. **Переменные окружения**
   - `STORAGE_TYPE=fs`
   - `STORAGE_ROOT=/data`
   - `LOG_LEVEL=INFO`
   - `LOG_FORMAT=json`
   - `REDIS_URL=redis://redis:6379/0`

4. **Команда запуска**
   - `CMD ["python", "-m", "api.worker"]`

### ✅ Обновить docker-compose.yml

Обновлен `docker-compose.yml` с отдельным worker сервисом:

1. **Сервис `dataprocessor-worker`**
   - Отдельный контейнер для worker процесса
   - Зависит от `redis` и `dataprocessor-api`
   - Использует те же volumes что и API

2. **Resource limits**
   - Memory limit: 4G (больше чем API, так как обрабатывает видео)
   - CPU limit: 4 cores
   - Memory reservation: 1G
   - CPU reservation: 1 core

3. **Масштабирование**
   - Можно запустить несколько worker'ов через `docker-compose up --scale dataprocessor-worker=3`
   - Каждый worker получает уникальный `WORKER_ID`

4. **Переменные окружения**
   - `WORKER_ID` - уникальный ID worker'а
   - `MAX_CONCURRENT_RUNS=2` - параллельных задач на worker
   - `SUBPROCESS_MEMORY_LIMIT_MB=8000` - лимит памяти для subprocess

### Архитектурные решения

1. **Отдельный процесс**
   - Worker запускается как отдельный процесс, не внутри API
   - Это позволяет масштабировать worker'ы независимо от API
   - API и Worker могут быть на разных хостах

2. **Graceful shutdown**
   - Обработка SIGTERM и SIGINT для корректного завершения
   - Остановка worker loop перед завершением
   - Закрытие Redis клиента

3. **Worker ID**
   - Генерируется автоматически из hostname + PID
   - Может быть переопределен через `WORKER_ID` environment variable
   - Используется для идентификации worker'а в consumer group

4. **Storage и KeyLayout**
   - Worker использует те же Storage и KeyLayout что и API
   - Это необходимо для checkpoint support
   - Инициализируется напрямую, без FastAPI Depends

## Использование

### Локальный запуск

```bash
# Запустить worker локально
python -m api.worker

# Или
python api/worker.py

# С указанием WORKER_ID
WORKER_ID=worker-1 python -m api.worker
```

### Docker Compose

```bash
# Запустить один worker
docker-compose up dataprocessor-worker

# Запустить несколько worker'ов
docker-compose up --scale dataprocessor-worker=3

# Запустить все сервисы (API + Worker + Redis)
docker-compose up -d
```

### Проверка работы

```bash
# Проверить логи worker'а
docker-compose logs -f dataprocessor-worker

# Проверить статус worker'ов
docker-compose ps dataprocessor-worker

# Проверить consumer group в Redis
docker-compose exec redis redis-cli XINFO GROUPS queue:normal
```
