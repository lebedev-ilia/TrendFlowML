## Выполненные задачи

### ✅ Настроить `slowapi`

Настроен `slowapi` для rate limiting:

1. **Добавление зависимости**
   - Добавлен `slowapi>=0.1.9` в `requirements-api.txt`
   - Версия выбрана для совместимости с FastAPI

2. **Инициализация Limiter**
   - Инициализация в `api/main.py` с проверкой доступности
   - Использование функции `get_backend_id()` для определения ключа rate limiting
   - Graceful degradation: если slowapi не установлен, rate limiting отключается с предупреждением

3. **Настройка exception handler**
   - Добавлен exception handler для `RateLimitExceeded`
   - Использование стандартного обработчика `_rate_limit_exceeded_handler` из slowapi
   - Автоматический возврат 429 Too Many Requests при превышении лимита

### ✅ Реализовать rate limit per backend

Реализован rate limit per backend instance:

1. **Использование `X-Backend-ID` header**
   - Создана функция `get_backend_id(request: Request)` в `api/main.py`
   - Извлечение backend ID из заголовка `X-Backend-ID`
   - Fallback на IP адрес клиента если заголовок отсутствует
   - Это позволяет ограничивать запросы от каждого backend instance отдельно

2. **Лимит "100/hour"**
   - Применен к `POST /api/v1/process` endpoint
   - Лимит: 100 запросов в час на backend instance
   - Используется sliding window алгоритм для подсчета запросов

3. **Интеграция в endpoint**
   - Добавлен декоратор `@limiter.limit("100/hour")` к `process_video()`
   - Добавлен параметр `http_request: Request` для работы rate limiter
   - Создана функция `apply_rate_limit()` для условного применения декоратора
   - Если slowapi не доступен, декоратор не применяется (graceful degradation)

## Технические детали

### Измененные файлы

1. **`requirements-api.txt`**
   - Добавлен `slowapi>=0.1.9` для rate limiting

2. **`api/main.py`**
   - Добавлен импорт slowapi с обработкой отсутствия библиотеки
   - Создана функция `get_backend_id()` для извлечения backend ID
   - Инициализация `Limiter` с `key_func=get_backend_id`
   - Настройка exception handler для `RateLimitExceeded`
   - Экспорт `limiter` для использования в endpoints

3. **`api/endpoints/process.py`**
   - Добавлен импорт `Request` из FastAPI
   - Добавлен импорт `limiter` из `api.main`
   - Создана функция `apply_rate_limit()` для условного применения декоратора
   - Добавлен декоратор `@apply_rate_limit` к `process_video()`
   - Добавлен параметр `http_request: Request` в функцию

### Архитектурные решения

1. **Rate limit per backend**
   - Использование заголовка `X-Backend-ID` для идентификации backend instance
   - Это позволяет ограничивать запросы от каждого backend instance отдельно
   - Fallback на IP адрес если заголовок отсутствует (для прямых запросов)

2. **Graceful degradation**
   - Если slowapi не установлен, rate limiting отключается
   - API продолжает работать без rate limiting
   - Логируется предупреждение о том, что rate limiting отключен

3. **Лимит "100/hour"**
   - Лимит выбран на основе требований архитектуры
   - Это позволяет обрабатывать до 100 запросов в час от каждого backend instance
   - Используется sliding window алгоритм для подсчета запросов

4. **Условное применение декоратора**
   - Создана функция `apply_rate_limit()` для условного применения декоратора
   - Если limiter не доступен, функция возвращается без изменений
   - Это позволяет работать с rate limiting опционально

## Использование

### Настройка rate limiting

Rate limiting работает автоматически при установке slowapi:

```bash
# Установить slowapi
pip install slowapi>=0.1.9

# Запустить API
python -m api.main
```

### Использование с X-Backend-ID

```bash
# Запрос с X-Backend-ID header
curl -X POST http://localhost:8001/api/v1/process \
  -H "X-API-Key: your-api-key" \
  -H "X-Backend-ID: backend-instance-1" \
  -H "Content-Type: application/json" \
  -d @request.json

# Rate limiting будет применен отдельно для каждого backend instance
```

### Превышение лимита

При превышении лимита (более 100 запросов в час) API вернет:

```json
{
  "error": "Rate limit exceeded: 100 per 1 hour"
}
```

HTTP статус: `429 Too Many Requests`

### Без slowapi

Если slowapi не установлен:
- API продолжает работать без rate limiting
- В логах будет предупреждение: "Rate limiting disabled (slowapi not available)"
- Все запросы будут обрабатываться без ограничений

## Известные проблемы и ограничения

1. **In-memory storage**
   - slowapi по умолчанию использует in-memory storage для хранения счетчиков
   - При перезапуске API счетчики сбрасываются
   - Для production может потребоваться Redis-based storage
   - TODO: Добавить поддержку Redis-based storage для rate limiting

2. **Один лимит для всех endpoints**
   - В текущей реализации лимит применяется только к `POST /api/v1/process`
   - Другие endpoints не имеют rate limiting
   - TODO: Добавить rate limiting для других endpoints при необходимости

3. **Отсутствие настраиваемых лимитов**
   - Лимит "100/hour" захардкожен в коде
   - Нет возможности настроить лимит через конфигурацию
   - TODO: Добавить настраиваемые лимиты через config

4. **Отсутствие метрик**
   - Нет метрик для отслеживания rate limit hits
   - Нельзя отследить сколько раз был превышен лимит
   - TODO: Добавить метрики для rate limiting

5. **Отсутствие разных лимитов для разных backend**
   - Все backend instances имеют одинаковый лимит
   - Нет возможности установить разные лимиты для разных backend
   - TODO: Добавить поддержку разных лимитов для разных backend
