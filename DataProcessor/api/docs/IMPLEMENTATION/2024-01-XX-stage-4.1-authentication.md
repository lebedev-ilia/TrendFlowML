## Выполненные задачи

### ✅ Реализовать API Key аутентификацию (MVP)

Реализована API Key аутентификация для защиты API endpoints:

1. **APIKeyHeader dependency**
   - Реализован в `api/security.py`
   - Использует `fastapi.security.APIKeyHeader` с именем "X-API-Key"
   - `auto_error=False` для кастомной обработки ошибок
   - Описание для OpenAPI документации

2. **verify_api_key() функция**
   - Проверка наличия API key (401 если отсутствует)
   - Проверка валидности API key (403 если невалиден)
   - Логирование попыток аутентификации (warning для неудачных попыток)
   - Development mode: разрешает доступ если API key не настроен (с предупреждением)
   - Возвращает API key при успешной проверке

3. **Проверка из env переменных**
   - Добавлено поле `api_key` в `APIConfig` из `DATAPROCESSOR_API_KEY`
   - Добавлено поле `auth_type` для будущего выбора типа аутентификации
   - Поддержка отсутствия API key для development режима

4. **Интеграция во все endpoints**
   - `POST /api/v1/process` - добавлена аутентификация
   - `GET /api/v1/runs/{run_id}` - добавлена аутентификация
   - `GET /api/v1/runs/{run_id}/status` - добавлена аутентификация
   - `GET /api/v1/runs/{run_id}/events` - добавлена аутентификация
   - `GET /api/v1/runs/{run_id}/manifest` - добавлена аутентификация
   - `GET /api/v1/runs/{run_id}/artifacts/{component}` - добавлена аутентификация
   - Health (`GET /api/v1/health`) и metrics (`GET /api/v1/metrics`) endpoints остаются без аутентификации (стандартная практика для мониторинга)

### Архитектурные решения

1. **Development mode**
   - Если API key не настроен (`DATAPROCESSOR_API_KEY` не установлен), доступ разрешается с предупреждением
   - Это позволяет работать в development без необходимости настройки API key
   - В production всегда должен быть установлен `DATAPROCESSOR_API_KEY`

2. **Health и metrics endpoints**
   - Health и metrics endpoints остаются без аутентификации
   - Это стандартная практика для мониторинга и health checks
   - Эти endpoints обычно защищаются на уровне инфраструктуры (firewall, reverse proxy)

3. **Обработка ошибок**
   - 401 Unauthorized: если API key не предоставлен
   - 403 Forbidden: если API key невалиден
   - Логирование всех попыток аутентификации (успешных и неудачных)

4. **mTLS инфраструктура**
   - Подготовлена структура для будущей реализации mTLS
   - Функция `verify_mtls()` содержит TODO комментарии с описанием необходимых шагов
   - Функция `get_auth_dependency()` позволяет легко переключиться на mTLS в будущем

## Использование

### Настройка API Key

```bash
# Установить API key через environment variable
export DATAPROCESSOR_API_KEY="your-secret-api-key-here"

# Или через .env файл
echo "DATAPROCESSOR_API_KEY=your-secret-api-key-here" >> .env
```

### Использование API с аутентификацией

```bash
# Запрос с API key в заголовке
curl -X POST http://localhost:8001/api/v1/process \
  -H "X-API-Key: your-secret-api-key-here" \
  -H "Content-Type: application/json" \
  -d @request.json

# Запрос без API key (вернет 401)
curl -X POST http://localhost:8001/api/v1/process \
  -H "Content-Type: application/json" \
  -d @request.json
# Response: {"detail": "API key required"}

# Запрос с невалидным API key (вернет 403)
curl -X POST http://localhost:8001/api/v1/process \
  -H "X-API-Key: invalid-key" \
  -H "Content-Type: application/json" \
  -d @request.json
# Response: {"detail": "Invalid API key"}
```

### Development режим

Если `DATAPROCESSOR_API_KEY` не установлен, API будет работать в development режиме:
- Все запросы будут разрешены
- В логах будет предупреждение о том, что API key не настроен
- Это позволяет работать без настройки аутентификации в development

### OpenAPI документация

FastAPI автоматически добавит информацию об аутентификации в OpenAPI схему:
- Endpoints будут помечены как требующие аутентификации
- В Swagger UI появится кнопка "Authorize" для ввода API key
- API key можно будет ввести один раз и использовать для всех запросов

## Известные проблемы и ограничения

1. **Один API key**
   - В текущей реализации поддерживается только один API key
   - Для production может потребоваться поддержка нескольких API keys или управление через БД
   - TODO: Добавить поддержку множественных API keys

2. **Отсутствие ротации ключей**
   - Нет механизма ротации API keys
   - Для production может потребоваться поддержка ротации ключей
   - TODO: Добавить поддержку ротации API keys

3. **Отсутствие rate limiting per key**
   - Нет ограничения количества запросов на API key
   - Это может быть реализовано в будущем
   - TODO: Добавить rate limiting per API key

4. **mTLS не реализован**
   - Функция `verify_mtls()` является заглушкой
   - Для production может потребоваться полная реализация mTLS
   - TODO: Реализовать полную поддержку mTLS

5. **Health и metrics без аутентификации**
   - Health и metrics endpoints остаются без аутентификации
   - Это может быть проблемой безопасности в некоторых случаях
   - Рекомендуется защищать эти endpoints на уровне инфраструктуры
