## Выполненные задачи

### ✅ Реализовать `GET /api/v1/runs/{run_id}/events`

Endpoint реализован в `api/endpoints/runs.py`:

1. **Основной endpoint**
   - Путь: `GET /api/v1/runs/{run_id}/events`
   - Использует `EventSourceResponse` из `sse-starlette`
   - Возвращает формат `text/event-stream`
   - Проверяет существование run'а перед стримингом
   - Обрабатывает ошибки (404, 410, 503)

2. **Проверка доступности**
   - Проверка существования run'а через `StateReader`
   - Проверка статуса run'а (410 если завершён и события недоступны)
   - Проверка доступности Redis (503 если не доступен)

### ✅ Использовать Redis Streams для событий

Интеграция с Redis Streams:

1. **Публикация событий**
   - Уже реализовано в `api/services/redis_schema.py`
   - Использует `XADD` для добавления событий в `stream:events:{run_id}`
   - TTL для stream: 1 день

2. **Чтение событий**
   - Использует `XREAD` для чтения новых событий в реальном времени
   - Блокирующее чтение с таймаутом (по умолчанию 5 секунд)
   - Поддержка чтения с определенного ID (для фильтрации по времени)

3. **Формат данных**
   - События хранятся в формате: `{"event_type": "...", "timestamp": "...", "data": "..."}`
   - Данные события хранятся как JSON строка в поле `data`

### ✅ Реализовать SSE streaming

SSE streaming реализован в `api/services/sse_service.py`:

1. **Формат SSE**
   - Формат: `event: <type>\ndata: <json>\n\n`
   - Поддержка стандартных SSE событий

2. **Типы событий**
   - `progress` - обновления прогресса обработки
   - `stage` - смена стадии обработки
   - `component_start` - начало обработки компонента
   - `component_complete` - завершение обработки компонента
   - `complete` - завершение всей обработки
   - `error` - ошибки обработки
   - `connected` - подключение к SSE stream
   - `disconnected` - отключение от SSE stream

3. **Маппинг типов событий**
   - Функция `_map_event_type_to_sse()` маппит типы событий из Redis на SSE event types
   - Поддержка префиксов для автоматического определения типа

4. **Keepalive**
   - Отправка keepalive сообщений для поддержания соединения
   - Небольшая задержка между чтениями для предотвращения busy loop

### ✅ Query параметры

Реализованы query параметры:

1. **`since=<timestamp>`**
   - ISO 8601 timestamp для фильтрации событий
   - Пример: `since=2024-01-01T12:00:00Z`
   - Использует `XRANGE` для поиска первого сообщения после указанного времени
   - Если не указан, читает с начала stream

2. **`component=<name>`**
   - Фильтр по компоненту
   - Пример: `component=core_clip`
   - Фильтрует события по полю `component` в данных события

### ✅ Ограничение количества соединений

Реализован `SSEConnectionManager`:

1. **Отслеживание соединений**
   - Словарь `_connections` для отслеживания активных соединений по run_id
   - Использование `asyncio.Lock` для thread-safety

2. **Методы управления**
   - `acquire(run_id)` - попытка получить соединение
   - `release(run_id)` - освобождение соединения
   - `get_connection_count(run_id)` - получение количества активных соединений

3. **Конфигурация**
   - `max_sse_connections_per_run` в `config.py` (по умолчанию 10)
   - Настраивается через переменную окружения `MAX_SSE_CONNECTIONS_PER_RUN`

4. **Обработка ошибок**
   - Выбрасывает `ValueError` если превышен лимит соединений
   - Endpoint возвращает ошибку в SSE stream

### Архитектурные решения

1. **SSE vs WebSocket**
   - Выбран SSE для простоты реализации и односторонней передачи данных
   - SSE поддерживается большинством браузеров и HTTP клиентов
   - Не требует дополнительных протоколов или библиотек на клиенте

2. **Redis Streams для событий**
   - Использование Redis Streams обеспечивает персистентность событий
   - Поддержка чтения с определенного ID для фильтрации по времени
   - Блокирующее чтение для real-time стриминга

3. **Ограничение соединений**
   - Предотвращает перегрузку сервера множественными SSE соединениями
   - Ограничение на уровне run_id для распределения нагрузки
   - Graceful degradation при превышении лимита

4. **Обработка ошибок**
   - Проверка существования run'а перед стримингом
   - Проверка доступности Redis
   - Отправка ошибок через SSE stream для информирования клиента

## Использование

### Пример запроса

```bash
# Базовый запрос
curl -N http://localhost:8001/api/v1/runs/{run_id}/events

# С фильтром по времени
curl -N "http://localhost:8001/api/v1/runs/{run_id}/events?since=2024-01-01T12:00:00Z"

# С фильтром по компоненту
curl -N "http://localhost:8001/api/v1/runs/{run_id}/events?component=core_clip"
```

### Пример клиента на Python

```python
import httpx
import json

async def stream_events(run_id: str):
    async with httpx.AsyncClient(timeout=None) as client:
        async with client.stream(
            "GET",
            f"http://localhost:8001/api/v1/runs/{run_id}/events"
        ) as response:
            async for line in response.aiter_lines():
                if line.startswith("event: "):
                    event_type = line[7:]
                elif line.startswith("data: "):
                    data = json.loads(line[6:])
                    print(f"Event {event_type}: {data}")
```

### Пример клиента на JavaScript

```javascript
const eventSource = new EventSource(`/api/v1/runs/${runId}/events`);

eventSource.addEventListener('progress', (e) => {
    const data = JSON.parse(e.data);
    console.log('Progress:', data);
});

eventSource.addEventListener('complete', (e) => {
    const data = JSON.parse(e.data);
    console.log('Complete:', data);
    eventSource.close();
});

eventSource.onerror = (error) => {
    console.error('SSE error:', error);
    eventSource.close();
};
```

## Известные проблемы и ограничения

1. **Таймауты**
   - Таймаут чтения из Redis Streams установлен в 5 секунд
   - Может быть недостаточно для медленных соединений
   - TODO: Настраиваемый таймаут через query параметр

2. **Фильтрация по времени**
   - Использует `XRANGE` для поиска сообщений после указанного времени
   - Может быть медленно для больших stream'ов
   - TODO: Оптимизация поиска через индексы

3. **Обработка завершенных run'ов**
   - Проверка доступности событий для завершенных run'ов
   - События могут быть удалены после TTL (1 день)
   - TODO: Более детальная обработка завершенных run'ов

4. **Масштабирование**
   - Ограничение соединений на уровне одного API инстанса
   - При масштабировании API нужно синхронизировать соединения
