# Руководство по документированию изменений

Этот документ описывает, как правильно документировать изменения в API модуле.

## Когда создавать документы

Создавайте документы в следующих случаях:

1. **Реализация нового этапа** → `IMPLEMENTATION/YYYY-MM-DD-stage-X.Y-description.md`
2. **Изменение API** (endpoint, schema, response) → `API_CHANGES/YYYY-MM-DD-change-description.md`
3. **Архитектурное изменение** → `ARCHITECTURE/YYYY-MM-DD-architecture-change-description.md`
4. **Устаревание функции** → Обновить `DEPRECATIONS.md`
5. **Любое значимое изменение** → Обновить `CHANGELOG.md`

## Процесс документирования

### 1. Реализация этапа

После завершения этапа из чеклиста:

1. Создайте отчет в `IMPLEMENTATION/`:
   ```bash
   # Пример имени файла
   IMPLEMENTATION/2024-01-15-stage-1.2-fastapi-app.md
   ```

2. Обновите `CHANGELOG.md`:
   - Добавьте запись в секцию `[Unreleased]`
   - Укажите тип изменения (Добавлено/Изменено/Исправлено)

3. Обновите `INDEX.md`:
   - Добавьте ссылку на новый отчет
   - Обновите соответствующие секции

### 2. Изменение API

При изменении endpoints, schemas, responses:

1. Создайте документ в `API_CHANGES/`:
   ```bash
   # Примеры имен файлов
   API_CHANGES/2024-01-15-endpoint-process-added.md
   API_CHANGES/2024-01-20-schema-processrequest-updated.md
   ```

2. Обновите `CHANGELOG.md`

3. Если есть breaking changes:
   - Обновите `DEPRECATIONS.md`
   - Добавьте информацию о миграции

### 3. Архитектурное изменение

При изменении архитектуры (Redis, Worker, State Machine и т.д.):

1. Создайте документ в `ARCHITECTURE/`:
   ```bash
   # Примеры имен файлов
   ARCHITECTURE/2024-01-25-redis-integration-added.md
   ARCHITECTURE/2024-02-01-worker-isolation-implemented.md
   ```

2. Обновите `CHANGELOG.md`

3. Обновите основной архитектурный документ (`DATAPROCESSOR_API_ARCHITECTURE.md`)

### 4. Устаревание функции

При объявлении функции устаревшей:

1. Обновите `DEPRECATIONS.md`:
   - Добавьте запись с форматом из примера
   - Укажите версию устаревания и удаления
   - Добавьте инструкции по миграции

2. Обновите `CHANGELOG.md` в секции "Устарело"

3. Добавьте deprecation warning в код (если применимо)

## Шаблоны документов

### Шаблон отчета о реализации

```markdown
# Отчет о реализации: Этап X.Y - Описание

**Дата**: YYYY-MM-DD  
**Этап**: X.Y Описание  
**Статус**: ✅ ЗАВЕРШЕНО / ⏳ В ПРОЦЕССЕ / ⏸️ ПРИОСТАНОВЛЕНО  
**Чеклист**: `DataProcessor/docs/API_DEVELOPMENT_CHECKLIST.md` (строки X-Y)

## Метаданные
- **Версия API**: X.Y.Z
- **Ссылка на архитектуру**: `DataProcessor/docs/DATAPROCESSOR_API_ARCHITECTURE.md` (строки X-Y)

## Выполненные задачи
- ✅ Задача 1
- ✅ Задача 2

## Технические детали
...

## Тестирование
...

## Известные проблемы
...

## Следующие шаги
...
```

### Шаблон изменения API

```markdown
# Изменение: Описание

**Дата**: YYYY-MM-DD  
**Версия API**: X.Y.Z  
**Тип**: Добавление / Изменение / Удаление

## Описание
Что изменилось и почему.

## Детали
- Старое поведение (если изменение)
- Новое поведение
- Примеры

## Миграция
Как мигрировать существующий код.

## Связанные изменения
- Ссылки на другие документы
```

## Обновление CHANGELOG.md

Формат записи в CHANGELOG:

```markdown
## [Unreleased]

### Добавлено
- Описание добавления

### Изменено
- Описание изменения

### Исправлено
- Описание исправления

### Устарело
- Описание устаревания

### Удалено
- Описание удаления
```

## Обновление INDEX.md

При добавлении нового документа:

1. Найдите соответствующую секцию в `INDEX.md`
2. Добавьте ссылку на новый документ
3. Обновите дату последнего обновления (если есть)

## Примеры

### Пример 1: Реализация этапа

После завершения этапа 1.2:

1. Создать: `IMPLEMENTATION/2024-01-15-stage-1.2-fastapi-app.md`
2. Обновить `CHANGELOG.md`:
   ```markdown
   ### Добавлено
   - FastAPI приложение с настройкой CORS и логирования (Этап 1.2)
   ```
3. Обновить `INDEX.md`:
   ```markdown
   - [Этап 1.2: FastAPI приложение](./IMPLEMENTATION/2024-01-15-stage-1.2-fastapi-app.md) ✅
   ```

### Пример 2: Изменение endpoint

При изменении формата ответа POST /process:

1. Создать: `API_CHANGES/2024-01-20-endpoint-process-response-updated.md`
2. Обновить `CHANGELOG.md`:
   ```markdown
   ### Изменено
   - Формат ответа POST /api/v1/process (добавлено поле estimated_duration)
   ```
3. Если breaking change → обновить `DEPRECATIONS.md`

## Проверка перед коммитом

Перед коммитом убедитесь:

- [ ] Создан соответствующий документ (если нужно)
- [ ] Обновлен `CHANGELOG.md`
- [ ] Обновлен `INDEX.md` (если добавлен новый документ)
- [ ] Обновлен `DEPRECATIONS.md` (если есть устаревания)
- [ ] Все ссылки работают
- [ ] Форматирование корректное
