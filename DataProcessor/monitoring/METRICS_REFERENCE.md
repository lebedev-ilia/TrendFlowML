# Справочник метрик Prometheus / Grafana (DataProcessor)

В **Explore** или **Metrics browser** Grafana видны **все** серии, попавшие в Prometheus — не только «бизнес»-метрики. Ниже: что относится к **DataProcessor**, что к **стандартной библиотеке** `prometheus_client` (процесс Python), и как читать типы.

**Важно:** при E2E два scrape target’а (API и worker) отдают **одни и те же имена** `dataprocessor_*`, но с разными лейблами `job` / `instance`. Расход `python_*` / `process_*` — это **тот Python-процесс**, который обслуживает данный endpoint (`/api/v1/metrics` у API, `/metrics` у worker).

> **Про ориентиры ниже:** цифры — **порядок величин** для типичного локального E2E / одной ноды. Они **не** SLA: длина видео, профиль, GPU, `MAX_CONCURRENT_RUNS`, Triton, диск **сильно** меняют «норму». Сравнивайте **с собой** по времени (до/после релиза) и при **стабильном** контенте.

## 1. Метрики продукта (`dataprocessor_*`)

Определены в `api/services/metrics.py`, регистрация в общем `REGISTRY` `prometheus_client`.

| Метрика | Тип | Лейблы | Что показывает | Примерный ориентир «норма» | Когда смотреть внимательнее |
|--------|-----|--------|----------------|----------------------------|----------------------------|
| `dataprocessor_queue_length` | Gauge | `priority` = high, normal, low | Сколько задач **сейчас** ждут в очереди с этим приоритетом. | **0** при простое; при постоянном потоке — **единицы–десятки** (очередь «дышит»). | Долго **>50–100** на приоритете: перегруз/затык воркера, смотрите `active_runs` и `processing_seconds`. **Всегда 0** при работающем потоке — тоже странно, если ожидаете бэклог. |
| `dataprocessor_queue_wait_seconds` | Histogram | — | Сколько **секунд** от постановки в очередь до старта обработки. | В простаивающем стеке **p95 ≈ 0–10 s**; под нагрузкой p95 **десятки секунд — несколько минут** возможны. | p95 **стабильно много минут/десятки минут** при пустой GPU и низком `active_runs` — логика очереди/блокировки; **растёт вместе с `queue_length`**. |
| `dataprocessor_processing_seconds` | Histogram | `processor`, `component` | Длительность **одного** этапа (наблюдение `observe` в **секундах**). Сквозной прогон часто `processor=pipeline`, `component=main_py`. | **Секунды–часы** в зависимости от длительности ролика и этапа; **полный `main_py` по длинному видео** — **десятки минут–часы** не редкость. p95 > p50 — длинный хвост (тяжёлые ролики). | `rate` **ошибок** + рост p95 **при том же контенте**; **0 наблюдений** при работающем worker — target не с той ноды / метрика не в том процессе. |
| `dataprocessor_failures_total` | Counter | `processor`, `component`, `error_type` | Суммарное число **зафиксированных** сбоев с момента старта процесса (счётчик). | **Стабильный** прогон: `rate` близок к **0** (нет новых инкрементов за окно). Редкие **единичные** инкремента — как отладочный сигнал. | **Постоянный** положительный `rate` по одной связке `processor/component/error_type` — повторяющаяся ошибка пайплайна/контента. |
| `dataprocessor_memory_bytes` | Gauge | `run_id` | Оценка/учёт **RSS-ориентированной** памяти, привязанной к `run_id` (если код обновляет). | **Сотни МБ – несколько ГБ** на тяжёлый run — типично для визуала+батчей; **зависит** от `run_id` и стадии. | **Безумный рост** на одном `run_id` + OOM в логах; **0** везде — метрика может не заполняться, не полагайтесь слепо. |
| `dataprocessor_active_runs` | Gauge | — | Сколько run **параллельно** в работе (внутренняя логика). | **0** когда нет задач; при лимите — **до** `MAX_CONCURRENT_RUNS` (часто **1–8**). | **Долго = лимит**, а `queue_length` большой — не успевает воркер; **0** при известных зависших job — проверить воркер/Redis. |
| `dataprocessor_crashed_runs_total` | Counter | — | Сколько раз учли run как **сорванный** (heartbeat/логика «crash»). | **Не растёт** в стабильной среде; общее значение **мало** относительно числа успешных. | **Рост** при сетевых/диске/Kill -9/необработанных исключениях в child-процессе — сопоставить с **логами** и `failures_total`. |
| `dataprocessor_component_stage_seconds` | Histogram | `worker_id`, `component`, `component_kind`, `status` | **По одному observe** на каждый компонент в `manifest.json` после завершения `main.py` (секунды, `duration_ms/1000`). Даёт разрез по **реальным** стадиям пайплайна, в отличие от одного `processing_seconds` с `pipeline/main_py`. | От **долей секунды** до **десятков минут** на тяжёлые модули; сравнивайте **p95 по component** между релизами. | **Нет** данных — worker не дошёл до чтения manifest (путь не совпал, ранний crash до flush). **Рост p95** при том же контенте — регресс по конкретному `component`. |
| `dataprocessor_host_cpu_usage_ratio` | Gauge | — | Доля CPU (0..1), **psutil.cpu_percent** при scrape `/metrics` (throttle ~3 с). | Зависит от **того же процесса**, что отдаёт `/metrics` (API или worker), не «весь хост» с идеальной точностью. | Ноль/странные ступени — первый замер `cpu_percent`; смотрите **тренд**, не абсолют. |
| `dataprocessor_host_load1` | Gauge | — | `load average` **1m** (Unix), если доступен. | Обычно **< число ядер** в спокойной системе. | **Долго >** количества CPU — хост перегружен. |
| `dataprocessor_host_num_cpus` | Gauge | — | Логические CPU (psutil). | Справочно для **интерпретации load**. | — |
| `dataprocessor_host_memory_available_bytes` | Gauge | — | «Свободно» в смысле `psutil.virtual_memory().available` (для аллокаций, не unmapped cache). | Сильно зависит от **ОЗУ** и **других** процессов. | **Низкое** значение + OOM/свап в логах — дефицит RAM. |
| `dataprocessor_host_disk_free_bytes` | Gauge | `mountpoint` | Свободное место на **/** и опционально на пути из `DATAPROCESSOR_HOST_DISK_PATH` (тот же scrape). | Зависит от **тома**, куда пишет result_store. | **Мало** — риск не записать артефакты. |

**Типы (кратко):**

- **Gauge** — мгновенное значение, может расти и уменьшаться.
- **Counter** — только рост; для «сколько за интервал» — `increase` / `rate` за **5m** (или 1m для отзывчивости).
- **Histogram** — `*_bucket{le=...}` кумулятивно, плюс `_count`, `_sum`. Квантиль: `histogram_quantile(0.95, sum(rate(...[5m])) by (le, ...))`. Подробно: [Histograms and summaries](https://prometheus.io/docs/practices/histograms/).

## 2. Служебные метрики Python (`python_*`)

Подключаются автоматически `prometheus_client` при `generate_latest()` (API) и `start_http_server()` (worker).

| Метрика | Тип | Лейблы | Что показывает | Примерный ориентир «норма» | Когда смотреть внимательнее |
|--------|-----|--------|----------------|----------------------------|----------------------------|
| `python_info` | Gauge | `implementation`, `version`, … | Метка **какой Python**; числовое value часто **1** — «серия живая». | Как **справка** при репортах: CPython 3.10/3.11/…; не метрика нагрузки. | **Нет** series — endpoint не тот / экспорт сломан. |
| `python_gc_objects_collected_total` | Counter | `generation` = 0, 1, 2 | Сколько объектов GC **собрал** с **старта процесса** (накапливается), по поколениям. | **Большие числа** (миллионы–миллиарды) на долгом процессе — **норм** при активных аллокациях; gen0 обычно **>>** gen1 **>>** gen2. | **Резкий** относительно baseline рост **без** роста полезной нагрузки — косвенно про «больше мусорных» аллокаций (после смены кода). |
| `python_gc_objects_uncollectable_total` | Counter | `generation` | Объекты, которые **нельзя** собрать (циклы с финализаторами и т.п.). | **0** или **несколько** и **стабильно** после прогрева. | **Рост** со временем (особенно >0 стабильно) — **подозрение на утечку/циклы**; смотреть `tracemalloc`/профили. |
| `python_gc_collections_total` | Counter | `generation` | Сколько **раз** запускали коллектор для данного **поколения**. | gen0: **самое большое** число запусков; gen1, gen2 — **меньше** на порядок. Частота по времени **само по себе** не критерий; важен **тренд** к своему baseline. | **Аномальный** всплеск `rate` при **плоской** нагрузке + рост **RSS** — давление по памяти, GC «частит». Сравнивайте **до/после** деплоя. |

*Поколения GC: gen0 — молодые объекты; gen1, gen2 — старше; см. [gc](https://docs.python.org/3/library/gc.html).*

## 3. Служебные метрики процесса ОС (`process_*`)

`ProcessCollector` (Linux) для **того же** PID, что и Python.

| Метрика | Тип | Что показывает | Примерный ориентир «норма» | Когда смотреть внимательнее |
|--------|-----|----------------|----------------------------|----------------------------|
| `process_cpu_seconds_total` | Counter | **Накопленное** CPU-время (user+system) с **старта** процесса, сек. | `rate()` **от долей до единиц CPU** в прогоне (0.1–2 cores) для одного воркера — типично; **API** часто **ниже**, чем **worker** под пайплайном. | `rate` **0** длительно при **известной** нагрузке — залипшая метрика/не тот target. **Скачок** `rate` **без** ожидаемой работы — busy loop, лишние логи, неоптимальный hot path. |
| `process_open_fds` | Gauge | Число **открытых** файловых дескрипторов. | **Десятки–сотни** в спокойном API; **сотни** у воркера с пулом, кешами, дескрипторов на видео/моделях. | Ползёт **к лимиту** `process_max_fds` — риск **«Too many open files»**; **скачок** вверх — незакрытые файлы/сокеты. |
| `process_max_fds` | Gauge | **rlimit** на FD (мягкий лимит, как правило). | **1024** (часто по умолчанию в shell) или **1 048 576** и выше, если увеличен — смотрите `ulimit -n` на хосте. | Сравнение с `open_fds`: «запас» **< 10%** max — **опасно** под пик. |
| `process_resident_memory_bytes` | Gauge | **RSS** — реально в RAM (см. `top` `RES`), байты. | **API:** часто **сотни МБ**; **worker** с PyTorch/кадрами: **1–8+ ГБ** и **пики** при батчах — не удивительны. | **Пила** — рост **без** падения после run (возможная утечка); **OOM** в логах + рост **RSS** вровень. |
| `process_virtual_memory_bytes` | Gauge | **VIRT** — виртуальное адресное пространство, байты. | **Заметно больше RSS** (карты mmap, **CUDA/PyTorch** резервируют объём) — **часто норм**. | **Взрыв** VIRT при **стабильном** RSS — реже критично, чем RSS; **аномалия** — вместе с OOM. |
| `process_start_time_seconds` | Gauge | **Unix time** момента старта процесса. | Не «норма/плохо» — **справка**: `time() - value` = **возраст** процесса. | **Сбрасывается** при рестарте воркера; после деплоя — **ожидаемо** «молодой» PID. |

## 4. Системные (Prometheus UI)

| Метрика | Что значит | Примерный ориентир | Интерпретация |
|--------|------------|--------------------|----------------|
| `up{job=..., instance=...}` | Успех **последнего** scrape. | **1** = endpoint отвечал; **0** = недоступен, таймаут, 5xx, сеть. | **0** на worker — нет `DP_WORKER_METRICS_PORT` или порт/фаервол; **0** везде — Prometheus/сеть. |
| `scrape_duration_seconds` | Длительность HTTP-запроса **к /metrics** со стороны Prometheus. | Обычно **< 0.1–0.5 s** при нормальном объёме серий. | **Секунды+** — тяжёлый **экспорт** или **тысячи** timeseries; реже — медленный сетевой путь. |
| `scrape_samples_scraped` | Сколько **сэмплов** записано при последнем scrape. | **Сотни–пары тысяч** — типично для Python + default collectors + `dataprocessor_*`. | **Резкий** рост без смены кода — **дубли** лейблов, кардинально новый тип метрик, разумно проверить `label_replace`/дубли `job`. |

**Alertmanager** в E2E-override может быть **не** поднят — отсутствие алертов/роутеров **не** считается сбоем.

## 5. Что смотреть в дашборде (сводка)

| Цель | Куда смотреть | Нормальный рисунок |
|------|---------------|---------------------|
| Очередь, активные/упавшие run, время и ошибки пайплайна | `dataprocessor_*` + worker **UP** | `queue` не клинит, `active_runs` < лимит, `failures` **rate** ~0, `processing` p95 **согласован** с длительностью роликов. |
| «Здоровье» Python / утечки | `process_resident_memory_bytes`, `python_gc_objects_uncollectable_total` | `uncollectable` **0**; `RSS` **возврат** после run или **тренд** к плато, не вечный рост. |
| GC «шум» | `python_gc_collections_total` | Сравнение **версия к версии**; **резкие** отклонения при той же нагрузке — повод смотреть аллокации. |

**Эти метрики не заменяют** проверку артефактов (NPZ, manifest) — см. [README.md](README.md) (NPZ / батч).

## 6. Фильтр в Grafana / Prometheus

Только бизнес-метрики:

```text
{__name__=~"dataprocessor_.*"}
```

Только служебные `prometheus_client` (без `dataprocessor`):

```text
{__name__=~"python_.*|process_.*"}
```

---

*Версия `prometheus_client` в теории может чуть менять `python_*` / `process_*` — смотрите `# HELP` / `# TYPE` в `curl` на `/metrics`.*
