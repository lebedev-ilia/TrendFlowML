# Подготовка к большому прогону (60+ видео)

**Дата:** 2026-04-15  
**Цель:** до массового e2e на **≥60** видео закрыть три блока: **оптимизации**, **курация набора** (чтобы выборка была показательной для **всех** компонентов), **качественный вывод фичей и метрик**.  
**Опора:** [AUDIT_4_CRITERIA_AND_PLAN.md](AUDIT_4_CRITERIA_AND_PLAN.md) (наборы **A/B/C**, уровни L1–L3), [RUN_LOG.md](RUN_LOG.md), [AUDIT_V4_2_L2_CROSS_PROCESSORS_SUMMARY.md](AUDIT_V4_2_L2_CROSS_PROCESSORS_SUMMARY.md), engineering logs [components/audit_4_2/README.md](components/audit_4_2/README.md).

## Блокеры до «честного» 60+

Имеет смысл снять или явно учесть **до** финализации списка видео:

| Блокер | Суть | Действие перед60+ |
|--------|------|-------------------|
| **TextProcessor e2e** | На текущем наборе **A+B** только **2/5** успешных `text_processor` | Починить корневую ошибку пайплайна / пересобрать `result_store`; иначе текстовые `tp_*` и зависимые визуальные ветки будут неполными |
| **Visual `micro_emotion`** | **blocked** на L2 (PCA на одном B-run) | Отдельно включить в набор видео с достаточной размерностью/качеством входа под PCA или ослабить/починить ветку — иначе метрики модуля на 60+ будут с дырой |
| **Git / репро** | Во многих строках `RUN_LOG` **git commit: TODO** | Зафиксировать commit + при необходимости `config_hash` для сравнения «до/после» 60+ |

## 1. Оптимизации (код и инфраструктура)

**Принцип:** не раздувать объём изменений перед60+; приоритет — **измеримые** узкие места и **безопасные** правки (без смены семантики NPZ без обновления схем/отчётов).

### 1.1 Где уже есть зацепки

- **Audio / часть Visual / Text:** журналы **Audit4.2** — [`components/audit_4_2/README.md`](components/audit_4_2/README.md) (профилирование, env-gated этапы, заметки «before/after»).
- **Сводка по подсистемам L2:** [AUDIT_V4_2_L2_CROSS_PROCESSORS_SUMMARY.md](AUDIT_V4_2_L2_CROSS_PROCESSORS_SUMMARY.md).

### 1.2 Чек-лист по компонентам (повторяемый)

Для каждого экстрактора / модуля, который попадёт в 60+:

1. Есть ли **engineering log 4.2** и зафиксированы ли **горячие** функции / стадии (`meta.stage_timings_ms` где внедрено)?
2. Включён ли **контролируемый** профиль (одинаковые env-флаги на стенде60+)?
3. Есть ли **дублирующиеся** тяжёлые вызовы (повторное чтение файла, повторный inference на тех же кадрах) — кандидат на кеш / ранний выход?
4. Меняется ли **размер** NPZ или состав ключей — если да, нужны обновление **schema / validate_npz** и строка в `RUN_LOG`.

Итог блока 1: короткая **таблица** «компонент → статус профиля → решение (оптимизировать / отложить / не трогать)»; критичные оптимизации — до60+, остальное — отдельной итерацией после первого среза метрик.

## 2. Курация 60+ видео: какие сценарии кому нужны

**Цель:** не «случайные 60», а покрытие **измеримых** осей, о которых уже сказано в отчётах L1/L2 и в плане §3 (**B** = diversity, **C** = edge).

### 2.1 Сквозные оси (минимум для всех процессоров)

Закрепить в таблице отбора (столбцы: `video_id`, длительность, жанр/тип контента, теги):

| Ось | Зачем |
|-----|--------|
| Длительность | короткое / среднее / длинное — влияет на сегменты, агрегаты, стоимость |
| Аудио: речь / музыка / тишина / смесь | AudioProcessor + `high_level_semantic`, VAD-зависимые ветки |
| Плотность текста: title/desc/comments/transcript/OCR | TextProcessor + `text_scoring`, корпусные top‑K |
| Кадры / движение / лица / сцены | Visual core + modules (face, flow, shot, scene, …) |
| **N** кадров Segmenter | согласование времени и размерностей T×F |
| Edge: пустые ветки, битый вход, near-empty | набор **C** плана v4 |

### 2.2 Откуда брать требования «по компонентам»

Не дублировать здесь все 66 строк; использовать как **источник требований**:

- **TextProcessor:** нумерованный список «Следующие шаги» в [AUDIT_V4_L1_TEXT_PROCESSOR_SUMMARY.md](AUDIT_V4_L1_TEXT_PROCESSOR_SUMMARY.md) (пустые хештеги, ≥2 чанка транскрипта, diar+ASR для `speaker_turn_embeddings_aggregator`, FAISS vs numpy, …).
- **AudioProcessor:** темы и таблица в [AUDIT_V4_L1_AUDIO_PROCESSOR_SUMMARY.md](AUDIT_V4_L1_AUDIO_PROCESSOR_SUMMARY.md); edge — тишина, короткий клип, сегментные режимы.
- **VisualProcessor:** риски и оценки в [VISUAL_PROCESSOR_AUDIT_V4_SUMMARY.md](VISUAL_PROCESSOR_AUDIT_V4_SUMMARY.md); явно: несколько клипов для **`action_recognition`**, данные под **`micro_emotion`**, лица для face-цепочки, OCR-плотность для `text_scoring`/OCR.
- **Кросс-L1:** [AUDIT_V4_L1_CROSS_PROCESSORS_SUMMARY.md](AUDIT_V4_L1_CROSS_PROCESSORS_SUMMARY.md) — § «До набора B» и сквозные темы.

**Практика:** завести матрицу **компонент × обязательный тег сценария** (хотя бы в spreadsheet или в этом репозитории отдельным CSV позже). Минимум **≥1** видео на каждый критичный тег; перекрытие допустимо (одно видео закрывает несколько тегов).

### 2.3 Размер выборки 60+

- **60+** — хороший минимум для **перцентилей и хвостов** по основным tabular/dense полям (см. §4 плана v4).
- Разбить явно: например **~50** diversity (**B**) + **~10** edge (**C**) или пропорция пориску; зафиксировать в `RUN_LOG` после утверждения списка `video_id`.

## 3. Качественный вывод фичей и метрик

### 3.1 Что уже есть

- По каждому компоненту с L2: **`storage/audit_v4/<name>_l2/*_audit_v4_stats.json`** (и figures у части Audio).
- Контрактные проверки: **validate_npz** / схемы (см. отчёты и [AUDIT_4_CRITERIA_AND_PLAN.md](AUDIT_4_CRITERIA_AND_PLAN.md) §4).

### 3.2 Что добавить для 60+

1. **Единый manifest прогона:** список путей `result_store` + `meta.status` + версия кода; тот же список — вход для всех L2-скриптов (или обёртки), чтобы не плодить расхождения.
2. **Агрегация метрик:** поверх множества JSON — сводная таблица: компонент, доля NaN по ключевым полям, диапазоны, число OK run, ссылки на выбросы (`video_id`).
3. **Витрина для Models:** какие ключи **model_facing** (по контрактам Models) — отдельный срез или документ «что уходит в encoder» на 60+.
4. **Регрессия:** после первого 60+ зафиксировать **§4.8**-подобные сигнатуры хотя бы для критичной цепочки (опционально пилот).

### 3.3 Критерий готовности блока 3

- На всех **60+** путях для обязательных компонентов: либо **OK NPZ**, либо явный **`empty`/`empty_reason`** (без молчаливого битого tabular).
- Сводный отчёт (Markdown или dashboard-spec) обновляется **одной командой** или одним скриптом из одного manifest.
