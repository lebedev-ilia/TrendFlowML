# Дизайн: appearance-embedding трекер (core_object_detections)

Статус: **спека к реализации** (2026-07-03). Автор: Claude (дизайн) → исполнитель: владелец/Cursor.
Контекст: в Audit v3 трекинг убрали из `core_object_detections` (суррогатные per-detection
`track_id`), из-за чего downstream (`action_recognition`) фрагментируется на длинном контенте.
Владелец выбрал вариант: **свой трекер на эмбеддингах боксов** + сверка боксов между кадрами.
Этот документ описывает алгоритм, контракт, параметры и критерии приёмки.

---

## 0. Идея и почему именно эмбеддинги

Кадры к компонентам приходят **разреженно** (union-sampling Segmenter'а), поэтому классический
motion/IoU-трекинг (ByteTrack/SORT) плохо работает: между двумя сэмплированными кадрами объект
успевает уйти далеко, IoU=0, трек рвётся. **Appearance-эмбеддинг устойчив к разрывам**: даже
если бокс сместился, косинусная близость эмбеддингов crop'ов остаётся высокой → ассоциация
переживает пропуски и повторную детекцию. Это и есть основной довод в пользу выбранного подхода;
motion остаётся **вторичным гейтом** (защита от «телепортов» и склейки разных людей рядом).

---

## 1. Где живёт

Отдельная **стадия внутри `core_object_detections`** (после детекции, до сериализации), под
флагом конфига `tracking.enabled`. Плюсы: единый source-of-truth (`detections.npz`), не плодим
артефакт, downstream читает `track_ids` из того же файла. Схема поднимается
`core_object_detections_npz_v2 → _v3` (добавляется одно поле, см. §4).

Альтернатива (не выбрана): отдельный модуль `core_object_tracking`, читающий `detections.npz`.
Отклонена ради простоты плюмбинга; если позже трекинг понадобится и другим потребителям с
другим бюджетом кадров — вынести, контракт §4 сохранится.

---

## 2. Эмбеддер бокса

Класс-специфичный: в MVP трекаем **только `person` (class_id 0)** — для него appearance-ReID
максимально осмысленен. Остальные классы (logo/text/…) при `track_ids` получают `-1` (или
IoU-fallback, см. §6) — их динамика downstream не нужна.

Кандидаты эмбеддера (по убыванию рекомендации):

| Вариант | Эмбеддинг | Плюсы | Минусы |
|---|---|---|---|
| **(A) OSNet (torchreid, market1501)** — рекоменд. | 512-d | заточен под person-ReID, ~2.2M params, быстрый, разделяет людей по одежде/облику | +1 вес в манифест |
| (B) CLIP image (уже в Triton) | 512/768-d | нулевая новая инфра | семантический, а не identity — хуже разделяет двух похожих людей |
| (C) YOLO-фичи (penultimate) | — | бесплатно (уже считаем детекцию) | не обучены на ReID, слабая разделимость |

**Решение:** (A) OSNet как основной; (B) CLIP — конфигурируемый fallback, если не хотим тянуть
новый вес на первый прогон. Эмбеддер выбирается `tracking.embedder: osnet|clip`.

Пайплайн эмбеддинга (на кадр, батч по всем валидным person-боксам):
1. crop по `boxes` (пиксельные) c паддингом `pad_frac=0.1`, клип к границам кадра;
2. resize к входу эмбеддера (OSNet: 256×128, CLIP: 224×224);
3. forward → L2-норм эмбеддинг `e ∈ R^d`.

Батчим все боксы кадра (и по возможности все кадры видео) одним прогоном — стоимость мала
относительно самой детекции.

## 3. Алгоритм ассоциации (online, покадрово)

Идём по сэмплированным кадрам в порядке `frame_indices` (возрастание). Состояние трека:
`{track_id, last_frame_idx, gallery: deque[emb] (cap G), repr_emb (EMA), last_box, age, hits}`.

Для кадра `t` с детекциями `D_t` (валидные person-боксы, эмбеддинги `e_i`):

1. **Матрица стоимости** `Cost[track, det]`:
   ```
   app  = 1 - cos(repr_emb[track], e_det)          # аппearance, ∈ [0,2]
   mot  = center_dist(last_box[track], box_det) / diag_norm   # нормир. смещение центров
   Cost = w_app * app + w_mot * mot
   ```
   с `w_app=0.7`, `w_mot=0.3` (дефолт).
2. **Гейтинг** (запрет матча) — ставим `Cost=+inf`, если:
   - `cos < sim_gate` (дефолт `0.5`) — разные люди; **или**
   - `mot > max_dist_gate * gap_scale`, где `gap_scale = 1 + (t - last_frame)/fps_sample`
     (чем больше разрыв кадров — тем шире допустимое смещение).
3. **Назначение**: `scipy.optimize.linear_sum_assignment` по гейтованной матрице; матчи с
   `Cost=inf` отбрасываем.
4. **Обновление**: matched-трек ← `gallery.append(e)`, `repr_emb = ema(repr_emb, e, α=0.5)`,
   `last_box/last_frame` обновить, `age=0`, `hits+=1`.
5. **Рождение**: unmatched-детекции — сначала пробуем **re-ID из «потерянных»** (см. §3.1);
   если нет — новый `track_id`.
6. **Старение**: unmatched-треки `age+=1`; при `age > max_age_steps` (дефолт `3` сэмпл-шага)
   → в пул «потерянных» (для re-ID); удаляем из пула после `max_lost_steps` (дефолт `10`).

### 3.1 Re-identification (защита от разрывов)
Перед созданием нового трека сверяем эмбеддинг детекции с `repr_emb` потерянных треков; при
`cos ≥ reid_sim_gate` (дефолт `0.6`, строже основного) — **воскрешаем** старый `track_id`
вместо нового. Это ключ к тому, чтобы человек, ушедший из кадра и вернувшийся, получил тот же id.

### 3.2 Пост-фильтр
Треки короче `min_track_len` (дефолт `2` детекции) помечаем как шум: их `track_ids` можно
оставить (downstream сам решит по длине) — не удаляем детекции, только маркируем в мете
`tracks_json[id].len`.

## 4. Контракт выхода (schema v3)

Добавляется **одно поле** в `detections.npz`:

| key | required | tier | dtype | shape | описание |
|---|---|---|---|---|---|
| `track_ids` | v3 | `model_facing` | `int32` | `(N, M)` | persistent id per detection-slot; `-1` = невалидный слот / не-трекаемый класс |

Плюс в `meta_json`:
```jsonc
"tracking": {
  "enabled": true, "embedder": "osnet", "classes": ["person"],
  "params": { "sim_gate":0.5, "w_app":0.7, "w_mot":0.3, "max_age_steps":3,
              "max_lost_steps":10, "reid_sim_gate":0.6, "gallery_size":16, "min_track_len":2 },
  "num_tracks": 12, "model_signature": "osnet_x1_0_market1501@<sha>"
},
"tracks_json": { "<id>": {"len": 7, "first_frame": 12, "last_frame": 340, "class": "person"} }
```

Обратная совместимость: при `tracking.enabled=false` поле `track_ids` не пишется, схема остаётся
v2 (валидатор допускает v2 ∨ v3). `allow_extra_keys` при v3 — добавить `track_ids` в whitelist.

Опционально (не в MVP): companion `track_embeddings (num_tracks, d)` — репрезентативный эмбеддинг
трека для downstream переиспользования (напр. person-consistency в behavioral). Пока downstream
(`action_recognition`) нужен только `track_ids`, не тянем.

## 5. Параметры (config `core_object_detections.tracking`)
```yaml
tracking:
  enabled: true
  embedder: osnet          # osnet | clip
  classes: [person]
  sim_gate: 0.5            # мин. cos для матча
  reid_sim_gate: 0.6       # мин. cos для воскрешения потерянного трека
  w_app: 0.7               # вес appearance в стоимости
  w_mot: 0.3               # вес motion
  max_dist_gate: 0.5       # макс. нормир. смещение центра (×gap_scale)
  max_age_steps: 3         # сколько сэмпл-кадров трек «жив» без матча
  max_lost_steps: 10       # сколько держим в пуле re-ID
  gallery_size: 16
  min_track_len: 2
  pad_frac: 0.1
```

---

## 6. Не-person классы
MVP: `track_ids=-1` для всех не-person. Если позже понадобятся треки logo/product — включить
**IoU-fallback** ветку (без эмбеддера): та же матрица, но `Cost=1-IoU`, `w_app=0`. Флаг
`tracking.classes` расширяется; person всегда через appearance.

---

## 7. Детерминизм и стоимость (200k)
- **Детерминизм:** эмбеддинги fp32 детерминированы; `linear_sum_assignment` детерминирован →
  golden-повтор одного видео должен давать идентичные `track_ids`. Добавить в приёмку.
- **Стоимость:** 1 forward эмбеддера на валидный бокс. Кап `M` (слотов/кадр) уже есть; добавить
  `tracking.max_boxes_per_frame` при желании. Ассоциация — O(Σ tracks×dets) на кадр, ничтожна.
  Общий overhead ≈ (число person-боксов)×(лёгкий эмбеддер) — малая доля от YOLO+инференса.

---

## 8. Критерии приёмки (нет GT-меток треков → прокси)
Валидируем без ручной разметки, по прокси-метрикам на реальных видео (RUN_SPEC v2):

1. **Когерентность vs длина:** на 4:35 и 8:00 роликах число треков **не** взрывается
   (было 205/199 одно-клиповых); `mean_track_len` заметно > 1; `tracks_with_multi_clips > 0`.
2. **Разделимость id (appearance):** внутри трека средний `cos` эмбеддингов **высокий**,
   между разными треками — **низкий** (intra ≫ inter). Посчитать и приложить гистограмму.
3. **Re-ID работает:** человек, вышедший и вернувшийся, сохраняет id (проверить на ролике с
   уходом из кадра; глазами по рендеру боксов с id).
4. **Контроль без людей:** `videoplayback.mp4` (2:47, без людей) → `num_tracks≈0`,
   `action_recognition` = валидный `empty (no_person_detections)`.
5. **Детерминизм:** повтор одного видео → идентичные `track_ids` (golden-compare).
6. **Стоимость:** overhead трекинга < X% времени стадии детекции (зафиксировать число).

## 9. Зависимость с Segmenter (важно)
Трекер сам по себе работает на любом наборе кадров, но **для action_recognition** нужны
непрерывные окна (см. `action_recognition` schema v3 спеку и RUN_SPEC v2): Segmenter должен
давать компоненту **плотные окна ≥ clip_len кадров**, иначе даже с идеальными треками SlowFast
получит по 1 клипу на трек. Трекинг и оконный сэмплинг — **две независимые правки**, обе нужны.
```
Segmenter (плотные окна) ─┐
                          ├─► core_object_detections (+ appearance-tracker → track_ids)
                          │            │
                          │            ▼
                          └─► action_recognition (per-clip stream v3, привязка clip→track_id)
```
