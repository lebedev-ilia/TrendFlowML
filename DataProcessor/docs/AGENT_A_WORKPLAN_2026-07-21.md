## Фаза 4 — Инфраструктурная оптимизация

Вопросы владельца: сколько параллельных потоков на одной машине (напр. та же RTX 2000 Ada 16GB)? Если
4 машины — как координировать? Может, меньше машин, но мощнее видеокарты?

- [ ] **Эмпирический stress-test параллелизма** — не считай теоретически по памяти GPU. На одной машине
      (можно на текущей конфигурации подешевле, RTX 2000 Ada) прогони матрицу: 1 / 2 / 3 / 4 параллельных
      воркера того же пайплайна одновременно на разных видео, замерь: деградацию wall-time на видео,
      пиковую GPU-память суммарно, CPU/RAM contention, диск I/O. Текущий пик 3575/16384 MiB — это
      **только на 7-компонентном подмножестве**; при добавлении всех 29 VP + AP + TP резидентная память
      Triton-моделей вырастет (больше моделей = больше VRAM даже в простое) — переизмерь на полном наборе
      компонентов, не экстраполируй с 9 на 29+21+22.
- [ ] Постой decision-фреймворк, а не готовое решение (у тебя ещё нет эмпирики): для каждой карты из
      списка владельца (RTX 2000 Ada 16GB, RTX 4090 24GB, RTX PRO 4000 24GB, RTX 4000 Ada 20GB,
      RTX A4000 16GB, RTX A4500 20GB, RTX A5000 24GB, L4 24GB, A40 48GB, RTX 3090 24GB) — собери
      $/час на RunPod, и посчитай метрику **$ за 1000 обработанных видео** = (стоимость часа) / (видео в
      час на этой карте с оптимальным числом параллельных потоков). Не любой ценой минимизировать $/час
      пода — минимизировать $/1000 видео с учётом реальной пропускной способности.
- [ ] Похоже (по факту использования 22% VRAM на 7-компонентном тесте), что для VP память может не быть
      узким местом — узкое место может быть CPU/RAM (много CPU-only компонентов: 15 из 29 VP, плюс почти
      весь TextProcessor CPU-only). Значит "больше видеокарта" не обязательно = "быстрее в 2 раза" —
      возможно оптимальнее взять карту подешевле (RTX 2000 Ada/A4000/L4) но с быстрым CPU/много RAM и
      несколько параллельных процессов. Проверь эту гипотезу эмпирически, не принимай как готовый вывод.
- [ ] Если понадобятся несколько машин — синка/координация: посмотри уже готовый паттерн шардирования
      из Fetcher (`stable_shard_slot` в `Fetcher/fetcher/dataset_collector/hf_coordination.py` —
      `hash(video_key) % N`, HF-based claims/done) — тот же принцип подойдёт для распределения корпуса по
      машинам DataProcessor без коллизий. Не изобретай велосипед, адаптируй.
- [ ] Учти прямое указание владельца: **для текущей аналитической/оптимизационной работы дорогие поды не
      нужны** — этот эмпирический stress-test делай на дешёвом/уже доступном железе, коротким прогоном
      (несколько видео × несколько уровней параллелизма достаточно для кривой деградации, не нужен полный
      корпус). Полноценный многомашинный тест — отдельная, следующая фаза (см. Фазу 7), не делай её
      преждевременно дорогой.

**Deliverable**: `DataProcessor/docs/corpus_run_report/INFRA_SCALING_PLAN.md` — таблица $/1000 видео по
GPU-опциям (с пометкой "оценка" vs "измерено"), рекомендуемое число параллельных процессов на машину (с

## Требование по всем фазам — прогноз затрат места (диска)

Действующая цифра: **~151 MB/видео** на 7 VP-компонентах (2400 NPZ / 300 видео = 8 файлов/видео, 54GB
итого). Это НЕ линейно масштабируется на 29+21+22 компонентов — некоторые (embeddings, spectrograms)
тяжелее одиночных скаляров/векторов.

- [ ] На каждую Фазу (5, 6) — добавляй оценку "МБ добавляется на видео" по новым компонентам (хотя бы
      грубую по типу выхода: скаляры/векторы малого размера vs. кадры/эмбеддинги/спектрограммы).
- [ ] Перед любым следующим тестом явно считай: **N видео × MB/видео (текущий набор компонентов) = итоговый
      объём**, и сверяй с доступным Network Volume (сейчас расширен владельцем до 120GB, per портфолио §3.1).
      Владелец предпочитает **выгружать результаты в HF, а не бесконечно расширять volume** — планируй
      промежуточные выгрузки/очистку по той же логике, что уже реализована в Fetcher
      (`Fetcher/fetcher/dataset_collector/hf_queues.py` — загрузка в HF + удаление локальной копии после
      подтверждённого коммита) — не изобретай новый паттерн, адаптируй существующий.
- [ ] Явно посчитай сценарий следующего теста владельца (см. Фазу 7 ниже) — 50 → 500 → 1000 видео на
      **всех** компонентах (VP 29 + AP 21 + TP 22) на 2-3 машинах: сколько это будет ГБ суммарно на каждом
      шаге, влезет ли в volume/сколько нужно выгружать в HF по ходу.

## Общие правила работы (коммуникация, git, бюджет)

- **VK**: коротко и по делу, отчёты по завершении каждой Фазы (не каждого чек-бокса). Вопросы владельцу —
  только реально важные, редко (напр. "нашёл спорную развилку по приоритету", не "как назвать переменную").
- **Git**: работай в `DataProcessor/**` (код компонентов, `DataProcessor/docs/**` для своих отчётов). НЕ
  трогай `Models/**` — это зона Agent B. Коммить часто, маленькими логическими шагами (легче ревьюить и
  откатывать), `git pull --rebase` перед каждым push (второй агент коммитит параллельно — избегай
  конфликтов, при рассинхроне — сообщи владельцу, не форсь push).
- **Бюджет/поды**: дорогие поды сейчас не нужны — работа в основном над уже собранными данными
  (CSV/JSON/логи) и кодом. Поды поднимай кратко и дёшево только для точечных экспериментов (профилирование,
  stress-test параллелизма, smoke-тесты новых компонентов), гаси сразу после. Текущие гейты
  (`max_pod_hourly_usd=0.30`, `max_pod_hourly_usd_short=0.60` в `automation/runner/settings.py`) уже
  подходят — не превышай их без явного согласования.
- **Качество превыше скорости отчёта**: если для честного вывода по какому-то пункту не хватает данных —
  так и напиши в отчёте ("недостаточно данных, нужно X"), не дописывай правдоподобную догадку как факт.
