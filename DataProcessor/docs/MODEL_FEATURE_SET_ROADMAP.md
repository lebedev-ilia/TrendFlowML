# Roadmap: Model Feature Set (основной фокус)

Дата фиксации: 2026-04-25  
Статус: рабочий документ ближайшего этапа

---

## 0) Контекст и цель этапа

Проект вошёл в ключевую стадию: не просто «собрать фичи», а подтвердить их **фактическую ценность** для задачи предсказания популярности.

Главная цель этапа:

- сформировать качественный **"Модельный набор фичей"** на разнообразии видео;
- не теряя полезные сигналы, разделить фичи на:
  - **Model** (для обучения и предсказания),
  - **Analytical** (для человеко-аналитики, мониторинга, интерпретации);
- после стабилизации набора — перейти к обучению.

## 1) Твоя зафиксированная позиция (согласовано)

Ниже зафиксировано то, что ты описал как ближайший roadmap.

1. Масштабные тесты на разнообразном наборе видео, включая большой прогон на 70 видео (см. `DataProcessor/docs/audit_v4/CHECKLIST_LARGE_VIDEO_BATCH_60PLUS.md`).
2. В процессе прогонов смотреть выходы алгоритмов и таблиц и оценивать каждую фичу по категориям:
  - **Model**: полезна для модели и предсказания популярности;
  - **Analytical**: может шуметь в модели, но полезна для ручной аналитики.
3. Не обязательно удалять фичи: можно переносить в "Analytical"; приоритет — собрать сильный "Model" набор.
4. По ходу итераций допустимо:
  - менять/дописывать алгоритмы,
  - внедрять новые модели,
  - добавлять новые компоненты.
5. Параллельно с QA:
  - чистка кода,
  - контроль времени выполнения,
  - улучшение архитектуры и sampling policy,
  - оптимизация без потери качества.

## 2) Расширение roadmap (рекомендации)

### 2.1 Управление фичами как продуктом

Ввести единый реестр статусов фич:

- `candidate` — новая/непроверенная;
- `model_trial` — в A/B/ablation для модели;
- `model_core` — подтверждённая для финального набора;
- `analytical` — полезна человеку, но не в train matrix;
- `deprecated` — временно/постоянно выключена из активного использования.

Это уменьшает хаос при постоянных доработках.

### 2.2 Явные критерии перевода Model ↔ Analytical

Переводить в `model_core`, если одновременно:

- стабильно проходит QA (nan/oor/drift в норме),
- даёт прирост на offline-метрике в ablation,
- не ухудшает робастность и вычислительную стоимость непропорционально.

Переводить в `analytical`, если:

- информативна для человека/диагностики,
- но не даёт прироста модели или даёт нестабильный/шумный вклад.

### 2.3 Две скорости разработки (чтобы не тормозить прогресс)

- **Fast lane (ежедневно):** небольшие изменения, короткие batch, быстрый QA.
- **Stable lane (еженедельно):** зафиксированный код/конфиг, полный weekly QA, сравнение с baseline/golden.

Это снижает риск «сломали всё ради одной гипотезы».

### 2.4 Обязательный контроль leakage до обучения

Перед любым train run:

- обновить `wide_batch_feature_manifest`,
- пересмотреть denylist leakage,
- собрать train matrix только после ручной валидации подозрительных колонок.

### 2.5 Управление стоимостью

Для каждого компонента регулярно фиксировать:

- вклад в качество (model gain),
- вклад в latency/cost.

Иметь право временно выключать дорогие низкополезные компоненты на пилотах.

## 3) Операционный цикл (повторяется итеративно)

1. Прогон batch на новом срезе видео (включая разнообразие).
2. Weekly QA (`feature_qa_pipeline`).
3. Обновление реестра инцидентов.
4. Triage shortlist + классификация фич (Model/Analytical).
5. Точечные правки алгоритмов/компонентов.
6. Повторный прогон и сравнение с baseline/golden.
7. Периодический freeze кандидатов в `model_core`.

---

## 4) Предлагаемые KPI этапа

### 4.1 Качество данных/фич

- доля `model_core` фич с `severity=low`,
- drift high-инциденты на batch,
- стабильность golden compare (доля mismatch).

### 4.2 Модельная ценность

- прирост к baseline по целевой offline-метрике,
- вклад компонент в ablation (Δ метрики),
- устойчивость прироста на разных срезах видео.

### 4.3 Производительность

- wall-time batch,
- p95/p99 времени по тяжёлым компонентам,
- ресурсная стоимость единицы data point.

---

## 5) Definition of Done для этого этапа

Этап считается завершённым, когда:

1. Сформирован и зафиксирован список `model_core` фич (версия набора).
2. Для `model_core` есть:
  - стабильный QA-профиль,
  - подтверждение полезности на ablation.
3. Остальные фичи классифицированы как `analytical`/`deprecated`/`candidate`.
4. Подготовлена стабильная train matrix (leakage-safe, time-split ready).
5. Можно запускать обучение как следующий отдельный этап.

---

## 6) Практические следующие шаги (сразу)

1. На базе текущего weekly QA:
  - провести triage `feature_shortlist`,
  - проставить первые статусы фич (Model/Analytical/Candidate).
2. Подготовить/актуализировать 70-video реестр и закрыть пункты чеклиста 60+.
3. Провести 1-2 крупных итерации:
  - правки алгоритмов,
  - повторный weekly QA,
  - сравнение drift/golden до/после.
4. После стабилизации — запустить первый model pilot на сборке `model_trial`.

---

## 7) Принцип принятия решений

Ключевой принцип этапа:

> Не максимизируем число фич.  
> Максимизируем качество и устойчивость **Model Feature Set** при контролируемой стоимости.
