# Критерии аудита baseline компонентов

**Версия**: v1.0  
**Дата**: 2025-01-XX  
**Применимо к**: Все baseline компоненты (Visual modules, Audio extractors, Core providers)

---

## Обзор

Этот документ определяет **обязательные критерии** для аудита каждого baseline компонента. Аудит должен подтвердить, что компонент:
1. Соответствует архитектурным требованиям baseline
2. Имеет измеренную производительность (время, память)
3. Имеет проверяемое качество выхода (для ML/аналитики)

---

## 1. Архитектурное соответствие

### 1.1 Наследование и интерфейсы

**Примечание**: Требования различаются для разных типов компонентов (Visual modules, Core providers, Audio extractors).

**Для Visual modules**:
- [ ] Компонент наследуется от `BaseModule` (`VisualProcessor/modules/base_module.py`)
- [ ] Реализован метод `process(frame_manager, frame_indices, config)` с правильной сигнатурой
- [ ] Реализован метод `required_dependencies()` → `List[str]` (возвращает список зависимостей, может быть пустым)
- [ ] Реализован метод `get_models_used(config, metadata)` → `List[Dict]` (если используются модели)

**Для Core providers**:
- [ ] Компонент реализует CLI интерфейс (`argparse`) или вызывается через оркестратор
- [ ] Читает `frame_indices` из `metadata.json[component_name.frame_indices]`
- [ ] Сохраняет NPZ с полным meta (см. раздел 1.2)

**Для Audio extractors**:
- [ ] Компонент реализует CLI интерфейс или вызывается через оркестратор
- [ ] Сохраняет NPZ в per-run структуре с полным meta
- [ ] Использует стандартные `empty_reason` значения (если применимо)

**Evidence**: Путь к файлу класса, номера строк методов, примеры вызова

### 1.2 Контракты входа/выхода

- [ ] Читает `frame_indices` только из metadata (не генерирует семплинг сам)
- [ ] Использует `union_timestamps_sec` как source-of-truth для временной оси
- [ ] Соблюдает RGB contract (для Visual компонентов): `FrameManager.get()` возвращает RGB uint8
- [ ] Сохраняет NPZ с полным meta, включая **все обязательные поля**:
  - **Базовые**: `producer`, `producer_version`, `schema_version`, `created_at`
  - **Run identity**: `platform_id`, `video_id`, `run_id`, `config_hash`, `sampling_policy_version`
  - **Версия пайплайна**: `dataprocessor_version` (обязательно, baseline допускает `"unknown"`)
  - **Статус**: `status` (`ok`/`empty`/`error`), `empty_reason` (если `status="empty"`)
  - **Модели** (если используются): `models_used[]`, `model_signature`
- [ ] Для per-frame / time-series артефактов (где есть `frame_indices`) сохраняет:
  - `times_s (float32)` строго из `union_timestamps_sec[frame_indices]` (no-fallback)
- [ ] Использует NaN для missing значений в численных массивах (не нули)
- [ ] Использует маски `*_present` для optional features (если применимо)
- [ ] `schema_version` соответствует каноническим значениям (см. `ARTIFACTS_AND_SCHEMAS.md`)
- [ ] `frame_indices` (если присутствует) отсортированы, уникальны, dtype int32

**Evidence**: Фрагменты кода, примеры NPZ meta, валидация через `artifact_validator.validate_npz()`

### 1.3 No-fallback policy

- [ ] При отсутствии обязательных зависимостей → `raise RuntimeError` (не fallback, не default values)
- [ ] При отсутствии `frame_indices` в metadata → `raise RuntimeError` (не генерирует семплинг сам)
- [ ] При отсутствии обязательных core provider artifacts → `raise RuntimeError` (не fallback на пустые данные)
- [ ] При невалидных входных данных → `raise RuntimeError` с понятным сообщением
- [ ] При отсутствии run identity keys в metadata → `raise RuntimeError` (platform_id, video_id, run_id, config_hash, sampling_policy_version)

**Исключение**: Valid empty outputs (см. раздел 1.6)

**Evidence**: Фрагменты кода с проверками и raise, отсутствие fallback логики

### 1.4 Per-run storage

- [ ] Сохраняет артефакты в `result_store/<platform_id>/<video_id>/<run_id>/<component_name>/`
- [ ] Имена файлов артефактов:
  - **Рекомендуется** фиксированное имя (например, `embeddings.npz` / `flow.npz` / `depth.npz`), так как уникальность обеспечивается `run_id` в пути
  - **Опционально** timestamped имена файлов, если компонент создаёт несколько артефактов в одном `run_id` (несколько профилей/вариантов)
- [ ] Manifest обновляется оркестратором (компонент не пишет manifest напрямую)
- [ ] Атомарное сохранение NPZ (через временный файл → `os.replace()`)

**Evidence**: Путь к сохранённому артефакту, структура директорий, код сохранения

### 1.5 Валидация артефактов

- [ ] Артефакт проходит валидацию через `artifact_validator.validate_npz()`
- [ ] Все обязательные meta поля присутствуют (см. `artifact_validator.py` для списка)
- [ ] `frame_indices` (если есть) валидны: отсортированы, уникальны, правильный dtype (int32)
- [ ] `schema_version` соответствует каноническим значениям из `ARTIFACTS_AND_SCHEMAS.md`

**Evidence**: Результат валидации, отсутствие ошибок в `ValidationIssue[]`

### 1.6 Valid empty outputs (валидная пустота)

- [ ] Если данных "нет" (нет аудио, нет лиц, нет текста) → компонент пишет NPZ со `status="empty"`
- [ ] `empty_reason` заполнен стандартным значением из каноничного словаря (см. `ARTIFACTS_AND_SCHEMAS.md`)
- [ ] Численные массивы содержат `NaN` (не нули, не пустые массивы)
- [ ] Есть булевые маски присутствия (`*_present`, `has_*`) где применимо
- [ ] `empty_reason` обязателен если `status="empty"`, иначе должен быть `null`

**Стандартные empty_reason значения**:
- `audio_missing_or_extract_failed`
- `no_faces_in_video`
- `no_text_available`
- `ocr_disabled_by_policy`
- `comments_missing_or_disabled`
- `video_too_short`
- `video_too_long`
- `dependency_missing`

**Evidence**: Примеры NPZ с `status="empty"`, код обработки пустых случаев

### 1.7 Документация требований к выборке (sampling requirements)

- [ ] В README компонента есть раздел **"Sampling / units-of-processing requirements"** (или аналогичный)
- [ ] Чётко описана стратегия выборки относительно всего видео:
  - Равномерная выборка (каждый N-й кадр)
  - Адаптивная выборка (с коэффициентом относительно длительности видео)
  - Умная стратегия (event-driven, stratified per-shot, coverage начало/середина/конец)
  - Минимальное/целевое/максимальное количество кадров
- [ ] Если используется **универсальная нелинейная кривая** (Segmenter-owned), в README фиксируются параметры кривой:
  - `k` (коэффициент замедления роста, \(0<k\le 1\); ближе к 1 → меньше замедление → больше units на длинных видео)
  - `min_units`, `max_units`
  - `linear_until_sec`, `cap_duration_sec`
  - ссылка на `docs/contracts/SEGMENTER_CONTRACT.md` как источник истины
- [ ] Указаны требования для разных диапазонов длительности видео (например, 120-72000 кадров)
- [ ] Описаны ограничения (min gap, max gap, coverage requirements)
- [ ] Указано, что Segmenter является единственным владельцем sampling (компонент не генерирует семплинг сам)

**Примеры описания**:
- "Равномерная выборка: каждый 3-й кадр, минимум 50 кадров, максимум 2000 кадров"
- "Адаптивная выборка: N = min(2000, max(50, total_frames / 10)), равномерно распределённые по времени"
- "Умная стратегия: coverage начало/середина/конец + event-driven по cuts, target=100-500 кадров"

**Evidence**: Раздел в README с описанием sampling requirements

### 1.8 Документация используемых моделей

- [ ] В README компонента есть раздел **"Models"** (или "Используемые модели")
- [ ] Все модели, используемые непосредственно в коде компонента, перечислены и разделены на:
  - **GPU модели** (с указанием устройства: `cuda:0`, `cuda:1`, etc.)
  - **CPU модели** (с указанием `cpu`)
- [ ] Для каждой GPU модели указано:
  - Находится ли модель в Triton (`DataProcessor/triton/models/<model_name>/`)
  - Если находиться, то какие есть ветки, например 256, 512, 768 и подобные. + какие есть сложности, например large, medium и тд. + есть ли препроцесс. 
  - Если в Triton: имя модели в Triton, версия, spec name (если используется ModelManager)
  - Если не в Triton: путь к локальным весам, способ загрузки
- [ ] Для каждой модели указаны:
  - Название модели
  - Версия (если применимо)
  - Runtime (`triton`/`inprocess`)
  - Engine (`torch`/`onnx`/`tensorrt`/`mediapipe`/etc.)
  - Precision (`fp16`/`fp32`)
- [ ] Указаны зависимости между моделями (если одна модель используется для инициализации другой)
- [ ] Консистентность ModelManager (`dp_models`) (если используется):
  - spec `precision/engine/runtime` соответствует фактическим артефактам (например, dtype весов ONNX / engine в Triton)
  - `runtime_params.triton_*` соответствуют реально развернутым моделям Triton (model_name/input_name/output_name/datatype)

### 1.9 Документация параллелизма

- [ ] В README компонента есть раздел **"Parallelization"** (или "Параллелизация")
- [ ] Чётко описано, как компонент может работать параллельно:
  - **Внутренний параллелизм** (внутри компонента):
    - Потоки (threading): количество потоков, какие операции параллельны
    - Батчинг: размер батча, как обрабатываются батчи
    - Асинхронная обработка: если используется
  - **Внешний параллелизм** (выше компонента):
    - Можно ли запускать несколько экземпляров компонента параллельно
    - Требования к изоляции (разные run_id, разные result_store пути)
    - Ограничения (shared resources, locks, etc.)
  - **Комбинированный подход**: если используется и внутренний, и внешний параллелизм
- [ ] Указаны ограничения и требования:
  - Thread-safety компонента
  - Требования к GPU (если несколько компонентов используют один GPU)
  - Требования к памяти (peak memory при параллельном выполнении)

**Примеры описания**:
- "Внутренний батчинг: обрабатывает кадры батчами по 8, может использовать несколько GPU потоков для Triton inference"
- "Внешний параллелизм: можно запускать несколько экземпляров параллельно на разных видео (разные run_id)"
- "Комбинированный: внутренний батчинг (batch_size=16) + внешний запуск на разных GPU (по одному компоненту на GPU)"

**Evidence**: Раздел в README с описанием параллелизма

### 1.10 Batching / scheduler contract (Triton)

Для GPU-only компонентов (Triton):
- [ ] `runtime=triton` (no-fallback)
- [ ] Batch size контролируется верхним scheduler:
  - компонент принимает `--batch-size` (или аналог) и использует его в inference-запросах
  - компонент не выбирает batch size сам (auto-batching внутри компонента запрещён)
- [ ] Triton модели batch-enabled:
  - `max_batch_size > 0`
  - входные `dims` в `config.pbtxt` не содержат фиксированный leading `1` (batch отделяется Triton)
  - ONNX/engine экспорт поддерживает dynamic batch (`batch_size>=1`)

### 1.11 Параметры конфигурации компонента (component config params)

- [ ] Компонент принимает **все** параметры настройки обработки через явные аргументы/конфиг
  (например: вариант модели `clip_image_224/clip_image_336`, `face_confidence`, `batch_size` и т.д.).
- [ ] Все параметры перечислены в README компонента:
  - имя параметра и назначение
  - допустимые значения / диапазоны
  - значение по умолчанию
  - влияет ли на качество/скорость/ресурсы
  - **влияние на скорость и цену**:
    - **Δ latency**: примерное изменение времени на 1 кадр (**ms/frame**) при включении/изменении параметра (в сравнении с baseline/default)
    - **Δ cost**: примерное изменение стоимости на 1 кадр (**₽/frame**) при включении/изменении параметра
    - источник/метод оценки: бенчмарк/профилирование/расчёт (должно быть воспроизводимо)
- [ ] Параметры подхватываются оркестратором из профиля анализа и передаются в компонент при запуске
  (без "скрытых" глобальных констант или hardcode).
- [ ] Есть пример блока конфигурации для компонента в README (минимум + расширенный вариант).

**Evidence**: README компонента с таблицей параметров и примером конфигурации, код чтения/передачи параметров.

### 1.12 Features contract (контроль выходных фич компонента)

**Цель**: компонент должен уметь контролируемо отдавать только нужные фичи, чтобы:
- снижать стоимость (CPU/GPU/IO/NPZ size),
- повышать наблюдаемость (явный список выходов),
- дать UI возможность включать/выключать фичи.

- [ ] Компонент имеет явный механизм выбора выходных фич через аргументы/конфиг (пример):
  - `--features <csv>` или `--features-json <json>` или `--feature-set <name>`
  - либо `features:` в конфиге компонента (но важно: оркестратор должен уметь это прокинуть).
- [ ] Есть **дефолтный набор фич** (если фичи не указаны), и он задокументирован.
- [ ] В README компонента есть раздел **"Features"**:
  - перечень всех возможных фич,
  - описание каждой фичи (что это, формат, единица: per-frame/per-run),
  - влияет ли на стоимость/качество,
  - какие фичи включены по умолчанию.
- [ ] В meta артефакта фиксируется, какие фичи были включены/выданы (например `features_enabled[]`, `features_produced[]`).
- [ ] Нет "скрытых" фич: если фича может появиться в output, она должна быть перечислена и управляемо включаться.

**Evidence**: README компонента + код парсинга аргументов/конфига + примеры запуска с разными feature sets + NPZ meta.

### 1.13 Промежуточный прогресс

Компонент должен уметь **репортить прогресс выполнения**, чтобы сайт мог показывать ход анализа в реальном времени.

- [ ] Компонент публикует прогресс в процессе работы (не только “start/end”):
  - либо как **процент** `progress ∈ [0..100]`,
  - либо как **stage-based** прогресс: текущий этап + статус этапов + общий процент (рекомендуется).
- [ ] Для компонентов с “основным циклом по кадрам” (пример: `core_object_detections`):
  - прогресс обновляется **по мере обработки кадров** (а не по таймеру),
  - обновления происходят **не реже 10 раз за run** (условно: каждые ~10% обработанных кадров).
- [ ] Для компонентов с несколькими логическими частями (пример: основной алгоритм → пост‑обработка → статистики):
  - этапы явно выделены и имеют: **id/name**, краткое **описание**, критерий **done**,
  - компонент репортит **переходы между этапами** и завершение этапов.
- [ ] Формат прогресса стабилен и машиночитаем (JSON/структура в metadata/лог‑события), чтобы оркестратор мог его проксировать в backend сайта.

**Evidence**: путь к коду репортинга прогресса + пример реального output (лог/JSON) для короткого run, где видно ≥10 обновлений или stage transitions.

### 1.14 Профилирование по стадиям (stage timings)

Компонент должен уметь **измерять и фиксировать время выполнения логических частей**, чтобы:
- выявлять bottlenecks,
- корректно оценивать стоимость фич/feature-gating,
- сравнивать presets и оптимизации.

- [ ] Компонент измеряет время ключевых стадий (пример: `load_deps`, `infer`, `postprocess`, `save_npz`)
- [ ] Тайминги сохраняются в артефакте (рекомендуется: `summary.stage_timings_ms` или `features.stage_*_ms`)
- [ ] Тайминги используются в документе аудита (раздел “Производительность компонента”)

**Evidence**: NPZ содержит `stage_timings_ms` (dict) + ссылка на код измерений.

## 2. Производительность компонента

### 2.1 Обязательные измерения

**Для каждого компонента должны быть измерены**:

1. **Latency** (время обработки):
   - Единица измерения (unit): `frame`, `frame_pair`, `segment`, etc.
   - Latency per unit: среднее время обработки одной единицы
   - Распределение: p50, p95, p99 (если доступно)
   - Spikes: наличие/отсутствие выбросов

2. **CPU RAM**:
   - Peak RSS (Resident Set Size) в MB
   - Измеряется: 1) во время полного прогона, 2) для 1 единицы измерения (`frame`, `frame_pair`, `segment`, etc.)

3. **GPU VRAM** (если используется GPU):
   - Peak VRAM в MB
   - Измеряется: 1) во время полного прогона, 2) для 1 единицы измерения (`frame`, `frame_pair`, `segment`, etc.)
   - Delta от baseline (если Triton используется)

**Дополнение (Triton VRAM method)**:
- [ ] Если компонент/модель работает через Triton, VRAM метрики фиксируются **по процессу `tritonserver`**:
  - `vram_triton_before_mb`, `vram_triton_peak_mb`, `vram_triton_after_mb`
  - `vram_triton_delta_run_mb` и `vram_triton_drift_mb`
  - если drift заметный → `restart_recommended=true` и причина

### 2.2 Где хранятся данные

**Источник правды**: `docs/models_docs/resource_costs/<component>_costs_v1.json`

**Структура JSON**:
```json
{
  "schema_version": "resource_costs_<component>_v1",
  "generated_at": "ISO timestamp",
  "component": "<component_name>",
  "unit": "<unit_of_processing>",
  "costs": [
    {
      "source_resolution": "WxH",
      "preset": "default|quality|fast",
      "metrics": {
        "latency_ms_mean_stable_per_unit": 25.5,
        "cpu_rss_peak_mb": 200.0,
        "gpu_vram_peak_mb": 650.0,
        "spikes": false
      }
    }
  ]
}
```

### 2.3 Что должно быть в README компонента

**Обязательные разделы**:

#### A) Performance characteristics

```markdown
## Performance characteristics

### Resource costs (measured)

**Источник данных**: `docs/models_docs/resource_costs/<component>_costs_v1.json`

**Единица обработки**: `<unit>` (например, `frame_pair`)

**Типичные значения (preset="default")**:

| Resolution | Latency per unit | CPU RAM peak | GPU VRAM peak | Notes |
|------------|------------------|--------------|---------------|-------|
| 320p | ~25 ms | ~200 MB | ~650 MB | typical |

**Для видео с N кадрами**: Total latency ≈ N × latency_per_unit

**Полные данные**: см. `docs/models_docs/resource_costs/<component>_costs_v1.json`
```

#### B) Sampling requirements (см. раздел 1.7)

#### C) Models (см. раздел 1.8)

#### D) Parallelization (см. раздел 1.9)

### 2.4 Что должно быть в документе аудита

**Обязательный раздел "Производительность компонента"**:

- Таблица с типичными значениями latency/RAM для разных разрешений
- Общая производительность для типичного baseline видео (например, 800 кадров)
- Ссылка на полные данные в `resource_costs/`

### 2.5 Unit-cost vs throughput (batch scaling)

Для baseline GPU моделей:
- [ ] **Обязательно**: unit-cost при `model_batch_size=1` (оценка “на 1 единицу” для scheduler)
- [ ] **Желательно**: throughput-run при `model_batch_size=8` (или другом целевом B), с фиксацией VRAM drift
  - цель: понять масштабирование, предельные B на конкретном GPU, и необходимость перезапуска Triton

## 3. Проверка качества выхода компонента

### 3.1 Цель проверки

Убедиться, что компонент корректно выполняет свою задачу и выходные данные полезны для:
- ML модели (baseline/v1/v2)
- Аналитики
- Downstream компонентов

### 3.2 Методы проверки качества

#### A) Автоматическая оценка (если доступна)

**Для компонентов с ground-truth или reference**:
- Precision/Recall/F1 метрики
- Сравнение с reference implementation или preset
- Статистические тесты на разумность значений

**Пример**: `cut_detection` имеет скрипт `eval_cut_detection_quality.py` для сравнения presets.

**Что фиксировать**:
- Путь к скрипту оценки качества
- Ожидаемые значения метрик (precision > X, recall > Y)
- Примеры запуска скрипта

#### B) Human-friendly визуализация (рекомендуется)

**Цель**: Представить выход компонента в формате, удобном для ручного просмотра и валидации.

**Что визуализировать** (зависит от типа компонента):

**Для cut_detection**:
- Timeline с отмеченными cuts
- Thumbnails кадров до/после cuts
- Графики сигналов детекции (hist_diff, ssim_drop, flow_mag)

**Для shot_quality**:
- Timeline с shot boundaries и quality scores
- Heatmap quality по времени
- Распределение quality scores

**Для core_clip**:
- t-SNE визуализация embeddings
- Примеры кадров с их embeddings
- Cosine similarity матрица между кадрами

**Для core_object_detections**:
- Кадры с нарисованными bounding boxes
- Timeline с количеством объектов по времени
- Треки объектов (если есть tracking)

**Рекомендуемый формат**:
- HTML страница с интерактивными элементами (timeline, графики, thumbnails)
- Или JSON с метаданными для внешней визуализации
- Или набор изображений для ручного просмотра

**Что проверять визуально**:
- ✅ Корректность детекции/вычисления (соответствует ожиданиям)
- ❌ False positives (ложные срабатывания)
- ❌ False negatives (пропущенные события)
- ✅ Разумность статистических фичей

#### C) Статистическая валидация

**Проверка разумности статистических фичей**:

- Диапазоны значений соответствуют ожиданиям для типичных видео
- Отсутствуют аномальные значения (NaN где не ожидается, бесконечности)
- Корреляция между связанными фичами разумна

**Пример**:
```python
# Для cut_detection
assert 0 < hard_cuts_per_minute < 50
assert 0.5 < avg_shot_length < 30
assert jump_cut_ratio_per_minute <= hard_cuts_per_minute
```

#### D) Интеграция с downstream компонентами

**Проверка**:
- Downstream компоненты корректно читают артефакты
- Формат данных соответствует ожиданиям downstream
- Shot boundaries/indices правильно выровнены

### 3.3 Что должно быть в README компонента

**Обязательный раздел "Quality validation & human-friendly inspection"**:

```markdown
## Quality validation & human-friendly inspection

### Как проверить качество выхода компонента

[Описание методов проверки качества]

```

### 3.4 Что должно быть в документе аудита

**Обязательный раздел "Проверка качества выхода компонента"**:

- Описание методов проверки качества

## 7. Приоритизация

Если компонент не соответствует всем критериям:

1. **Критично** (блокирует baseline):
   - Архитектурные требования (BaseModule, контракты)
   - No-fallback policy

2. **Важно** (нужно для полного baseline):
   - Измерения производительности
   - Методы проверки качества

3. **Желательно** (улучшения):
   - Human-friendly визуализация
   - Детальная статистическая валидация


Очень важно что компонент должен выдавать промежуточные результаты своей работы что бы backend на сайте мог выводить прогресс выполнения компонента. Также у компонента должен быть функционал (может отдельный файл, может просто блок кода) который преобразует выход компонента в формат который можно паказать пользователю на сайте, например распределения значений, значения для построения графиков и тд.
