## 2. Резюме

`topk_similar_titles_extractor` — **retrieval-компонент похожих заголовков**: находит top-k ближайших заголовков из
референс-корпуса (similar_titles_corpus_v1) через FAISS/numpy → top1_score, topk_mean_score + ID похожих. Продуктовый
смысл ценный — **«какие видео с похожим заголовком уже есть»**: бенчмарк/конкурентный анализ, find-similar, и
потенциальный **kNN-prior** (похожие заголовки → похожие просмотры) для предсказания. **На данных сигнал ПУСТ:**
`present=1` у всех 22, механизм работает (title-эмбеддинг реальный, норма 1.0), но **референс-корпус — zero-stub**:
`embeddings.npy` содержит **8 полностью НУЛЕВЫХ эмбеддингов** (все нормы 0.0) → каждый косинус = 0.0 → `top1_score=0`,
`topk_mean=0` у **всех 22**. Прямой аналог визуального `similarity_metrics` (пустой reference pack). Инженерия отлична
(FAISS/HNSW/кэш/версионирование), блокер — незаполненный корпус.

## 3. Функционал

Работает после title-эмбеддера:

1. **Загрузка корпуса** (embeddings.npy + ids.json через dp_models, с weights_digest).
2. **Построение индекса** — FAISS HNSW (если доступен и корпус большой) или numpy brute-force (fallback); index-кэш
   (TTL, LRU).
3. **kNN-запрос** — L2-норм title-эмбеддинг → top-k косинусов к корпусу → top1_score, topk_mean_score, top-k ID.
4. **Export-режимы** — ids_only / ids_and_scores / none; max_export_k с truncation-флагом.
5. Пишет 29 tp_topktitles_* (scores + present/backend/corpus/config/6 диагност-флагов).

**Зачем продукту:** «покажи видео с похожими заголовками» — **бенчмарк-инструмент**: сравнить свой заголовок с
существующими, оценить насыщенность ниши, а по перформансу соседей — **прайор для предсказания** (kNN-регрессия:
похожие заголовки историй ↔ похожие просмотры). Основа конкурентной аналитики заголовков.

## 4. Вход

- **title-эмбеддинг** (bge, реальный по значению, но mock-текст).
- **референс-корпус** similar_titles_v1 — **8 НУЛЕВЫХ эмбеддингов** (zero-stub placeholder).
- Нет title/корпуса → present=0.

## 5. Выход

- **29 tp_topktitles_* фич:** top1_score, topk_mean_score, k, corpus_size, dim, backend_faiss/faiss_available,
  export-режимы one-hot + export_k_used/truncated, config (cache/hnsw/require-faiss), 5 диагност-флагов
  (unsafe_relpath/title_embed_missing/dim_mismatch/zero_norm/nan_inf).
- **top-k ID похожих** (при export-режиме) — в артефактах.
- **NaN-политика:** нет входа → present=0; здесь scores=0 (нулевой корпус, не NaN).

## 6. Фичи (важное/неочевидное)

- **top1_score=0, topk_mean=0 у всех 22 (главное)** — корпус zero-stub (8 нулевых эмбеддингов) → все косинусы 0.0.
  Нулевой сигнал по всему корпусу.
- **corpus_size=8** — крошечный placeholder-корпус (реальный бенчмарк требует тысяч заголовков).
- **backend_faiss=0, faiss_available=0** — FAISS не установлен → numpy brute-force (на 8 векторах не важно).
- **title-эмбеддинг реальный (норма 1.0)** — запрос корректен; проблема исключительно в корпусе.
- **export_k_used=5** — механизм экспорта работает (просто возвращает нулевые скоры).
- **Диагност-флаги все 0** — zero_norm корпуса НЕ поднят (проверяется норма запроса, не корпуса — мелкий недочёт:
  нулевой корпус не флагуется).

## 7. Архитектура / алгоритм

- **kNN-retrieval** (FAISS HNSW или numpy brute-force) по корпусу заголовков. Своей модели нет.
- **Index-кэш** (модульный `_INDEX_CACHE`, TTL, LRU) — корпус индексируется раз.
- **Строгая загрузка** корпуса через dp_models (weights_digest, версии).
- **Сложность:** O(corpus_size×dim) numpy или HNSW-логарифм; сейчас тривиально (8 векторов).
- **Детерминизм:** детерминирован (numpy-путь).

## 8. Оптимизации

- **FAISS HNSW** (m/ef_construction/ef_search) — масштаб на большие корпусы; numpy-fallback.
- **Index-кэш** (TTL/LRU) — не переиндексировать корпус на каждое видео.
- **Export-режимы** (ids/scores/none) + max_export_k — контроль объёма/приватности.
- **require_faiss_above_corpus_size** — авто-выбор бэкенда по размеру.
- **Версионирование корпуса** (weights_digest) — воспроизводимость retrieval.

## 9. Слабые места

- **Корпус zero-stub (главный блокер)** — 8 нулевых эмбеддингов → 0 сигнала. Нужен реальный корпус заголовков
  (тысячи, желательно с известным перформансом).
- **Нулевой корпус не флагуется** — zero_norm проверяется для запроса, не для корпуса; тихо возвращает 0 вместо
  диагностики «корпус пуст/нулевой».
- **corpus_size=8** — даже ненулевой был бы статистически бесполезен для бенчмарка.
- **Аналог similarity_metrics** — та же болезнь пустой reference-базы (визуальная сторона).
- **Mock-title** — даже с реальным корпусом запрос по mock-заголовку полу-фиктивен.
- **Leakage-риск** (kNN-prior по перформансу) — если корпус включает будущие/те же видео.

## 10. Рекомендации по улучшению алгоритма (приоритизированные)

1. **[выс., блокер] Наполнить корпус** реальными заголовками (тысячи), желательно с метаданными перформанса
   (просмотры/лайки) — для бенчмарка и kNN-prior.
2. **[выс.] Флагувать нулевой/пустой корпус** — поднимать диагност-флаг (сейчас тихо 0), чтобы не путать «нет похожих»
   с «корпус сломан».
3. **[сред.] kNN-prior для предсказания** — при реальном корпусе с перформансом добавить целевой прайор (средние
   просмотры соседей) — сильный признак.
4. **[сред.] Leakage-контроль** — исключать само видео/будущие из корпуса.
5. **[низ.] Установить FAISS** — при большом корпусе (сейчас numpy на 8 ок).

## 11. Рекомендации по архитектуре / связям

- **Общий title-корпус с `embedding_source_id`/FAISS-индексом** — единый vector store «похожих видео» (title + иные

- **kNN-prior → Fusion/baseline** — средний перформанс соседей как признак (при реальном корпусе с таргетами).
- **Бенчмарк-аналитика** — «ваш заголовок похож на эти N видео (их просмотры…)».
