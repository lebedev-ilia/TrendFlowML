## 2. Резюме

`frames_composition` — **CV-анализатор композиции кадра** (без нейросетей), сшивающий три core-провайдера
(объекты, лица, глубина) в композиционные признаки: правило третей, якоря внимания, баланс/симметрия,
негативное пространство, сложность/плотность краёв, ведущие линии, глубина/боке и грубые «стили» кадра
(minimalist/cinematic/vlog/product). Отдаёт **32 per-frame фичи** `(N,32)` + `frame_feature_present_ratio` +
**217 video-level агрегатов** (mean/std/p10/p50/p90/min/max на фичу + style-распределение). Valid-empty при
отсутствии лиц (`no_faces_in_video`). Прод-готов по контракту: все U1–U6/C1–C4 PASS, golden diff=0 (чистый
numpy/opencv). Но, как и у соседних компонентов, **реальный storage-корпус — пре-фикс**: два логических бага
исправлены 2026-07-13 в коде, но батч ими не пере-прогнан → на реальных данных `style_dominant_id ≡ cinematic`
на всех 4 OK-видео (мёртвая различимость стиля).

## 3. Функционал

Стоит в Tier-2, после трёх core-провайдеров (детекции/лица/глубина, все на одной Segmenter-выборке). Для
каждого кадра (после эвристического letterbox-crop от чёрных полос) считает 32 фичи по группам:
`anchors` (якорь внимания, тип, дистанция), `balance` (LR-баланс негативного пространства), `symmetry`
(h/v/общая), `negative_space`, `complexity` (edge_density, texture_entropy), `leading_lines` (сила/число/
сходимость линий), `depth` (mean/std/p05/p95 из MiDaS), `objects` (число/покрытие/макс-площадь из детектора),
`faces` (центр/площадь/правило третей из landmarks), `style` (4 эвристические вероятности). Затем — 217
video-агрегатов.

**Зачем продукту:** композиция — **язык профессиональной съёмки**. Правило третей, баланс, ведущие линии,
негативное пространство отличают «снято грамотно» от любительского кадра и влияют на удержание/восприятие
качества. Компонент даёт и model-вход (32 фичи/кадр), и понятную креатору аналитику («кадр несбалансирован»,
«лицо не по третям», «сложный/минималистичный кадр»).

## 4. Вход

- **`core_object_detections`** (hard, valid-empty ок) — боксы/классы для objects/anchors/balance.
- **`core_face_landmarks`** (hard, valid-empty → `no_faces_in_video`) — лица для face-фич/третей/якорей.
- **`core_depth_midas`** (hard, **должен быть status=ok**, no-fallback) — карта глубины для depth/боке/стиля.
- **`frame_indices`** (Segmenter, строго) + **`union_timestamps_sec`** — оси; mismatch с любой зависимостью → error.
- Все три dep выровнены по одному frame_indices (shared sampling group).

## 5. Выход

- **Model-facing per-frame:** `frame_feature_values (N,32)` + `frame_feature_names (32,)` +
  **`frame_feature_present_ratio (32,)`** — доля finite по столбцу (помогает модели трактовать NaN).
- **Model-facing video-level:** `feature_values (217,)` + `feature_names (217,)` — 7 квантильных агрегатов
  на каждую фичу + `style_dominant_id` и style-распределение.
- **Ось:** `frame_indices (N,)`, `times_s (N,)`.
- **Empty-путь:** `status=empty`, `empty_reason=no_faces_in_video`, D=1/F=1, все ключи присутствуют.
- **NaN-политика:** face-зависимые столбцы (`face_center_x/y`, `face_area_ratio`, `anchor_*`, `thirds_alignment`)
  = NaN на кадрах без лица (доля ≈ 1−face_present_ratio); `neg_space_balance_lr`=1.0 при нет-объектов (by design после фикса).

## 6. Фичи (важное/неочевидное)

- **`frame_feature_present_ratio`** — умное решение: вместо того чтобы прятать NaN, компонент явно отдаёт долю
  определённости каждого столбца → модель знает, что `face_center_x` определён на 55% кадров, а не «сломан».
- **Letterbox-crop перед image-метриками** — edge_density/symmetry/leading_lines считаются на кадре без чёрных
  полос, иначе полосы завышали бы симметрию/занижали плотность краёв (осознанная робастность).
- **`neg_space_balance_lr`=1.0 при нет-объектов** (bug #1 фикс) — семантика «нет объектов = идеальный баланс»,
  а не 0.0. Тонкий, но важный семантический выбор.
- **`style_*` — эвристические линейные комбинации** нормированных фич (minimalist=f(complexity,neg_space,obj);
  cinematic=f(depth_std,center_offset,symmetry,bokeh); vlog=f(face_centering,complexity); product=f(obj,thirds,
  bokeh)), softmax-подобная нормировка. Грубые «объяснялки», не выученные классы.
- **depth_std / bokeh в стиле нормированы CV/scale** (bug #2 фикс): `ds=clip01(depth_std/depth_mean)`,
  `bokeh=clip01((p95−p05)/1024)` — иначе сырые MiDaS-единицы (~250) всегда давали 1.0 и заваливали всё в cinematic.

## 7. Алгоритм / архитектура

- **Чистый CPU** (numpy/opencv): Sobel/Canny (края/линии), симметрия отражением, saliency-центр, KMeans нет.
  Потребляет готовые core-массивы (depth/objects/faces) — не считает их сам.
- **Параллелизм:** per-frame `_work` через workers (дефолт из `--num-workers`); 32 фичи на кадр.
- **Сложность:** ~10.8 c/видео (N=215, workers=8) на CPU; для 200k ~555 ч → нужен horizontal scaling, но сам
  дёшев (GPU держат deps).
- **Детерминизм:** golden diff=0 без пиннинга (чистый numpy/opencv, нет стохастики) — чище, чем color_light.

## 8. Оптимизации

- **Потребитель core-провайдеров** — не дублирует depth/detections/landmarks, берёт готовое (reuse тяжёлого GPU).
- **`present_ratio`** вместо сокрытия NaN — дешёвая, но ценная мета для модели.
- **Letterbox-crop** — устраняет систематический шум от чёрных полос.
- **Feature gating** (`--feature-set default|ml|all`, `--features anchors,balance,...`) — можно считать не все
  группы (экономия на проде/UI).
- **Параллельные воркеры** per-frame.

## 9. Слабые места

- **Storage пре-фикс: стиль мёртв на реальных данных (главное).** Все 4 OK-видео в storage имеют
  `style_dominant_id ≡ 1.0 (cinematic)` — это симптом bug #2 (сырой depth_std всегда 1.0 → cinematic
  доминирует). Фикс в коде (CV-нормализация) есть, но **батч не пере-прогнан** → style-распределение как
  аналитическая/модельная фича сейчас бесполезно (константа). Тот же паттерн «фикс в git, данные старые».
- **Оба фикса требовали git-фиксации** (REPORT §8) — надо убедиться, что закоммичены в основной репо (иначе
  200k получит те же баги).
- **Стили — грубые эвристики** без калибровки/разметки: линейные веса подобраны вручную, «minimalist/cinematic/
  vlog/product» — сигнальные ярлыки, не выученные классы; их предсказательная ценность сомнительна.
- **Жёсткая зависимость от `core_depth_midas.status=ok`** (no-fallback) — если depth упал, весь компонент =
  error, хотя многие фичи (симметрия/линии/лица) от глубины не зависят.
- **Много NaN на видео без/с редкими лицами** — face-столбцы NaN 45–65% на реальных OK-видео; на no-faces видео
  весь компонент empty (D=1) — теряется вся композиционная аналитика нелицевого контента.
- **217 агрегатов — раздутость** (7 квантилей × 31 фича): сильная корреляция, много малоинформативных полей.
- **`_style_probs` использует `face_centering`, которого нет в сигнатуре вызова явно** — читаемость/риск дрейфа.

## 10. Рекомендации по улучшению алгоритма (приоритизированные)

1. **[выс.] Пере-прогнать storage на пост-фикс коде** — сейчас `style_dominant_id` константен (cinematic) на
   всех реальных видео из-за не-пере-прогнанного bug #2; различимость стиля есть только на тест-подах.
2. **[выс.] Подтвердить git-коммит обоих фиксов** (neg_space_balance_lr, depth-норм) в основной репо DataProcessor.
3. **[сред.] Смягчить no-fallback на depth** — при depth≠ok считать depth-независимые фичи (симметрия/линии/
   объекты/лица), а depth-поля пометить NaN + present_ratio, вместо полного error.
4. **[сред.] Калибровать/заменить стили** — либо обучить лёгкий классификатор композиционного стиля, либо
   задокументировать, что style_* — грубая эвристика (сейчас выглядит как «класс», а это веса на глаз).
5. **[низ.] Проредить 217 агрегатов** — убрать коррелирующие квантили, оставить информативное ядро.

## 11. Рекомендации по архитектуре / связям

- **Reuse трёх core-провайдеров закреплён** (depth/objects/faces) — это правильный паттерн; убедиться в
  shared sampling group (общий frame_indices), иначе mismatch → error.
- **Композиция + shot_quality + color_light** сильно пересекаются (все три — «качество кадра»): рассмотреть
  общий per-frame CV-слой/кэш градиентов/симметрии, чтобы не считать Sobel/edge трижды.
- **present_ratio-паттерн** стоит распространить на другие компоненты с NaN by design (color_light, landmarks).

## 13. Интерпретируемость

- **Сильная сторона:** почти все фичи человекочитаемы (правило третей, баланс, симметрия, ведущие линии,
  негативное пространство); `render.py` есть; style_* задуманы именно как «explainability»-ярлыки.
- **Добавить:** overlay на кадре (сетка третей + якорь внимания + ведущие линии), словесная сводка «кадр
  сбалансирован/лицо не по третям/сложная композиция»; но style-ярлыки честно подписать как эвристику.

## 14. Польза для моделей

**Умеренная.** 32 per-frame композиционных фичи + present_ratio — осмысленный, различимый (C2 CV до 0.91)
model-вход: композиционная грамотность правдоподобно коррелирует с воспринимаемым качеством и удержанием.
present_ratio — полезная мета для трактовки NaN. Но: много NaN на нелицевых кадрах, style-фичи на реальных
данных константны (пре-фикс) и эвристичны, жёсткая зависимость от depth. Геометрическое ядро ценно, style —
пока балласт. Крепкое «приемлемо».

## 15. Польза для аналитиков

**Высокая по потенциалу.** Композиция — понятный про-инсайт: «кадр несбалансирован», «лицо не по правилу
третей», «нет ведущих линий», «слишком сложный/пустой кадр». C2-фичи (edge_density, line_strength,
negative_space, symmetry, face_present) реально различают видео и наглядны. Ограничения: style_dominant на
реальных данных константен (пре-фикс), нелицевой контент → empty (нет аналитики), 217 полей избыточны.
