## 2. Резюме

`mfcc_extractor` — **MFCC** (мел-частотные кепстральные коэффициенты): классическое компактное представление
речи/тембра (13 коэффициентов = DCT логарифма мел-спектра). Отдаёт **`mfcc_mean_by_segment (N,13)`** — компактный
аудио-seq + `delta_mean_by_segment` (дельта-MFCC, динамика) + 13 табличных фич. **Полностью живой** (все 6 видео
ok, 0 NaN, golden diff **строго 0.0**), но **слабо-информативный по скалярам:** различимость едва проходит порог
(mfcc_energy CV=14%, bandwidth CV=12%), а **2 из 4 производных мертвы** — `mfcc_stability` ≈ константа 0.501 (CV≈0),
`mfcc_centroid` ≈ 0.0. MFCC — **самый избыточный из STFT-семейства** (буквально DCT логарифма mel из
`mel_extractor`). Ценность — компактный (13-d) seq-токен; скалярная часть слаба/частично мертва.

## 3. Функционал

Работает после Segmenter (аудио). torchaudio MFCC:

1. STFT → mel-спектр → log → **DCT → 13 MFCC** → усреднение по сегменту → `mfcc_mean_by_segment (N,13)`.
2. **Delta-MFCC** (первая производная во времени) → `delta_mean_by_segment` (динамика тембра).
3. Статистики `mfcc_mean/std/min/max` + скаляры `mfcc_energy`, `mfcc_centroid`, `mfcc_bandwidth`, `mfcc_stability`.

**Зачем продукту:** MFCC — **исторически ключевая фича распознавания речи/тембра** (компактное описание огибающей
спектра). Различает голоса/тембры/типы звука в 13 числах (vs 128 у mel). Для модели — лёгкий аудио-seq; для
аналитика — полностью опаковый (кепстральные коэффициенты).

## 4. Вход

- **Аудио** (Segmenter) — нет аудио → `status=empty` (`audio_missing`, n_mfcc=NaN by design).
- **MFCC-параметры** (`n_mfcc=13`, `n_fft`, `hop_length`, `n_mels`, `fmin/fmax`).

## 5. Выход

- **Model-facing seq:** `mfcc_mean_by_segment (N,13)` + `delta_mean_by_segment` (динамика).
- **Статистики:** `mfcc_mean/std/min/max` (по коэффициентам), `mfcc_energy_by_segment`.
- **13 табличных фич** + `segment_*_sec`/`segment_mask`.
- **NaN-политика:** empty → мета-only NaN; `mfcc_mean_by_segment`=zeros (не NaN).

## 6. Фичи (важное/неочевидное)

- **`mfcc_mean_by_segment (N,13)` — несущая ценность** — компактный кепстральный seq (13-d): лёгкий аудио-токен,
  классика ASR/тембра. Плюс `delta_mean` — динамика (изменение тембра во времени).
- **`mfcc_stability` = константа 0.501 (МЁРТВАЯ)** — CV≈0 на всех видео; заявлена «by design», но постоянное 0.50
  на любом входе — вырожденная/бесполезная фича (красный флаг, как by-design-константы в других компонентах).
- **`mfcc_centroid` ≈ 0.0 (МЁРТВАЯ)** — near-zero mean, неинформативна (MFCC центрированы → центроид ~0 by nature).
- **`mfcc_energy`/`bandwidth` — слабо различимы** (CV 14%/12%, едва >10%-порога) — коэффициенты нормированы, разброс мал.
- **Избыточность с mel (максимальная в семействе)** — MFCC = DCT(log(mel)); формально производная от `mel_extractor`.
  По сути то же представление в сжатом ортогональном базисе.

## 7. Архитектура / алгоритм

- **torchaudio MFCC** (STFT → mel → log → DCT-II, 13 коэфф.) + delta (numpy), агрегация.
- **Сложность:** как mel + DCT; дёшев, CPU.
- **Детерминизм:** golden max|Δ|=**0.0** (torchaudio CPU, побайтово, 10 прогонов).

## 8. Оптимизации

- **Компактность (13-d)** — лёгкий seq vs 128-d mel (экономия для baseline).
- **Delta-MFCC** — динамика без доп. модели.
- **torchaudio** — детерминированный CPU-путь.
- **Empty → zeros** (не NaN) для seq — стабильная форма.

## 9. Слабые места

- **Максимальная избыточность с mel (главное)** — MFCC = DCT(log-mel); формально вычислимо из `mel_extractor`.
  Отдельный компонент + отдельный STFT = дублирование вычислений и представления.
- **2 из 4 скаляров мертвы** — `mfcc_stability`≡0.501, `mfcc_centroid`≈0. Занимают контракт, 0 информации.
- **Слабая различимость живых скаляров** — energy/bandwidth CV 12–14% (едва над порогом); MFCC-скаляры в принципе
  малоинформативны (ценность в самих коэффициентах-seq, не в их агрегатах).
- **Полная опаковость для аналитика** — кепстральные коэффициенты не интерпретируемы человеком вообще.
- **Общий STFT не шэрится** (spectral/mel/mfcc/chroma независимо) — техдолг эффективности.

## 10. Рекомендации по улучшению алгоритма (приоритизированные)

1. **[выс.] Решить избыточность mel↔mfcc** — либо считать MFCC из общего mel (не второй STFT), либо оставить один
   из представлений в model-контракте (Encoder вряд ли нужны оба + CLAP).
2. **[сред.] Убрать мёртвые скаляры** `mfcc_stability` (≡0.5) и `mfcc_centroid` (≈0) — фича-мусор.
3. **[сред.] Общий STFT-кэш** для всего спектрального семейства.
4. **[низ.] Оставить `mfcc_mean_by_segment`+`delta` как компактный аудио-seq** для лёгких baseline; агрегаты — минимизировать.

## 11. Рекомендации по архитектуре / связям

- **Единый спектральный слой** (STFT → mel → {spectral, mfcc, chroma}) — не считать спектр 4 раза, не дублировать.
- **Определить model-контракт аудио** — mel vs mfcc vs CLAP; вероятно не нужны все три как seq.
- MFCC — кандидат на роль лёгкого резерва (13-d) при отказе от тяжёлого CLAP.

## 13. Интерпретируемость

- **Нулевая для человека** — кепстральные коэффициенты не интерпретируемы вообще (даже спектрограммой).
- Показывать нечего; MFCC — чисто модельная фича.

## 14. Польза для моделей

**Умеренная.** `mfcc_mean_by_segment (N,13)` + delta — компактный (13-d) кепстральный аудио-seq, классика ASR/
тембра; чист и детерминирован. Но: максимально избыточен с mel (DCT log-mel) и, вероятно, с CLAP; скалярные
агрегаты слабы/частично мертвы. Полезен как лёгкий резерв аудио-seq, но добавляет мало сверх mel/CLAP.

## 15. Польза для аналитиков

**Пренебрежимая.** Кепстральные коэффициенты полностью опаковы, скаляры слабы/мертвы. Ни одной actionable/
интерпретируемой величины для креатора. Чисто служебная модельная фича.
