## 2. Резюме

`mel_extractor` — **мел-спектрограмма** аудио: перцептивно-масштабированное частотно-временное представление
(128 мел-полос). Отдаёт **`mel_mean_by_segment (N,128)`** — dense seq-токен (базовое аудио-представление для
Encoder) + **`mel_stats_vector (512=4×128)`** (video-level сводка mean/std/min/max по полосам) + 22 табличные фичи
(mel_energy/centroid/bandwidth/flatness/stability/entropy) + per-segment массивы. **Полностью живой и сильно
различимый:** все 6 видео ok, 0 NaN, golden diff **строго 0.0** (побайтово), mel_energy CV=**159%** (0.5–37.8 —
очень различимо), баг validate (n_mels NaN на empty) исправлен. Мел-спектрограмма — **фундаментальное аудио-
представление** (вход большинства аудио-моделей), но пересекается с `spectral_extractor` (те же дескрипторы) и,
частично, с обучаемым `clap_extractor` (вопрос — что использует Encoder: сырой mel или CLAP-эмбеддинг).

## 3. Функционал

Работает после Segmenter (аудио). librosa mel-спектрограмма:

1. **Mel-спектрограмма** (128 полос по перцептивной мел-шкале) → усреднение по сегменту → `mel_mean_by_segment (N,128)`.
2. **Статистики:** `mel_mean/std/min/max` по полосам → `mel_stats_vector (512)`.
3. **Дескрипторы:** `mel_energy`, `mel_centroid`, `mel_bandwidth`, `mel_rolloff`, `mel_flatness`, `mel_stability`,
   `mel_spectrogram_entropy/contrast` (mean/std).
4. Per-segment: `mel_energy/centroid/bandwidth_mean_by_segment`.

**Зачем продукту:** мел-спектрограмма — **стандартное низкоуровневое аудио-представление** (перцептивно-
взвешенное под слух), вход аудио-нейросетей. Кодирует энергию/распределение по частотам во времени → музыка/речь/
шум, характер саундтрека. Для модели — базовый аудио-seq; для аналитика — опаковый (128 полос).

## 4. Вход

- **Аудио** (Segmenter) — нет аудио → `status=empty` (`audio_missing_or_extract_failed`, fn=6 мета, NaN).
- **Mel-параметры** (`n_fft`, `hop_length`, `n_mels=128`, `fmin/fmax`, `power`).

## 5. Выход

- **Model-facing seq:** `mel_mean_by_segment (N,128)` — мел-спектрограмма по сегментам (dense аудио-токен).
- **Video-level:** `mel_stats_vector (512)`, `mel_mean/std/min/max` (по полосам).
- **22 табличные фичи** + per-segment `mel_*_by_segment` + `segment_*_sec`/`segment_mask`.
- **NaN-политика:** empty → мета-only (6 фич NaN), segments_count=0.

## 6. Фичи (важное/неочевидное)

- **`mel_mean_by_segment (N,128)` — несущая ценность** — dense мел-спектрограмма-seq: базовый аудио-вход Encoder.
  128 полос × N сегментов = богатое частотно-временное представление (в отличие от скаляров spectral).
- **`mel_energy` — сильнейшая различимость** (CV=159%): -Q6fnPIy=37.8 (громкий басовый — музыка/гейминг) vs 0.5–6.7
  у остальных. Согласуется со spectral (тот же ролик: centroid низкий, flatness~0 — басовый тон).
- **`mel_stats_vector (512)`** — фикс-размерная video-сводка mel (4 статистики × 128 полос) — удобно для tabular-head.
- **`mel_stability`** (0.83–1.0) — устойчивость спектра во времени (ровное vs меняющееся аудио).
- **Пересечение с spectral/clap** — mel_centroid/bandwidth/flatness дублируют spectral_extractor; сырой mel
  частично перекрывается с обучаемым CLAP (см. §9/§11).

## 7. Архитектура / алгоритм

- **librosa mel-спектрограмма** (STFT → mel-фильтр-банк 128) → per-segment усреднение, статистики (numpy).
- **Сложность:** O(samples·log) STFT + mel-проекция; дёшев, CPU.
- **Детерминизм:** golden max|Δ|=**0.0** (побайтово, 10 прогонов) — строгий.

## 8. Оптимизации

- **Чистый librosa/numpy** — без моделей/GPU.
- **Seq + stats_vector + tabular** — три уровня (модель/сводка/аналитика).
- **Empty-путь мета-only** — компактно.
- **128 полос** — стандарт (баланс разрешение/размер).

## 9. Слабые места

- **Пересечение со spectral_extractor (главное)** — mel_centroid/bandwidth/flatness/rolloff дублируют спектральные
  дескрипторы (оба из STFT). Дублирование вычислений и фич.
- **Возможная избыточность с CLAP** — Encoder-аудио-ветка, вероятно, использует обучаемый CLAP-эмбеддинг, а не
  сырой mel; тогда `mel_mean_by_segment` — альтернатива/резерв, а не основной вход. Нужно уточнить контракт Encoder.
- **Опаковость для аналитика** — 128-полосная спектрограмма/сырые дескрипторы непонятны креатору.
- **Смешивает источники** — один mel на речь+музыку+шум.
- **Общий STFT не переиспользуется** — spectral/mel/mfcc/chroma считают STFT независимо (техдолг эффективности).

## 10. Рекомендации по улучшению алгоритма (приоритизированные)

1. **[выс.] Уточнить контракт Encoder-аудио** — использует ли он сырой `mel_mean_by_segment` или CLAP-эмбеддинг?
   Если CLAP — mel как model-seq избыточен (оставить только stats/дескрипторы).
2. **[сред.] Общий STFT-кэш** для spectral/mel/mfcc/chroma — не считать спектр 4 раза.
3. **[сред.] Убрать дубли дескрипторов** — mel_centroid/bandwidth совпадают со spectral; консолидировать.
4. **[низ.] Качественные ярлыки** (басовый/яркий/энергичный) для аналитика.

## 11. Рекомендации по архитектуре / связям

- **Единый аудио-спектральный слой** — общий STFT → mel/spectral/mfcc/chroma из одного спектра.
- **Определить роль mel vs CLAP** в Encoder — не дублировать аудио-семантику.
- **mel_stats_vector (512)** — хороший фикс-вход для baseline tabular-head.

## 13. Интерпретируемость

- **Низкая** — 128-полосная спектрограмма/сырые дескрипторы непонятны человеку.
- **Добавить:** mel-спектрограмма как картинка в UI (визуально понятно «где энергия»); ярлыки басовый/яркий/энергичный.

## 14. Польза для моделей

**Умеренно-высокая.** Мел-спектрограмма — **фундаментальное аудио-представление**: `mel_mean_by_segment (N,128)` —
богатый dense аудио-seq, `mel_stats_vector (512)` — фикс-сводка; сильно различимы (energy CV 159%), чисты, строго
детерминированы. Ограничивают: пересечение со spectral (дубли) и вопрос избыточности с CLAP (если Encoder на CLAP,
сырой mel второстепенен). Как основа/резерв аудио-входа — ценно.

## 15. Польза для аналитиков

**Низкая.** Мел-спектрограмма и 128-полосные дескрипторы опаковы для креатора. Различимы (energy), но не actionable.
Ценность появится только через визуализацию (спектрограмма-картинка) или качественные ярлыки; сейчас — служебный.
