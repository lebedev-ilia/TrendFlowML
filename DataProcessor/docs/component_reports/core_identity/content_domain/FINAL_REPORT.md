## 2. Резюме

`content_domain` — **классификатор типа контента** (live_action / animation / game / anime / cartoon /
screen_recording…) через **CLIP zero-shot text-retrieval**: для каждой доменной метки эмбеддит текстовые промпты
(CLIP text, Triton), усредняет в label-эмбеддинг и считает косинус к frame-эмбеддингам `core_clip`; top-K + флаги
уверенности по `threshold_global`. **Это первый компонент core_identity, дающий реальный выход на storage-данных**
(все 6 видео `status=ok`, 5/6 → `live_action` — корректно для реального футажа). **Но с двумя ограничениями:**
(1) в базе меток только **2 домена** (`live_action`/`animation` — стаб `db_digest='e2e_stub_v1'`, хотя дизайн
предполагает game/anime/cartoon/screen_recording); (2) **оценки жмутся к порогу** (0.23–0.27 при threshold=0.23) —
тонкая маржа, 5/22 confident, 1 вероятная ошибка (music-studio → animation). Zero-shot, без дообучения. v0.2.

## 3. Функционал

Tier-2, semantic head, на общей выборке `core_clip.frame_indices`:

1. Для каждой доменной метки — набор текстовых промптов; **CLIP text embed** (Triton), усреднение → label-эмбеддинг (A,512) L2.
2. **Косинус** frame-эмбеддингов `core_clip` к label-эмбеддингам → per-frame top-K.
3. Агрегация по видео → per-track (весь ролик = 1 трек) top-K домен + `is_confident_top1` (score ≥ threshold_global).
4. top-K **никогда не гейтится** — пороги только выставляют `is_confident`-флаги (мягкая политика).

**Зачем продукту:** тип контента (реальное видео / анимация / игра / скринкаст) — **важный высокоуровневый
признак формата**: определяет ожидания и паттерны вовлечённости, помогает нормализовать сравнение (анимация vs
live-action — разные распределения). Model-вход (домен как контекст) + аналитика («это анимация/реальный футаж»).

## 4. Вход

- **`core_clip`** (hard) — `frame_embeddings` + `frame_indices` (общая выборка Segmenter).
- **CLIP text (Triton)** — эмбеддинги промптов доменных меток.
- **Label-база** (`db_name=content_domain`, `db_digest=e2e_stub_v1`) — набор доменов + промпты (сейчас 2).
- **`union_timestamps_sec`** — ось.

## 5. Выход

- **Label-space:** `semantic_label_names (A)` (`"0:live_action", "1:animation"`), `threshold_per_label_arr`.
- **Per-frame:** `frame_topk_ids/scores`, `frame_is_confident_top1`.
- **Per-track (видео):** `track_ids` (=1), `track_topk_ids/scores`, `track_is_confident_top1`, `track_present_mask`.
- **Ось:** `frame_indices`, `times_s`. Всегда `status=ok` (нет empty-пути — домен всегда присваивается).

## 6. Фичи (важное/неочевидное)

- **Zero-shot text-retrieval, не object-DB** — принципиально отличается от place/brand/car: не ищет объекты в
  базе, а сравнивает кадр с **текстовым описанием домена**. Поэтому работает без курируемой базы эмбеддингов
  (в отличие от stub-DB place/brand) — и даёт реальный выход.
- **top-K не гейтится** — домен присваивается всегда (даже при низком score), пороги только помечают уверенность.
  Разумно для «типа контента», который есть всегда.
- **Только 2 метки (стаб)** — `live_action`/`animation`; дизайн предполагает больше (game/anime/cartoon/
  screen_recording), но загружены 2 → фактически бинарный классификатор.
- **Оценки у порога** (0.23–0.27, thr=0.23) — CLIP-косинус к тексту низкий по природе; маржа тонкая → часть меток
  борд��рлайн (17/22 conf=False по отчёту). 1 вероятная ошибка (music_studio → animation, 0.248).
- **Промпт-усреднение** — label-эмбеддинг = среднее нескольких промптов (робастнее одного).

## 7. Алгоритм / архитектура

- **CLIP text (Triton) промпты → label-эмбеддинги → cosine к core_clip frame-эмбеддингам** (numpy). Reuse
  core_clip (не эмбеддит кадры заново).
- Zero-shot — не обучается; качество ∝ качеству промптов + разделимости доменов в CLIP-пространстве.
- **Детерминизм:** U5 SKIP (Triton offline), но логика numpy-детерминирована. **v0.2.**

## 8. Оптимизации

- **Reuse core_clip frame-эмбеддингов** — не гоняет CLIP-image заново (общий с scene/place).
- **Батчинг промптов в Triton** — flatten всех промптов в один запрос (не 1 request/промпт).
- **Zero-shot** — не требует обучения/разметки; новый домен = новые промпты (расширяемо).
- **top-K без гейтинга** — простая мягкая политика уверенности.

## 9. Слабые места

- **Только 2 домена (стаб-база, главное).** `live_action`/`animation` вместо задуманных game/anime/cartoon/
  screen_recording (`db_digest=e2e_stub_v1`). Фактически бинарный классификатор — теряется бо́льшая часть
  ценности (различение игр/скринкастов/аниме).
- **Тонкая маржа у порога** — scores 0.23–0.27 при thr=0.23; 17/22 conf=False (бордерлайн), 1 вероятная ошибка.
  Zero-shot CLIP-косинус к тексту низкий → слабая разделимость, порог хрупок.
- **Нет калибровки порога** — `threshold_global=0.23` глобальный, per-label не используется; выбран вручную.
- **v0.2** — сырая; U5/U6 не проверены (Triton offline).
- **Зависимость от Triton CLIP text** — инфра-точка отказа.

## 10. Рекомендации по улучшению алгоритма (приоритизированные)

1. **[выс.] Загрузить полную доменную таксономию** — game/anime/cartoon/screen_recording/… вместо стаб-2;
   это раскроет реальную ценность (сейчас бинарно).
2. **[выс.] Калибровать пороги per-label** — на размеченных примерах доменов; текущий глобальный 0.23 хрупок.
3. **[сред.] Улучшить промпты** — несколько разнообразных промптов на домен (ensemble) для лучшей разделимости.
4. **[сред.] Проверить точность** на видео разных доменов (анимация/игра/скринкаст), U5/U6 при живом Triton.
5. **[низ.] Fine-tune лёгкий классификатор** поверх CLIP-эмбеддингов, если zero-shot маржа останется тонкой.

## 11. Рекомендации по архитектуре / связям

- **Reuse core_clip закреплён** — общий frame-эмбеддинг для content_domain/scene/place; не эмбеддить трижды.
- **Домен как контекст для нормализации** — Encoder/аналитика могут условно интерпретировать другие фичи по домену
  (анимация vs live-action — разные нормы motion/faces).
- **Общий CLIP-text-retrieval паттерн** с scene advanced-семантиками — единый zero-shot слой.

## 14. Польза для моделей

**Умеренная (реальная, в отличие от place/brand/car).** Тип контента — полезный высокоуровневый контекст-признак
(нормализация live-action vs animation), и он **реально присутствует** на данных (не пусто). Ограничивают:
2-метки (стаб) → мало различения, тонкая маржа у порога (шумный сигнал у бордерлайна). После полной таксономии +
калибровки — ценность вырастет. Крепкое «приемлемо».

## 15. Польза для аналитиков

**Умеренная.** «Реальное видео vs анимация» — понятный ярлык, но бинарность ограничивает (не различает игры/
скринкасты/аниме, что интереснее). На реальных данных работает (5/6 корректно). После полной таксономии станет
наглядным «тип контента» для сегментации/сравнения.
