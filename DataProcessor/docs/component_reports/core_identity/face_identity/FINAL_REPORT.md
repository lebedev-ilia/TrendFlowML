## 2. Резюме

`core_face_identity` — **распознавание личности по лицу**: эмбеддит каждое лицо через **ArcFace** и ищет
совпадение в базе известных лиц через **Embedding Service**, возвращая id/имя/similarity + bbox. Это **самый
работающий из retrieval-компонентов core_identity**: на реальном корпусе 4/6 видео `status=ok` с **настоящими
ArcFace-similarity 0.54–0.80**, база содержит реальные записи (`A=3–8`, не стаб-`A=1`), 2 видео без лиц → корректный
empty. **Но:** (1) база — **HF-seed тестовые лица**, не реальная база известных людей (celebrities — зона
владельца); (2) `threshold=0.0` → **нет гейтинга уверенности** (возвращает топ-K независимо от score); (3)
**privacy-critical** — распознавание индивидов требует governance (GDPR/согласие), и большинство YouTube-креаторов
в принципе нет ни в одной базе. 4 бага починены; U5/U6 SKIP. v0.2.

## 3. Функционал

Tier-2, semantic head, гейтится лицами:

1. Берёт лица из `core_face_landmarks` (кадры с лицами).
2. **ArcFace-эмбеддинг** каждого лица (state-of-the-art face recognition).
3. **Embedding Service kNN** vs face-DB (`category=face`, top-K).
4. Возвращает per-face `face_ids`/`face_names`/`face_similarities` + `face_bbox_xyxy` (нормализ. [0,1]).

**Зачем продукту:** узнавание конкретных людей — присутствие знаменитостей/приглашённых гостей/самого автора.
Для некоторого контента (интервью, реакции, коллаборации) наличие известной персоны — сильный фактор охвата.
Но ниша + тяжёлая privacy-нагрузка.

## 4. Вход

- **Лица** (core_face_landmarks) — нет лиц → `status=empty, no_faces_in_video`.
- **ArcFace** — эмбеддинг лиц.
- **Embedding Service** — face-DB (`db_digest`); empty_reason разделён (баг-фикс): `no_faces_in_video` /
  `embedding_service_unavailable` / `no_match_in_db`.
- **`union_timestamps_sec`** + `frame_indices` — ось.

## 5. Выход

- **Label-space:** `semantic_label_names (A)` (имена в базе), `semantic_object_ids` (UUID, дедуп после фикса).
- **Per-face:** `face_ids (∈[-1,A))`, `face_names`, `face_similarities (∈[0,1])`, `face_bbox_xyxy` (норм., строки
  all-NaN или all-finite — фикс C2).
- **Ось:** `frame_indices`, `times_s`. Empty → `fi=[]`, `face_ids=(0,5)`.

## 6. Фичи (важное/неочевидное)

- **ArcFace — SOTA face recognition** (не generic CLIP): специализированные лицевые эмбеддинги → реальные
  различающие similarity (0.54–0.80 на данных). Это делает механизм рабочим (в отличие от stub-retrieval place/brand/car).
- **База с реальными записями** (`A=3–8`, варьируется per-video) — но это **HF-seed тестовые лица**, не реальные
  идентичности. Значения sim реальны, но «личности» тестовые.
- **`threshold=0.0` — нет гейтинга** — возвращает топ-K даже при низком score; downstream должен сам фильтровать
  по similarity (иначе «узнал» кого-то при sim 0.1).
- **bbox-локализация** — где лицо на кадре (для UI/privacy-blur).
- **Privacy by design** — компонент по природе PII; требует хранения/согласия/удаления по запросу (GDPR).
  Реальная база намеренно вынесена в зону владельца.

## 7. Архитектура / алгоритм

- **ArcFace embed → Embedding Service kNN (face category) → per-face id/name/sim/bbox** (numpy сборка).
- Retrieval, не обучается; качество ∝ ArcFace (высокое) × покрытие face-DB (тестовое).
- **Детерминизм:** U5 SKIP (ES offline). **v0.2.**

## 8. Оптимизации

- **ArcFace** (специализированный) вместо generic CLIP — лучшая разделимость лиц.
- **Reuse лиц** из core_face_landmarks.
- **Дедупликация label-space по UUID** (баг-фикс) — чистое пространство меток.
- **Гранулярный empty_reason** (баг-фикс) — различает нет-лиц / сервис-офлайн / нет-матча.
- **bbox нормализован** — переносимость между разрешениями.

## 9. Слабые места

- **База — HF-seed тестовые лица (главное).** `A=3–8` реальных записей, но это тест-сид, не реальные идентичности.
  Similarity реальны (0.54–0.80), но «узнанные личности» бессмысленны продуктово. Реальная база — зона владельца, не проверена.
- **`threshold=0.0` — нет гейтинга уверенности** — возвращает матчи независимо от score; риск ложных «узнаваний»
  без фильтрации downstream. Нужен осмысленный порог.
- **Privacy-critical, governance не адресована в компоненте** — распознавание индивидов требует согласия/хранения/
  удаления (GDPR); для продукта это юридический, не только технический вопрос.
- **Ниша + непокрытие** — большинство YouTube-креаторов нет ни в одной базе → узнавание неприменимо к массе контента.
- **U5/U6 SKIP** — детерминизм/масштаб не проверены (ES offline).
- **v0.2** — сырая; 4 бага чинились в этой сессии (валидатор, empty_reason, empty-path, schema, дедуп).

## 10. Рекомендации по улучшению алгоритма (приоритизированные)

1. **[выс.] Ввести осмысленный similarity-порог** — `threshold=0.0` даёт «узнавания» при любом score; калибровать
   ArcFace-порог (обычно ~0.4–0.6 cosine для уверенного матча).
2. **[выс.] Адресовать privacy-governance** — политика согласия/хранения/удаления PII, opt-out; юридический review.
3. **[выс.] Реальная база известных лиц** (celebrities/публичные персоны) — зона владельца; без неё продуктового смысла нет.
4. **[сред.] Проверить точность/детерминизм** на реальной базе (U5/U6) при живом ES.
5. **[сред.] Оценить scope** — узнавание применимо лишь к контенту с известными персонами; приоритет ниже.

## 11. Рекомендации по архитектуре / связям

- **Reuse лиц** из core_face_landmarks/face-стека (общий source-of-truth лиц) — не детектить заново.
- **Общий Embedding Service** с place/brand/car/franchise; face — отдельная privacy-зона в базе.
- **bbox → privacy-blur pipeline** — если лица не узнаны/не согласованы, размывать в UI.

## 13. Интерпретируемость

- **Отличная** (при реальной базе): «в видео: <имя>, 0.8» + bbox — прямой факт. **Privacy:** показывать только
  согласованные/публичные лица.
- **Сейчас** — тестовые имена, бессмысленны; нужен реальный DB + порог.

## 14. Польза для моделей

**Нишевая, механизм работает.** Присутствие известной персоны — сильный охват-фактор для узкого контента
(интервью/реакции), и ArcFace реально различает лица (0.54–0.80). Но: DB тестовая, порог 0, применимо лишь к
контенту с известными людьми (меньшинство), + privacy. Фактическая польза ограничена; выше нуля (механизм рабочий),
но продуктово не реализована.

## 15. Польза для аналитиков

**Нишевая + privacy-ограниченная.** «Кто в видео» ценно для интервью/коллабораций, но: тестовая база, нет реальных
идентичностей, тяжёлая privacy-нагрузка, непокрытие массы креаторов. Показывать можно только согласованные/публичные лица.
