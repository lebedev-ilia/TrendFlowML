## 2. Ось 1 — Корректность
- **U1 (валидатор rc=0):** `validate_core_depth_midas_npz.py --struct --ranges --qa` → **rc=0 на всех 7**. `--struct` и `--ranges` — «✅ OK». `--qa` штатно пропускается: опциональный конфиг `storage/result_store/view_csv_feature_qa.json` **отсутствует во всём репозитории** (аналитический CSV-view слой ещё не материализован) → валидатор корректно печатает «QA: пропуск» и возвращает rc=0. Это не дефект компонента. *Рекомендация инфры: доразместить QA-ranges для core_depth_midas, когда появится view-слой.*
- **U2 (ось времени):** `times_s = union_timestamps_sec[frame_indices]` (строгий no-fallback в `_require_union_times_s`), len(times_s)=N на всех, times_s неубывающий на всех (`ts_nondec=true`), frame_indices строго возрастает (`fi_strict_inc=true`).
- **U3 (health/finite):** `depth_maps` finite, raw NaN=0 / Inf=0 по всему корпусу (983 кадра); не константа (per-frame complexity>0, range_robust≫0). main.py при невалидной карте (`isfinite.any()==False`) падает штатно → nan_rate raw = 0 by-design.
- **shape/dtype/range:** карты (N,256,256) float32; per-frame агрегаты (N,) float32; preview (10,256,256). Всё соответствует SCHEMA v3.

## 3. Ось 2 — Стабильность / Golden
- **U5 / C4 (golden на GPU):** 2 прогона на GPU, побайтовое сравнение sha256 ключевых массивов (`depth_maps, depth_maps_norm, depth_mean/std/p05/p95, complexity, fg_bg, frame_indices, times_s`).
  **Результат: identical=true на всех 7, diff_keys=[], max|Δ| по depth_maps = 0.0 (глобально).**
  Bicubic-интерполяция MiDaS (`F.interpolate mode=bicubic`) + `inference_mode`/`eval` оказались полностью детерминированы на этом GPU → критерий выполнен строгим нулём, лучше согласованного порога 1e-3.

## 4. Ось 3 — Различимость (C2/C3)
Кросс-видео (7 роликов, mean per-video):
- `depth_complexity_score`: min=0.00234, max=0.00479, mean=0.00404, **std=0.00076, CV=18.8%**.
- `foreground_background_separation_proxy`: min=2.574, max=3.152, mean=3.010, std=0.194, CV=6.4%.
- `depth_range_robust`: min=369.7, max=704.2, mean=604.3, **std=105.7, CV=17.5%**.

- **C3 — PASS:** fg_bg и range_robust finite на всём корпусе и хорошо различимы (range_robust std=105.7 — сильный сигнал; fg_bg std=0.194).
- **C2 — PASS по сути / FAIL по букве порога.** complexity ∈ [0.00222, 0.00820] ⊂ [0,1] на всех, различима между роликами (CV=18.8% — самый высокий относительный разброс из трёх прокси; per-frame размах до 3.6×). НО согласованный в брифинге **абсолютный** порог `std между роликами > 0.01` НЕ выполняется: факт std=0.00076.
  **Причина:** метрика = средний модуль градиента по нормированной [0,1] карте 256², её естественная абсолютная шкала ~0.004; порог 0.01 задан в абсолютных единицах ДО данных и мискалиброван. Сигнал реален (constant-однокадровый MMhLOCzZSmY=0.00234 vs динамичные ролики до 0.0082).
  **Предлагаю переопределить C2** на масштабо-инвариантный критерий: `CV(complexity) между роликами ≥ 0.10` (факт 0.188) ИЛИ абсолютный `std ≥ 5e-4` (факт 7.6e-4). Прошу решение владельца.

## 5. Ось 4 — Model-fit (Encoder) и аналитики
- **Для Models/Encoder** практическая seq — **dense per-frame time-series** согласованной длины N по оси `times_s` (union): `depth_mean, depth_std, depth_p05, depth_p95, depth_range_robust, depth_complexity_score, foreground_background_separation_proxy` — все (N,) finite, различимы, с корректной осью. `depth_maps_norm` (N,256,256, model_facing по схеме) технически тоже seq, но полные карты 65 536-мерны на кадр — тяжелы для трансформер-Encoder; практический путь — агрегаты (+опц. downsampled карты). Полные карты идут в backend/shot_quality/debug.
