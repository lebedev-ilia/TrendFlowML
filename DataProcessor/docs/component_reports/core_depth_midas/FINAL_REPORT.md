## 2. Резюме

`core_depth_midas` — **Tier-0 провайдер карт глубины** визуальной цепочки. На той же primary-выборке кадров,
что и core_clip (владелец выборки — Segmenter), он прогоняет MiDaS через Triton и выдаёт **относительную**
(не метрическую) карту глубины на каждый кадр `depth_maps (N,H,W)`, нормализованную версию `depth_maps_norm`
∈[0,1], набор per-frame агрегатов (mean/std/p05/p95/range/complexity/fg-bg) и preview-подмножество из K=10
карт для бэкенда. Его выход потребляют минимум три модуля (`shot_quality`, `frames_composition`,
`optical_flow`) плюс сам Encoder. Компонент прод-готов: контракт стабилен на 5+ прогонах Audit v4, golden
**побайтово детерминирован (max|Δ|=0.0 на 7 видео)**, на 26 реальных артефактах — 0 NaN/Inf, norm строго в [0,1].

## 3. Функционал

Стоит в начале визуального пайплайна (Tier-0, после Segmenter, параллельно core_clip). Делает одно: для
каждого sampled-кадра оценивает **относительную глубину** — насколько каждый пиксель «ближе/дальше» внутри
кадра. Из сырой карты считает робастную нормализацию (по p05/p95) и набор скалярных прокси, описывающих
геометрию сцены: насколько она «объёмная» vs «плоская», есть ли выраженное разделение передний/задний план,
насколько сложен рельеф глубины.

**Зачем продукту:** глубина — это единственный источник **3D-геометрии сцены** в системе, где всё остальное
2D. Она отвечает на вопросы, которые сильно коррелируют с «продающейся» картинкой: есть ли выразительный
объект на переднем плане (портрет/товар) против плоского фона, «глубокий» кинематографичный кадр против
плоского скриншота. Это прямое сырьё для оценки качества кадрирования (`shot_quality`) и композиции
(`frames_composition`), а также самостоятельная фича Encoder'а.

## 4. Вход

Контракт строгий, **no-fallback** (компонент падает, а не молчит):

- **Кадры** — `FrameManager.get(idx)` из `frames_dir`, RGB uint8 по умолчанию (`--frames-bgr` если BGR).
- **`metadata.json.core_depth_midas.frame_indices`** (обяз.) — какие кадры брать. Владелец выборки —
  **Segmenter**; провайдер сам не семплирует. Пустой/отсутствующий список → `RuntimeError`.
- **`metadata.json.union_timestamps_sec`** (обяз.) — source-of-truth ось времени; `times_s =
  union_timestamps_sec[frame_indices]`, строгий no-fallback (`_require_union_times_s`).
- **run identity** (обяз.): `platform_id`, `video_id`, `run_id`, `config_hash`, `sampling_policy_version`,
  `dataprocessor_version` — при отсутствии падение.
- **`--batch-size`** (обяз.) — задаётся scheduler/DynamicBatch, авто-подбор внутри запрещён.
- **Triton** (обяз., runtime строго `triton`): `--triton-model-spec` через ModelManager (рекомендуется,
  single source-of-truth) либо явные `--triton-http-url`/`--triton-model-name`. Preset `midas_{256,384,512}`.

**Важная связка:** `core_clip`, `core_depth_midas`, `core_object_detections`, `core_face_landmarks`
обязаны работать на **одном и том же** primary `frame_indices` — иначе downstream (`shot_quality`,
`frames_composition`), который выравнивает индексы четырёх core-провайдеров, падает по mismatch.

## 5. Выход

NPZ `depth.npz`, `allow_extra_keys=false`. Смысловые классы ключей (tier в схеме):

- **model-facing (для Encoder):** `depth_maps_norm (N,H,W)`, `frame_indices (N,)`, `times_s (N,)`.
  Практическая seq для трансформер-Encoder — **не** полные карты (65 536-мерны на кадр при 256²!), а
  per-frame агрегаты; полные norm-карты объявлены model_facing, но реально идут в backend/shot_quality/debug.
- **analytics (per-frame time-series):** `depth_mean`, `depth_std`, `depth_p05`, `depth_p95`,
  `depth_range_robust`, `depth_complexity_score`, `foreground_background_separation_proxy` — все (N,) float32,
  осмысленный dense-ряд по `times_s`.
- **debug / backend-preview:** `depth_maps (N,H,W)` (сырая, tier=debug), `preview_depth_maps` /
  `preview_depth_maps_norm (K,H,W)`, `preview_frame_indices/times_s (K,)`, `meta`. K=10 равномерно по
  таймлайну — именно их бэкенд рисует (норм-карты магмой/viridis), сырые наружу не отдаются.

Размерности стабильны: **H×W=256×256** (валидация/прод midas_256, preset определяет; out по умолчанию 384²),
**K=10**. Единицы: глубина **относительная безразмерная** (не метры!), `depth_maps_norm` ∈[0,1], `times_s` — сек.

## 6. Фичи (важное/неочевидное)

- **`depth_maps` относительная, не метрическая.** Абсолютные значения нельзя сравнивать между видео
  (у MiDaS сырой диапазон плавает, реально ~0…976). Сравнимость появляется только после робастной
  нормализации по p05/p95 → `depth_maps_norm` ∈[0,1]. Это ключевой источник ошибок при неаккуратной подаче.
- **`depth_range_robust = p95 − p05`** — робастный «размах глубины» кадра. Физически: насколько сцена
  объёмна (большой диапазон = есть и близкий передний план, и далёкий фон). По 26 реальным видео per-video
  mean **754**, диапазон 536…770, CV **0.064** — умеренно различима.
- **`foreground_background_separation_proxy = range_robust / (std+ε)`** — «есть ли резкое разделение FG/BG»
  (высокое значение = бимодальная глубина: чёткий объект на фоне). По реальным данным mean **2.96**, CV
  **0.038** — самая стабильная из трёх (слабо различает видео, но численно здорова).
- **`depth_complexity_score`** — средний модуль градиента по нормированной карте: «изрезанность» рельефа
  глубины (много планов/объектов vs плоскость). Естественная абсолютная шкала ~0.002…0.008 (маленькая,
  т.к. градиент по [0,1]-карте). **Самая различимая между видео фича: CV=0.445 на 26 реальных видео**
  (в валидации на 7 было 0.188). Исходный абсолютный порог `std>0.01` был мискалиброван (см. §9).
- **`depth_mean`/`depth_std`** — средняя/разброс глубины кадра; `shot_quality` пересчитывает их же по
  ROI-областям (лицо/объект), а не берёт готовые — т.е. глобальные агрегаты и локальные считаются отдельно.

## 7. Алгоритм / архитектура

- **Модель:** Intel **MiDaS** (monocular relative depth). Прод — ONNX через Triton ensemble
  `midas_{256,384,512}` (CPU-preprocess `preprocess_midas_*` → GPU-inference `midas_*_onnx`). Валидация —
  inprocess `torch.hub intel-isl/MiDaS → MiDaS_small`. Внешняя предобученная сеть, не обучается в проекте.
- **Препроцессинг:** клиент делает **только** resize до S×S и отдаёт **UINT8 NHWC RGB**; полная
  нормализация/перевод в FP32 NCHW живёт в Triton ensemble (baseline-контракт).
- **Инференс:** батчами `--batch-size`; выход (B,1,h,w) или (B,h,w) → resize bicubic до out_h×out_w →
  per-frame статистики (векторизованные mean/std, поштучные percentiles).
- **Где идёт:** Triton GPU (прод, no-network через ModelManager). Локальный torch запрещён политикой
  (Triton-only).
- **Сложность:** линейна по N. Чистый инференс MiDaS_small **0.77–0.80 мс/кадр** на 200-кадровых роликах
  (RTX 2000 Ada); Triton midas_256 batch=16 на 100+ кадрах даёт ~2–3× throughput. Post-processing (resize
  каждой карты + percentiles) — последовательный, потенциальное узкое место на больших N.

## 8. Оптимизации

- **Preprocess в Triton, а не в клиенте** — клиент шлёт лёгкий uint8, тяжёлая нормализация на стороне
  сервиса. Осознанное baseline-решение (общий паттерн core-провайдеров).
- **Векторизованные per-batch статистики** (`np.mean/std axis=1`, предвычисленная `isfinite`-маска) —
  ускорение single-video 1.2–1.5×, batch 2–5× (README). Формулы не изменились.
- **Preview K=10 вместо всех карт наружу** — компактный backend-payload; сырые карты (N,256,256, до
  десятков МБ) не отдаются на сайт (тех. необходимость + осознанный контракт Audit v3).
- **Робастная нормализация p05/p95** вместо min/max — устойчивость к выбросам глубины; даёт стабильную
  [0,1]-шкалу для визуализации.
- **Атомарная запись NPZ** (tmp + `os.replace`) + пост-валидация схемы с удалением артефакта при провале.
- **batch_size honest-tuning:** документировано, что при N≤10 `batch_size=1` быстрее (CPU-preprocess
  bottleneck на больших батчах) — редкий пример честной анти-рекомендации батчинга.
- Env-gated `resource_profile_before` (VP_RESOURCE_PROFILE) — наблюдаемость без изменения контракта.

## 9. Слабые места

- **MiDaS_small / midas_256 — слабый край семейства.** Патч/разрешение 256² теряют мелкую геометрию;
  midas_384/512 точнее, но дороже (161 c против 24 c на 304 кадрах при N-масштабе). Качество↔скорость в
  пользу скорости.
- **Относительная глубина не сравнима между видео** без нормализации — легко сделать ошибочный кросс-видео
  вывод по сырым `depth_maps`/`depth_mean`. Норм-карты сравнимы только по форме, не по абсолюту.
- **Порог различимости C2 был мискалиброван** (абсолютный `std>0.01` при естественной шкале метрики ~0.004).
  Переопределён на масштабо-инвариантный `CV≥0.10` (факт 0.188 на 7 видео, 0.445 на 26) — багом это не было,
  но урок: пороги задавать в CV, а не в абсолюте.
- **Полные карты тяжелы для Encoder** (65 536 значений/кадр при 256²) — model_facing по схеме, но реально в
  модель идут агрегаты; норм-карты требуют downsampling если понадобятся модели.
- **Короткие видео / малый N** — медиана N на реальных данных = **12** (как у core_clip): статистика
  распределения глубины по 12 кадрам бедная, complexity/fg-bg шумят.
- **Golden не был закрыт в Audit v4** (§4.6/§4.8 DoD) — но закрыт отдельным прогоном 2026-07-06 (max|Δ|=0.0
  ×7 видео, побайтово). Формальная golden-signature по набору A всё же не зафиксирована.
- **QA-ranges конфига нет в репозитории** (`view_csv_feature_qa.json` отсутствует) — валидатор `--qa`
  graceful-skip; аналитический QA-слой для depth ещё не материализован.
- Отдельного `LOGIC_ERRORS_FOR_CLAUDE.md` L-номера по core_depth_midas не заведено (багов не зафиксировано).

## 10. Рекомендации по улучшению алгоритма (приоритизированные)

1. **[выс.] Профиль midas_384/512 для «quality»-тарифа.** Заметный прирост геометрической детализации на
   тонких сценах; оставить midas_256 дефолтом. Стоимость ↑ ~2–5×, но осознанный выбор тарифа. Схема
   переживёт (H×W уже параметризованы).
2. **[выс.] Рассмотреть Depth Anything V2 вместо MiDaS.** SOTA-monocular depth, заметно лучше на реальном
   видео/сжатых кадрах, есть ONNX — прямая замена в Triton ensemble при том же контракте выхода.
3. **[сред.] Нормировать/логировать `depth_complexity_score` в CV-шкале** и переопределить QA-диапазоны
   через перцентили реальных данных, а не абсолютные пороги (урок C2).
4. **[сред.] Downsampled depth-карта для Encoder** (например 16×16 или 32×32 pooled) как компактный
   model_facing-тензор — компромисс между «одним скаляром» и «65 536 значений».
5. **[низ.] Temporal-smoothing sanity-флаг** — по резким скачкам `depth_mean` между соседними кадрами
   помечать возможный рассинхрон sampling (уже описано в render-доке как диагностика, но не автоматизировано).

## 11. Рекомендации по архитектуре / связям

- **Закрепить shared sampling group** (`core_clip`/`core_depth_midas`/`core_object_detections`/
  `core_face_landmarks` на одном `frame_indices`) на уровне Segmenter/SamplingPolicy, а не как «требование
  компонента» — сейчас mismatch ловится downstream падением, лучше гарантировать на входе.
- **shot_quality пересчитывает depth_mean/std/grad по ROI из сырых `depth_maps`** — это правильно (нужна
  локальная глубина объекта, а не глобальная), но стоит задокументировать, что глобальные агрегаты
  core_depth и локальные shot_quality — **разные величины**, чтобы не путать на дашборде.
- **Отдавать в frames_composition готовые p05/p95** (уже так) — не дублировать перцентили; хороший пример
  reuse. Расширить: если появятся ещё потребители «объёмности», централизовать `fg_bg`/`range_robust` как
  канонические, чтобы не считали заново.
- **Preview-карты — общий формат с core_clip thumbnails** для единого backend-рендера «что видела модель».

## 14. Польза для моделей

`depth_maps (N,H,W)` объявлена model-facing во [`FEATURE_ENCODER_CONTRACT`](../../../../Models/docs/source_migrations/FEATURE_ENCODER_CONTRACT.md),
но практический seq-вход Encoder'а — **per-frame агрегаты** (mean/std/p05/p95/range/complexity/fg_bg): dense
time-series согласованной длины N по оси `times_s`. Глубина даёт модели **единственный сигнал 3D-геометрии**
поверх 2D-фичей CLIP: «объёмность» кадра, наличие выраженного переднего плана, сложность сцены — правдоподобно
коррелирует с воспринимаемым качеством/«дороговизной» картинки, а значит с просмотрами/лайками. Прямых данных
о feature-importance пока нет (модель в разработке), гипотеза умеренно сильная — глубина скорее **дополняющий**
сигнал к CLIP, чем несущий.

## 15. Польза для аналитиков

- **`depth_complexity_score`** → «насколько сцена геометрически насыщена» (самый различимый прокси, CV=0.45).
- **`depth_range_robust` / `foreground_background_separation_proxy`** → «объёмность» кадра, есть ли выраженный
  передний план (портрет/товар против плоского фона) — понятно и сравнимо между роликами (после нормализации).
- **Preview depth-карты** → визуальное «модель видит объём так» — сильный интерпретируемый артефакт для дашборда.
- Оговорка: абсолютную глубину между видео сравнивать нельзя, только нормированные/прокси-величины.

| 6. Фичи | 4 | Осмысленные прокси, но относительность глубины легко неверно трактовать |
| 8. Оптимизации | 4 | Preprocess-в-Triton, векторизация, preview, честный batch-tuning; post-processing последователен |
| 9. Слабые места (инверсно) | 3 | midas_256 слабоват, относительность, малый N, QA-слой не материализован |
| 12. Результаты тестов | 4 | Golden 0.0, 26 реальных видео чисты; нет формальной golden-sig/QA-конфига |
| 13. Интерпретируемость | 3 | База (render + preview-карты в NPZ) есть, но overlay/персонализация в TODO |
| 14. Польза для моделей | 4 | Уникальный 3D-сигнал, дополняющий CLIP; агрегаты — практический seq |
| 15. Польза для аналитиков | 4 | Объёмность/сложность сцены + preview-карты наглядны, требуют пояснения про относительность |

### Итоговые оценки

- **Польза для моделей: 4/5.** Даёт Encoder'у уникальную ось — 3D-геометрию сцены, которой нет больше нигде;
  per-frame агрегаты — готовый dense-seq. Не несущий сигнал (в отличие от CLIP), а важное дополнение, плюс
  детерминирован и стабилен.
- **Польза для аналитиков: 4/5.** «Объёмность/сложность кадра» и preview depth-карты — наглядный,
  интерпретируемый выход для сравнения контента; снижают оценку только относительность глубины (нужна
  аккуратная подача) и малый N на коротких видео.
