# FINAL REPORT — `clip_times_s` / `clip_frame_indices` (action_recognition v3 time-axis)

> Глубокий разбор по [`COMPONENT_DEEP_DIVE_PROTOCOL.md`](../../COMPONENT_DEEP_DIVE_PROTOCOL.md).
> **Что это на самом деле:** не отдельный модуль, а **ось времени model-facing стрима `action_recognition`
> v3** — сестра [`clip_embeddings`](../clip_embeddings/FINAL_REPORT.md). В деп-дайв-чеклист попала отдельной
> строкой, потому что `stamped_components()` подхватил её из фич-леджера валидации. Разбирается как фича-группа
> оси; родитель — `action_recognition` (SlowFast v3).

## 2. Резюме

`clip_times_s` и `clip_frame_indices` — **ось времени action-токена**. Для каждого клипа (окно вокруг трека
человека) фиксируется его **центральный кадр** в union-домене (`clip_frame_indices`, int32) и соответствующее
время `clip_times_s = union_timestamps_sec[center]` (float32, сек). Весь стрим отсортирован по времени
(монотонно ↑), так что `clip_embeddings[i]` ↔ `clip_times_s[i]` ↔ `clip_frame_indices[i]` — согласованная
тройка. Без этой оси Encoder не может ни упорядочить клипы, ни построить позиционное/временное кодирование
(`t_center/duration`), а аналитик — не может нарисовать ленту действий. Валидатор жёстко проверяет: длины ==C,
монотонность `clip_times_s`, `clip_frame_indices` ∈ [0, union_len). Как и `clip_embeddings`, в реальном
storage-корпусе v3-стрим ещё не материализован (артефакты v2, но per-track аналоги `clip_center_times_s`/
`clip_center_frame_indices` там есть).

## 3. Функционал

Ось строится в конце v3-пайплайна (`build_v3_arrays`):

1. При нарезке клипов `action_recognition_slowfast.py` для каждого клипа хранит **центральный кадр**
   (`clip_center_indices`) и полный список кадров окна.
2. `clip_times_s = union_timestamps_sec[center]` — время центра клипа по общей оси Segmenter.
3. `build_v3_arrays` делает **stable argsort по времени** и переставляет ВСЕ параллельные массивы
   (`clip_embeddings`, `clip_track_id`, `clip_frame_indices`, `clip_times_s`, `clip_segment_id`) одним `order`
   — гарантия выравнивания тройки.

**Зачем продукту:** action-токены приходят по разным трекам вперемешку; ось делает из них **упорядоченную по
времени последовательность** — именно то, что потребляет трансформер-Encoder (позиционное кодирование от
`t_center/duration`). Для аналитика это «когда в видео что происходило» — timeline действий. Это «клей», без
которого `clip_embeddings` — просто мешок векторов без порядка.

## 4. Вход

- **`union_timestamps_sec`** (Segmenter, hard) — единственный источник времени; `clip_times_s` — выборка из
  него по индексам центров. Валидатор входа падает, если `union_timestamps_sec` пуст/отсутствует.
- **центры клипов** (`clip_center_indices`) — из нарезки окон SlowFast, в union-домене.
- Опосредованно зависит от person-детекций/треков (нет клипов → пустая ось, `(0,)`).

## 5. Выход

- **`clip_frame_indices (C,)` int32** — **центральный кадр** каждого клипа (union-домен). ⚠ Семантика
  сменилась между версиями: в **v2** ключ `clip_frame_indices` = полный список кадров окна (list-of-lists,
  32 кадра), а **v3** `clip_frame_indices` = **скаляр-центр на клип**. Полное окно в v3 наружу не выносится.
- **`clip_times_s (C,)` float32** — время центра, сек, ⊆ `union_timestamps_sec`, **монотонно ↑** (стрим
  отсортирован). Единицы — секунды от начала видео.
- Обе — model_facing hard (нужны Encoder'у для порядка и позиционного кодирования).

## 6. Фичи (важное/неочевидное)

- **Согласованность тройки** `emb↔time↔frame` через единый `order` — ключевое свойство: любой клип можно
  привязать и ко времени, и к кадру, и к треку. Ошибка сортировки здесь ломала бы весь токен-стрим.
- **`clip_times_s` — центр, а не начало/конец окна.** Для длинного окна (v3.2 = 96 кадров) центр — разумный
  «момент действия»; но при быстром действии в начале окна центр может «смазать» тайминг.
- **Равномерность центров.** На реальных данных центры идут с шагом окна: напр. видео `-3Mbinqzig4` — клипы
  на кадрах [40, 80, 120] → времена [4.07, 8.13, 12.20] с (шаг ровно 40 кадров/≈4.07 с). Ось регулярна, т.к.
  окна нарезаются со stride.
- **union-домен, не локальный.** `clip_frame_indices` индексирует ОБЩУЮ ось Segmenter (как у всех visual-фич),
  что позволяет кросс-компонентное выравнивание (action ↔ depth ↔ CLIP по времени).

## 7. Алгоритм / архитектура

- Чистая numpy-логика (в `build_v3_arrays`): индексация `union_timestamps_sec`, `np.argsort(kind="stable")`,
  перестановка массивов. Модели/GPU не требует, детерминизм тривиален (golden ×2 идентичны).
- Сложность O(C log C) на сортировку, C = число клипов (десятки). Незаметна на фоне SlowFast-инференса.

## 8. Оптимизации

- **Stable-sort** (а не обычный) — при равных временах сохраняет исходный порядок → воспроизводимость golden.
- **Скаляр-центр вместо полного окна в v3** — компактнее NPZ (не хранит list-of-lists по 32 кадра на клип);
  полное окно осталось только во внутренних метриках/рендере. Осознанное упрощение контракта наружу.
- **Общая ось Segmenter** — не заводит свою систему координат, переиспользует `union_timestamps_sec`.

## 9. Слабые места

- **v3-оси нет в реальном storage** (тот же корпус-разрыв, что у `clip_embeddings`): артефакты — v2, где ось
  живёт как per-track `clip_center_times_s`/`clip_center_frame_indices` внутри `results_json`. Плоских
  `clip_times_s`/`clip_frame_indices (C,)` в батче пока нет — нужен пере-прогон на v3.
- **Семантический дрейф имени `clip_frame_indices`** (v2 окно ↔ v3 центр) — потенциальный источник путаницы
  для потребителя, если он читает старые артефакты по новому контракту. Требует явной проверки `schema_version`.
- **Паддинг окна в v2** (`clip_frame_indices` первого трека `-0InsUQNw`: `[...,33,33,33,...]` — хвост забит
  повтором) — артефакт старой нарезки, починен в v3.1, но в storage ещё виден.
- **Центр может смазывать тайминг** резких коротких действий внутри длинного окна (96 кадров).
- **Пустая ось на не-people-видео** — нет людей → нет клипов → `(0,)` (наследует зависимость родителя).

## 10. Рекомендации по улучшению алгоритма (приоритизированные)

1. **[выс.] Пере-прогнать корпус на v3** — материализовать плоскую ось (общий блокер с `clip_embeddings`).
2. **[сред.] Задокументировать/защитить семантику `clip_frame_indices`** — потребитель обязан проверять
   `meta.schema_version` (v2 окно vs v3 центр); в идеале переименовать v3-ключ в `clip_center_frame_idx`,
   чтобы имя не совпадало с v2-семантикой.
3. **[низ.] Опционально добавить `clip_start_s`/`clip_end_s`** (границы окна) рядом с центром — для аналитики
   «длительность действия», без раздувания model-входа.

## 11. Рекомендации по архитектуре / связям

- **Кросс-компонентное выравнивание по времени:** т.к. ось в union-домене, Encoder/аналитика могут
  синхронизировать action-клипы с depth/CLIP/motion по `clip_times_s` — закрепить это как общий паттерн
  time-join визуальных фич.
- **Единый time-embedding контракт с Models:** `t_center/duration_sec` — Encoder уже так и планирует
  позиционное кодирование; ось `clip_times_s` — прямой вход для него, подтвердить в FEATURE_ENCODER_CONTRACT.

## 12. Результаты тестов и оценка

| Прогон | Результат | Что говорит |
|---|---|---|
| Валидатор оси (v3) | len==C, `clip_times_s` монотонно ↑, `clip_frame_indices`∈[0,union_len) | контракт оси жёстко проверяется |
| golden ×2 (v3.2) | идентичны | stable-sort детерминирован |

Вывод: ось **логически корректна и детерминирована** (гейты зелёные, golden=0), но как плоский v3-артефакт в
реальном корпусе не материализована.

## 13. Интерпретируемость

- Сама по себе ось — служебная, но именно она делает возможной **ленту действий по времени** (timeline с
  метками Kinetics-классов на `clip_times_s`) — самый понятный пользователю вид выхода action_recognition.
- Предложение: в кабинете показывать шкалу видео с точками «в 4с — X, в 8с — Y» (данные уже есть в оси+классах).

## 14. Польза для моделей

**Косвенная, но необходимая.** Сами по себе времена/индексы не предсказывают популярность, но без них
`clip_embeddings` нельзя ни упорядочить, ни снабдить позиционным кодированием — а именно временная траектория
действия (что и когда) несёт сигнал. Ось — enabling-инфраструктура токена: `t_center/duration` идёт прямо в
time-embedding Encoder'а. Оценка держится средней, потому что ценность целиком производна от `clip_embeddings`.

## 15. Польза для аналитиков

**Прямая и наглядная.** `clip_times_s` + классы = timeline действий («когда что происходило») — один из самых
понятных выходов для креатора. `clip_frame_indices` позволяет ткнуть в конкретный кадр-превью момента действия.
Ограничения: пусто без людей; на реальных данных пока v2 (ось внутри per-track структуры, не плоская).

### Итоговые оценки

- **Польза для моделей: 3/5.** Ось — обязательная инфраструктура: даёт Encoder'у порядок клипов и вход для
  time-embedding (`t_center/duration`). Собственного предсказательного сигнала не несёт (ценность производна
  от `clip_embeddings`), поэтому балл средний, а не выше.
- **Польза для аналитиков: 4/5.** Прямо превращается в timeline действий («когда в видео что происходило») —
  один из самых понятных креатору выходов; ограничивают пустота на не-people-контенте и незрелость данных (v2).
