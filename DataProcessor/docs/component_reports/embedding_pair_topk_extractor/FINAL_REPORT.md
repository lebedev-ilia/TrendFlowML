## 2. Резюме

`embedding_pair_topk_extractor` — **derived-компонент top-k близости заголовка к сегментам видео**: считает (1)
title↔desc косинус и (2) **title↔транскрипт-ЧАНКИ top-k** — какие сегменты транскрипта ближе всего к заголовку
(top1..8 + max/mean), с FAISS-режимом для больших корпусов. Продуктовый смысл хороший и **гранулярнее cosine_metrics**:
не «весь ли контент соответствует заголовку», а **какой именно сегмент лучше отрабатывает обещание заголовка**. **На
данных:** present=1 у всех 22 (title↔desc всегда), но **title↔desc = 0.841/0.853 mock-константа (идентична
cosine_metrics)**, а title↔transcript top-k считается у 5, из которых **4/5 имеют лишь 1 чанк → только top1 (нет
ранжирования)**; настоящий top-k (top1+top2) даёт лишь 21a279af (3 чанка). Проблемы: **сильное дублирование
cosine_metrics** (title_desc + title_transcript), **69 полей с дуальным namespace** (embpair_+pairtopk_) и массой NaN-
слотов, вырождение на 1-чанковых транскриптах.

## 3. Функционал

Работает после эмбеддеров и chunk-эмбеддера:

1. Грузит эмбеддинги title/desc + матрицу чанк-эмбеддингов транскрипта (приоритет источников + legacy-fallback).
2. **title↔desc** косинус.
3. **title↔транскрипт-чанки:** косинус к каждому чанку → сортировка → top-k (top1..8 + индексы) + max/mean.
4. **FAISS-режим** (auto/never/always, min_corpus) — для больших корпусов ускоряет поиск ближайших.
5. Пишет 69 полей (дуальный namespace) + present/флаги/config.

**Зачем продукту:** «какой сегмент видео лучше всего доставляет обещание заголовка» — **best-matching-segment**:
можно показать креатору момент, где контент совпадает с тайтлом (или что НИ ОДИН сегмент не совпадает = кликбейт).
Гранулярнее, чем title↔transcript одним числом.

## 4. Вход

- **title/description эмбеддинги** (bge, **mock**).
- **chunk-эмбеддинги транскрипта** (whisper, реальные у 5, но чаще 1 чанк).
- Нет входа → соответствующие метрики NaN.

## 5. Выход

- **69 полей (дуальный namespace embpair_/pairtopk_):** title_desc_cosine, title_transcript top1..8 (+ idx + dup),
  topk_max/mean, present-флаги, n_chunks, source one-hot, FAISS-режим one-hot, config (top_k/slots/clamp), 5 диагност-
  флагов.
- **NaN-политика:** нет чанков → top-k NaN; пустые слоты NaN.

## 6. Фичи (важное/неочевидное)

- **title↔desc = 0.841/0.853 mock-константа (главное)** — идентична cosine_metrics.title_desc; чистое дублирование.
- **title↔transcript top-k вырожден** — 5 видео с чанками, но 4/5 имеют **1 чанк** → top1 только (top2..8 NaN); нет
  реального ранжирования сегментов. Лишь 21a279af (3 чанка): top1=0.798, top2=0.755 — единственный настоящий top-k.
- **top1 значения 0.73–0.798** — по сути совпадают с cosine_metrics.title_transcript (дублирование сигнала).
- **69 полей, дуальный namespace** — каждое значение продублировано в tp_embpair_ и tp_pairtopk_; 8 слотов × (val+
  dup+idx) в основном NaN. Сильная раздутость контракта.
- **FAISS-режим** — не задействован (корпус мал, min_corpus=512); заготовка под масштаб.
- **Диагност-флаги все 0** — чисто.

## 7. Архитектура / алгоритм

- **Чистый numpy** (+ опц. FAISS) — косинус к каждому чанку, сортировка top-k. Своей модели нет.
- **Безопасность:** safe-join relpath, finite/zero-norm/dim проверки.
- **Сложность:** O(n_chunks×dim); FAISS для больших корпусов.
- **Детерминизм:** тривиально детерминирован.

## 8. Оптимизации

- **FAISS-режим** (auto по размеру корпуса) — масштабируемый поиск ближайших (задел).
- **Reuse эмбеддингов** (title/desc/chunks).
- **Slot-clamp** + export-gates (slots/indices/summary) — контроль объёма вывода.
- **Приоритет источников** + legacy-fallback.

## 9. Слабые места

- **Дублирует cosine_metrics (главное)** — title_desc идентичен, title_transcript top1 ≈ тот же сигнал. Два
  компонента дают перекрывающийся вывод.
- **top-k вырожден на 1-чанковых транскриптах** — best-segment-ранжирование мёртво (4/5 → top1 only); общий блокер
  нарезки чанков.
- **69 полей + дуальный namespace** — сильная раздутость; каждое значение ×2 префикса + NaN-слоты.
- **title mock** — title↔* полу-фиктивны (операнд mock).
- **FAISS не задействован** — сложность режима без пользы на текущем корпусе.

## 10. Рекомендации по улучшению алгоритма (приоритизированные)

1. **[выс.] Консолидировать с cosine_metrics** — title_desc + title_transcript дублируются; оставить один источник
   пары, здесь — только уникальное (top-k по сегментам).
2. **[выс., системный] Мельче нарезать чанки** — оживит top-k best-segment (4/5 сейчас 1 чанк).
3. **[сред.] Убрать дуальный namespace** (embpair_/pairtopk_) — вдвое меньше полей.
4. **[низ.] FAISS — включать только при реальном большом корпусе** (сейчас балласт-конфиг).

## 11. Рекомендации по архитектуре / связям

- **Единый «title-alignment» блок** — слить cosine_metrics (пары) + pair_topk (сегменты) + title_to_hashtag; убрать
  дублирование, оставить: пары-косинусы + best-segment top-k.
- **best-segment → UI** — «момент видео, лучше всего отвечающий заголовку» (при мелкой нарезке).
- **transcript_chunk_embedder → pair_topk** — согласовать гранулярность (общий барьер derived-группы).

## 14. Польза для моделей

**Низкая.** Дублирует cosine_metrics (title_desc mock-константа + title_transcript top1 тот же сигнал); уникальный
top-k по сегментам вырожден (4/5 = 1 чанк). 69 полей дуального namespace — почти все NaN. Модель почти не получает
нового сверх cosine_metrics. Потенциал (best-segment) — при мелкой нарезке.

## 15. Польза для аналитиков

**Умеренная (концепт), низкая (факт).** «Какой сегмент видео лучше отрабатывает заголовок» — ценный гранулярный
инсайт (лучше, чем одно число cosine_metrics), интерпретируемый и actionable. Но на данных вырожден (1-чанк top-k),
title mock, дублирует пары cosine_metrics. После мелкой нарезки — сильная сегментная аналитика; сейчас — почти то же,
что cosine_metrics.
