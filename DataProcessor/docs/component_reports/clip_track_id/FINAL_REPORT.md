# FINAL REPORT — `clip_track_id` (action_recognition v3 person-link)

> Глубокий разбор по [`COMPONENT_DEEP_DIVE_PROTOCOL.md`](../../COMPONENT_DEEP_DIVE_PROTOCOL.md).
> **Что это на самом деле:** не отдельный модуль, а **фича-группа привязки клипа к персонажу** в
> model-facing стриме `action_recognition` v3 — третья сестра [`clip_embeddings`](../clip_embeddings/FINAL_REPORT.md)
> и [`clip_times_s/clip_frame_indices`](../clip_times_s_clip_frame_indices/FINAL_REPORT.md). В деп-дайв-чеклист
> попала отдельной строкой из-за `stamped_components()`.

## 2. Резюме

`clip_track_id` привязывает каждый action-клип к **конкретному человеку в кадре**. Источник — свой
**appearance-embedding трекер** (эмбеддинг crop'а бокса + косинусная ассоциация между разреженными кадрами),
живущий внутри `core_object_detections` и пишущий `track_ids` в `detections.npz`. `action_recognition` строит
клипы **по трекам** (`frame_indices_per_person`), поэтому `clip_track_id` = владелец окна (мода `track_ids` в
окне; `-1` если трека нет). Это то, что превращает набор клипов в **траектории действия отдельных персонажей**:
Encoder может собрать «что делал человек #2 во времени», аналитик — «сколько людей и как долго в кадре».
Именно ради этой привязки v3 завёл appearance-трекер — он починил фрагментацию (v1: 205 треков → v3: 11–16).
**Оговорка данных:** в реальном storage-корпусе (v2) `mean_clips_per_track ≈ 1` — фрагментация ещё видна;
целевые 4.0 достигнуты только на v3.2-тест-прогонах.

## 3. Функционал

- **Appearance-трекер** (в `core_object_detections`): для каждого person-бокса берёт crop → эмбеддинг
  (OSNet ReID 512-d рекоменд., CLIP-fallback) → L2 → косинусная ассоциация с существующими треками; motion/IoU
  — вторичный гейт против «телепортов». Устойчив к разрывам разреженного union-сэмплинга (главный довод дизайна).
- **`action_recognition`**: группирует кадры по `track_ids` (`frame_indices_per_person`), нарезает клипы внутри
  трека, `clip_owner` = id трека. `build_v3_arrays` кладёт это в `clip_track_id`, считает `num_tracks` и
  `mean_clips_per_track`.

**Зачем продукту:** без привязки action-клипы — обезличенный поток. `clip_track_id` даёт **субъектность**:
кто именно совершает действие и как его действие меняется во времени (траектория на трек ~4–5 клипов). Это
поднимает качество action-сигнала (Encoder видит согласованную траекторию, а не разрозненные метки) и даёт
аналитику «сколько персонажей, как долго каждый в кадре».

## 4. Вход

- **`track_ids`** (`core_object_detections/detections.npz`, schema v3, `(N,MAX) int32`, `-1`=невалидный слот) —
  hard для связи клип↔трек. Это и есть выход appearance-трекера.
- **person-детекции** — hard (нет person → `empty`, `clip_track_id=(0,)`).
- **общий `frame_indices`** (shared sampling group) — должен совпадать с сэмплом action_recognition.

## 5. Выход

- **`clip_track_id (C,) int32`** — id трека-владельца каждого клипа (согласован с `clip_embeddings`/`clip_times_s`
  через единый `order`), `-1` если клип без трека. Model_facing (soft по контракту: Encoder может и без него,
  но он повышает качество).
- **Агрегаты-спутники:** `num_tracks` (сколько уникальных персонажей), `mean_clips_per_track` (плотность
  траектории — главный критерий качества v3).

## 6. Фичи (важное/неочевидное)

- **`clip_track_id` = владелец окна, а не мода по кадрам** (в v3 клипы уже нарезаны *внутри* трека, поэтому
  владелец однозначен). Контракт допускает моду `track_ids` в окне при пограничных случаях.
- **`-1` — легитимное значение** (клип без надёжного трека), а не ошибка; Encoder/аналитика должны его
  отфильтровывать при пер-трек агрегации.
- **`mean_clips_per_track` — метрика здоровья трекинга.** =1 означает фрагментацию (каждый трек = 1 клип, нет
  траектории); v3.2 поднял до **4.0/4.06** косто-нейтрально (окно ×3). На реальном v2-корпусе — всё ещё ≈1.
- **Внутри-трековая когерентность:** на реальных данных `stability=1.0`, `num_switches=0` — трекер не «прыгает»
  между личностями внутри трека (это про качество identity, отдельно от фрагментации между треками).
- **Только person** (class_id 0): остальные классы получают `-1` — appearance-ReID осмыслен только для людей.

## 7. Алгоритм / архитектура

- **Трекер:** appearance-embedding (OSNet ReID 512-d рекоменд. / CLIP-fallback), косинусная ассоциация +
  motion-гейт, в `core_object_detections` под `tracking.enabled`. Детерминизм → golden ×2 идентичны.
- **Привязка в action_recognition:** чистая группировка по `track_ids` + `build_v3_arrays` (numpy).
- **Сложность:** эмбеддинг бокса — доп. forward на каждый person-crop в детекторе; ассоциация O(боксы×треки).
  В action_recognition — почти бесплатно (индексация).

## 8. Оптимизации

- **Appearance вместо IoU/motion** — осознанный выбор под разреженный сэмплинг (IoU рвётся между кадрами).
- **Трекер внутри детектора** (не отдельный модуль) — единый source-of-truth `detections.npz`, не плодит артефакт.
- **Косто-нейтральные окна v3.2** — подняли плотность клипов/трек 1→4 без роста бюджета кадров.
- **OSNet опционален** (сейчас histogram/CLIP-fallback) — не тянуть новый вес на первый прогон.

## 9. Слабые места

- **Фрагментация всё ещё в реальном корпусе.** v2-storage: `mean_clips_per_track ≈ 1` (11 из 12 треков — по
  1 клипу; только `-3Mbinqzig4` дал 3). Целевые 4.0 — только на v3.2-тест-прогонах → **батч не пере-прогнан
  на v3-трекер** (общий разрыв с `clip_embeddings`/осью).
- **OSNet не включён в прод** — ветка готова, но требует `torchreid`; сейчас более слабый histogram/CLIP-эмбеддер
  → на толпном/длинном контенте разделение id хуже (по метрике intra/inter).
- **Только person** — действия не-людей (машины, животные) не трекаются и не получают action-траектории.
- **`-1`-клипы** уменьшают полезный сигнал, если трекер не смог ассоциировать (тихая потеря субъектности).
- **Зависимость от качества person-детекций** — ложные/пропущенные person-боксы → битые треки.
- Историческая первопричина фрагментации (суррогатные per-detection track_id в Audit v3) закрыта дизайном
  EMBEDDING_TRACKER, но реальные артефакты его ещё не отражают.

## 10. Рекомендации по улучшению алгоритма (приоритизированные)

1. **[выс.] Пере-прогнать корпус на v3-трекере** — материализовать реальные `mean_clips_per_track≈4` вместо
   фрагментированного v2≈1 (общий блокер линейки action).
2. **[сред.] Включить OSNet ReID** в прод (добавить `torchreid`) — измеримо лучше histogram по intra/inter на
   толпном контенте; сейчас разделение id субоптимально.
3. **[сред.] Метрика доли `-1`-клипов** в meta — мониторить, сколько субъектности теряется на реальных данных.
4. **[низ.] Расширить трекинг на ключевые не-person классы** (напр. авто для авто-контента) — если появится
   спрос downstream.

## 11. Рекомендации по архитектуре / связям

- **Reuse `track_ids`:** appearance-трекер уже отдаёт их в `detections.npz`, а action_recognition переиспользует
  — закрепить, не считать трекинг дважды; тот же `track_ids` могут использовать другие person-компоненты
  (emotion/behavioral) для пер-персонажной агрегации.
- **Shared sampling group** (core_object_detections ↔ action_recognition) — гарантировать совпадение
  `frame_indices` на Segmenter, иначе привязка клип↔трек рассыпается.
- **Единый id-контракт персонажа** по всей visual-ветке — если несколько компонентов агрегируют по людям,
  им нужен один и тот же `track_id` (потенциальный общий «person-id» слой).

## 13. Интерпретируемость

- `clip_track_id` + `num_tracks` → «сколько людей в видео»; `mean_clips_per_track` → «как долго каждый в кадре».
- Предложение для пользователя: подсветка «в видео N персонажей», лента действий **по каждому человеку**
  (эмбеддинг+классы, сгруппированные по `clip_track_id`) — понятный про-инсайт «кто что делает».

## 14. Польза для моделей

**Косвенная, качество-повышающая.** Сам id не предсказывает популярность, но он позволяет Encoder'у собрать
**согласованную траекторию действия по персонажу** (а не разрозненные клипы) — это делает `clip_embeddings`
информативнее (динамика действия конкретного субъекта). По контракту soft: Encoder работает и без него, но с
ним action-ветка богаче. Ограничивают фрагментация в реальных данных (v2≈1 клип/трек) и слабый прод-эмбеддер
(не OSNet). Оценка средняя — ценность производна и пока не материализована.

## 15. Польза для аналитиков

**Прямая.** `num_tracks`/`mean_clips_per_track` = «сколько персонажей и как долго в кадре» — понятная метрика
контента (соло-влог vs групповое видео); группировка действий по `clip_track_id` = «кто что делает».
Ограничения: только люди; на реальных данных пока фрагментировано (v2), `-1`-клипы теряют часть субъектности.

### Итоговые оценки

- **Польза для моделей: 3/5.** Даёт action-ветке субъектность — согласованную траекторию действия конкретного
  человека, что усиливает информативность `clip_embeddings`. Но по контракту soft, ценность производна, а в
  реальном корпусе привязка ещё фрагментирована (v2 ≈1 клип/трек) и прод-эмбеддер слабее OSNet — отсюда средний балл.
- **Польза для аналитиков: 4/5.** «Сколько людей в кадре и кто что делает» — наглядная, понятная креатору
  метрика (соло vs групповой контент, лента действий по персонажу). Ограничивают «только person», `-1`-клипы
  и незрелость реальных данных.
