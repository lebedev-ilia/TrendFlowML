## 2. Резюме

`description_embedder` — **семантический эмбеддинг описания видео** (bge-large, 1024-d, L2) — более развитый
родственник `title_embedder`: умеет **чанкинг длинного текста** (разбивает описание на куски, эмбеддит каждый,
**length-weighted pooling** → 1024-d) + богатая телеметрия (encode/chunk/pool_ms, cache, fp16, device, model_digest).
**Механизм образцов:** 1024-d, L2-норма=1.0, present=1, детерминизм golden=0 (19 прогонов). **Но данные — та же MOCK-
проблема, что у title:** 5/6 видео получили **один фикстур-описание** (110 символов autogen) → эмбеддинги идентичны
(косинус 1.0), только -Q6fnPIy отличается (0.84); `n_chunks=1` (mock короткий — реальное описание давало бы несколько
чанков). Описание — **более слабый предиктор CTR, чем заголовок** (title читают, description просматривают), но
несёт ключевые слова/ссылки/контекст.

## 3. Функционал

Работает в TextProcessor:

1. Берёт описание (из метаданных Fetcher; в E2E — autogen-фикстура).
2. **Чанкинг** (avg_chunk_tokens≈25, n_chunks по длине) → **bge-large** эмбеддинг каждого чанка → **length-weighted
   pooling** → 1024-d → **L2**.
3. Пишет `_artifacts/description_embedding.npy` + tp_descemb_* мета (dim, present, l2, chunks, тайминги).

**Зачем продукту:** описание — **вторичный текст-сигнал**: ключевые слова, ссылки, хэштеги, контекст, CTA. Слабее
заголовка для CTR, но обогащает семантический профиль (SEO/тема). Model-facing 1024-d токен для Fusion.

## 4. Вход

- **description** (метаданные) — есть → чанкинг+эмбеддинг; нет → graceful (present=0). В батче — mock-фикстура.
- **bge-large checkpoint**.

## 5. Выход

- **`_artifacts/description_embedding.npy` (1024,)** L2.
- **tp_descemb_* мета:** `dim=1024`, `present`, `l2_norm`, `n_chunks`, `avg_chunk_tokens`, `pooling_length_weighted`,
  тайминги (encode/chunk/pool_ms), `cache_*`, `fp16`, `device_cuda`, `model_digest_u24`, `artifact_written`.
- **NaN-политика:** нет description → present=0, graceful.

## 6. Фичи (важное/неочевидное)

- **Чанкинг + length-weighted pooling** — ключевое отличие от title: длинное описание разбивается на чанки,
  эмбеддится, взвешенно усредняется. Правильная обработка длинного текста (title всегда 1 чанк).
- **5/6 идентичны (MOCK)** — тот же фикстур-текст на 5 видео → косинус 1.0. `n_chunks=1` (mock короткий); реальное
  описание дало бы multi-chunk.
- **Богатая телеметрия** (encode/chunk/pool_ms, cache, model_digest) — хорошая наблюдаемость производительности.
- **Английская bge-large** — RU-описания субоптимальны (как title).
- **Детерминизм golden=0** — воспроизводимо.
- **model_digest_u24** — провенанс модели в фичах (воспроизводимость).

## 7. Архитектура / алгоритм

- **bge-large + chunking + length-weighted mean-pooling → 1024-d L2.**
- **Сложность:** encode 448–1164 мс (CV 34%); чанкинг для длинных текстов.
- **Детерминизм:** golden max|Δ|=0.0.

## 8. Оптимизации

- **Чанкинг длинного текста** — не обрезает описание (в отличие от наивного truncate).
- **Length-weighted pooling** — длинные чанки весомее (осмысленно).
- **Cache-инфраструктура** (сейчас off) — задел под кэш эмбеддингов.
- **fp16/device-флаги** — гибкость прод-исполнения.
- **Детерминизм CPU**.

## 9. Слабые места

- **MOCK-описания в батче (главное)** — 5/6 идентичны (та же проблема, что title); реальные описания из Fetcher
  не поданы. n_chunks=1 (mock короткий) — реальный чанкинг не задействован.
- **Английская модель** — RU/смешанные описания субоптимальны.
- **Слабее заголовка** — описание менее предиктивно для CTR; вторичный сигнал.
- **Различимость не проверена** на реальных данных (mock скрывает).
- **1024-d непрозрачен** — нужны производные (тема/ключевые слова).

## 10. Рекомендации по улучшению алгоритма (приоритизированные)

1. **[выс.] Подать реальные описания** (Fetcher) — сейчас 5/6 mock-идентичны; без реальных данных бесполезно.
2. **[выс.] Мультиязычная модель** (bge-m3) — для RU/смешанных описаний.
3. **[сред.] Проверить чанкинг** на длинных реальных описаниях (n_chunks>1, pooling) — сейчас n_chunks=1 mock.
4. **[низ.] Включить кэш** эмбеддингов (инфра готова) — для прод-масштаба.

## 11. Рекомендации по архитектуре / связям

- **Единый эмбеддер** для title/description/comments (одна bge-модель, общее пространство) — согласованность.
- **description-эмбеддинг → Fusion** (текст-ветка) как вторичный текст-сигнал.
- **Reuse** для semantic_cluster/cosine_metrics (сравнение title↔description).
