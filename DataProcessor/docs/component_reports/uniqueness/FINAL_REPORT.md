## 2. Резюме

`uniqueness` — анализатор **внутривидео-разнообразия/повторяемости**: из CLIP-эмбеддингов строит матрицу
косинус-сходства N×N и считает, насколько видео «повторяет само себя» vs показывает разнообразный контент:
`diversity_score`, `repetition_ratio`, `effective_unique_frames`, `max_sim_to_other`, `cos_dist_next` + Otsu-
авто-порог повторов. **Это один из самых здоровых Visual-компонентов на реальных данных:** 0 NaN, все 23 NPZ
`status=ok`, метрики **различимы** (diversity_score 0.033–0.314, CV≈0.5 на уникальных видео; наглядно отделяет
монотонное короткое видео (div=0.03, eff_unique=0) от разнообразного (div=0.31, eff_unique=30)), golden Δ=0,
баг L3 (all-NaN Otsu) исправлен. **Важная оговорка именования:** это **внутривидео**-разнообразие, НЕ уникальность
относительно корпуса (последняя — в мёртвой reference-части `similarity_metrics`).

## 3. Функционал

Tier-3, после core_clip. Из N кадровых CLIP-эмбеддингов:

1. **Матрица косинус-сходства N×N** (max_frames=200 → ≤40K элементов).
2. **`max_sim_to_other`** (N,) — для каждого кадра макс. сходство с любым другим (насколько кадр «не уникален»).
3. **`repetition_ratio`** — доля кадров, чей max_sim выше Otsu-порога (повторяющиеся).
4. **`diversity_score`** — общая мера разнообразия (обратная повторяемости).
5. **`effective_unique_frames`** — сколько «эффективно уникальных» кадров (после дедупа похожих).
6. **`cos_dist_next`** (N−1,) — покадровая новизна; **Otsu-авто-порог** калибрует границу «повтор» из распределения.

**Зачем продукту:** повторяемость/монотонность — **фактор удержания**: визуально монотонное видео (одна сцена,
статика) теряет зрителя; разнообразное удерживает. `effective_unique_frames` = сколько реально разного контента.
Аналитику — «ваше видео повторяющееся/разнообразное»; модели — ось монотонности.

## 4. Вход

- **`core_clip`** (hard, no-fallback) — `frame_embeddings` + `frame_indices`; нет → `FileNotFoundError`.
- **`union_timestamps_sec`** + `frame_indices` — ось.
- Пустой frame_indices → `RuntimeError` (нет empty-пути by design; batch → status=empty без файла).

## 5. Выход

- **Скаляры (F=20):** `diversity_score`, `repetition_ratio`, `effective_unique_frames`, `effective_unique_ratio`,
  `max_sim_to_other_mean/p95`, `pairwise_sim_mean/p95`, `cos_dist_next_mean/p95`, `temporal_change_mean`,
  `n_frames`, `max_frames` + Otsu-поля (`repeat_threshold_is_otsu/used/raw/quality/min/max/bins`).
- **Массивы:** `max_sim_to_other (N)`, `cos_dist_next (N−1)`. Ось: `frame_indices`, `times_s`.
- **0 NaN** на всех видео (полностью финитный выход — редкость среди разобранных).

## 6. Фичи (важное/неочевидное)

- **`diversity_score`** — несущая фича: 0.033 (монотонное) … 0.314 (разнообразное). CV≈0.5 на уникальных видео →
  **различима** (в отличие от слабой centroid_sim из similarity_metrics, CV 4.8%). Более информативное кадрирование
  того же CLIP-сигнала.
- **`effective_unique_frames`** — «сколько реально разного»: N=12 повторяющееся видео → 0; N=119 разнообразное →
  30 (при этом видно высокую избыточность — из 119 кадров лишь 30 «уникальны»). Наглядный сигнал.
- **`repetition_ratio`** 0.565–1.0 — доля повторяющихся кадров; 1.0 = всё повторяется (статичное видео).
- **Otsu-авто-порог** — калибрует границу «повтор» из самого распределения (не фикс-порог) — адаптивно; но
  вырождается на all-identical (баг L3, исправлен guard'ом `return 0.97, 0.0`).
- **Оговорка «uniqueness»** — здесь это **внутривидео**-разнообразие, а НЕ «уникальность vs другие видео»
  (последнее — в `similarity_metrics.uniqueness_*`, мёртвом). Имя вводит в заблуждение (см. §11).

## 7. Архитектура / алгоритм

- **Чистый CPU** (numpy): матрица N×N cosine, Otsu-порог, редукции. N≤200 → ≤40K элементов (~160KB) — ничтожно.
- **Детерминизм:** golden max|Δ|=0.0 на 23 парах (чистый numpy).
- **Reuse core_clip** — эмбеддинги считаются один раз для всех downstream.
- L3-фикс: guard против all-NaN sigma_b2 в Otsu.

## 8. Оптимизации

- **Reuse CLIP-эмбеддингов** — не эмбеддит кадры заново.
- **max_frames=200 cap** — ограничивает N×N (квадратичную) стоимость.
- **Otsu-авто-порог** — адаптивная калибровка без ручного порога.
- **Полностью финитный выход** (0 NaN) — чистый контракт для Encoder (не нужна маска).

## 9. Слабые места

- **Именование vs `similarity_metrics` (главное концептуальное).** «uniqueness» здесь = **внутривидео**-
  разнообразие, а не уникальность относительно корпуса. Настоящая корпус-уникальность (`similarity_metrics.
  uniqueness_*`) мертва (нет reference pack). Два компонента с «uniqueness» в разных смыслах → путаница/дубль.
- **Один концептуальный сигнал** — по сути одна ось (разнообразие/повторяемость) в 20 производных фичах; много
  коррелирующих величин (diversity vs repetition vs effective_unique — сильно связаны).
- **Только визуал (CLIP)** — не учитывает разнообразие звука/темпа/сюжета; «монотонность» лишь картиночная.
- **max_frames=200 cap** — на длинных видео диверсити считается по первым/сэмплированным 200 кадрам.
- **Otsu на малом N** — на N=12 порог/diversity шумноваты (короткое видео = мало точек для распределения).

## 10. Рекомендации по улучшению алгоритма (приоритизированные)

1. **[выс.] Развести именование** — переименовать в `intra_video_diversity` (или явно задокументировать), чтобы
   не путать с корпус-уникальностью `similarity_metrics`; объединить/разграничить два «uniqueness».
2. **[сред.] Проредить 20 коррелирующих фич** — оставить ядро (diversity_score, effective_unique_ratio,
   repetition_ratio, temporal_change) + Otsu-мета; убрать дубли.
3. **[сред.] Мультимодальное разнообразие** — расширить на audio/pacing (монотонность не только визуальная).
4. **[низ.] Адаптивный max_frames** для длинных видео (или сегментное diversity).
5. **[низ.] low_confidence-флаг** на коротких N (Otsu шумит).

## 11. Рекомендации по архитектуре / связям

- **Объединить с `similarity_metrics`** — оба про similarity над CLIP; uniqueness = intra-video half,
  similarity_metrics = coherence + reference. Единый similarity-движок убрал бы дубль uniqueness-концепции.
- **Reuse core_clip закреплён** — правильно.
- **Diversity как контекст** — Encoder/аналитика могут связывать монотонность с удержанием.

## 13. Интерпретируемость

- **Отличная:** «повторяющееся vs разнообразное видео», «из N кадров лишь M уникальны» — прямой понятный инсайт.
- **Добавить:** словами «ваше видео визуально монотонное (повтор 85%)»; timeline новизны (cos_dist_next) с
  пиками смены; сравнение с медианой ниши (когда будет reference).

## 14. Польза для моделей

**Умеренная.** Монотонность/разнообразие — правдоподобный фактор удержания (монотонное → drop-off), сигнал
**различим** (CV~0.5) и **чист** (0 NaN, детерминирован) — хорошая форма для Encoder. Ограничивает: одна
концептуальная ось (несколько коррелирующих фич), только визуал. Крепкое «приемлемо-хорошо» как одна из осей.

## 15. Польза для аналитиков

**Хорошая.** «Ваше видео повторяющееся/разнообразное», «сколько уникального контента» — понятный, различимый,
**действенный** инсайт (можно посоветовать разнообразить). Работает на реальных данных (в отличие от корпус-
уникальности similarity_metrics). Ограничивает: только визуальная монотонность + путаница с «vs другие видео».
