## 2. Резюме

`loudness_extractor` — **измеритель громкости** аудио: RMS, peak, dBFS и (опционально) LUFS + per-segment/per-frame
RMS-статистики (mean/std/median/p10/p90). **Чистый, простой, полностью живой** компонент: все 6 видео ok, метрики
**различимы** (dBFS −10.6…−21.1 — от громкого к тихому), 18 фич, golden diff 6.7e-8 (float32), багов не найдено.
**Единственный пробел:** `loudness_lufs` = **NaN на всех видео** — библиотека `pyloudnorm` не установлена
(`lufs_present=False`). Иронично: **LUFS — самая продуктово-важная метрика громкости** (вещательный перцептивный
стандарт, к которому YouTube нормализует аудио ~−14 LUFS), и именно она недоступна. Базовые RMS/dBFS/peak работают.

## 3. Функционал

Работает после Segmenter (аудио). DSP-подсчёт:

1. **Video-level:** `loudness_rms`, `loudness_peak`, `loudness_dbfs`, `loudness_lufs` (опц.).
2. **Frame-level RMS-статистики:** `frame_rms_mean/std/median/p10/p90`, `frames_count`.
3. **Segment-level:** `segment_rms/dbfs/lufs/peak` (массивы) + агрегаты `segment_rms_mean/std/median/p10/p90`.
4. `duration_sec`, `sample_rate`, `segments_count`.

**Зачем продукту:** громкость — **фактор качества и удержания**: слишком тихое аудио = плохой опыт (зритель
уходит/крутит громкость), пере-компрессированное/громкое = утомляет. YouTube нормализует к ~−14 LUFS — отклонение
влияет на воспринимаемое качество. `segment_rms_std` — динамика громкости (ровное vs скачущее). Прямой actionable
инсайт креатору («ваше аудио тихое»).

## 4. Вход

- **Аудио** (Segmenter) — нет аудио → `status=empty` (`audio_missing_or_extract_failed`).
- **pyloudnorm** (опц.) — для LUFS; не установлена → `loudness_lufs=NaN`, `lufs_present=False`.

## 5. Выход

- **18 фич** (video + frame + segment RMS-статистики), см. §3.
- **Массивы сегментов:** `segment_rms/dbfs/lufs/peak`, `segment_start/center/end_sec`, `segment_mask`.
- **`lufs_present`** флаг. Ось: `segment_*_sec`.
- **NaN-политика:** `loudness_lufs`/`segment_lufs` = NaN (pyloudnorm off); empty → fv NaN×18.

## 6. Фичи (важное/неочевидное)

- **`loudness_dbfs`** — основная различимая метрика: −10.6 (громкое) … −21.1 (тихое) на корпусе. Понятна, actionable.
- **`loudness_peak`** — пик амплитуды (0.19…1.0); peak=1.0 = клиппинг/полная шкала (нормально, но 1.0 + rms высокий = возможна пере-компрессия).
- **`segment_rms_std`** — **динамика громкости**: 0.0003 (тихое ровное) … 0.099 (скачущая громкость). Различает
  ровный vs динамичный звук — полезнее среднего RMS.
- **`loudness_lufs` = NaN by design (pyloudnorm off)** — перцептивная громкость (учитывает частотную
  чувствительность уха), стандарт вещания. Её отсутствие — главный пробел (см. §9). `lufs_present`-флаг честно сигналит.
- **Frame vs segment RMS** — две гранулярности (кадры ~мс, сегменты ~1с); frame_rms_std ловит кратковременные всплески.

## 7. Архитектура / алгоритм

- **Чистый DSP** (numpy): RMS = √mean(x²), dBFS = 20·log10(rms), peak = max|x|; LUFS через pyloudnorm (K-weighting).
- **Сложность:** O(samples), тривиально дёшев, CPU.
- **Детерминизм:** golden max|Δ|=6.7e-8 (float32-арифметика без OMP-пиннинга — как у librosa-компонентов; на
  качество не влияет).

## 8. Оптимизации

- **Чистый numpy** — GPU/модели не нужны; тривиальная стоимость.
- **Три гранулярности** (video/frame/segment) — богатый профиль без переусложнения.
- **pyloudnorm опционален** — не блокирует пайплайн, честный `lufs_present`-флаг вместо падения.

## 9. Слабые места

- **LUFS мёртв на всех данных (главное).** `pyloudnorm` не установлена → `loudness_lufs`/`segment_lufs`=NaN.
  Это **самая ценная метрика** (перцептивная, стандарт YouTube-нормализации −14 LUFS) — и именно она недоступна.
  Установка одной библиотеки разблокирует её.
- **Одномерность концепции** — 18 фич вокруг одной оси (уровень громкости) + гранулярности; сильно коррелируют
  (rms↔dbfs — монотонная связь).
- **Peak=1.0 не различает** норму от клиппинга без доп. метрики (true-peak/clipping-ratio).
- **Только уровень, не качество** — не ловит искажения/шум/частотный баланс (это в quality/spectral-экстракторах).
- **golden не строго 0** (6.7e-8) — норма float32, но для побитового golden нужен OMP-пиннинг.

## 10. Рекомендации по улучшению алгоритма (приоритизированные)

1. **[выс.] Установить pyloudnorm** — разблокировать LUFS (integrated + short-term); это самая важная громкостная
   метрика (YouTube −14 LUFS), сейчас NaN из-за одной отсутствующей зависимости.
2. **[сред.] Добавить отклонение от −14 LUFS** как готовый actionable-скаляр («ваше аудио на X dB тише нормы»).
3. **[сред.] True-peak / clipping-ratio** — отличать peak=1.0 клиппинг от чистого пика.
4. **[низ.] Проредить коррелирующие RMS-фичи** — rms/dbfs/median сильно связаны.

## 11. Рекомендации по архитектуре / связям

- **Reuse аудио-декода** с другими audio-экстракторами (общий 16/48kHz буфер).
- **Связка с quality_extractor/spectral** — громкость + качество + спектр = полный аудио-технический профиль.
- **LUFS → UI-нормализация** — показать креатору «громкость vs стандарт платформы».

## 13. Интерпретируемость

- **Отличная:** «аудио тихое/громкое (dBFS)», «ровное/скачущее (rms_std)» — прямо понятно и actionable.
- **Добавить:** «громкость −18 dB, ниже стандарта YouTube на 4 dB» (после LUFS); индикатор клиппинга.

## 14. Польза для моделей

**Умеренная.** Уровень громкости и его динамика — модест-сигнал качества/удержания (тихое/пере-компрессированное
аудио ↔ опыт). Различим и чист. Ограничивает: одна концептуальная ось (коррелирующие фичи), отсутствие LUFS
(перцептивная норма). Полезно как техническая аудио-фича, не сильная.

## 15. Польза для аналитиков

**Хорошая (и была бы отличной с LUFS).** Громкость — прямой actionable инсайт («ваше аудио тихое / скачет по
громкости»). dBFS/RMS работают и понятны. Главный пробел — LUFS (стандарт нормализации YouTube), самая ценная
для креатора метрика, сейчас NaN. После установки pyloudnorm — одна из наглядных технических аналитик.
