## 2. Резюме

`speaker_diarization_extractor` — **диаризация дикторов** («кто говорит когда») на **pyannote**: определяет число
говорящих, размечает **реплики (turns)** с диктором, считает баланс/доминирование, плотность и переходы между
дикторами. На реальном корпусе **функционален и различим**: все 6 видео ok, speaker_count ∈ {0,1,2} чётко
разделяет **диалог** (sc=2, 3 реплики, 1 переход, balance 0.96), **монолог** (sc=1) и **не-речь/музыку** (sc=0,
dominant=−1); turn-разметка побайтово стабильна между прогонами (golden PASS). **Ключевое ограничение:**
**эмбеддинги дикторов ОТКЛЮЧЕНЫ** (Audit v3 no-op) — нет voice-print для downstream (связывание дикторов между
сегментами/видео). Плюс на этом корпусе мало дикторов (talking-head/музыка, диалогов почти нет) и инфра-хрупкость
(pyannote/torch конфликты, модель не на Network Volume).

## 3. Функционал

Работает после Segmenter (аудио). pyannote выдаёт:

1. **Число дикторов** `speaker_count` + разметку реплик `turn_start/end_sec` + `turn_speaker_id`.
2. **Баланс/доминирование:** `speaker_balance_score` (насколько равномерно распределено время), `dominant_speaker_id`,
   `speaker_time_ratio`, `speaker_duration_sec`, `speaker_turns_count_by_speaker`.
3. **Динамика:** `speaker_turns_count`, `speaker_turns_density`, `speaker_transitions_count`.
4. **Аудио-стат:** rms, peak, duration, sample_rate (16kHz).

**Зачем продукту:** структура диалога — **сильный сигнал формата и вовлечённости**: интервью/подкасты (много
дикторов, частые переходы) vs монолог-влог (1 диктор) vs музыка (0). Баланс дикторов — кто доминирует в разговоре.
Плотность реплик — динамичность беседы. Кросс-модально связывается с лицами (кто на экране говорит).

## 4. Вход

- **Аудио** (Segmenter, 16kHz) — нет аудио → `status=empty` (`audio_missing`).
- **pyannote checkpoint** (community-1, inprocess через ModelManager).

## 5. Выход

- **Табличные (10):** `speaker_count`, `duration_sec`, `sample_rate`, `rms`, `peak`, `speaker_balance_score`,
  `dominant_speaker_id`, `speaker_turns_count`, `speaker_turns_density`, `speaker_transitions_count`.
- **Turn-level:** `turn_start/end_sec`, `turn_speaker_id`, `turn_mask`; `speaker_ids`, `speaker_duration_sec`,
  `speaker_time_ratio`, `speaker_turns_count_by_speaker`.
- **Ось сегментов:** `segment_start/center/end_sec`, `segment_mask`.
- **`segment_speaker_embeddings` — ОТКЛЮЧЕНЫ** (Audit v3 no-op) — по контракту есть, но не заполняются.
- **NaN/degenerate:** sc=0 → K=0/S=0, dominant=−1, balance=1.0 (дефолт).

## 6. Фичи (важное/неочевидное)

- **`speaker_count` — несущая** — 0/1/2 на корпусе: чётко «музыка / монолог / диалог». Дискриминативна для формата.
- **`speaker_transitions_count` + `turns_density`** — динамика беседы: интервью с частыми перебивками vs
  размеренный диалог. На корпусе 0–1 (мало диалогов).
- **`speaker_balance_score`** — распределение времени между дикторами; **осмысленна только при sc≥2** (при sc≤1
  = 1.0 дефолт, вырождена — не путать с «идеальным балансом»).
- **`dominant_speaker_id=−1` при sc=0** — корректный маркер «нет дикторов» (не диктор с id −1).
- **Эмбеддинги дикторов отключены** — `segment_speaker_embeddings` no-op: **нет voice-print** → нельзя связать
  «этот голос = диктор X» между видео или с лицами. Существенное ограничение для мультимодального speaker-linking.

## 7. Архитектура / алгоритм

- **pyannote/speaker-diarization-community-1** (сегментация + кластеризация голосов), inprocess, ~1x realtime GPU.
- **Детерминизм:** sc=0 идентичен между прогонами; sc>0 — turn-времена побайтово стабильны (golden PASS).
- **Сложность:** ~160с на 149с аудио (1x realtime); повторная загрузка модели доминирует на коротких (5.5x на 28с).

## 8. Оптимизации

- **community-1 (33MB)** — компактная модель.
- **Turn-level + agg** — и разметка реплик, и сводные метрики.
- **inprocess ModelManager** — без сетевой инференс-зависимости.
- **Segment-mask/degenerate-дефолты** — чистая обработка sc=0.

## 9. Слабые места

- **Эмбеддинги дикторов отключены (главное).** `segment_speaker_embeddings` = no-op (Audit v3) → нет voice-print;
  downstream (`speaker_turn_embeddings_aggregator` в Text, связывание голос↔лицо) не может идентифицировать/связывать
  дикторов. Теряется мультимодальный потенциал.
- **Мало дикторов на корпусе** — sc 0–2 (talking-head/музыка); диалоговых/подкаст-видео почти нет → transitions/
  density слабо нагружены; различимость динамики беседы на этих данных бедна.
- **`balance_score` вырождён при sc<2** (=1.0 дефолт) — легко принять за «идеальный баланс»; нужен NaN/флаг.
- **Инфра-хрупкость** — pyannote 4.0.7 ↔ torch 2.13 ↔ torchvision конфликты; модель не на Network Volume (scp
  вручную). Риск для прод-масштаба.
- **Повторная загрузка модели** на каждый видео (5.5x realtime на коротких) — нет прогретого воркера.
- **Только аудио** — не связано с лицами (кто на экране) без voice-print.

## 10. Рекомендации по улучшению алгоритма (приоритизированные)

1. **[выс.] Включить эмбеддинги дикторов** — voice-print для speaker-linking (голос↔лицо, диктор между видео);
   сейчас no-op обесценивает мультимодальную ветку.
2. **[выс.] Прогретый pyannote-воркер** для прода — сейчас перезагрузка модели доминирует (5.5x realtime на коротких).
3. **[сред.] `balance_score`=NaN/флаг при sc<2** — не выдавать вырожденную 1.0 как реальный баланс.
4. **[сред.] Стабилизировать инфру** — зафиксировать pyannote/torch/torchvision версии, положить модель в bundled.
5. **[низ.] Связать с лицами** (core_face_landmarks/emotion) — «кто из говорящих на экране» через voice-print + bbox.

## 11. Рекомендации по архитектуре / связям

- **Voice-print → speaker_turn_embeddings (Text) + face-linking** — включить эмбеддинги и закрепить контракт.
- **Reuse аудио** с другими audio-экстракторами (общий декод 16kHz).
- **Связка с ASR** — diarization + ASR = «кто что сказал» (turn_speaker_id × транскрипт) — сильный совместный сигнал.

## 13. Интерпретируемость

- **Отличная:** «в видео N говорящих», «диалог/монолог», «кто доминирует» — прямо понятно.
- **Добавить:** таймлайн реплик по дикторам (turn_speaker_id по времени); «интервью: 2 спикера, 50/50» —
  наглядная диаграмма; связка «говорит человек на экране» (после voice-print).

## 14. Польза для моделей

**Умеренная.** Число дикторов, динамика реплик, баланс — осмысленные сигналы формата (подкаст/интервью/влог),
различимые на корпусе. Ограничивают: мало дикторов на данных (0–2), отключённые эмбеддинги (нет voice-фичи для
Encoder), вырожденный balance при sc<2. Полезно как формат-признак, не более, в текущем виде.

## 15. Польза для аналитиков

**Хорошая.** «Сколько говорящих, диалог vs монолог, кто доминирует, динамика беседы» — понятные и полезные метрики
(особенно для интервью/подкастов). Ограничивают бедность диалогов в корпусе и отсутствие voice-linking «кто есть кто».
