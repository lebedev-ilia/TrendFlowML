## 2. Резюме

`comments_embedder` — **семантический эмбеддинг комментариев** видео (multilingual-e5-large, 1024-d, L2) → `(N,1024)`
эмбеддинги отобранных комментариев; потребитель — `comments_aggregator`. Механизм солидный (мультиязычный, L2,
golden=0) с **продуманным дизайном взвешивания** (маски authority/likes/recency — вес комментария по авторитету
автора/лайкам/свежести). **Но данные — MOCK** (как title/desc/hashtag): 3 фикстур-комментария на каждом видео
(count=3) → **все 6 эмбеддингов идентичны** (косинус 1.0). Комментарии — **сигнал реакции аудитории** (что думают
зрители, сентимент, качество вовлечённости), богатый источник. **Важная оговорка:** комментарии появляются **после
публикации** → **риск утечки (leakage)** при предсказании просмотров/лайков (комментарии — следствие просмотров).
Нужны реальные комментарии + leakage-осознанное использование.

## 3. Функционал

Работает в TextProcessor, после Fetcher (комментарии):

1. Отбирает комментарии (top-N по важности) → `comments_selected_indices`.
2. **multilingual-e5-large** эмбеддинг каждого → 1024-d → **L2** → `(N,1024)`.
3. **Взвешивание** (задел): маски authority/likes/recency для агрегации в comments_aggregator.
4. Пишет `comments_embeddings.npy` + tp_commentsemb_* мета.

**Зачем продукту:** комментарии — **голос аудитории**: сентимент (позитив/негатив/споры), темы обсуждения, качество
реакции. Богатый сигнал восприятия. Взвешивание по лайкам/авторитету → важные комментарии весомее. Кормит
comments_aggregator → Fusion.

## 4. Вход

- **комментарии** (Fetcher) — есть → эмбеддинг; нет → present=0. В батче — 3 mock-комментария.
- **multilingual-e5-large checkpoint**.

## 5. Выход

- **`_artifacts/comments_embeddings.npy` (N,1024)** L2 + `comments_selected_indices.npy`.
- **tp_commentsemb_*/tp_comments_agg_* мета:** present, dim=1024, count=3, compute_mean/median/std,
  weights_mask_{authority,likes,recency}, weights_applied.
- **NaN-политика:** нет комментариев → present=0; extra-ключи NaN by design (emit_extra=False).

## 6. Фичи (важное/неочевидное)

- **Все 6 идентичны (MOCK)** — 3 фикстур-комментария на всех → косинус 1.0. Нулевой реальный сигнал (как title/desc/hashtag).
- **Дизайн взвешивания (authority/likes/recency)** — продуманно: вес комментария по авторитету автора, лайкам,
  свежести. `weights_applied=0` на mock, но инфраструктура готова — сильная идея (важные комментарии весомее).
- **multilingual-e5-large** — RU/EN-комментарии корректно (как transcript; лучше bge-en).
- **top-N отбор** (`comments_selected_indices`) — не все комментарии, а важнейшие (масштабируемость).
- **Consumer comments_aggregator** — этот компонент даёт сырые эмбеддинги, агрегация отдельно.

## 7. Архитектура / алгоритм

- **multilingual-e5-large** (1024-d, L2) на отобранных комментариях + задел взвешивания.
- **Детерминизм:** golden max|Δ|=0.0 (19 прогонов + 2 инстанса).
- **Сложность:** e5-large на N комментариях; CPU.

## 8. Оптимизации

- **Мультиязычная модель** — RU/EN.
- **Взвешивание** (authority/likes/recency) — приоритет важных комментариев.
- **top-N отбор** — не эмбеддить все (масштаб).
- **Детерминизм**, отдельный артефакт.

## 9. Слабые места

- **MOCK-данные (главное)** — все 6 идентичны (3 фикстур-комментария); реальные комментарии из Fetcher не поданы.
- **LEAKAGE-риск (важно)** — комментарии появляются **после** публикации; при предсказании просмотров/лайков это
  утечка (комментарии — следствие метрик). Нужна leakage-осознанная политика (только ранние комментарии на момент прогноза, или исключить).
- **Взвешивание не задействовано** (weights_applied=0 на mock) — сильный дизайн не проверен.
- **1024-d непрозрачен** — ценность через сентимент/темы (в comments_aggregator/производных).
- **Зависит от Fetcher** — качество/объём комментариев вариативны.

## 10. Рекомендации по улучшению алгоритма (приоритизированные)

1. **[выс.] Подать реальные комментарии** (Fetcher) — сейчас mock (все идентичны).
2. **[выс.] Leakage-политика** — определить, какие комментарии доступны на момент прогноза (2/3 недели); исключить
   пост-прогнозные, иначе утечка в предсказание просмотров/лайков.
3. **[сред.] Задействовать взвешивание** (authority/likes/recency) на реальных данных — важные комментарии весомее.
4. **[низ.] Сентимент/темы** из эмбеддинга для аналитика.

## 11. Рекомендации по архитектуре / связям

- **comments_embedder → comments_aggregator → Fusion** — с leakage-контролем.
- **Единая multilingual-модель** для transcript/comments (уже e5) — согласованность; распространить на title/desc.
- **Взвешивание по лайкам/авторитету** — закрепить как модель важности комментариев.

## 14. Польза для моделей

**Низкая-умеренная (потенциал), нулевая (факт).** Комментарии — голос аудитории (сентимент/реакция), богатый сигнал,
с продуманным взвешиванием. **Но** данные mock (все идентичны) → нулевой вклад, а **leakage-риск** (комментарии
пост-публикация) делает их опасными для предсказания просмотров без временно́й фильтрации. Реальная польза требует
и реальных данных, и leakage-политики.

## 15. Польза для аналитиков

**Умеренная (потенциал).** «Что думает аудитория» (сентимент/темы комментариев) — ценнейший инсайт для креатора
(реакция на контент). Взвешивание по лайкам/авторитету — умно. Но на mock недоступно; leakage для аналитики не
проблема (это ретроспектива). После реальных комментариев — сильная аналитика восприятия.
