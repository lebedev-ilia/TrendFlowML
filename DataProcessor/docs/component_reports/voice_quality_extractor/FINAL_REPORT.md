## 2. Резюме

`voice_quality_extractor` — **измеритель качества голоса**: клинические метрики **jitter** (пертурбация высоты =
хрипота), **shimmer** (пертурбация амплитуды), **HNR** (harmonic-to-noise ratio = придыхательность/шумность
голоса), vq_f0 и композитный quality (29 фич). **Ключевая находка — расхождение отчёта и данных:** алгоритм
**работает** (в отчёте re-run с yin+16000 дал реальные значения на 5 видео: jitter 0.0002–0.454, vq_f0 75.9–440 Hz,
HNR ~−0.15 dB, quality до 0.83), **но все 6 storage-NPZ ПУСТЫ** (`voice_quality_all_segments_failed`, 24/29 NaN).
То есть в реальном батче компонент **упал на каждом сегменте всех видео** — фактического сигнала нет; отчёт
валидировал пере-прогоном с рабочим методом. Клинически осмысленный дизайн (хрипота/придыхательность — реальный
голосовой сигнал), но нулевой факт в данных.

## 3. Функционал

Работает после Segmenter (аудио). DSP-анализ голоса:

1. **F0-трекинг** (yin) → период → **jitter** (вариация периода = хрипота/нестабильность высоты).
2. **Амплитудная огибающая** → **shimmer** (вариация амплитуды = дрожание громкости).
3. **HNR** — отношение гармонической энергии к шумовой (высокий = чистый голос, низкий = придыхательный/шумный).
4. Композит `voice_quality` + per-segment `jitter/shimmer/hnr_by_segment`.

**Зачем продукту:** качество голоса — **сигнал приятности/профессиональности речи**: хриплый/дрожащий/придыхательный
голос менее приятен, чистый стабильный — располагает. Клинические метрики (jitter/shimmer/HNR) — стандарт оценки
голоса. Для говорящих-голов (влоги/лекции) — качество подачи.

## 4. Вход

- **Аудио** (Segmenter) — два empty-пути: нет аудио и **`voice_quality_all_segments_failed`** (все сегменты упали

## 5. Выход

- **29 табличных фич:** jitter/shimmer/HNR (+ статистики), vq_f0, voice_quality + мета.
- **Per-segment:** `jitter/shimmer/hnr_by_segment`, `segment_*_sec`, `segment_mask`.
- **NaN-политика:** all_segments_failed → 24/29 NaN (на storage — все видео).

## 6. Фичи (важное/неочевидное)

- **jitter/shimmer/HNR — клинические метрики голоса** (стандарт фониатрии): jitter = хрипота, shimmer = дрожание,
  HNR = придыхательность. Осмысленный, специализированный набор — **не избыточен** со спектральными фичами.
- **В storage все NaN (all_segments_failed)** — на реальном корпусе компонент не дал ни одной валидной оценки.
- **Алгоритм работает при рабочем методе (re-run)** — отчёт: vq_f0 75.9–440 Hz (совпадает с pitch: -4WRepA 76≈78 Hz,
  -Q6fnPIy 440≈441 Hz), jitter 0.0002 (стабильный тон -Q6) … 0.454 (шумная речь). То есть логика верна, но батч сломан.
- **HNR ~−0.15 dB (в re-run)** — низкий/отрицательный HNR подозрителен (обычно голос имеет положительный HNR); возможна
  проблема калибровки/метода даже в re-run.

## 7. Архитектура / алгоритм

- **librosa/DSP:** yin F0 → jitter (RAP/PPQ), амплитуда → shimmer (APQ), гармоники/шум → HNR.
- **Сложность:** дёшев, CPU.
- **Детерминизм:** заявлен PASS (в re-run).
- **Батч-отказ:** все сегменты падают на storage-конфиге (метод/параметры отличаются от рабочего re-run yin+16000).

## 8. Оптимизации

- **Чистый DSP** — без GPU.
- **Per-segment + agg** — качество голоса по времени.
- **Клинический набор** (jitter/shimmer/HNR) — специализированный, не дубль.
- **Гранулярный empty-reason** (`all_segments_failed`) — честный маркер отказа.

## 9. Слабые места

- **100% empty в реальном батче (главное).** Все 6 storage-видео `voice_quality_all_segments_failed`, 24/29 NaN.
  Компонент **не произвёл ни одной валидной оценки** на корпусе — батч-исполнение сломано (отчёт валидировал
  пере-прогоном с другим методом). Фактическая польза = 0 до починки батч-конфига.
- **Расхождение отчёт↔данные** — штамп основан на re-run (yin+16000), а storage-артефакты пусты; штамп не отражает
  реального состояния данных.
- **Подозрительный HNR** (~−0.15 dB даже в re-run) — отрицательный HNR для голоса нефизичен; возможна проблема метода.
- **Клиническая опаковость** — jitter/shimmer/HNR непонятны креатору без перевода в «хриплый/чистый голос».
- **Только вокализованное аудио** — на музыке/не-речи неопределён.
