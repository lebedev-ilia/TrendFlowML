## Что исправлено относительно v1 (главное) ✅

- **Фрагментация треков устранена.** Было 205/199 суррогат-треков (все 1-клиповые) на synthetic-длинных;
  стало **26/31** реальных appearance-треков со средней длиной **14.7/16.0** детекций. `frac_single_len`
  0.19/0.13 (мало одно-кадровых). Корневая причина (убранный трекинг) закрыта твоим embedding-трекером.
- **Контроль без людей → валидный `empty`** (в v1 был невалиден: детектор находил person). Теперь
  на реальном people-free ролике — 0 треков, корректный `no_person_detections`.
- **Плоский v3-контракт проходит** component-scoped валидатор на всех прогонах (вкл. valid-empty).
  `clip_embeddings (C,256)` L2, `clip_times_s` монотонны, `clip_track_id` присутствует.
- **Классы Kinetics работают:** `classes_available=true` — значит тап `raw[:, :400]` реально снял
  логиты головы SlowFast (softmax→top-k без хирургии над forward, как и планировалось).
- **Стоимость трекера мала:** overhead ~4% от стадии детекции (2.3 s / 53.6 s на 4:35). Разделимость
  id хорошая: intra ≫ inter (0.58 vs 0.21 на 4:35).

## Оценка по 4 осям

- **Корректность:** ✅ контракт v3 pass, эмбеддинги L2/finite, времена ⊆ union, классы снимаются.
- **Стабильность:** ✅ golden идентичен (детерминизм трекера + fp32).

## Проблемы (найденные, по данным NPZ)

1. **🟡 `mean_clips_per_track=1.0` + разреженное покрытие.** `clip_track_id=[0..21]` — по одному клипу
   на трек, всего 22 клипа на 4:35. Причина двойная: (а) окно = `clip_len` (32) → **1 клип на окно**;
   (б) `max_windows=48` → покрытие таймлайна разрежено (~48×1.3 s из 275 s). Для **per-clip stream**
   (наш дизайн — per-track динамику депрецировали) это не фатально: 22 action-токена по видео —
   валидный разреженный поток. Но плотность действий низкая для аналитики/модели.
2. **🟠 Паддинг коротких клипов размывает сигнал.** Только **4 из 26** треков имеют ≥32 кадров
   (`60,58,47,37`); остальные короче → `_make_clips` **паддит** (повтор последнего кадра) до 32.
   Значит **18 из 22** клипов построены из <32 реальных кадров (часть — почти статичные) → их
   action-эмбеддинг/логиты шумны. Окна ставятся по всему таймлайну независимо от людей, поэтому
   человек попадает лишь в часть окна → короткий трек → padded-клип.
3. **🟠 Метки Kinetics — placeholder** (`action_<id>`). `dominant_action_ids=[225,244,232,233,221,…]`
   без имён нельзя семантически сверить с содержимым (владелец не подтвердит правдоподобие классов).
4. **🟠 Диск для 200k.** Плотные окна + union-кадры → ~10 GB/видео кэша frames (8:00 упал на 100% fill).
   Для 200k нужна policy: не хранить frames после NPZ / 2-проходная плотность только вокруг людей.
5. **🟡 Embedder histogram** достаточен по cosine-разделимости на 4:35, но на 8:00 (inter 0.30)
   OSNet вероятно улучшит ReID; на длинном/толпном контенте риск слияния id.

## Рекомендации (доработка, до штампа)

**R1 (плотность/паддинг — главное).** Развязать клипы и окна + гарантировать реальные кадры:
- сэмплировать **несколько перекрывающихся клипов на окно** (stride < clip_len) там, где трек длинный;
- **минимум реальных кадров на клип** (напр. ≥16/32); иначе клип не эмитить (или флажок `padded=true`);
- **2-проходность окон вокруг людей:** сначала детекция/треки, затем плотные окна **центрируются на
  person-присутствии** (чтобы трек занимал полное 32-кадровое окно, а не хвост) — убирает и паддинг,
  и лишние people-free окна (экономит диск).
- `max_windows` — сделать **адаптивным к длительности** (для 8:00 больше окон), с явным cost-cap.

**R2 (метки Kinetics).** Положить `kinetics400_labels.txt` (400 строк) в
`dp_models/visual/action_recognition/` → перепрогнать сборку `class_names` → владелец сверяет
top-действия с роликами (единственное, что сейчас блокирует семантическую проверку).

**R3 (диск/200k).** Policy «frames удаляются после NPZ» + 2-проходные окна (R1) резко снижают кэш.
Зафиксировать в LOAD_AND_SCALING_PLAN как обязательное для dense-компонентов.

**R4 (embedder).** Оставить histogram для короткого/среднего; для длинного/толпного — прогнать OSNet
и сравнить inter_cos. Решение по дефолту — по метрике.

**R5 (baseline-ablation).** Оценить предсказательную ценность action-стрима (со/без) на baseline —
отдельная задача Models-workstream.
