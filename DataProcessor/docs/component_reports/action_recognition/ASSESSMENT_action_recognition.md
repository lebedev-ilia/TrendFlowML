## 1. Что улучшить до «почти идеального»

### 1.1 🔴 Главное: качество эмбеддинга-токена (для Encoder)
Сейчас 256-d `clip_embeddings` = `L2(embedding_proj(raw))`, где `embedding_proj` —
**необученный** `nn.Linear(2048→256)` (xavier), а `raw` = выход головы SlowFast = **400 логитов
Kinetics**, дополненные нулями до 2048. То есть токен модели — это **фиксированная случайная
проекция 400 логитов**. Он детерминирован (ок для golden), но **несёт не больше информации, чем
сами логиты**, и теряет богатые визуально-моторные признаки backbone.
**Фикс:** снимать **penultimate-фичи** SlowFast (pooled features ДО классификационной головы,
~2304-d для slowfast_r50) через forward-hook и отдавать их (или обученную проекцию) как эмбеддинг.
Это даёт модели настоящий представленческий токен, а не перемешанные логиты. Логиты/классы
оставить отдельным семантическим сигналом (уже есть). **Наивысший приоритет для Models.**

### 1.2 🟠 Плотность/семантика клипов (переосмысление `mean_clips_per_track`)
`mean_clips_per_track=1.0` — не дефект для **плоского per-clip stream** (мы осознанно
депрецировали per-track динамику): 35 токенов на 4:35 = ~1 токен/8 с, клипы — полные 32-кадровые
окна (не padded, т.к. треки теперь длинные), классы осмысленны. Что реально улучшить:
- **temporal action localization** вместо равномерных окон: ставить клипы на границы действий
  (сменах активности), а не по сетке — плотнее там, где «что-то происходит».
- **per-person tubelet**: сейчас SlowFast видит **весь кадр** окна, а привязка к треку — только
  выбор окна. Для истинного «действия конкретного человека» — кропать облако боксов трека
  (tubelet) и гнать SlowFast по нему. Тогда `clip_track_id` становится семантически точным, и на
  групповых сценах различаются действия разных людей.
- если Models захотят **per-track траекторию** — включить multi-clip скольжением внутри длинного
  трека (инфраструктура готова: `stride`, `clip_track_id`).

### 1.3 🟠 ReID-эмбеддер трекера
Histogram-HSV достаточен на 4:35 (intra 0.58 ≫ inter 0.21), но на 8:00 inter=0.30 (толпа/похожая
одежда) — риск слияния id. **OSNet/TransReID** заметно усилит разделимость на длинном/многолюдном
контенте. Ветка `--track-embedder osnet` уже предусмотрена (сейчас fallback→histogram).

### 1.4 🟡 Прочее
- Порог `min_person_confidence=0.3` и `min_clip_real_frames=16` — вынести в конфиг/провалидировать
  на размеченных примерах (сейчас эвристики).
- Мультимасштабность действий: короткие микро-действия (<32 кадров) теряются — опц. второй масштаб
  clip_len (16) для быстрых движений.

## 2. Оптимизация (GPU/CPU, масштаб, параллельность, алгоритмы)

**Что уже хорошо:**
- Батч-инференс клипов на GPU (`batch_size`, fp32) + per-clip fallback при OOM. ✓
- Трекер: overhead ~4% от детекции, ассоциация O(T·K) на кадр — дёшево. Детерминирован. ✓
- Компонент **stateless на видео** → горизонтально масштабируется (k8s несколько воркеров +
  DynamicBatch). ✓

**Что оптимизировать:**
- 🔴 **Стоимость dense-детекции (↑3–5×) + диск 11 GB/видео.** Выравнивание детекции под окна дало
  качество, но дорого для 200k. Оптимальнее: **2-проходно** — сначала разреженная детекция → находим
  person-интервалы → плотные окна СТАВИМ только вокруг них (а не по всему таймлайну). Убирает
  people-free плотные окна и снижает и compute, и диск. Плюс **policy «frames удаляются после NPZ»**
  (обязательно для 200k — сейчас кэш не чистится автоматически).
- 🟠 **fp16/bf16 инференс** SlowFast (сейчас fp32) — ~1.5–2× скорость на GPU без заметной потери.
- 🟠 **Векторизовать histogram-эмбеддер** (сейчас python-цикл по боксам; cv2.calcHist на батче/ROI).
- 🟠 **Единый декод кадров:** dense-окна декодируются и детекцией, и action — организовать общий
  кэш кадров окна (уже частично через union), не читать дважды.
- 🟡 **Батч-путь `process_batch_frames`** (`action_recognition_batch`) **НЕ обновлён под v3** — при
  батч-режиме выход разойдётся со schema v3. Нужно синхронизировать (сейчас v3 гарантирован только
  в одиночном `run()`).
- 🟡 Кап `max_windows=48` — сделать адаптивным к длительности с явным cost-budget на 200k.

## 4. Рекомендации по замене моделей/алгоритмов (на будущее)

Если при обучении наших моделей текущие признаки окажутся слабыми:

**Backbone действий (SlowFast R50 →):**
- **VideoMAE v2 / VideoMAEv2-g**, **Hiera**, **InternVideo2**, **UniFormerV2** — сильнее на Kinetics
  и дают **богатые self-supervised фичи** (лучше как токен для Encoder, чем логиты SlowFast).
  Провайдер `transformers_pretrained` + spec `videomae_kinetics400_inprocess` **уже есть** — переход
  малозатратен; поменяется I/O (16 кадров, эмбеддинг вместо 400-логитной головы).
- Критерий выбора — **baseline-ablation**: какой backbone даёт больший прирост Spearman на таргете.

**Эмбеддер трекера (histogram →):** **OSNet / TransReID / CLIP-ReID** — при появлении толпных/
длинных видео; выбор по метрике intra/inter cos на нашем датасете.

**Локализация действий (равномерные окна →):** **ActionFormer / TriDet** (temporal action
detection) — если важны точные границы и типы действий, а не равномерная выборка.

**Мульти-персона:** per-person **tubelet** (кроп по треку) + action-head — если нужны действия
отдельных людей на групповых сценах.

**Голова классов:** дообучить лёгкую голову под **наши** таргеты/домен (YouTube), а не только
Kinetics-400 — Kinetics-классы могут не покрывать релевантные для популярности действия.

## Итог
Логика компонента **рабочая и осмысленная** (сильнейшее подтверждение — семантически верные классы
на реальных роликах). До «почти идеального» ключевое: **(1.1) настоящий penultimate-эмбеддинг вместо
случайной проекции логитов**, **(2) 2-проходная детекция + очистка frames для 200k**, **(3) input-
валидатор + Prometheus-метрики + синхронизация батч-пути**. Модельные апгрейды (VideoMAEv2, OSNet,
tubelet, action localization) — по результатам baseline-ablation. Рекомендую: перед штампом v3
закрыть **1.1** (это меняет контракт эмбеддинга с Models) и **3 (input-validator + доки)**; остальное —
плановыми итерациями.
