## Оценка по 4 осям

- **Корректность:** эмбеддинги `(num_clips,256)` L2, 100% finite; статусы `ok`; схема соблюдена. ✅
- **Стабильность:** golden-повтор не гоняли (нужно добавить). Эмбеддинги детерминированы (fp32). ⚠ проверить.
- **Различимость:** per-track метрики динамики (`stability`, `temporal_jump`, `num_switches`)

- **Предсказательная ценность:** не измерена (нужен ablation на baseline). ⏳

## Пригодность для Models (§0.1)

- ✅ Эмбеддинг 256-d, finite, L2 — валидный token для Encoder.
- ✅ Времена клипов существуют: `clip_center_frame_indices` + `clip_center_times_s`.
- ⚠ **Но seq выдаётся per-track внутри `results_json` (object-dict)**, а не чистыми

## Проблемы (найденные)

1. **🔴 Контроль «без людей» невалиден.** В `ar_10s_control_no_people` детектор реально
   нашёл person (class_id 0, score 0.91) → это НЕ people-free ролик (матрица собрана из
   hf-клипов с людьми). Пустой путь (`no_person_detections`) **не протестирован**.
2. **🔴 Фрагментация треков на длинном/склеенном контенте** (110/205/199 треков,
   `tracks_with_multi_clips=0`). Первопричина: в Audit v3 **трекинг убран из
   `core_object_detections`** (суррогатные `track_id` per-detection). Без реального
   трекинга per-track динамика (`temporal_jump/stability`) на длинном видео **недостоверна**,
   а модель получает много одно-клиповых обрывков вместо когерентных треков.
   (Усугублено тем, что длинные ролики — synthetic-склейки; на реальных длинных нужно перепроверить.)
3. **🟡 §0.2 (`e2e_validate_output_quality`) не работает пер-компонентно** — ждёт full-E2E
   manifest, падает на всех run. Нет component-scoped проверки контракта NPZ.
4. **🟡 Per-track метрики не доходят до `feature_quality_audit`** (object-массивы vs wide-CSV)
   → ось «различимость» фактически не измерена.
5. **🟡 Матрица длинных видео — synthetic** (повторы коротких), длительность/трекинг
   нерепрезентативны для реального длинного контента.
6. **🟡 Стоимость ∝ track_count**: process-stage до ~29 s (4m/8m), batch avg 17 s;
   на реальных длинных видео с людьми может дорожать. Для 200k — лимиты клипов/треков.

## Рекомендации (вплоть до переработки)

**R1. Переструктурировать выход в плоский per-clip stream (schema `action_recognition_npz_v3`):**
- `clip_embeddings (C, 256)` float32 L2 — единый time-ordered поток (для Encoder);
- `clip_times_s (C,)`, `clip_frame_indices (C,)` — ось времени ⊆ `union_timestamps_sec`;
- `clip_topk_action_ids (C, K)` + `clip_topk_probs (C, K)` + `class_names` — **классы Kinetics** (аналитику+модели);
- `clip_track_id (C,)` — привязка к суррогатному треку (опц.).
Это чинит и model-fit (плоский seq), и «оба выхода» (эмбеддинг+классы), и делает per-track
динамику вторичной (агрегат из stream).

**R2. Решить вопрос трекинга.** Без реального трекинга per-track динамика бессмысленна.
Варианты: (а) вернуть лёгкий трекинг (ByteTrack/IoU) для action-треков; (б) **отказаться
от per-track динамики**, перейти на per-clip stream (R1) + опц. агрегаты по близости.
Рекомендую (б) — проще, надёжнее, дружелюбнее модели. Метрики `stability/temporal_jump`
пересчитать поверх stream или депрецировать.

**R3. Подгонка Segmenter под компонент.** Сейчас union-кадры разрежены → SlowFast (clip_len=32)
получает по 1 клипу на суррогат-трек. Нужно семплить **непрерывные окна ≥32 кадров вокруг
person-детекций** (или снизить clip_len с корректным ресемплингом). Это ключевая причина
одно-клиповых треков; правится в `Segmenter/_build_default_component_budgets` + оконный сэмплинг.

**R4. Валидный контроль + FP детектора.** Взять реально people-free ролик → проверить путь
`empty (no_person_detections)`. Отдельно измерить FP детектора person (влияет на пустые кейсы).

**R5. Тулинг:** сделать component-scoped output-validator (или починить §0.2 под one-component
manifest) + флэттенить per-track метрики в wide-CSV, чтобы аудит их видел.

**R6. Реальные длинные видео** (не synthetic) для достоверного вердикта по длинному контенту.

**R7. Стоимость 200k:** лимит клипов/треков на видео + батч-инференс клипов.
