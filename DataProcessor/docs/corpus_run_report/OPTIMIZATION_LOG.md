## OPT-1 — cut_detection: ленивый импорт torch/torchvision (✅ принято)

**Диагностика (cProfile на реальном прогоне):** из 118с прогона cut_detection **~74с — это ИМПОРТ**, а не
обработка (реальная работа `_process_video_frames` = 39.5с). Разбивка импорта:
`torch` 27.5с + `torchvision` 22с + чтение .py/.pyc с mfs (`posix.stat` 19.6с, `BufferedReader.read` 28.7с).
Причина: venv на **сетевом FS (mfs `/workspace`)** → импорт тысяч мелких файлов медленный, и `torch`/
`torchvision`/`clip` импортировались на уровне модуля БЕЗУСЛОВНО, хотя нужны только для deep/CLIP-фич,
которые в baseline (`--no-use-clip`, no-network policy) **выключены и не вызываются**. `clip` — вообще
dead-import (нигде не используется).

**Фикс** (`modules/cut_detection/utils/cut_detection.py`): torch/torchvision/clip → `None` на уровне модуля +
`_lazy_import_torch()` on-demand в `get_embedding_model` (deep-path). Гард `torch is not None` в определении
device (`__init__`, строка ~1960) — в baseline GPU не используется → 'cpu'.

**Замер (одно и то же видео, тот же под):**

| | wall | CPU% | NPZ фичи |
|---|---|---|---|
| ДО | 121.6с | 45% | baseline |
| ПОСЛЕ | 68.6с | 75% | **битово идентичны** |

- **−53с (−44%)**, CPU 45%→75% (ушло I/O-ожидание импортов, стало compute-bound).
- **NPZ идентичны:** features 6/6 + model_facing 37/37 фич-массивов совпали побитово; расходятся только
  `meta.stage_timings_ms` и `meta.created_at` (таймстемпы — ожидаемая недетерминированность, не фичи).
- **Масштаб:** ~53с/видео × 300 = ~**4.4 часа** экономии только на cut_detection.

**Обобщение (для след. шагов):** тот же паттерн — тяжёлые неиспользуемые module-level импорты из venv на
сетевом FS — вероятно есть и у других компонентов. Аудит остальных экстракторов на dead/heavy импорты +
инфра-опция «venv на локальном SSD пода» (Фаза 4) дадут аналогичный выигрыш по всему пайплайну.
