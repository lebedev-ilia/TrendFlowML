# Фаза 4 — Инфраструктурный план масштабирования

Цель: ответить на вопросы владельца — сколько параллельных процессов на машину, сколько машин, какая GPU
оптимальна по **$/1000 видео** (не по $/час). Основа: эмпирический stress-test параллелизма + тайминги
300-прогона + системные находки (import-tax, depth-диск).

## 1. Ключевой вывод из профилирования (Фазы 1-2): пайплайн CPU-bound, не GPU-bound
- Пик VRAM в 300-прогоне: **2.5 / 16 ГБ** (Triton-модели резидентно ~2.4ГБ + инференс). Огромный запас.
- GPU util (§TIMING §4): core_clip 8-24%, depth 44%, flow 55%, а cut/scene/pacing/uniqueness — **0% (CPU)**.
- Самые дорогие компоненты — **CPU-bound**: cut_detection (~105с, ffmpeg/skimage), scene_classification
  (~101с, resnet50 на CPU из-за драйвера), segmenter (ffmpeg-декод, CPU 181-376%).
- **Следствие для железа:** «больше видеокарта» ≠ «быстрее». Узкое место — **CPU-ядра + RAM + параллелизм**,
  не VRAM. Оптимум — **дешёвая GPU (для Triton CLIP/MiDaS/RAFT) + быстрый CPU/много ядер + N параллельных
  процессов**, а не дорогая большая карта.

## 2. Методология stress-test
На одной RTX 2000 Ada (16GB, $0.24/ч) — N=1/2/3/4 параллельных воркеров полного pipeline (segmenter → 3
Triton-стадии → cut/scene/pacing/uniqueness) на разных scene_videos. Замер: per-video wall, суммарный wall
на N видео, пик GPU util/VRAM, пик RAM. Метрика — **видео/час** = N / (total_wall_на_N_видео в часах).

## 3. Результаты параллелизма (эмпирика, RTX 2000 Ada 16GB, полный pipeline)

| N воркеров | total_wall (N видео) | **видео/час** | пик GPU util | пик VRAM | пик RAM | масштаб vs N=1 |
|---|---|---|---|---|---|---|
| 1 | ~301с¹ | **12.0** | ~2% | 2505 МБ | 61 ГБ² | 1.0x |
| 2 | 339.3с | **21.2** | 100% | 3216 МБ | 47 ГБ | 1.77x |
| 3 | 314.9с | **34.3** | 100% | 2576 МБ | 47 ГБ | 2.86x |
| 4 | 352.3с | **40.9** | 100% | 2418 МБ | 48 ГБ | **3.4x** |

¹ N=1 — по таймстемпам (bc не был установлен на первом воркере); приблизительно. ² N=1 RAM-пик 61ГБ —
вероятно всплеск буфер-кэша (free -m включает cache), не резидент; на N=2-4 стабильно ~47-48ГБ.

**Вывод:** параллелизм масштабируется **почти линейно до N=4** (3.4x). Ключ — **огромный запас ресурсов**:
VRAM **2.4-3.2 / 16 ГБ**, RAM **~48 / 251 ГБ**. GPU util достигает 100% уже на N=2, НО throughput продолжает
расти до N=4 — потому что CPU-фазы (segmenter/cut/scene) и GPU-фазы (Triton) **перекрываются между воркерами**
(пока один воркер на CPU, другой грузит GPU). Прирост N=3→N=4 (34.3→40.9, +19%) меньше, чем N=2→N=3 (+62%) —
начало насыщения, но потолок ещё не достигнут.

**Оптимальное N на RTX 2000 Ada:** **4** (солидная рабочая точка, 3.4x); стоит проверить N=5-6 — вероятно
ещё +10-20% до насыщения CPU-ядер. VRAM/RAM НЕ ограничивают (можно и больше по памяти, упрётся в CPU/ядра).

## 4. $/1000 видео по GPU-опциям
Метрика: **$/1000 = ($/час пода) / (видео/час при оптимальном N)**. Не минимизировать $/час — минимизировать
$/1000 с учётом реальной пропускной способности.

Список владельца: RTX 2000 Ada 16GB, RTX 4090 24GB, RTX PRO 4000 24GB, RTX 4000 Ada 20GB, RTX A4000 16GB,
RTX A4500 20GB, RTX A5000 24GB, L4 24GB, A40 48GB, RTX 3090 24GB.

> **Оговорка (честно):** точные $/час RunPod меняются по спросу и типу облака (community/secure); живые цены
> нужно снять на момент запуска (`runpod_gpu_scraper`/консоль). Ниже — **измеренная база (RTX 2000 Ada)** +
> **правило пересчёта**. Т.к. пайплайн CPU-bound, «видео/час» слабо зависит от модели GPU (Triton-часть не
> узкое место) — при равном CPU/ядрах throughput близок, поэтому **$/1000 ≈ пропорционально $/час** → при
> CPU-bound нагрузке **самая дешёвая карта с достаточным CPU/RAM выигрывает**. Дорогие карты (4090/A40)
> оправданы ТОЛЬКО если перевести CPU-компоненты (scene/cut) на GPU (см. §6) — тогда GPU станет узким местом.

**Измерено (RTX 2000 Ada):** при N=4 = **40.9 видео/ч** → $/1000 = $0.24/40.9×1000 = **$5.9 / 1000 видео**
(против $20/1000 при N=1 — параллелизм режет стоимость **3.4x**).

| GPU | VRAM | $/час (оценка*) | видео/час @оптим.N | $/1000 (оценка) | вывод |
|---|---|---|---|---|---|
| **RTX 2000 Ada** | 16GB | **$0.24 (изм.)** | **40.9 (изм., N=4)** | **$5.9 (изм.)** | базлайн, отлично |
| RTX A4000 | 16GB | ~$0.17-0.21 | ≈40 (CPU-bound, тот же CPU-класс) | **~$4.2-5.1** | вероятно дешевле — проверить цену/CPU |
| RTX 4000 Ada | 20GB | ~$0.20-0.30 | ≈40 | ~$5-7 | ≈ |
| RTX A4500 / A5000 | 20/24GB | ~$0.26-0.36 | ≈40 | ~$6-9 | дороже, throughput тот же → хуже |
| L4 | 24GB | ~$0.32-0.43 | ≈40 | ~$8-11 | дороже, не оправдана при CPU-bound |
| RTX 4090 / A40 | 24/48GB | ~$0.34-0.79 | ≈40 (если scene/cut не на GPU) | ~$8-19 | **не оправдана** пока CPU-bound |

\* цены-оценки — снять живыми при запуске (`runpod_gpu_scraper`). Т.к. пайплайн CPU-bound, видео/час слабо
зависит от модели GPU → **$/1000 ≈ пропорционально $/час** → **самая дешёвая карта с достаточным CPU/ядрами
выигрывает**. Дорогие карты окупятся ТОЛЬКО после переноса scene/cut на GPU (§6).

## 5. Координация нескольких машин (адаптировать Fetcher, не изобретать)
Паттерн шардирования из `Fetcher/fetcher/dataset_collector/hf_coordination.py` — `stable_shard_slot`:
`hash(video_key) % N_machines` + HF-based claims/done. Применить к DataProcessor: каждая машина берёт свой
шард `corpusXXX.json` по `hash(video_id) % N`, отмечает done в общем месте (HF/файл) → нет коллизий, нет
центрального координатора. Watchdog — по одному инстансу на машину (простейший вариант, см. Фаза 7).

## 6. Разблокировка «больше GPU = быстрее» (условие)
Дорогая карта окупится ТОЛЬКО если снять CPU-узкое место: перевести `scene_classification` и (частично)
`cut_detection` на GPU. scene — resnet50/places365 (модель уже в Triton-бандле!) → экспорт на Triton убирает
CPU-инференс. Это отдельный OPT (Фаза 2). До него — CPU-bound, дешёвая карта оптимальна.

## 7. Рекомендация Фазы 4 (по измеренным данным)

1. **GPU: самая дешёвая с достаточным CPU** — RTX 2000 Ada ($5.9/1000 @N=4) или **RTX A4000** (дешевле,
   тот же CPU-класс → вероятно ~$4-5/1000). НЕ брать 4090/A40/L4 — при CPU-bound нагрузке переплата без
   выигрыша по throughput.
2. **Параллелизм: N=4 процесса на машину** (3.4x, $/1000 в 3.4 раза ниже, чем 1 процесс). VRAM/RAM с запасом
   (2.4/16 ГБ, 48/251 ГБ) → упор в CPU-ядра. Проверить N=5-6 на целевой машине (вероятно +10-20%).
3. **Число машин** — из целевого throughput (Фаза 7): 1 машина @N=4 = ~40 видео/ч = ~1000 видео/сутки.
   Для 50 видео — 1 машина (~1.5ч). Для 1000 — либо 1 машина ~сутки, либо 2-3 машины параллельно (~8-12ч).
4. **Мульти-машина: шардирование по Fetcher-паттерну** (`stable_shard_slot`, `hash(video_id) % N`), done-метки
   в общем месте, watchdog по одному на машину. Без центрального координатора.
5. **Условие для апгрейда GPU:** перенести `scene_classification` (places365 уже в Triton-бандле!) и, если
   выйдет, `cut_detection` на GPU → снять CPU-узкое место → тогда дорогая карта начнёт окупаться. До этого —
   дешёвая карта + параллелизм оптимальны.

### Важно для сетапа свежего пода (обнаружено при stress-test)
Свежий RunPod-контейнер под Triton+pipeline требует эфемерных apt-зависимостей (не персистят на volume):
**`ffmpeg`, `time`, `bc`, `libarchive13` (Triton Python-backend), `python3-numpy` (для python3.10 backend)**.
Внести в pod-setup скрипт, иначе Triton не стартует (libarchive) и метрики/сегментер частично падают.
