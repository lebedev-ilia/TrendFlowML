# Прогноз затрат диска (сквозной, все фазы)

Измеренная база + прогноз масштабирования NPZ на весь набор компонентов, сверка с томом (120 ГБ),
план выгрузки в HF (владелец предпочитает выгрузку, а не бесконечное расширение тома).

## Измерено (7 VP-компонентов, 300 видео)
- **151 МБ/видео** (2400 NPZ / 300 = 8 файлов/видео), **54 ГБ** итого. Это после per-video очистки
  seg-кадров (оставляются только `rs/<component>/*.npz`). Без очистки seg было бы ~1 ГБ/видео.

## Прогноз по типу выхода компонента
| Тип выхода | компоненты (примеры) | ~МБ/видео каждый |
|---|---|---|
| Скаляры/малые векторы | video_pacing, uniqueness, color_light, behavioral, cut_detection, content_domain | 2-10 |
| Средние эмбеддинги/карты | core_clip, core_depth, core_optical_flow, scene_classification, title/desc embed | 10-25 |
| Тяжёлые (спектрограммы/per-frame/per-face) | mel/spectral (AP), core_face_landmarks, emotion_face, action_recognition, similarity | 15-50 |

## Прогноз на видео по наборам
| Набор компонентов | ~МБ/видео | источник |
|---|---|---|
| 7 VP (текущий) | **151** | измерено |
| Полный VP (29, минус заблок.) | ~300-450 | прогноз (VP_ROLLOUT_FORECAST) |
| VP + AP (21) + TP (22) | **~400-550** | прогноз (AP_TP_ROLLOUT_FORECAST) |

## Сверка с томом 120 ГБ (сценарии владельца: 50 → 500 → 1000 видео, ВСЕ процессоры)
При **~500 МБ/видео** (полный набор):

| Видео | Объём NPZ | Влезает в 120 ГБ? | Действие |
|---|---|---|---|
| 50 | ~25 ГБ | ✅ (с запасом на venv 27ГБ + бандл 12ГБ + репо) | локально, выгрузка не обязательна |
| 500 | ~250 ГБ | ❌ (>120 ГБ) | **обязательна выгрузка в HF по ходу** |
| 1000 | ~500 ГБ | ❌❌ | **потоковая выгрузка + удаление** (на диске держать только «в работе») |

Плюс постоянный оверхед тома: `venv` 13ГБ (visual) + `.ap_venv` 7.4ГБ + `.tp_venv` 6.8ГБ + Triton-бандл 12ГБ
+ репо 6ГБ ≈ **~45 ГБ базы** → под NPZ реально доступно ~75 ГБ → потолок **~150 видео** полного набора без выгрузки.

## План выгрузки в HF (адаптировать Fetcher, не изобретать)
Паттерн из `Fetcher/fetcher/dataset_collector/hf_queues.py`: `upload_local_files_commit` (батч-коммит в HF-репо)
+ `delete_local_file` (удаление локальной копии ПОСЛЕ подтверждённого коммита). Адаптация для DataProcessor:
1. После завершения видео (`_summary` в metrics) — поставить его `rs/`-папку в очередь на HF-upload (в
   отдельный dataset-репо, напр. `Ilialebedev/dp_results_<run>`), путь `<video_id>/<component>/*.npz`.
2. Батч-коммит N видео → при успехе удалить локальные `rs/` этих видео (как Fetcher удаляет mp4 после аплоада).
3. На диске в любой момент — только «в работе» + буфер (напр. последние 20-50 видео) → диск не растёт линейно.
4. Резюмируемость: очередь upload + done-keys на диске (как у Fetcher) → переживает обрывы/рестарты пода.

## Рекомендации
- **50 видео (первый gate):** выгрузка не обязательна, но включить её сразу для проверки пайплайна выгрузки.
- **500/1000:** выгрузка в HF обязательна и должна работать ПОТОКОВО (не «накопить потом залить») — иначе
  том переполнится на ~150 видео (как уже было с квотой в 300-прогоне — см. BATCH_RUN_SUMMARY §4).
- Держать в `run_corpus`/batch-обёртке буфер-лимит: если свободно на томе < X ГБ — приостановить новые видео
  до завершения выгрузки (backpressure), как backpressure в Fetcher.
