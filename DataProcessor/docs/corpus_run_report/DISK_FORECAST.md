# Прогноз затрат диска (сквозной, все фазы)

Измеренная база + прогноз масштабирования NPZ на весь набор компонентов, сверка с томом (120 ГБ),
план выгрузки в HF (владелец предпочитает выгрузку, а не бесконечное расширение тома).

## Измерено (7 VP-компонентов, 300 видео)
- **~159 МБ/видео** (54 ГБ / 300). После per-video очистки seg-кадров (оставляются только `rs/*.npz`).

### 🔴 ГЛАВНАЯ находка: `core_depth_midas` = **151 МБ из 159 (95% всего диска!)**
Разбивка на одном видео (165 кадров):
| Компонент | размер | | Компонент | размер |
|---|---|---|---|---|
| **core_depth_midas** | **151 МБ** | | scene_classification | 1.4 МБ |
| core_clip | 3.0 МБ | | cut_detection | 1.0 МБ |
| core_optical_flow | 1.2 МБ | | video_pacing | 1.0 МБ |
| | | | uniqueness | 1.0 МБ |

Внутри `depth.npz`: `depth_maps` (165×384×384 f32 = **97 МБ**) + `depth_maps_norm` (**97 МБ**, дубликат в
другой нормировке, выводится из depth_maps) = ~194 МБ сырых (сжато ~151). А **pooled-статистики уже посчитаны
и крошечные** (`depth_mean/std/p05/p95/range_robust/complexity_score/fg_bg_separation`, по 165 f32) + есть
`preview_depth_maps` (10 кадров, 6 МБ) для интерпретируемости.

**Модели полные карты НЕ используют** (Agent B их пропустил в exp_0005, планирует pooled-статистики). Значит
хранение двух полных копий карт = ~95% диска впустую.

### 🟢 #1 disk-оптимизация (рекомендация, требует координации — меняет shared feature-контракт)
- **Не хранить полные `depth_maps`+`depth_maps_norm`**, оставить pooled-статистики (уже есть) + `preview_*`
  → depth.npz с 151 МБ до **~6 МБ**, per-video с **159 → ~14 МБ (11x)**.
- Минимальный безрисковый шаг: убрать `depth_maps_norm` (выводится из depth_maps + p05/p95) → −97 МБ (−50%),
  ноль потери информации.
- В коде: нет флага «не хранить карты» (`core_depth_midas/main.py`), есть только `--out-width/--out-height`
  (дефолт 384 → снизить до 64-128 даёт 9-36x, но меняет разрешение хранимых карт).
- **Уточнение по потребителям (проверено grep):** полные `depth_maps` читают `shot_quality`
  (`depth_metrics` покадрово) и `frames_composition` — НО оба **не входили в 300-прогон** (это Phase-5
  доп.компоненты). Значит:
  - В **текущем 7-компонентном наборе** полные карты — чистый мусор (нет потребителя) → удалять безопасно.
  - В **полном наборе** (с shot_quality/frames_composition) карты нужны, но как **транзитный интермедиат**:
    потребить в пайплайне → НЕ персистить в финальном depth.npz (как seg-кадры чистятся после сегментации).
    Итоговый depth.npz = pooled + preview.
- **Реализация OPT-3:** флаг `--persist-depth-maps` (default off): по умолчанию сохранять только pooled+preview;
  полные карты держать на диске лишь пока их читают shot_quality/frames_composition (в пределах одного run),
  потом удалять. Модель-контракт (pooled) не меняется. Требует: (а) флаг в core_depth_midas, (б) порядок
  запуска depth→shot_quality→cleanup, (в) sign-off Agent B (Encoder использует pooled — Agent B уже пропускает
  карты). НЕ менял молча.

**Влияние на прогноз ниже:** если применить, все оценки МБ/видео падают ~10x → HF-выгрузка становится
почти не нужна (1000 видео полного набора без depth-карт ≈ 40-60 ГБ вместо ~500 ГБ).

## Сверка с томом 120 ГБ (сценарии владельца: 50 → 500 → 1000 видео, ВСЕ процессоры)
При **~500 МБ/видео** (полный набор):

| Видео | Объём NPZ | Влезает в 120 ГБ? | Действие |
|---|---|---|---|
| 50 | ~25 ГБ | ✅ (с запасом на venv 27ГБ + бандл 12ГБ + репо) | локально, выгрузка не обязательна |
| 500 | ~250 ГБ | ❌ (>120 ГБ) | **обязательна выгрузка в HF по ходу** |
| 1000 | ~500 ГБ | ❌❌ | **потоковая выгрузка + удаление** (на диске держать только «в работе») |

Плюс постоянный оверхед тома: `venv` 13ГБ (visual) + `.ap_venv` 7.4ГБ + `.tp_venv` 6.8ГБ + Triton-бандл 12ГБ
+ репо 6ГБ ≈ **~45 ГБ базы** → под NPZ реально доступно ~75 ГБ → потолок **~150 видео** полного набора без выгрузки.

## План выгрузки в HF (адаптировать Fetcher, не изобретать)
Паттерн из `Fetcher/fetcher/dataset_collector/hf_queues.py`: `upload_local_files_commit` (батч-коммит в HF-репо)
+ `delete_local_file` (удаление локальной копии ПОСЛЕ подтверждённого коммита). Адаптация для DataProcessor:
1. После завершения видео (`_summary` в metrics) — поставить его `rs/`-папку в очередь на HF-upload (в
   отдельный dataset-репо, напр. `Ilialebedev/dp_results_<run>`), путь `<video_id>/<component>/*.npz`.
2. Батч-коммит N видео → при успехе удалить локальные `rs/` этих видео (как Fetcher удаляет mp4 после аплоада).
3. На диске в любой момент — только «в работе» + буфер (напр. последние 20-50 видео) → диск не растёт линейно.
4. Резюмируемость: очередь upload + done-keys на диске (как у Fetcher) → переживает обрывы/рестарты пода.

## Рекомендации
- **50 видео (первый gate):** выгрузка не обязательна, но включить её сразу для проверки пайплайна выгрузки.
- **500/1000:** выгрузка в HF обязательна и должна работать ПОТОКОВО (не «накопить потом залить») — иначе
  том переполнится на ~150 видео (как уже было с квотой в 300-прогоне — см. BATCH_RUN_SUMMARY §4).
- Держать в `run_corpus`/batch-обёртке буфер-лимит: если свободно на томе < X ГБ — приостановить новые видео
  до завершения выгрузки (backpressure), как backpressure в Fetcher.
