# План прогона 10 000 видео — все процессоры, оптимизировано, валидно

> Цель владельца (неделя): **10k видео × все процессоры (VP+AP+TP) × максимально оптимизировано × всё
> даёт валидный качественный результат.** Ниже — исполнимый критический путь. Статус: pod-free подготовка
> начата (расширение пула + дизайн оптимизаций); pod-часть гейтится бюджетом.

## 1. Пул кандидатов (10k) — ✅ В РАБОТЕ (pod-free)
`select_corpus_candidates.py --shards 0..21 --per-cell 250 --channel-cap 5` (22 шарда HF `pre_final_data`,
метаданные-only, резюмируемо). Цель ~12-15k кандидатов → ~10-12k в `video_index` (скачиваемы) → выбрать
сбалансированные 10k (как corpus1000: round-robin по 62 ячейкам dur×views×lang). Индекс: 103 926 видео.

## 2. Триаж состава — «все ВАЛИДНЫЕ компоненты» (ключевое честное место)
«Всё валидно» невозможно для компонентов, у которых **нет данных/идёт переразметка** — их гоним как
known-limited или исключаем (не выдаём пустышку за «валидно»). Детали: `VP_ROLLOUT_FORECAST.md`,
`AP_TP_ROLLOUT_FORECAST.md`, портфолио §8.

**✅ Включаем (валидны / чинибельны за неделю):**
- **VP (валидированы):** core_clip, core_depth_midas, core_optical_flow, cut_detection, scene_classification,
  video_pacing, uniqueness (+ фиксы L12/L13).
- **VP (дешёвые готовые):** content_domain, detalize_face, behavioral, color_light, story_structure.
- **VP (после весов/GPU-решения):** shot_quality, frames_composition, high_level_semantic, emotion_face,
  action_recognition — валидировать в smoke, включить если дают валидный выход.
- **AP:** clap, tempo, loudness, asr + производные (mel/mfcc/spectral/chroma/pitch/onset/rhythmic/hpss/key)
  — после фиксов voice_quality (config-drift), WavLM-кэша (emotion_diarization), pyannote config (speaker_diar).
- **TP:** все — мост AP→doc.asr→TP **уже подключён** (`_autogen_text_input_from_asr`), нужна e2e-валидация +
  реальные title/description в документ.

**🚫 Исключаем (невалидны сейчас — решения владельца §8):**
- `core_object_detections` — идёт ручная переразметка YOLO (пере-прогон бессмыслен).
- `brand/car/franchise/place/face_identity` semantics — stub-БД (данных нет).
- `ocr_extractor`, `text_scoring` — detector-баги. `micro_emotion` — Docker-блокер на RunPod.

> Итог: реальный набор ≈ **VP ~17 + AP ~20 + TP ~22 ≈ 59 валидных компонентов** (не 75). Точный список — после smoke §5.

## 3. Оптимизации (ОБЯЗАТЕЛЬНЫ до 10k) — дизайн готов к деплою
### 3.1 🔴 Кадры на локальный SSD (главный рычаг — измерено, что mfs-запись биндит при N=8)
segmenter пишет `<--output>/<vid>/video/batch_*.npy` (строка 476). Сейчас `--output` = mfs.
**Правка `run_smoke.sh`:** `--output /root/frames_run` (локальный SSD); компоненты `--frames-dir
/root/frames_run/$VID/video`; `--rs-path` (финальные NPZ ~24МБ) остаётся на mfs; после видео `rm -rf
/root/frames_run/$VID`. Ожидаемо segmenter ~160с→~12с + быстрее чтение кадров компонентами. Замерить.
### 3.2 venv на локальном SSD (import 18x) — держим (`cp -a /workspace/venv /root/venv` в pod-setup).
### 3.3 🔴 Потоковая HF-выгрузка (иначе диск 10k×полный ≈ 0.7-1.5ТБ >> 120ГБ тома)
Пер-видео post-step: после `rs/*.npz` → `HfApi().upload_folder(rs → HF dataset)` → `rm -rf` локально.
Backpressure по диску (если >N ГБ — ждать выгрузку). Переиспользовать `HfApi` (Fetcher уже использует).
Построить+проверить на Gate 500 ДО 10k.
### 3.4 N-тюнинг + мультимашина: сравнить N=4/8/12 на полном наборе (при frames-on-SSD баланс сместится).
Network Volume attach к 1 поду → мультимашина требует либо отдельных томов, либо шардить корпус по машинам
с независимыми томами (не общий corpus_smoke).

## 4. Диск / компьют / стоимость (порядок величины)
- **Диск:** VP ~24МБ + AP ~30-80МБ + TP ~15-40МБ ≈ **70-150МБ/видео** → 10k = **0.7-1.5ТБ** → §3.3 обязательна.
- **Компьют:** полный набох тяжелее 7-VP; грубо ~30-60 видео/ч/машина оптимизированно → 10k ≈ **170-330 ч** →
  **~$40-80** (1 машина, $0.24/ч) или меньше времени на нескольких машинах (та же сумма). **Точно — после smoke §5.**

## 5. Последовательность (критический путь)
1. **Пул 10k** (§1, идёт) + **сбор corpus10k.json** (после расширения).
2. **Оптимизации** (§3.1 frames-SSD + §3.2 venv-SSD + §3.3 HF-выгрузка) — реализовать+проверить.
3. **AP+TP bringup**: venv на под, фиксы voice_quality/WavLM/pyannote, e2e ASR-мост.
4. **Smoke полного набора 20-50 видео** → аудит валидности КАЖДОГО компонента (замок §2 «всё валидно») +
   реальные throughput/диск → заменить оценки §4.
5. **Gate 500** на полном валидном наборе с выгрузкой (стабильность+диск на масштабе).
6. **10 000** (при пополненном бюджете) с выгрузкой/мультимашиной → полный мультимодальный датасет для Agent B.
