# Фаза 7 — План следующего большого теста (синтез фаз 1-6)

Параметры владельца: **все 3 процессора, все компоненты**; 2-3 машины; 2-3 (проверено — до 4) параллельных
процесса на машину; старт с **50 видео**, затем **500**, затем **1000** при стабильной работе. Ниже — план
на основе измерений 300-прогона + оптимизаций + прогнозов (VP/AP/TP/disk/infra).

## 1. Набор компонентов
- **VP:** 7 (прогнанные) + 6 «дешёвых готовых» (content_domain, detalize_face, behavioral, color_light,
  optical_flow, story_structure) сразу; + 5 «медленных на CPU» после решения torch-GPU; emotion_face/
  action_recognition после весов; **micro_emotion — отложить** (Docker-блокер на RunPod).
  **Исключить:** core_object_detections, ocr_extractor, text_scoring (detector-баг), brand/car/franchise/
  place/face_identity (stub-БД). См. `VP_ROLLOUT_FORECAST.md`.
- **AP (21):** после фиксов voice_quality (config-drift) + WavLM-кэш + pyannote config. `AP_TP_ROLLOUT_FORECAST.md`.
- **TP (22):** **ТОЛЬКО после** выстраивания `AP→doc.asr→TP` (иначе 7 TP-компонентов пусты) + передачи реальных
  title/description в TP-документ.

## 2. Инфраструктура (из `INFRA_SCALING_PLAN.md`)
- **GPU:** дешёвая (RTX 2000 Ada $0.24/ч или RTX A4000 дешевле) — пайплайн CPU-bound, дорогая карта не
  окупается. **N=4 процесса/машину** (3.4x throughput, VRAM/RAM с запасом, упор в CPU-ядра).
- **Throughput (измерено, 7 VP-комп.):** ~**40.9 видео/ч @N=4** на RTX 2000 Ada. **Полный набор (VP+AP+TP)
  будет медленнее** (больше компонентов) — точный throughput ТРЕБУЕТ смоука (не экстраполировать).
- **Мульти-машина:** шардирование `hash(video_id) % N_машин` (Fetcher `stable_shard_slot`), done-метки общие,
  watchdog по одному инстансу на машину.

## 4. Прогноз диска (из `DISK_FORECAST.md`) — с учётом depth-оптимизации
- Без OPT-3 (depth): полный набор ~400-550 МБ/видео → **500 видео >120ГБ тома** → **обязательна потоковая
  HF-выгрузка** (адаптировать Fetcher `hf_queues`: upload+delete после коммита).
- **С OPT-3 (не персистить полные depth-карты, 95% диска):** ~40-60 МБ/видео → 1000 видео ≈ 40-60 ГБ →
  влезает без выгрузки. **Сильно рекомендуется применить OPT-3 перед 500/1000** (координация с Agent B: он
  полные карты не использует).
- Постоянная база тома: venv (13+7.4+6.8) + бандл 12 + репо 6 ≈ **45 ГБ** → под NPZ ~75 ГБ.

## 6. Gate-фазы (критерии перехода)
**Gate 1 — 50 видео (валидация пайплайна):**
- Критерий перехода на 500: **≥95% видео complete (0 систематических фейлов)**, время в пределах прогноза
  **±30%**, диск/выгрузка работают, все включённые компоненты дают NPZ (не пустые систематически).
- Что проверяем: связка AP→TP (ASR не пустой), voice_quality не падает, HF-выгрузка коммитит+чистит.

**Gate 2 — 500 видео (масштаб):**
- Критерий перехода на 1000: **≥97% complete**, throughput стабилен (нет деградации во времени — утечки
  памяти/диска), мульти-машинное шардирование без коллизий/дублей, watchdog отработал ≥1 авто-рестарт чисто.

**Gate 3 — 1000 видео (прод-масштаб):**
- Успех: **≥97% complete**, полный отчёт метрик (как `BATCH_REPORT`), NPZ выгружены в HF, том не переполнен.

> Критерии согласовать с владельцем ДО запуска (а не по факту). «Систематический фейл» = один и тот же
> компонент/причина на >5% видео (как было с leading-dash до фикса).

## 7. Watchdog на нескольких машинах
Текущий `watchdog.py` — одноподовый (проверен: авто-рестарт батча/Triton). Для 2-3 машин — **N независимых
инстансов** (по одному на машину, простейшее рабочее решение; проверено надёжным в 300-прогоне). Централизация
не нужна — шардирование делает машины независимыми. Каждый watchdog: батч + Triton своей машины + resumable
(skip по `_summary`).
