# План консолидации избыточных семей фич (§3.8 портфолио-оценки)

**Дата:** 2026-07-19. **Автор:** deepdive-агент (автономно). **Статус:** предложение + разбивка на
безопасные/отложенные шаги. **Источник:** §3.8 [`COMPONENT_DEEP_DIVE_PORTFOLIO_ASSESSMENT.md`](COMPONENT_DEEP_DIVE_PORTFOLIO_ASSESSMENT.md)
+ разделы «Рекомендации по архитектуре/связям» FINAL_REPORT'ов затронутых компонентов.

> Зачем: перед фиксацией финального feature-сета Encoder несколько «семей» фич описывают одни и те же
> 2-3 физические оси через 15+ коррелированных признаков и считают один и тот же спектр/огибающую по
> 3-4 раза. Это technical debt: лишняя размерность входа без прироста информации + лишний CPU. Ошибка
> «просто удалить» — потерять сигнал; поэтому план разделён на два слоя с разной ценой ошибки.

---

## Два слоя консолидации (важно не путать)

- **Layer 1 — Consolidation of COMPUTE (инфра-оптимизация).** Считать общий STFT / общую onset-огибающую /
  музыкальность-gate ОДИН раз и переиспользовать. **Значения фич НЕ меняются** — меняется только то, сколько
  раз считается спектр. Цена ошибки низкая (регресс-тест на побитовое совпадение фич). Можно делать
  инкрементально уже сейчас.
- **Layer 2 — Consolidation of FEATURE-SET для Encoder (отбор входа).** Не подавать в Encoder заведомо
  коррелированные дубли. Реализуется **маской входа Encoder / выбором подмножества**, а НЕ удалением
  экстракторов (обратимо, аналитике agg-поля остаются). Цена ошибки выше — можно случайно выкинуть ось,
  которая различает контент. **Финализировать только после реальных корреляций на корпусе §9** — на текущем
  mock/talking-head-корпусе почти всё вырождено (музыкальные фичи плоские), корреляции недостоверны.

---

## A. Аудио: единый СПЕКТРАЛЬНЫЙ слой (Layer 1 + Layer 2)

**Проблема.** `spectral_extractor` (46 фич), `mel_extractor` (128-мел + 512-стат), `mfcc_extractor`
(13 кепстр.), `chroma_extractor` (12 pitch-class), `band_energy_extractor` (3 полосы),
`spectral_entropy_extractor` (2 фичи), `hpss_extractor` (12 спектр. дескрипторов на стемах) — **каждый
считает свой STFT независимо** (отмечено в mel/mfcc/spectral_entropy/band_energy отчётах). Плюс прямые
дубли по смыслу: `spectral_entropy` ≡ `spectral_flatness` (spectral) ≡ `chroma_entropy` — ранжируют видео
идентично; `band_energy` (3-бин) — грубая версия `spectral_centroid/rolloff`; `mfcc` = DCT(log-mel), т.е.
формально производная от `mel`.

**Layer 1 (compute).** Ввести общий `stft_provider` (одно окно/hop как контракт), от него питать
{spectral, mel, mfcc, chroma, band_energy, spectral_entropy}. Экономия: 6× STFT → 1×. Значения фич не
трогаем (сверить golden побитово до/после).

**Layer 2 (feature-set).**
- **`spectral_entropy_extractor` → в поле `spectral_extractor`.** Отдельный компонент ради 1 содержательной
  фичи, которая дублирует `spectral_flatness`. В Encoder оставить ОДНУ из {flatness, spectral_entropy,
  chroma_entropy}, две другие снять маской.
- **`band_energy` → поле поверх общего STFT.** Как EQ-инсайт (bass/mid/treble) полезнее сырого centroid ДЛЯ
  UI — оставить для аналитики; в Encoder-вход band_share коррелирует с centroid — кандидат снять.
- **`mfcc` vs `mel` vs `CLAP` — определить аудио-model-контракт.** Вероятно не нужны все три как seq для
  Encoder. Гипотеза: CLAP (обучаемый семантический хаб, 5/5) — основной seq; `mel` — dense-резерв; `mfcc`
  (13-d, компактный) — лёгкий fallback при отказе от CLAP. Финальный выбор — за Encoder-командой на реальных
  корреляциях; сейчас зафиксировать как открытый вопрос, не выкидывать вслепую.

## B. Аудио: единый РИТМ-модуль (Layer 1 + Layer 2)

**Проблема.** `tempo_extractor` / `onset_extractor` / `rhythmic_extractor` — тройной анализ ритма.
`rhythmic.tempo_bpm ≈ tempo.tempo` (оба фиксируются на спурьёзном дефолте ~129 BPM на речи),
`rhythmic.beats_count ≈ onset.count`. `onset` — самый здоровый (работает и на речи, и на музыке).

**Layer 1 (compute).** Общий `onset_envelope_provider` → beat-tracking один раз → {tempo, density,
regularity, syncopation}. Одна **музыкальность-gate** (`rhythm_regularity` из rhythmic честно отделяет
музыку 0.56 от речи 0.02-0.07) — определять «есть ли бит» ОДИН раз и переиспользовать; это чинит и дефект
`tempo_confidence≈0.9 везде` (§3.7).

**Layer 2 (feature-set).** В Encoder-вход из ритм-трио оставить: `onset` (event-rate, работает везде) +
`rhythm_regularity`/`tempo_variation` (музыкальность) + `tempo_bpm` ОДИН раз (из общего трекинга, под
gate). Снять дубли `rhythmic.tempo_bpm`, `rhythmic.beats_count`. Аналитике agg-поля оставить.

## C. Аудио: ТОНАЛЬНЫЙ модуль (уже почти консолидирован — образец)

`chroma → key` уже переиспользует хрому (`chroma_reused=True`) — это правильный паттерн и эталон для
остального аудио. Действие: **распространить паттерн reuse** на A/B (key показывает, что так можно). Плюс
честная `key.confidence` (low на речи = warning) — образец против спурьёзных значений tempo.

## D. Текст: единый ALIGNMENT-блок (Layer 2)

**Проблема.** `cosine_metrics_extractor` (5 кросс-косинусов), `embedding_pair_topk_extractor` (title↔desc +
title↔chunks top-k), `title_to_hashtag_cosine_extractor` (title↔hashtag) — три независимых способа считать
похожесть текстовых полей; `pair_topk.title_desc` дублирует `cosine_metrics.title_desc`.

**План.** Объединить в один `text_alignment`-блок с осями: title↔desc, title↔transcript (золотой
кликбейт-детектор), title↔hashtag (SEO-релевантность), transcript↔comments. `embedding_pair_topk`
оставляет уникальное «best-segment» (какой чанк лучше отвечает заголовку) — сохранить как под-фичу, не
дублируя базовый title↔transcript. Убрать дуальные namespace-префиксы (`tp_*` дубли). **Блокер:** все
операнды сейчас mock (cosine=1.0) — реальную избыточность видно только на реальных title/desc/hashtag
(§7.2 owner-решение про Fetcher→DataProcessor + корпус §9). До этого — только зафиксировать структуру, не
удалять.

## Порядок исполнения (приоритет = польза/риск)

1. **Layer 1 A+B (общий STFT + общая onset-огибающая)** — самое безопасное (golden побитово), реальная
   экономия CPU на прод-масштабе 200k видео. Можно делать инкрементально, каждый шаг под регресс-тест.
2. **Слить микро-компоненты `spectral_entropy` (2 фичи) и `band_energy` (3 фичи) как поля** —
   малый риск, чистит реестр компонентов.
3. **Определить аудио-model-контракт (mel/mfcc/CLAP как seq)** — открытый вопрос Encoder-команде,
   решать на реальных корреляциях корпуса §9.
4. **Layer 2 (маски входа Encoder для ритм/тональность/alignment)** — ТОЛЬКО после корпуса §9: посчитать
   реальную корреляционную матрицу фич, снять дубли эмпирически (|corr|>0.95 на разнообразном корпусе),
   а не по эвристике на mock-данных.

## Что НЕ делаем сейчас (и почему)

- Не рвём заштампованные экстракторы на части до фиксации Encoder-контракта — потеряем валидацию без выигрыша.
- Не выкидываем «музыкальные» фичи (chroma/key/hpss/tempo) по корреляциям с текущего talking-head-корпуса —
  там они вырожденно-плоские; на музыкальном контенте они расходятся. Ждём корпус §9.
- Layer 2 удаления реализуем маской Encoder-входа (обратимо), а не удалением полей NPZ (аналитике нужны).

## Итог для владельца (1 абзац)

Консолидация распадается на дешёвую безопасную часть (считать спектр/ритм один раз — Layer 1, делаю
инкрементально под golden) и чувствительную часть (какие коррелированные фичи не подавать в Encoder —
Layer 2), которую честно нельзя финализировать на mock-корпусе: нужны реальные корреляции с разнообразного
корпуса §9. Поэтому план сознательно ставит Layer 1 вперёд, а Layer 2 — после сбора корпуса.
