# Протокол автоматического подбора тест-корпуса (vision-driven)

> Решает разрыв из [`COMPONENT_DEEP_DIVE_PORTFOLIO_ASSESSMENT.md`](COMPONENT_DEEP_DIVE_PORTFOLIO_ASSESSMENT.md)
> §7.2/§9: сейчас DataProcessor валидируется на 5-26 тестовых видео с фиктивными
> заголовками/описаниями, а реальный собранный корпус (Fetcher + этот HF-датасет) никак не
> используется. Задача агента: САМОМУ, без владельца, собрать разнообразный набор реальных видео
> для полного прогона системы, используя собственное зрение (Claude видит изображения через Read)
> вместо отдельного vision-сервиса или ручного отбора.

Источники данных (HF, аккаунт владельца):
- Текстовые метаданные: `Ilialebedev/pre_final_data`, файлы `main_ready/data_00.json` …
  `data_21.json` (22 шарда, ~390МБ каждый, ~100k видео суммарно).
- Сами видеофайлы: `Ilialebedev/videos`, `videos2`, … `videos11` (11 датасетов).

---

## Шаг 0 — Разведка схемы (обязательно, один раз)

Прежде чем писать логику отбора, скачать НЕБОЛЬШОЙ кусок одного шарда (не файл целиком — 390МБ) и
понять реальную структуру:
- Как называется поле с `video_id` (или его аналог) — по нему видео ищутся в `Ilialebedev/videos*`.
- Какие метрики есть (просмотры/лайки/комментарии/категория/канал/дата) и как называются точно.
- Формат самого JSON — единый массив объектов или JSON Lines. При 390МБ загружать в память как
  один `json.load()` может быть тяжело — если это массив, читать потоково (`ijson`) или через
  `pandas.read_json(..., chunksize=...)`, если JSONL — построчно.
- Как названы файлы видео внутри `Ilialebedev/videos*` (по video_id? по индексу?) и как понять,
  в каком именно из 11 датасетов лежит нужное видео (искать по всем по очереди, либо в текстовых
  метаданных может быть прямое поле с именем датасета/файла — проверить).

Использовать `huggingface_hub` (`HfApi().list_repo_files(...)`, `hf_hub_download(...)`,
`load_dataset(..., streaming=True)` если это оформлено как HF Dataset, а не голые файлы) — то же,
что уже стоит в `Fetcher/.fetcher_venv`/окружении раннера (проверить наличие, доставить при
необходимости). Не гадать по названиям полей заранее — фиксировать то, что реально увидел.

## Шаг 1 — Отбор кандидатов по метаданным (дёшево, без скачивания видео)

Скачать метаданные (можно не все 22 шарда сразу — начать с одного-двух, расширять по необходимости).
Составить пул кандидатов, покрывающий разные срезы, а не просто «топ по просмотрам» — цель этого
корпуса: полный прогон ВСЕЙ системы, значит нужно РАЗНООБРАЗИЕ обстоятельств, которые ломают
компоненты (см. портфолио-оценку §3 — валидность зависит от наличия лиц/людей/музыки/речи/движения):
- Разные категории (если поле категории есть).
- Разные диапазоны длительности (короткие ролики уже не раз были причиной NaN/edge-case в отчётах —
  специально включить и короткие, и длинные).
- Разные диапазоны просмотров/лайков/комментариев (не только хиты — и середина, и низкие).
- Разные каналы/авторов (не 10 видео одного канала).

Взять на этом шаге кандидатов С ЗАПАСОМ (например 400-600), не сразу 200 — часть отсеется на шаге 2/3.

## Шаг 2 — Скачать видео кандидатов

Скачать сами mp4 по video_id из `Ilialebedev/videos`…`videos11` (искать по всем, если явного
указания датасета нет в метаданных). Не скачивать всё сразу под завязку диска — разумными партиями,
удалять то, что не прошло отбор на шаге 3, сразу после разбора на кадры (аналогично тому, как Fetcher
не копит видео на подах — см. `automation/fetcher/README.md`).

## Шаг 3 — Кадры + САМОСТОЯТЕЛЬНОЕ описание через зрение

Для каждого видео-кандидата:
1. `ffmpeg` — вырезать 10–100 кадров (равномерно по таймлайну; для коротких видео — меньше, для
   длинных — больше, но не более ~100, чтобы не раздувать стоимость).
2. Инструментом `Read` посмотреть КАЖДЫЙ (или представительную выборку, если кадров много —
   например каждый 3-й) кадр НАПРЯМУЮ — Read показывает изображение нативно, отдельный
   vision-сервис не нужен.
3. По увиденному — коротко зафиксировать для видео: есть ли люди (сколько, крупный/дальний план),
   есть ли узнаваемые объекты (авто, техника, еда, текст на экране, логотипы), есть ли явная
   динамика/движение по кадрам, статична ли сцена, какой примерный жанр (говорящая голова, влог,
   игровой стрим, монтажный клип, анимация и т.п.), есть ли явная музыка/танцы (нельзя увидеть
   напрямую, но иногда очевидно по контексту — уточнить по названию/описанию).

## Шаг 4 — Собрать финальный сбалансированный набор

Из отсмотренных кандидатов выбрать финальные ~150-250 (или сколько владелец попросит) так, чтобы
покрыть КОМБИНАЦИИ признаков, а не просто «200 случайных»:
- С людьми (разное количество лиц: 0, 1, несколько) — закрывает face/emotion-ветку.
- Явно без людей (природа, техника, скринкасты) — закрывает contra-случай (valid-empty пути).
- Высокая динамика (спорт, экшен) и низкая (интервью, лекция) — закрывает motion/pacing-ветку.
- С музыкой и с чистой речью (насколько можно понять до реального прогона ASR/CLAP) — закрывает
  аудио-ветку.
- Разные длительности (очень короткие ролики специально включить — именно они чаще всего ломают
  `min_frames`-подобные зависимости, см. портфолио-оценку).
- Явный текст/логотипы в кадре (если попадаются) — единственный шанс проверить `ocr_extractor`/
  `text_scoring` не на пустом месте (хотя корень их проблемы — веса детектора, не корпус, см. §3.4
  портфолио-оценки — не трогать `core_object_detections`, пока не завершится ручная переразметка).

Составить короткую сводную таблицу (video_id, что на кадрах, почему выбран) — она и есть обоснование
подбора, не только список ID.

## Шаг 5 — Прогнать систему и передать результат

Передать финальный список video_id в DataProcessor batch-валидацию (тот же контур, что для
`e2e_run_hf_videos11.py`, см. `backend/scripts/`) — по существующей инфраструктуре подов
(`pod_control`/`manager`), с обычным подтверждением владельца на поднятие GPU-пода. Результат —
реальные NPZ на разнообразном корпусе вместо мок-фикстур, на которых можно честно пере-оценить
компоненты, помеченные в портфолио-оценке как «мертвые из-за mock/stub данных».

## Стоимость и ограничения

- Просмотр кадров через `Read` — обычные вызовы модели, стоимость сессии растёт с числом кадров;
  не смотреть 100 кадров на КАЖДОЕ из 400-600 кандидатов на шаге 3 — сначала можно прорядить
  кандидатов по метаданным (шаг 1) агрессивнее, если бюджет сессии важен, либо смотреть меньше
  кадров на кандидата на первом проходе (5-10) и только для финалистов — полные 10-100.
- Скачивание видео — трафик и место на диске, чистить после разбора на кадры (шаг 2 note).
- Если внутри одной сессии не укладывается по `max_turns`/бюджету — разбить на несколько сессий
  (агент и так восстанавливается после обрыва, см. `deepdive_agent.py`), сохраняя промежуточный
  список кандидатов на диск между сессиями, чтобы не начинать с нуля.
