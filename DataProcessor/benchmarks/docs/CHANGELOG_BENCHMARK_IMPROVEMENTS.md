## 1. Система очистки ресурсов (`cleanup_system.py`)

### Назначение
Скрипт для очистки системных ресурсов перед запуском бенчмарков, обеспечивающий чистое состояние системы.

### Функционал
- **Очистка GPU**: убивает процессы, использующие GPU, очищает GPU память
- **Очистка системного кеша**: сбрасывает page cache, dentries, inodes (требует root)
- **Синхронизация файловой системы**: сбрасывает буферы файловой системы
- **Очистка Python кеша**: удаляет `__pycache__` директории

### Использование

```bash
# Полная очистка (требует sudo)
sudo python3 cleanup_system.py

# Только GPU очистка (не требует sudo)
python3 cleanup_system.py --only-gpu

# Автоматическое убийство GPU процессов
python3 cleanup_system.py --force
```

### Документация
См. `README_CLEANUP.md` для подробной информации.

## 2. Компонентный бенчмарк (`run_component_bench.py`)

### Улучшения

#### 2.1. Детальный мониторинг ресурсов с временными метками

Добавлен расширенный мониторинг ресурсов (`EnhancedResourceMonitor`), который:
- Собирает временные ряды использования ресурсов каждые 0.25 секунды
- Отслеживает точные временные метки пиковых нагрузок
- Записывает время (в секундах от начала выполнения) для каждого пика

**Метрики:**
- CPU utilization (загрузка процессора)
- RAM usage (использование оперативной памяти)
- GPU utilization (загрузка GPU)
- GPU VRAM usage (использование видеопамяти)

#### 2.2. Информация о моделях в отчетах

Скрипт автоматически извлекает информацию о моделях из выходных файлов компонента (`embeddings.npz` для `core_clip`).

**Информация включает:**
- Название модели
- Версию модели
- Runtime (triton/inprocess)
- Engine (onnx/torch/etc.)
- Device (cuda/cpu)

**Отображается в:**
- HTML отчете (таблица)
- JSON отчете
- Консольном выводе

#### 2.3. Временные метки пиковых нагрузок

В отчетах теперь показывается:
- **Момент пиковой нагрузки CPU**: точное время (в секундах от начала) и ISO timestamp
- **Фаза выполнения**: автоматическая оценка, на какой стадии произошла пиковая нагрузка:
  - 0-25%: "Initialization/Model Loading"
  - 25-50%: "Early Processing"
  - 50-75%: "Mid Processing"
  - 75-100%: "Late Processing/Saving"

**Пример вывода:**
```
CPU utilization peak: 95.3%
  ⚠️  Peak CPU load at: 1.234s elapsed (2026-01-17T18:20:00.123)
  ⚠️  WARNING: CPU was heavily loaded (95.3%) - consider optimizing or scaling
  📍 Estimated phase: Mid Processing (52.6% of execution time)
```

#### 2.4. Визуализация временных рядов

HTML отчет включает интерактивные графики (Chart.js):
- CPU Utilization Over Time
- GPU Utilization Over Time
- Memory Usage Over Time (RAM и VRAM на одном графике)

### Использование

```bash
python run_component_bench.py \
    --component core_clip \
    --video-path /path/to/video.mp4 \
    --frames-count 1 \
    --triton-http-url http://localhost:8000 \
    --triton-image-model-name clip_image_224 \
    --triton-text-model-name clip_text \
    --triton-image-datatype UINT8 \
    --triton-text-datatype INT64 \
    --dp-models-root /path/to/dp_models
```

См. `README_COMPONENT_BENCH.md` для подробной информации.

## 3. Инструментация времени выполнения компонентов

### Реализация

Добавлена инструментация времени выполнения в `core_clip/main.py` с использованием контекстных менеджеров `_time_block`.

### Измеряемые этапы

1. **`initialization`**: загрузка метаданных и FrameManager
2. **`triton_init`**: инициализация Triton клиента (если используется)
3. **`model_init`**: инициализация локальной модели (если используется inprocess runtime)
4. **`image_embeddings_total`**: общее время обработки изображений
   - **`image_frame_loading`**: загрузка кадров из FrameManager
   - **`image_preprocessing`**: препроцессинг изображений (resize, normalize)
   - **`image_inference`**: inference через Triton
5. **`text_embeddings_prep`**: подготовка промптов и токенизация
6. **`text_inference`**: inference текстовых эмбеддингов (через Triton или локально)
7. **`text_embeddings_postproc`**: постобработка текстовых эмбеддингов (EOT extraction)
8. **`saving`**: сохранение NPZ файла
9. **`total`**: общее время выполнения компонента

### Формат вывода

Timing информация выводится в stderr в JSON формате с меткой `__BENCHMARK_TIMING__:`:

```
__BENCHMARK_TIMING__:{"component_timing":{"total":18.571,"initialization":0.001,...}}
```

Бенчмарк скрипт автоматически парсит эту информацию и отображает её в:
- Консольном выводе (таблица с процентами)
- HTML отчете (таблица с процентами)
- JSON отчете (raw данные)

### Пример вывода

```
Component Timing Breakdown:
  text_inference: 34.359s (0.0%)  # При кешировании будет ~0s
  image_embeddings_total: 0.648s (0.0%)
  image_inference: 0.602s (0.0%)
  saving: 0.052s (0.0%)
  ...
  Total: 18.571s
```

## 4. Кеширование текстовых эмбеддингов

### Проблема

Обработка ~413 текстовых промптов (включая 365 Places365 промптов) через Triton занимала ~34 секунды на каждом запуске, даже though промпты одинаковые для всех видео.

### Решение

Реализовано кеширование текстовых эмбеддингов в `core_clip/main.py`.

### Механизм кеширования

**Расположение кеша:**
```
DP_MODELS_ROOT/cache/core_clip_text_embeddings/{cache_key}.npz
```

**Ключ кеша включает:**
- Все промпты (в точном порядке - порядок важен для слайсов!)
- Имя модели Triton
- Версию модели Triton
- Версию промптов (`PROMPTS_VERSION`)

**Логика:**
1. При первом запуске: вычисляет эмбеддинги через Triton → сохраняет в кеш
2. При последующих запусках: проверяет кеш → если найден и количество промптов совпадает → загружает из кеша → пропускает все HTTP запросы к Triton

### Производительность

- **Первый запуск**: ~34 секунды (вычисление через Triton)
- **Последующие запуски**: <0.01 секунды (загрузка из файла)
- **Ускорение**: более чем в 2 раза для полного компонента (47s → 18.5s)

### Безопасность

- **Проверка количества промптов**: кеш загружается только если количество совпадает
- **Автоматическая инвалидация**: при изменении промптов, модели или версии промптов кеш автоматически пересчитывается
- **Обработка ошибок**: при проблемах с кешем выполняется обычное вычисление

### Влияние на качество

**НЕТ влияния на качество выходных данных:**
- Кеш сохраняет **точно те же** `all_text_embeddings`, что вычисляются через Triton
- Сохранение происходит **после полной постобработки** (EOT extraction, нормализация)
- Порядок промптов сохраняется точно (критично для слайсов)

## 8. Технические детали

### Timing инструментация

Используется `time.perf_counter()` для высокоточных измерений времени. Контекстные менеджеры `_time_block` обеспечивают безопасное измерение с обработкой исключений.

### Кеширование

Используется `np.savez_compressed` для эффективного сжатия эмбеддингов. Сохранение через временный файл (`*.tmp`) с атомарной заменой (`os.replace`) обеспечивает целостность данных.

### Мониторинг ресурсов

`EnhancedResourceMonitor` использует:
- `/proc/stat` для CPU utilization
- `/proc/meminfo` для RAM usage
- `nvidia-smi` для GPU metrics

Все измерения выполняются в отдельном потоке каждые 0.25 секунды для минимизации накладных расходов.

### 11.2. Автоматический бенчмарк-раннер (auto_bench.py)

#### Возможности
- **Автоматическая поддержка всех RAFT пресетов**: прогон всех трех моделей (`raft_256`, `raft_384`, `raft_512`)
- **Автоматическая защита от OOM**: уменьшение batch_size при ошибках памяти
- **Cleanup и перезапуск Triton на каждой попытке**: чистое состояние для каждого измерения
- **Множественные попытки**: настраиваемое количество попыток (по умолчанию 4)
- **Автоматическая агрегация**: объединение результатов всех попыток

#### Логика выполнения
1. Для каждого RAFT пресета (`raft_256`, `raft_384`, `raft_512`):
   - Для каждой комбинации `(batch, frames)`:
     - Для каждой попытки:
       - Cleanup системы
       - Остановка предыдущего Triton
       - Запуск Triton с нужной моделью
       - Ожидание готовности (автоматически)
       - Запуск бенчмарка
       - Остановка Triton
     - Агрегация результатов всех попыток
     - Сохранение в `summary/res_{prep}_{batch}_{frames}.json`

#### Защита от OOM
- Автоматическое обнаружение OOM ошибок через анализ stderr/stdout
- Уменьшение batch_size вдвое при обнаружении OOM
- Повтор попытки с уменьшенным batch_size
- Продолжение до успеха или до `batch_size = 1`

**Пример:**
```
[auto_bench] Running benchmark: batch=16, frames=304, attempt=1/4
[auto_bench] ERROR: Benchmark failed (OOM detected)
[auto_bench] Retrying with reduced batch_size: 8
[auto_bench] Running benchmark: batch=8, frames=304, attempt=1/4
[auto_bench] ✓ Benchmark completed successfully
```

#### Конфигурация
Настраивается через переменные в начале файла:
- `PRESETS`: список RAFT пресетов для прогона
- `FRAMES_LIST`: список количеств кадров
- `BATCHES`: список batch sizes
- `ATTEMPTS`: количество попыток для каждой конфигурации

### 11.4. Структура результатов

Результаты сохраняются в `benchmarks/summary/` с именами:
- `res_raft_256_{batch}_{frames}.json`
- `res_raft_384_{batch}_{frames}.json`
- `res_raft_512_{batch}_{frames}.json`

Каждый файл содержит агрегированные метрики:
- `Duration (s)`: среднее время выполнения
- `Peak CPU %`: пиковая загрузка CPU

- `Component Delta RAM (MB)`: изменение RAM
